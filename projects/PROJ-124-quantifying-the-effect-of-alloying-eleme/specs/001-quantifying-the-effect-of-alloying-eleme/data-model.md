# Data Model: Quantifying the Effect of Alloying Elements on the Glass-Forming Ability of Metallic Glasses

## 1. Overview

This document defines the data structures, schemas, and transformations used throughout the pipeline. All data flows from raw CSV ingestion to feature engineering, model training, and final candidate screening.

## 2. Data Flow

1.  **Raw Data**: Downloaded from HuggingFace (CSV).
2.  **Processed Data**: Normalized compositions + Pymatgen descriptors (CSV).
3.  **Model Artifacts**: Pickle files (`.pkl`).
4.  **Results**: JSON (SHAP, verification requests) and CSV (candidates).

## 3. Schemas

### 3.1 Raw Input Schema (`data/raw/gfa_raw.csv`)
*Source*: HuggingFace `GFA-D2/pilot_flags`

| Column | Type | Description |
|--------|------|-------------|
| `composition` | string | Elemental composition string (e.g., "Zr50Cu40Al10") |
| `Rc` | float | Critical cooling rate (K/s) |
| `log10_Rc` | float | Log-transformed Rc (if present, else computed) |

*Note*: If `log10_Rc` is missing, it is computed as `np.log10(Rc)`. **Pipeline fails if neither column exists.**

### 3.2 Processed Feature Schema (`data/processed/features.csv`)
*Generated by*: `code/data/features.py`

| Column | Type | Description |
|--------|------|-------------|
| `source_row_id` | integer | Original row index in raw data (for traceability) |
| `composition` | string | Original composition string |
| `log10_Rc` | float | Target variable |
| `atomic_radius_mean` | float | Weighted mean of atomic radii |
| `atomic_radius_var` | float | Variance of atomic radii |
| `electronegativity_mean` | float | Weighted mean of electronegativity |
| `electronegativity_var` | float | Variance of electronegativity |
| `VEC_avg` | float | Weighted mean of valence electrons |
| `VEC_raw` | float | Sum of valence electrons |
| `pairwise_size_mismatch` | float | Pairwise size mismatch descriptor |
| `element_1` | string | First element |
| `element_2` | string | Second element |
| `element_3` | string | Third element |
| ... | ... | Fraction columns (e.g., `Zr_frac`, `Cu_frac`) |

### 3.3 Candidate Output Schema (`output/candidates.csv`)
*Generated by*: `code/models/predict.py`
*Contract*: `contracts/candidates_csv.schema.yaml`

| Column | Type | Description |
|--------|------|-------------|
| `composition` | string | Predicted composition (e.g., "Zr50Cu30Al20") |
| `predicted_log10_Rc` | float | Model prediction |
| `ci_lower` | float | 2.5% percentile of bootstrapped predictions |
| `ci_upper` | float | 97.5% percentile of bootstrapped predictions |
| `doa_risk` | string | "low", "medium", "high" (based on Mahalanobis distance) |
| `penalty_applied` | float | Proportional penalty (0.0 or 1.0 as per FR-009) |
| `final_score` | float | `predicted_log10_Rc` + `penalty_applied` |
| `novelty_status` | string | "novel", "known", "unverified_external" |
| `risk_score` | float | Mahalanobis distance (raw value, for re-analysis) |

## 4. Transformation Logic

### 4.1 Normalization
- **Input**: Elemental fractions from composition string.
- **Logic**: Sum fractions. If $|Sum - 1.0| \le 0.01$, normalize to exactly 1.0. Else, log error and drop row.

### 4.2 Feature Engineering
- **Library**: `pymatgen`
- **Descriptors**:
  - `atomic_radius`: `Element(symbol).atomic_radius`
  - `electronegativity`: `Element(symbol).electronegativity`
  - `valence_electrons`: `Element(symbol).valence_electrons`
- **Interaction**:
  - Variance: `np.var([r1, r2, r3], ddof=0)`
  - Weighted Mean: $\sum (fraction_i \times property_i)$
  - Pairwise: Calculated for unique pairs (e.g., $|r_1 - r_2|$).

### 4.3 DoA Calculation
- **Method**: Mahalanobis distance $D_M = \sqrt{(x - \mu)^T \Sigma^{-1} (x - \mu)}$.
- **Preprocessing**: Features reduced via PCA (retaining [deferred] variance) to mitigate curse of dimensionality.
- **Threshold**: 95th percentile of training distances (on PCA-reduced features).
- **Penalty**: If $D_M > Threshold$, add **+1.0** to prediction (FR-009). `risk_score` is stored as $D_M$.

### 4.4 Heteroscedasticity Correction
- **Step 1**: Fit initial model. Compute residuals.
- **Step 2**: Perform Breusch-Pagan test.
- **Step 3**: If heteroscedastic:
  -   **Attempt**: Bin residuals by feature-space quantiles (min 50 samples/bin).
  -   **Fallback**: If bins are unstable, fit a global log-variance model or use Huber loss.
- **Step 4**: Retrain with weights (inverse variance) or robust loss.

### 4.5 Traceability
- **Mechanism**: Each row in `data/processed/features.csv` includes `source_row_id` mapping to the raw data index. This ensures every result in `paper/` or `output/` can be traced back to a specific row in `data/processed/`.

## 5. Data Hygiene & Versioning

- **Checksums**: SHA-256 of raw files recorded in `state/artifact_hashes.yaml`.
- **Immutability**: Raw files are never modified. Derived files have new names (e.g., `features_v1.csv`).
- **PII**: No PII expected in metallic glass datasets.
