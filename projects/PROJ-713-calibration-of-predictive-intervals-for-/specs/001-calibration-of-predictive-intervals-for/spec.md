## User Scenarios & Testing

### User Story 1 - Empirical Coverage Assessment on Benchmark Datasets (Priority: P1)

As a researcher, I want to compute the empirical coverage rates of predictive intervals ([deferred] and [deferred]) generated by ARIMA, Prophet, and a lightweight LSTM on the M4 and UCI Electricity datasets, so that I can determine if these standard methods achieve their nominal coverage levels.

**Why this priority**: This is the core hypothesis test of the project. Without establishing whether the intervals are calibrated, subsequent analysis of scoring rules or conformal adjustments has no baseline.

**Independent Test**: Can be fully tested by running the data acquisition, model fitting, and coverage calculation scripts on a single dataset (e.g., M4) and verifying that the output table lists the empirical coverage percentage for each model against the nominal target ([deferred] or [deferred]).

**Acceptance Scenarios**:

1. **Given** the M4 dataset is downloaded and pre-processed (80/20 split), **When** the ARIMA, Prophet, and LSTM models are fitted and their [deferred] predictive intervals are generated, **Then** the system outputs a table showing the empirical coverage for each model (e.g., "ARIMA: [deferred]", "Prophet: [deferred]", "LSTM: [deferred]").
2. **Given** the UCI Electricity dataset is loaded, **When** the coverage calculation is executed for the [deferred] interval level, **Then** the system reports the proportion of observed test values falling within the predicted bounds for all three models.

---

### User Story 2 - Calibration Diagnostics via PIT and CRPS (Priority: P2)

As a researcher, I want to generate Probability Integral Transform (PIT) histograms and calculate the Continuous Ranked Probability Score (CRPS) for each model, so that I can assess the shape of the calibration (uniformity) and the overall quality of the probabilistic forecasts beyond simple coverage.

**Why this priority**: Coverage alone is a coarse metric; PIT and CRPS provide deeper insight into distributional accuracy and are required to fully evaluate the "calibration" claim in the research question.

**Independent Test**: Can be tested independently by generating the diagnostic plots (PIT histograms) and numerical CRPS values from the model residuals and comparing them against the theoretical uniform distribution and baseline scores.

**Acceptance Scenarios**:

1. **Given** the forecast residuals and predictive distributions are available, **When** the PIT values are computed and binned, **Then** the system generates a histogram where a perfectly calibrated model shows a flat (uniform) distribution, and the Kolmogorov–Smirnov test p-value is reported.
2. **Given** the forecast distributions and observed values, **When** the `properscoring.crps_ensemble` function is applied, **Then** the system outputs a CRPS value for each model, allowing for a ranking of probabilistic forecast quality.

---

### User Story 3 - Statistical Significance and Conformal Baseline Comparison (Priority: P3)

As a researcher, I want to perform paired bootstrap tests to determine if coverage deviations are statistically significant (p < 0.05) and optionally apply a conformal prediction wrapper to assess if post-hoc calibration improves results, so that I can conclude whether the mis-calibration is systematic and if remediation is viable.

**Why this priority**: This addresses the "falsify the hypothesis" requirement and explores the "post-hoc calibration" motivation mentioned in the idea. It is lower priority because it depends on the results of the first two stories.

**Independent Test**: Can be tested by running the bootstrap resampling routine on the coverage data and reporting the p-values for pairwise model comparisons, and by executing the conformal wrapper on a subset of data to verify it shifts coverage toward the nominal level.

**Acceptance Scenarios**:

1. **Given** the empirical coverage rates for ARIMA and Prophet, **When** a paired bootstrap test is performed, **Then** the system outputs a p-value indicating whether the difference in coverage is statistically significant (p < 0.05).
2. **Given** the LSTM model outputs, **When** a conformal prediction wrapper (based on *Self-Calibrating Conformal Prediction*) is applied, **Then** the system reports the new empirical coverage, demonstrating if the adjustment moves the interval closer to the nominal target.

---

### Edge Cases

- **Data Availability**: What happens if the M4 or UCI Electricity dataset download fails or the file format is corrupted? The system must handle the error gracefully and log a specific failure message rather than crashing silently.
- **Model Convergence**: How does the system handle cases where the ARIMA model fails to converge or the LSTM training diverges? The system must catch the exception, mark the specific series as "failed," and proceed with the remaining series to ensure partial results are generated.
- **Zero Variance**: What happens if a time series in the test set has zero variance (constant values)? The system must detect this and either skip the PIT calculation for that series (as division by zero would occur) or handle it as a special case with a defined coverage of [deferred].

## Requirements

### Functional Requirements

- **FR-001**: The system MUST download the M4 and UCI Electricity datasets from their official repositories and verify file integrity before processing. (See US-1)
- **FR-002**: The system MUST split each time series into a training window (first [deferred]) and a hold-out test window (last [deferred]) and standardize the data to zero mean and unit variance. (See US-1)
- **FR-003**: The system MUST fit three distinct models (ARIMA via `statsmodels`, Prophet, and a lightweight LSTM with 32 hidden units via `tensorflow.keras`) on the training data. (See US-1)
- **FR-004**: The system MUST generate predictive intervals at [deferred] and [deferred] nominal levels for each model, using model-specific methods (conditional variance for ARIMA, `uncertainty_samples` for Prophet, residual-based Gaussian for LSTM). (See US-1)
- **FR-005**: The system MUST compute empirical coverage rates, PIT values, and CRPS scores for all models and datasets. (See US-2)
- **FR-006**: The system MUST perform paired bootstrap tests to compare coverage deviations and report p-values for statistical significance (threshold p < 0.05). (See US-3)
- **FR-007**: The system MUST implement a conformal prediction wrapper for the LSTM model to serve as a calibration baseline. (See US-3)
- **FR-008**: The system MUST output results in a structured format (CSV/JSON) and generate visualization artifacts (PIT histograms) for all models. (See US-2)

### Key Entities

- **TimeSeries**: A sequence of historical observations with a timestamp, split into training and test sets.
- **ForecastInterval**: A tuple representing the lower and upper bounds of a predictive interval for a specific time step and confidence level.
- **CalibrationMetric**: A record containing the empirical coverage, PIT statistic, and CRPS value for a specific model and dataset.

## Success Criteria

### Measurable Outcomes

> Planning docs state *what* will be measured and the *source/reference* it is measured against; defer specific empirical values (counts, dataset sizes, measured quantities, percentages) to the implementation/research phase.

- **SC-001**: Empirical coverage deviation from nominal levels ([deferred] and [deferred]) is measured against the target nominal coverage for each model. (See US-1)
- **SC-002**: Uniformity of PIT histograms is measured against the theoretical uniform distribution using the Kolmogorov–Smirnov test. (See US-2)
- **SC-003**: Probabilistic forecast quality is measured via the Continuous Ranked Probability Score (CRPS) against a baseline of random noise or a naive forecast. (See US-2)
- **SC-004**: Statistical significance of coverage differences is measured against the p < 0.05 threshold using paired bootstrap tests. (See US-3)
- **SC-005**: Improvement in calibration after conformal adjustment is measured as the reduction in absolute coverage deviation compared to the uncalibrated LSTM baseline. (See US-3)

## Assumptions

- The M4 and UCI Electricity datasets are available via direct HTTP download and fit within the 7 GB RAM / 14 GB disk constraints of the CI runner when processed in chunks or sampled if necessary.
- The `prophet` and `tensorflow` libraries are compatible with the CPU-only environment of the GitHub Actions free-tier runner and do not require CUDA acceleration.
- The LSTM model is trained with a small number of epochs and a single hidden layer (32 units) to ensure training completes within the 6-hour job limit on 2 CPU cores.
- The "lightweight" LSTM implementation uses default precision (float32) rather than 8-bit quantization, as the latter requires CUDA.
- The conformal prediction wrapper uses a split-conformal approach that does not require retraining the underlying LSTM model, relying instead on a calibration set.
- The M4 dataset contains sufficient series to perform meaningful statistical aggregation; if a specific series is too short for the 80/20 split, it is excluded from the analysis.
- The "nominal coverage" targets are strictly 80% and 95%; no other confidence levels are evaluated in this scope.
