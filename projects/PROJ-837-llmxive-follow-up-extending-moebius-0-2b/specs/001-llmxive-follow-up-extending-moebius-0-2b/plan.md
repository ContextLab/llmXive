# Implementation Plan: llmXive follow-up: extending "Moebius: 0.2B Lightweight Image Inpainting Framework with 10B-Level Pe"

**Branch**: `001-llmxive-moebius-dynamic` | **Date**: 2026-07-12 | **Spec**: `specs/001-llmxive-moebius-dynamic/spec.md`

## Summary

This project implements a dynamic rank-adjustment mechanism for image inpainting, extending the "Moebius" framework. The core innovation is a lightweight convolutional gating head that predicts the structural complexity of a masked region (based on human-rated ground truth) and dynamically modulates the rank of $L\lambda MI$ linear matrices during the forward pass. The system is designed to reduce inference latency on low-complexity regions by up to 30-40% on CPU-only edge devices while maintaining fidelity comparable to static high-capacity baselines.

**Critical Constraint & Mode Distinction**:
- **CI Mode (Simulation)**: Runs on GitHub Actions free-tier (multi-core CPU, 7GB RAM). Uses a "Moebius-Tiny" model (~15M params) and a **Decoupled Synthetic Ground Truth** (randomly generated labels independent of training metrics) to validate the *mechanism* without circularity. Results are labeled "Simulation-Only".
- **Research Mode (Human-Grounded)**: Requires external human-annotated data (N=50+). Uses the full architecture (or a larger variant) and real human labels. This mode is **not** guaranteed to run in CI due to resource constraints but is the only mode that supports the "Human-Grounded" claim.

The plan strictly adheres to the constraint that all CI training and evaluation must run on a GitHub Actions free-tier runner without GPU acceleration.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
| :--- | :--- | :--- |
| **Dynamic Rank Modulation** | Required to achieve the 30-40% latency reduction hypothesis (US-3). | Static models cannot adapt compute to input complexity, failing the efficiency goal. |
| **Decoupled Synthetic Ground Truth** | Required for CI to avoid circular validation (FR-007) while testing the mechanism. | Using the same metrics for training and validation creates a tautology. |
| **Human-Grounded Complexity (External)** | Required for the final research claim (Constitution VI). | Simulated data cannot support the "Human-Grounded" claim. |
| **CPU-Only Optimization** | Required by Constitution VII and CI constraints. | GPU-based methods are invalid for edge-deployment claims and would fail CI. |
| **Moebius-Tiny Architecture** | Required to fit training in 7GB RAM on 2 cores. | Full 0.2B model exceeds CI memory limits. |

## Phased Implementation Plan

### Phase 0: Data Preparation & Ground Truth Generation (US-1)
*Goal: Generate masked dataset and ground truth labels (Human or Decoupled Synthetic).*
1.  **Download & Verify**: Fetch Places365 subset from verified HuggingFace URL (`mit-places/Places365`). Verify checksum.
2.  **Synthetic Mask Generation**: Implement `mask_generator.py` to create masks with varying complexity (gradient variance, texture entropy). Record metrics.
3.  **Mock Crowdsourcing Interface**: Implement `annotator.py` to provide a CLI/JSON interface that simulates the *structure* of a crowdsourcing study (FR-002).
    -   **CI Mode**: Generate a "Decoupled Synthetic Ground Truth" (random Likert scores, independent of gradient/entropy metrics) to avoid circularity.
    -   **Research Mode**: Load external human-annotated CSV (if available).
4.  **Inter-Rater Reliability (IR)**: If multiple raters are simulated or external data has them, calculate Krippendorff's alpha. If single-rater simulation, flag as "Single-Rater (Simulation)".
5.  **Validation**: Verify independence of labels (no model inference used). Check sample size ≥ 50 (or CI equivalent).
*Dependencies*: None.
*Output*: `data/processed/masked_images/`, `data/annotations/human_scores.csv` (or `decoupled_scores.csv`).

### Phase 1: Synthetic Proxy Validation (US-4)
*Goal: Validate that synthetic metrics correlate with ground truth.*
1.  **Correlation Analysis**: Compute Pearson correlation between synthetic metrics (gradient, entropy) and ground truth (human or decoupled).
2.  **Gate Check**: If r < 0.7 (and using human data), flag dataset for re-sampling. If using decoupled data, expect low correlation. and log this as expected behavior for CI Mode.
*Dependencies*: Phase 0.
*Output*: `data/results/proxy_validation.json`.

### Phase 2: Dynamic Rank Adjustment Mechanism (US-2)
*Goal: Implement and train the Moebius-Dynamic architecture (Moebius-Tiny for CI).*
1.  **Gating Head Implementation**: Build lightweight conv head (≤15M params total model) to output scalar complexity.
2.  **Moebius Integration**: Implement rank modulation logic for $L\lambda MI$ matrices.
3.  **Training Loop**: Train on multi-task loss (reconstruction + regression on ground truth + rank classification).
    -   **CI Mode**: Train on decoupled synthetic labels.
    -   **Research Mode**: Train on human labels.
4.  **Permutation Test**: Run permutation test on gating inputs to verify no overfitting (FR-008). Shuffle labels and re-evaluate. Record p-value.
*Dependencies*: Phase 1.
*Output*: `code/models/moebius_dynamic.pt`, `data/results/gating_weights.pt`, `data/results/permutation_test.json`.

### Phase 3: Efficiency & Fidelity Evaluation (US-3)
*Goal: Benchmark dynamic vs. static models on CPU.*
1.  **Power Analysis**: Calculate statistical power for the latency reduction claim (SC-005). If power < 0.8, flag result as "Underpowered".
2.  **Baseline Setup**: Load static "Moebius-Tiny" baseline models.
3.  **Inference Benchmark**: Run inference on test set. Measure wall-clock latency (ms) per complexity bin.
4.  **Spearman Correlation**: Calculate Spearman's rank correlation between ground truth complexity and the *effective* model rank used (SC-001).
5.  **Quality Metrics**: Compute FID and LPIPS.
6.  **Statistical Testing**: Perform paired t-tests for quality differences.
*Dependencies*: Phase 2.
*Output*: `data/results/evaluation_report.json`, `data/results/figures/`.

### Phase 3.5: Ablation & Causal Isolation (US-3)
*Goal: Isolate the efficiency gain from the prediction mechanism.*
1.  **Counterfactual Run**: Run static model with *forced* low rank (simulating the dynamic outcome without the prediction cost).
2.  **Comparison**: Compare (Dynamic Model) vs. (Static Low Rank) vs. (Static High Rank).
3.  **Analysis**: Determine if the *prediction* overhead is offset by the *reduction* gain.
*Dependencies*: Phase 3.
*Output*: `data/results/ablation_report.json`.

### Phase 4: Reporting & Paper Generation
*Goal: Compile results into `paper/` artifacts.*
1.  **Data Traceability**: Ensure all numbers trace to `data/results`.
2.  **Mode Labeling**: Explicitly label results as "CI Simulation" or "Research Mode" in the paper.
3.  **Drafting**: Generate paper sections with embedded metrics.
4.  **Review**: Self-review against Success Criteria (SC-001 to SC-006).
*Dependencies*: Phase 3.5.
*Output*: `paper/draft.md`.

## Risk Mitigation

- **Dataset Mismatch**: If Places365 subset lacks required diversity, fallback to CelebA-HQ (verified source) or synthetic generation with higher variance.
- **CPU Time Limit**: If training exceeds a substantial duration, reduce dataset size. (sample more aggressively) or simplify the gating head architecture.
- **Proxy Failure**: If synthetic metrics fail correlation (r < 0.7) with human data, the plan explicitly flags this as a "Blocking Coverage Gap" and halts training, forcing a re-evaluation of the synthetic generation parameters.
- **Memory Overflow**: Implement chunked processing for FID/LPIPS calculation to stay within 7GB RAM.
- **Circular Validation**: Mitigated by using Decoupled Synthetic Ground Truth in CI Mode and external human data in Research Mode.
