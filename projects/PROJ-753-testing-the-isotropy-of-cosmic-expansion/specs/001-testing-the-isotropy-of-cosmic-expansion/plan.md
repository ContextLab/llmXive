# Implementation Plan: Testing the Isotropy of Cosmic Expansion with Type Ia Supernova Data

**Branch**: `001-testing-isotropy-cosmic-expansion` | **Date**: 2026-06-22 | **Spec**: `specs/001-testing-the-isotropy-of-cosmic-expansion/spec.md`
**Input**: Feature specification from `/specs/001-testing-the-isotropy-of-cosmic-expansion/spec.md`

## Summary

This project implements a statistical test for the isotropy of cosmic expansion using the Pantheon+ Type Ia Supernova dataset. The technical approach involves ingesting the Pantheon+ release, calculating distance-modulus residuals against a flat ΛCDM model, and performing a **Maximum Likelihood Estimation (MLE)** directly on the sparse point-source residuals to extract dipole and quadrupole amplitudes. Statistical significance is assessed by comparing observed amplitudes against a null distribution generated from **synthetic isotropic Gaussian Random Fields (GRF)** that are sampled at the static observed coordinates to correctly apply the survey selection function.

### Constitution Implementation Details

This section explicitly maps the plan's design to the project's Constitutional Principles:

- **Principle I (Reproducibility)**: Random seeds are pinned in `code/likelihood_analysis.py` and `code/simulations.py`. The Pantheon+ version is recorded in `data/metadata.json`.
- **Principle II (Verified Accuracy)**: All dataset references point to verified, reachable URLs (see `research.md`).
- **Principle III (Data Hygiene)**: Raw data is checksummed; derived data is written to new files; PII scan passed.
- **Principle IV (Single Source of Truth)**: All figures/stats trace to `data/` and `code/`; no hand-typed numbers.
- **Principle V (Versioning Discipline)**: Artifacts carry content hashes; `state.yaml` updated on change.
- **Principle VI (Anisotropy Signal Quantification)**: The plan satisfies this by implementing a **Maximum Likelihood Estimator (MLE)** to directly fit the dipole ($A_1$) and quadrupole ($A_2$) amplitudes to the distance-modulus residuals. This method is robust for sparse point-source data and avoids the biases of pixel-based harmonic decomposition. The specific calculation logic is defined in `code/likelihood_analysis.py` via the `compute_likelihood` function.
- **Principle VII (Pantheon+ Data Versioning)**: The specific release version (Pantheon+ v1.0) and its checksum are recorded in `data/metadata.json` during the ingestion phase (FR-001). This file is included in the artifact hash calculation to ensure the residuals correspond to the correct cosmological baseline.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
| :--- | :--- | :--- |
| **Maximum Likelihood Estimator (MLE)** | Required for sparse point-source data ([deferred] SN) where pixel-based methods (MASTER) fail due to aliasing and noise bias. | Pixel-based harmonic decomposition (Nside=32) is numerically unstable with <0.5 SN/pixel. MLE directly models the point-source likelihood. |
| **Synthetic GRF Null Simulation** | Required to correctly simulate an isotropic universe with the survey's selection effects, avoiding the circularity of rotating observed data. | Rotating observed data preserves the mask-induced dipole, leading to a biased null distribution. GRF simulation + static mask correctly isolates the signal. |
| **Nside=16 Grid** | Used for visualization and mask definition only. | Nside=32 is too sparse for stable pixel means; Nside=16 provides ~768 pixels for [deferred] SN (approx 2 SN/pixel), improving stability for visualization. |
| **Streaming Simulation Storage** | Required to fit 10,000 simulations within the 7GB RAM / 14GB disk constraint. | Storing full maps for [deferred] runs would exceed disk limits. Storing only scalar amplitudes (2 floats/run) reduces storage to <1MB. |
| **Parallelization (2 workers)** | Required to meet the 6-hour runtime goal on a 2-CPU runner. | Serial execution of [deferred] MLE fits on [deferred] points each may exceed the 6-hour limit. `multiprocessing.Pool` with a parallel worker pool ensures feasibility. |

## Compute Feasibility

- **Environment**: GitHub Actions Free Tier (multiple CPU cores, 7GB RAM, 14 GB disk).
- **Strategy**:
 - **Streaming Simulation**: The simulation loop computes the dipole/quadrupole amplitudes for each iteration and writes **only the scalar values** (2 floats) to a CSV. It does not store the full synthetic maps. This reduces the disk footprint of [deferred] runs to <1MB and RAM usage to negligible levels.
  - **Parallelization**: Use `multiprocessing.Pool` with `processes=2` (matching the runner's CPU cores) to parallelize the simulation loop in `code/simulations.py`.
  - **Libraries**: `healpy` (CPU optimized), `scipy` (CPU native). No GPU required.
  - **Runtime**: Estimated runtime for a substantial number of simulations (MLE on a substantial dataset) is expected to be within the project's temporal constraints on 2 cores.
