# Feature Specification: Reproduce & validate: Toward Generalist Autonomous Research via Hypothesis-Tree Refinement

**Feature Branch**: `001-reproduce-arbor`  
**Created**: 2024-05-21  
**Status**: Draft  
**Input**: User description: "Reproduce & validate: Toward Generalist Autonomous Research via Hypothesis-Tree Refinement"

## User Scenarios & Testing

### User Story 1 - Verify Execution & Artifact Generation (Priority: P1)

The primary objective is to confirm that the vendored `Arbor` codebase can be executed in the CI environment and produces tangible, non-fabricated output artifacts (e.g., tree JSON, evaluation logs, or result reports) without crashing or hanging indefinitely.

**Why this priority**: This is the foundational "smoke test." If the code cannot run or produce output, the project cannot proceed to validation or scientific claims. It validates the "Compute Feasibility" constraint.

**Independent Test**: Execute the primary CLI entry point (`arbor run` or equivalent demo script) on a minimal task configuration. Verify that the process exits with code 0 and that specific output files (defined in the repo's `docs/outputs-and-resume.md`) are created with non-zero byte sizes.

**Acceptance Scenarios**:

1. **Given** the `Arbor` submodule is cloned and dependencies are installed, **When** the user runs the minimal demo task (e.g., `examples/dashboard_demo.py`), **Then** the process completes within 60 minutes and generates `tree.json` and `eval_results.json` in the output directory.
2. **Given** a valid task configuration exists, **When** the `arbor run` command is invoked with `--dry-run` or a single-step limit, **Then** the system logs the initialization of the coordinator and executor without raising an unhandled exception.
3. **Given** the CI environment has no GPU, **When** the system initializes its model loaders, **Then** it defaults to CPU-only inference paths and does not attempt to load CUDA-specific libraries.

---

### User Story 2 - Validate Hypothesis-Tree Refinement Logic (Priority: P2)

Confirm that the core mechanism—Hypothesis Tree Refinement (HTR)—is functioning as described: the system must create a tree structure, link hypotheses to evidence, and update the tree state based on execution results.

**Why this priority**: This validates the "scientific" contribution of the paper. It ensures the agent isn't just running code but is actually maintaining the persistent state required for long-horizon research.

**Independent Test**: Inspect the generated `tree.json` or the HTML visualization (`arbor-tree-demo.html`) to verify that nodes are created with distinct types (Hypothesis, Artifact, Evidence) and that parent-child relationships reflect the execution flow.

**Acceptance Scenarios**:

1. **Given** a task requiring iterative improvement, **When** the agent executes a hypothesis and receives a result, **Then** a new "Evidence" node is appended to the tree with a link to the "Hypothesis" node, and the tree version number increments.
2. **Given** multiple hypotheses are tested, **When** the coordinator synthesizes results, **Then** the tree reflects a "distilled insight" node that references the successful artifacts and their corresponding metrics.
3. **Given** a failed execution, **When** the error is captured, **Then** the tree records the failure state and the coordinator generates a refined hypothesis node branching from the failed attempt.

---

### User Story 3 - Reproduce Paper Claims (Priority: P3)

Execute a full benchmark task (e.g., `algotune_knn`) and compare the final held-out performance metric against the baseline reported in the paper (or a known baseline if the paper's exact environment cannot be perfectly replicated).

**Why this priority**: This is the ultimate validation of the research. It confirms that the vendored code actually achieves the results claimed in the abstract (e.g., "2.5x gain").

**Independent Test**: Run the `algotune_knn` benchmark using the provided `eval.py`. Extract the final score and compare it against the baseline score defined in the task's `task.py` or the paper's reported baseline.

**Acceptance Scenarios**:

1. **Given** the `algotune_knn` task is configured, **When** the full optimization loop completes, **Then** the `eval.py` script outputs a final score that is logged and stored in `results/benchmark_score.txt`.
2. **Given** the baseline score is available (e.g., from `task.py` defaults), **When** the final score is calculated, **Then** the system logs a comparison (e.g., "Improvement: +X%") to the console and the final report.
3. **Given** the run completes, **When** the user reviews the `report.md` generated by `arbor report`, **Then** it contains a summary of the tree depth, number of hypotheses, and the final metric value.

### Edge Cases

- **Timeout Handling**: What happens if an executor task hangs (e.g., waiting for a model response) for longer than the configured timeout? The system must kill the process, log the timeout, and update the tree node to "Failed" rather than hanging the entire CI job.
- **Resource Exhaustion**: If the tree grows too large or the executor consumes >7GB RAM, the system must gracefully fail with a specific error code (e.g., `RESOURCE_EXHAUSTED`) and output a partial tree state for debugging.
- **Missing Dependencies**: If a required Python package in `requirements.txt` is missing or incompatible with the runner's OS, the installation step must fail immediately with a clear error message, preventing a partial run.

## Requirements

### Functional Requirements

- **FR-001**: The system MUST execute the `arbor` CLI entry point with a specified task configuration and terminate within 6 hours (See US-1).
- **FR-002**: The system MUST generate a `tree.json` artifact containing nodes for Hypotheses, Artifacts, and Evidence with valid parent-child relationships (See US-2).
- **FR-003**: The system MUST run on a CPU-only environment without attempting to load CUDA libraries or require GPU hardware (See US-1).
- **FR-004**: The system MUST capture execution logs and error messages for every hypothesis attempt and link them to the corresponding tree node (See US-2).
- **FR-005**: The system MUST produce a final evaluation report (e.g., `results/summary.md`) containing the final metric score and a comparison to the baseline (See US-3).
- **FR-006**: The system MUST handle executor timeouts by terminating the hanging process and marking the node as "Failed" rather than blocking the coordinator (See Edge Cases).

### Key Entities

- **Hypothesis Node**: A tree node representing a proposed research direction or code change.
- **Evidence Node**: A tree node containing the output (stdout, stderr, metrics) of an executed hypothesis.
- **Coordinator**: The long-lived agent managing the tree structure and strategy.
- **Executor**: The short-lived agent instance responsible for running code in an isolated environment.

## Success Criteria

### Measurable Outcomes

> Note: As this is a reproduction project, success criteria focus on the *process* of validation and the *existence* of artifacts, rather than new scientific discovery.

- **SC-001**: Artifact Generation Rate: [deferred] of required output files (`tree.json`, `eval_results.json`, `summary.md`) MUST be present and non-empty after a successful run (See US-1).
- **SC-002**: Execution Completeness: The system MUST complete the full benchmark loop (init -> execute -> evaluate -> report) without unhandled exceptions or timeouts (See US-3).
- **SC-003**: Resource Compliance: The peak memory usage of the CI job MUST remain within acceptable RAM limits and the total runtime MUST be [deferred]. (See US-1).
- **SC-004**: Tree Integrity: The generated `tree.json` MUST contain at least one root node, one hypothesis node, and one evidence node with valid linkage (See US-2).
- **SC-005**: Baseline Comparison: The final report MUST explicitly state the achieved metric and the baseline metric, allowing for a direct numerical comparison (See US-3).

## Assumptions

- **Assumption about Environment**: The GitHub Actions free-tier runner (multiple CPU cores, ~7 GB RAM) is sufficient to run the `algotune_knn` demo or a similarly small-scale task, provided the model is a small CPU-tractable variant (e.g., Llama-8B in 4-bit/8-bit quantization if supported on CPU, or a smaller model like Phi-2).
- **Assumption about Dependencies**: The vendored `Arbor` repository's `requirements.txt` and `pyproject.toml` are compatible with the Python version (+) available on the standard GitHub Actions runner.
- **Assumption about Model Access**: The reproduction assumes access to a free or pre-configured API key for the LLM provider (e.g., OpenAI, Anthropic, or a local model if vendored) required by the `arbor` configuration; if no key is provided, the system will fail to execute, which is an expected failure mode for this spec.
- **Assumption about Task Scope**: The "reproduction" is scoped to running the provided `examples/algotune_knn` or `examples/dashboard_demo` tasks, not the full multi-task benchmark suite described in the paper, to ensure feasibility within the 6-hour CI limit.
- **Assumption about Quantization**: If the code requires 8-bit quantization (bitsandbytes), it is assumed that the CPU-only runner supports this via `accelerate` or similar, or that the configuration is adjusted to use standard float32/16 precision to avoid CUDA dependency errors.
