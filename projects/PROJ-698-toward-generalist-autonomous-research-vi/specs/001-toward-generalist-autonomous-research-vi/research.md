# Research: Reproduce & Validate Arbor (Hypothesis-Tree Refinement)

## 1. Executive Summary

This research validates the feasibility of reproducing the "Arbor" autonomous research system on a CPU-only GitHub Actions runner. The core challenge is adapting the system's LLM inference and resource usage to fit within 7GB RAM and 2 CPU cores without GPU acceleration. The chosen benchmark task, `algotune_knn`, is selected for its low computational overhead (tabular data optimization) compared to large-scale NLP tasks.

**Critical Scope Clarification**: This study is a **Reproduction of Mechanism**, not a **Validation of Scientific Efficacy**. Due to the substitution of the paper's large model (likely 7B+) with a CPU-tractable model (e.g., Phi-2) and the limited sample size (Iris, N=150), we **cannot** causally validate the paper's claim of a "2.5x gain". The goal is to verify that the HTR *mechanism* (tree growth, hypothesis generation, evidence linking) functions correctly and to compare the HTR algorithm against a local Random Search baseline under identical constraints.

## 2. Dataset Strategy

### 2.1. Verified Datasets
The `algotune_knn` task in the Arbor codebase typically uses standard benchmark datasets for hyperparameter tuning. Based on the paper's context and standard ML practices:

| Dataset Name | Source/URL | Verification Status | Relevance to Arbor |
| :--- | :--- | :--- | :--- |
| **Iris** | `sklearn.datasets.load_iris()` (Built-in) | Verified (Standard Lib) | Primary dataset for `algotune_knn` demo. Small (150 rows), fits easily in RAM. |
| **Wine** | `sklearn.datasets.load_wine()` (Built-in) | Verified (Standard Lib) | Alternative small tabular dataset for robustness checks (N=178). |
| **UCI Adult** | `https://archive.ics.uci.edu/ml/datasets/adult` (Potential) | **NOT VERIFIED** for this specific run | Too large for quick demo; not used in the minimal reproduction scope. |

**Decision**: The plan will rely exclusively on **built-in scikit-learn datasets** (Iris/Wine) for the `algotune_knn` task. This eliminates the need for external downloads, ensures reproducibility, and guarantees the data fits within the 7GB RAM constraint. No external HuggingFace datasets are required for this specific reproduction scope.

### 2.2. Variable Fit
- **Predictors**: Features from the Iris/Wine dataset.
- **Outcome**: Species classification.
- **Covariates**: None (supervised learning).
- **Fit**: The dataset contains all necessary variables for the KNN algorithm. No missing data handling is required as the built-in dataset is clean.

### 2.3. Power Analysis & Dataset Limitation
- **Sample Size**: N=150 (Iris) is **insufficient** to detect nuanced performance improvements (e.g., 2.5x gain) with statistical power. The variance in KNN performance on such a small dataset is likely dominated by random seed effects.
- **Implication**: This study is **not** powered to reject the null hypothesis that "HTR performs no better than random search" in a generalizable sense.
- **Mitigation**: The goal is **Mechanism Validation** (does the tree grow? does the loop run?). We will use the Wine dataset (N=178) as a secondary robustness check, but the primary focus remains on the *process* of HTR, not the *magnitude* of the gain.

## 3. Statistical & Methodological Rigor

### 3.1. Reproduction Scope & Null Hypothesis
This is a **reproduction study** of the *software mechanism*, not a new hypothesis test for scientific discovery.

- **Null Hypothesis ($H_0$)**: The HTR mechanism does not improve KNN performance over a Random Search baseline (5 random hyperparameter sets) when run on the same data split and environment.
- **Alternative Hypothesis ($H_1$)**: The HTR mechanism yields a higher mean accuracy than Random Search.
- **Test**: Comparison of mean accuracy (HTR run vs. Random Search run) on the Iris dataset.
- **Limitation**: Due to N=150 and model substitution, any observed difference cannot be attributed to the *scientific novelty* of HTR (as claimed in the paper) but only to the *algorithmic difference* between HTR and Random Search in this specific environment.

### 3.2. Model Substitution Confound Analysis
- **Confound**: The paper likely uses a large-scale parameter model. We are using a Phi-2 or smaller model.
- **Impact**: Any performance difference (or lack thereof) is confounded by model capacity. We cannot claim the HTR *mechanism* failed if the score is low; it may simply be the model's inability to generate good hypotheses.
- **Mitigation**: We explicitly acknowledge this limitation. The comparison is between "HTR with Small Model" and "Random Search with Small Model". This validates the *search strategy*, not the *LLM's research capability*.

### 3.3. Measurement Validity
- **Metrics**: Accuracy/F1 are standard and validated by scikit-learn.
- **Baseline**: The baseline is **not** a hardcoded number. It is generated by a separate run of a Random Search script using the *exact same* data split and seed to ensure a fair comparison.

## 4. Compute Feasibility Analysis

### 4.1. Resource Constraints
- **RAM**: 7 GB.
- **CPU**: 2 Cores.
- **Time**: 6 Hours.

### 4.2. Model Strategy
- **Strategy**: The `research.md` mandates checking the `arbor/config` for the model name.
- **Fallback**: If the config specifies a large model, the plan includes a **patch step** to override it with `microsoft/phi-2` (approx 2.7B params) or `google/gemma-2b` in float32, which can run on CPU (albeit slowly) or a smaller distilled model like `TinyLlama/TinyLlama-1.1B-Chat-v1.0`.
- **Quantization**: **Avoid** `bitsandbytes`. Use standard `float32` or `float16` (if supported on CPU) to prevent CUDA errors. If the code forces 8-bit, the plan will modify the loader to skip quantization.

### 4.3. Execution Plan
1.  **Install**: `pip install torch --index-url https://download.pytorch.org/whl/cpu` (Ensures CPU-only wheel).
2.  **Config**: Override `model_name` in `config.yaml` to a small CPU-tractable model.
3.  **Run**: Execute `examples/algotune_knn.py` with a `--max-steps=5` limit to ensure the 6-hour window is respected.
4.  **Baseline**: Run a separate `random_search_knn.py` script with multiple iterations.
5.  **Monitor**: Use `memory_profiler` to log peak RAM.

## 5. Risk Assessment

| Risk | Probability | Impact | Mitigation |
| :--- | :--- | :--- | :--- |
| **CUDA Dependency** | High | Blocking | Explicitly install CPU-only PyTorch; patch code to remove `device_map="cuda"`. |
| **Memory Overflow** | Medium | Blocking | Use small datasets (Iris); stream logs; avoid loading full model weights if possible. |
| **Timeout (6h)** | Medium | Partial Failure | Limit `max_hypotheses` to a small, manageable range; enforce 300s timeout per step. |
| **Missing Dataset** | Low | Blocking | Use built-in `sklearn` datasets as fallback. |
| **Statistical Power** | High | Limitation | Acknowledged in scope; focus on mechanism validation. |

## 6. Decision Log

| Decision | Rationale |
| :--- | :--- |
| **Use Iris Dataset** | Built-in, small, fits RAM, standard for KNN demos. |
| **CPU-Only PyTorch** | Required by CI constraints; avoids CUDA errors. |
| **Small Model Substitution** | Paper model too large; substitution allows mechanism validation. |
| **No Quantization** | `bitsandbytes` often fails on CPU; standard precision is safer. |
| **Random Search Baseline** | Required for a fair, local comparison of the search algorithm. |
| **Dynamic Baseline** | Ensures fair comparison (same split, same seed) rather than hardcoded values. |
