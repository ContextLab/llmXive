## Unresolved concerns

- Task 3.4 proposes 'Leave-One-Repository-Out (LOO-CV)' for a dataset of ~100 repositories. LOO-CV requires fitting the mixed-effects model 100 times. Given the 6-hour runtime constraint (FR-009) and the computational cost of LMM convergence, this approach is likely infeasible and risks timeout or OOM, invalidating the validation strategy. A K-fold (e.g., 5 or 10) CV stratified by repository size is a more robust and tractable alternative for this sample size.
- Task 2.2 plans to fit Log-Normal and Weibull distributions using MLE. However, the plan does not specify a method to handle the heavy right-skew and potential zero/negative values (if any remain after cleaning) that often cause MLE convergence failures in `scipy.stats`. Without a defined fallback (e.g., robust initialization, bounded optimization, or exclusion of extreme tails), the 'fit quality metrics' (KS/AIC) may be unreliable or uncomputable, failing US-2.
- Task 3.2 mandates VIF calculation. VIF is undefined for categorical variables (like 'labels' or 'language') unless they are one-hot encoded. The plan does not specify the encoding strategy for these predictors. If one-hot encoding is used, the resulting high dimensionality (many labels) will likely cause perfect multicollinearity or near-singular matrices, making VIF calculation impossible without a dimensionality reduction step (e.g., grouping rare labels) which is not mentioned.
- Task 4.1 proposes a sensitivity analysis sweeping thresholds {0.01, 0.05, 0.1} to report 'False Positive (Type I) and False Negative (Type II) rates'. In an observational study with no ground truth labels for 'true' significance, Type I/II rates cannot be empirically calculated. The plan must clarify that this will be a 'stability analysis' (proportion of bootstrap resamples significant) rather than a calculation of actual error rates, otherwise the metric is scientifically invalid.
- Task 1.2 flags issues with 'resolution_time > 30 days' as outliers. The plan does not define the statistical method for outlier detection (e.g., IQR, Z-score, or a fixed threshold). A fixed 30-day threshold is arbitrary and may remove valid long-tail data (e.g., complex bugs) that is part of the distribution's shape. The methodology should justify this threshold or use a data-driven method to avoid biasing the distribution fit.
- FR-001 (Rate Limit Wait) and US-1 Acceptance Scenario 3 require a specific '≥60 seconds' wait logic. Task 1.2 mentions 'Explicitly implement the ≥60 seconds wait logic' but the plan text provided in the input does not contain a Task 1.2 with this description; it only lists Task 1.1, 1.2 (generic), 1.3. The specific constraint is missing from the explicit task description in the plan, creating a gap between the spec requirement and the plan element.

- The plan requires a sensitivity analysis to report 'False Positive (Type I) and False Negative (Type II) rates' for different alpha thresholds. The plan conflates 'sensitivity of the test' (power) with 'false negative rate' in a way that requires a known truth set. The only valid sensitivity analysis here is to report how the *set of significant findings* changes (stability), not the error rates. Claiming to calculate FP/FN rates without a ground truth is a category error.
- The plan specifies 'Leave-One-Repository-Out (LOO-CV)' for a dataset of ~100 repositories. The plan does not address the computational complexity or provide a fallback (e.g., K-fold with K=10) if LOO-CV exceeds time/memory limits, risking a failed validation step.

The task description should be revised to 'Report the number of hypotheses changing significance status' or 'Report stability proportion' to align with the observational nature of the data.
