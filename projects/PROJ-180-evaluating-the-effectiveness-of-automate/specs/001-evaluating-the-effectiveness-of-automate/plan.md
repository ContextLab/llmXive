# Implementation Plan: Evaluating Automated Code Review Tools Effectiveness

**Branch**: `001-evaluating-code-review-tools` | **Date**: 2024-01-15 | **Spec**: `specs/001-evaluating-the-effectiveness-of-automate/spec.md`
**Input**: Feature specification from `/specs/001-evaluating-the-effectiveness-of-automate/spec.md`

## Summary

This feature implements a reproducible pipeline to evaluate the effectiveness of three automated code review tools (SonarQube Scanner, DeepSource CLI, CodeClimate Engine) against human review baselines on a representative sample of open-source GitHub repositories. The approach involves: (1) stratified repository acquisition, (2) parallel tool execution, (3) **candidate generation** using keyword heuristics (as per FR-004) and semantic search, (4) **expert validation** of a stratified sample to establish a **Gold Standard** (breaking circularity), (5) **semantic alignment** (AST + embeddings) with strict confidence thresholds (line tolerance removed as a matching method), and (6) statistical analysis (Precision/Recall of *Estimated Defects* via Capture-Recapture, Wilcoxon signed-rank, Fixed Effects with Cluster-Robust SEs). The plan explicitly addresses metric limitations (Recall of Commented Defects vs. Estimated Total Recall) and includes a capture-recapture sensitivity analysis.

### Methodological Reconciliation: FR-004 vs. Validity
FR-004 mandates parsing comments via "keyword heuristics." This plan interprets this as the **retrieval mechanism** (candidate generation), not the definition of ground truth. The **Ground Truth** is defined *exclusively* by expert validation of the candidate set. This satisfies FR-004 (heuristic extraction) while ensuring validity (truth defined by human expert, not keyword overlap).

## Detailed Implementation Phases

### Phase 1: Data Acquisition (FR-001, FR-002, FR-003)
- **Strategy**: Query GitHub API for repos matching language/activity criteria. Apply PESTO filters (license, CI, issues).
- **Execution**: Clone repos, run SonarQube, DeepSource, CodeClimate via Docker (versions from `code/versions.yaml`).
- **Output**: JSON reports for each tool, repo metadata.

### Phase 2: Human Baseline & Gold Standard Construction (FR-004, FR-011, FR-012, FR-013)
- **Candidate Generation**:
 1. Keyword heuristics (bug, security, style) to retrieve PR comments (satisfies FR-004).
 2. **Semantic Search**: Use lightweight CPU embeddings (`all-MiniLM-L6-v2`) to find comments semantically similar to defect descriptions.
- **Expert Validation**: Experts validate a stratified random sample of the *combined* candidate set to confirm **actual defects**. This forms the **Gold Standard**.
 - **Circularity Break**: The heuristic retrieves candidates; the expert defines truth.
- **Metrics**: Cohen's κ on validation. Sensitivity analysis on keyword thresholds.
- **Capture-Recapture**: Estimate total defect population size using overlap between tool findings and human comments to contextualize Recall.

### Phase 3: Semantic Alignment (FR-005)
- **Primary Method**: AST-based diff matching (if available) or Code Embedding similarity (issue description vs. comment text).
- **Threshold**: Match only if confidence > 0.85.
- **Failure Mode**: If confidence < 0.85, mark as **unmatched**. **Line tolerance is NOT used for matching** to avoid bias against coarse-grained tools. Line tolerance is only used for sensitivity analysis (error reporting).
- **Output**: Aligned pairs with match confidence and status (matched, unmatched).

### Phase 4: Metrics & Statistical Analysis (FR-006, FR-007, FR-008, FR-009, FR-010)
- **Metrics**:
 - **Precision**: TP / (TP + FP)
 - **Recall**: **Estimated Recall** (via Capture-Recapture) / Total Estimated Defects.
 - **RCD (Recall of Commented Defects)**: Reported separately as a lower-bound metric.
- **Statistical Tests**:
 - Wilcoxon signed-rank (paired by project) for primary inference.
 - Fixed-effects regression (`metric ~ tool + language + project_size`) with **Cluster-Robust Standard Errors (CRSE)** to handle hierarchical data (issues nested in repos).
 - **Collinearity Diagnostics**: VIF calculation. If VIF > 5, report with caution or switch to Ridge regression.
 - **Power Analysis**: Post-hoc power check. If power < 0.8 for interaction terms, regression is used for descriptive trends only; Wilcoxon tests are primary.
 - **Multiple Comparisons**: Bonferroni correction.
- **Output**: CSV/PNG artifacts.

### Phase 5: Versioning & Artifacts (Constitution V)
- **Hashing**: `code/utils/hasher.py` generates SHA-256 for all data/code artifacts.
- **State Update**: Automated update of `state/projects/PROJ-180-evaluating-the-effectiveness-of-automate.yaml` with new hashes.

## Assumptions

- GitHub REST API rate limits will not prevent retrieval of PR comments for a representative set of repositories within a fixed time budget..
- Static analysis tools can be executed via Docker on CPU-only runners.
- Keyword heuristics + semantic search will capture a representative candidate set for expert validation.
- **Metric Limitation**: Recall is measured as "Estimated Recall" via capture-recapture. RCD is reported as a lower bound.
- **Statistical Power**: A small number of repositories may be underpowered. for complex interactions; Wilcoxon tests are primary, regression is descriptive.
- **Hierarchy**: Cluster-Robust SEs are used to handle hierarchical data without switching to Random Effects models.
- **Collinearity**: Language and size may be correlated; VIF diagnostics will be reported and remediation (Ridge) applied if needed.
- No post-task psychological variables are required.

## Success Criteria

- **SC-001**: Per-tool precision and **estimated recall** metrics computed against the expert-validated Gold Standard.
- **SC-002**: Statistical significance measured against p<0.05 with Bonferroni correction.
- **SC-003**: Pipeline runtime ≤ 5.5 hours.
- **SC-004**: Memory usage ≤ 6 GB.
- **SC-005**: Alignment accuracy ≥90% on expert-validated sample (using AST/semantic match).
- **SC-006**: Mixed-effects model fit (with CRSE) reported with collinearity diagnostics.
