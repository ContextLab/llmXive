# Research: Evaluating Automated Code Review Tools Effectiveness

## Research Questions

1. **Primary**: How do automated code review tools (SonarQube, DeepSource, CodeClimate) compare in terms of precision, recall, and F1 score against a human-review baseline?
2. **Secondary**: Do project characteristics (language, size, activity level) significantly influence tool performance?
3. **Exploratory**: What is the sensitivity of keyword-based heuristic extraction to threshold variations?

## Methodological Decisions

### 1. Tool Execution Environment
- **Decision**: Execute all tools via Docker containers on CPU-only runners.
- **Rationale**: Ensures version pinning (Constitution Principle VI) and reproducibility. Avoids dependency conflicts. CPU-only execution is feasible within the project's time budget for a representative subset of repositories.
- **Reference**: Tool documentation (SonarQube, DeepSource, CodeClimate) confirms Docker support.

### 2. Human Annotation Extraction & Ground Truth
- **Decision**: Use a **dual-source** ground truth: (A) keyword heuristics validated by experts, and (B) a stratified random sample of PR comments manually reviewed regardless of keywords.
- **Rationale**: This breaks the circular validation loop where tools are penalized for detecting bugs missed by the heuristic. The random sample ensures coverage of defects not captured by keywords.
- **Sample Size**:
 - **Heuristic Validation**: A sufficient number of annotations to validate the heuristic's precision/recall.
 - **Random Ground Truth**: A sufficient number of annotations to establish an independent baseline.
 - **Total Ground Truth**: Union of both streams (target ≥ 500 validated annotations).
- **Reference**: Prior studies on code review annotation (e.g., *Beller et al., 2017*) use similar heuristic-based approaches, but this study adds random sampling to mitigate bias.

### 3. Alignment Strategy
- **Decision**: Use AST-based alignment where available; fallback to diff-based or ±5 line tolerance.
- **Rationale**: AST provides semantic accuracy; fallback ensures coverage for languages/tools without AST support (FR-005).
- **Reference**: *Allamanis et al., 2018* on code alignment techniques.

### 4. Statistical Analysis
- **Decision**: Use Wilcoxon signed-rank tests for paired comparisons, permutation-based p-values, and **Mixed-Effects Models (LMM)**.
- **Rationale**: Non-parametric tests handle small sample sizes and non-normal distributions. Mixed-effects models control for project-level heterogeneity and avoid overfitting with small cluster sizes.
- **Reference**: *Cohen, 1988* on statistical power; *Gelman & Hill, 2006* on hierarchical models.

### 5. Multiple Comparisons Correction
- **Decision**: Apply Bonferroni correction to all hypothesis tests *after* permutation-based p-value calculation.
- **Rationale**: Controls family-wise error rate (FR-009, SC-002). The order is: Permutation Test → Raw P-value → Bonferroni Correction.
- **Reference**: *Dunn, 1961* on multiple comparisons.

### 6. Collinearity Handling
- **Decision**: Calculate Variance Inflation Factor (VIF) for all predictors. If VIF > 5, apply Ridge Regression or PCA.
- **Rationale**: Project characteristics (language, size) are likely correlated. This prevents unstable regression coefficients.
- **Reference**: *O'Brien, 2007* on VIF diagnostics.

## Statistical Rigor & Limitations

- **Power Limitation**: Sample size of repositories may limit statistical power. Mixed-effects models are used to mitigate this.
- **Causal Claims**: All findings are framed as associational due to the observational nature of the study. No randomization of tool usage.
- **Collinearity**: Project characteristics (language, size) may be correlated. Diagnostics (VIF) will be included in regression analysis. If high, Ridge Regression is used.
- **Measurement Error**: Keyword heuristics may introduce false positives/negatives. Expert validation and sensitivity analysis address this. The dual-source ground truth ensures tools are not penalized for heuristic misses.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|:--- |:--- |:--- |:--- |
| **Tool Execution Failure** | Medium | High | Retry mechanism (limited attempts); exclude and log failures. |
| **GitHub API Rate Limits** | Medium | Medium | Use authenticated requests; implement exponential backoff. |
| **Memory Overflow** | Low | High | Process repos in chunks; exclude large repos (>7GB RAM). |
| **Alignment Ambiguity** | Medium | Medium | Use tolerance window; flag for manual review. |
| **Low Expert Validation Rate** | Low | High | Automate validation workflow; prioritize high-confidence annotations. |

All external sources are verified, and the data strategy ensures reproducibility.
