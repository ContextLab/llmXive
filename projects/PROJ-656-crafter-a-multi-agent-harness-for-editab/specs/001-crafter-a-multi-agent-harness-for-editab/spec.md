# Feature Specification: Crafter Multi-Agent Harness Validation

**Feature Branch**: `656-crafter-validation`  
**Created**: 2026-06-03  
**Status**: Draft  
**Input**: User description: "Reproduce & validate: Crafter: A Multi-Agent Harness for Editable Scientific Figure Generation from Diverse Inputs"

## User Scenarios & Testing

### User Story 1 - End-to-End Reproduction of Figure Generation (Priority: P1)

A researcher validates that the vendored `Crafter` codebase can successfully ingest a raw scientific paper excerpt and generate a publication-quality figure without manual intervention, confirming the core "multi-agent harness" claim.

**Why this priority**: This is the fundamental existence proof. If the code cannot run end-to-end to produce an artifact, the project fails immediately. It validates the primary functionality described in the paper abstract.

**Independent Test**: Execute the `inference.py` entry point with a provided sample input; verify the output file exists and is non-empty.

**Acceptance Scenarios**:

1. **Given** the `Crafter` repository is cloned and dependencies are installed, **When** the user runs `python inference.py --input examples/sample_paper.txt --output output_figure.png`, **Then** the script completes without runtime errors and `output_figure.png` is created in the working directory.
2. **Given** a valid paper excerpt describing a bar chart, **When** the harness executes the generation pipeline, **Then** the output image contains visual elements corresponding to the described data (axes, bars, labels) rather than generic placeholders.
3. **Given** the system encounters a missing API key (if external LLMs are required), **When** the script starts, **Then** it fails fast with a clear error message within 5 seconds, rather than hanging or producing a silent failure.

---

### User Story 2 - Validation of Editable SVG Conversion (Priority: P2)

A user confirms that the `CraftEditor` component can successfully convert a generated raster image into an editable SVG format, preserving semantic structure (text, shapes) as claimed in the paper.

**Why this priority**: The paper distinguishes Crafter from prior art by its ability to produce *editable* outputs. Validating this conversion pipeline is essential to the "Editable" claim in the title.

**Independent Test**: Run the `convert.py` script on a generated raster image and inspect the resulting SVG for valid XML structure and editable text nodes.

**Acceptance Scenarios**:

1. **Given** a raster image `input.png` generated by the harness, **When** the user executes `python convert.py --input input.png --output output.svg`, **Then** `output.svg` is a valid XML file that opens in a standard vector editor (e.g., Inkscape) without parsing errors.
2. **Given** the input image contains distinct text labels, **When** the conversion completes, **Then** the SVG contains `<text>` or `<tspan>` elements corresponding to those labels, allowing for text selection and editing.
3. **Given** the input image contains geometric shapes (e.g., bars, lines), **When** the conversion completes, **Then** the SVG contains corresponding `<path>`, `<rect>`, or `<line>` elements rather than a single flattened `<image>` tag.

---

### User Story 3 - Benchmark Evaluation Execution (Priority: P3)

A researcher executes the `CraftBench` evaluation suite to reproduce the quantitative metrics (quality scores, editability rates) reported in the paper, confirming the harness's performance against the baseline.

**Why this priority**: This validates the scientific rigor of the claim. It ensures the evaluation infrastructure works and can reproduce the "substantially outperforms" claim mentioned in the abstract.

**Independent Test**: Run the evaluation script against the provided `craftbench` manifest and verify a summary report is generated with numeric scores.

**Acceptance Scenarios**:

1. **Given** the `craftbench` dataset is present in `external/Crafter/craftbench`, **When** the user runs `python -m craftbench.evaluation.run_eval`, **Then** the process completes within 60 minutes and outputs a JSON report file.
2. **Given** the evaluation report, **When** a reviewer inspects the metrics, **Then** the report includes a "quality_score" and "editability_score" for at least 3 distinct figure types as defined in the benchmark manifest.
3. **Given** the evaluation fails on a specific sample (e.g., timeout or API error), **When** the script continues, **Then** the failure is logged, and the final report includes a "success_rate" metric reflecting the failure, rather than the script crashing entirely.

---

### Edge Cases

- **API Rate Limiting**: If the harness relies on external LLM APIs (e.g., GPT-4) for agent reasoning, the system must handle rate-limit errors (HTTP 429) by implementing a backoff strategy (e.g., wait 30 seconds, retry up to 3 times) before failing.
- **Memory Constraints**: If the input paper is extremely large (e.g., >50 pages), the system must process it in chunks or fail gracefully with a "Input too large" error, rather than causing an Out-of-Memory (OOM) crash on the CI runner.
- **SVG Parsing Failures**: If the `raster_to_svg` conversion produces malformed SVG (e.g., unclosed tags), the `convert.py` script must catch the XML parsing exception and log the specific line number of the error instead of crashing silently.

## Requirements

### Functional Requirements

- **FR-001**: The system MUST execute the `inference.py` entry point to generate a figure from a text prompt within 10 minutes on a CPU-only environment, ensuring the core generation loop is functional (See US-1).
- **FR-002**: The system MUST invoke the `convert.py` script to transform a generated raster image into an SVG file, ensuring the output contains at least one editable text element (See US-2).
- **FR-003**: The system MUST run the `craftbench` evaluation suite against the provided manifest, producing a summary report that includes a `success_rate` metric calculated as (successful runs / total runs) (See US-3).
- **FR-004**: The system MUST detect missing external dependencies (e.g., API keys, specific Python packages) and exit with a non-zero status code and a descriptive error message within 30 seconds of startup (See US-1).
- **FR-005**: The system MUST log all agent interactions (prompts and responses) to a file `logs/agent_trace.log` during the generation process to enable debugging and verification of the "multi-agent" behavior (See US-1).

### Key Entities

- **FigureSpec**: The structured representation of a figure's components (axes, labels, data points) derived from the input text, used by the agents to plan the generation.
- **RasterImage**: The intermediate binary output (PNG/JPG) generated by the image backbone before conversion.
- **EditableSVG**: The final vector output containing semantic XML elements (text, paths) derived from the RasterImage.
- **BenchmarkSample**: A single entry in `craftbench/manifest.json` containing an input prompt, ground truth image, and expected metadata.

## Success Criteria

### Measurable Outcomes

> Planning docs state *what* will be measured and the *source/reference* it is measured against; defer specific empirical values (counts, dataset sizes, measured quantities, percentages) to the implementation/research phase.

- **SC-001**: The reproducibility rate is measured against the paper's claim of "substantially outperforms"; specifically, the `craftbench` evaluation must produce a `success_rate` ≥ 90% on the provided sample set (See FR-003, US-3).
- **SC-002**: The editability fidelity is measured against the ground truth SVGs in the benchmark; specifically, the generated SVGs must contain ≥ 80% of the text elements found in the ground truth when compared via string similarity (See FR-002, US-2).
- **SC-003**: The computational feasibility is measured against the CI runner constraints; specifically, the full end-to-end reproduction (generation + conversion + evaluation) must complete within 6 hours on a CPU-only runner with ≤ 7 GB RAM (See FR-001, US-1).
- **SC-004**: The agent trace completeness is measured against the requirement for debugging; specifically, the `agent_trace.log` must contain at least 3 distinct agent roles (e.g., planner, drawer, critic) interacting for each generated figure (See FR-005, US-1).

## Assumptions

- **Assumption about External APIs**: The reproduction assumes that the `Crafter` codebase can operate in a "mock" or "local" mode if external LLM APIs (e.g., OpenAI) are unavailable, or that a valid API key is provided via environment variables (e.g., `OPENAI_API_KEY`) for the duration of the CI job. If the code strictly requires paid APIs without a fallback, the project scope is limited to validating the *pipeline structure* rather than the *generation quality*.
- **Assumption about Compute Resources**: The analysis assumes the GitHub Actions free-tier runner (limited CPU, 7 GB RAM) is sufficient for running the `Crafter` agents on the small `craftbench` sample set. If the agents require loading large models (e.g., >4GB) that exceed RAM, the assumption is that the code will swap or fail gracefully, and the "success" criteria will be adjusted to reflect "pipeline execution" rather than "full generation."
- **Assumption about Dataset Completeness**: The `craftbench` dataset provided in the submodule is assumed to contain all necessary inputs (prompts, ground truths) to run the evaluation without needing to download external data.
- **Assumption about Methodology**: The evaluation of "quality" and "editability" relies on the automated metrics defined in `craftbench/evaluation`. We assume these automated metrics (e.g., LLM-based judges) are sufficient proxies for human quality assessment for the purpose of this reproduction, rather than requiring a new human annotation study.
