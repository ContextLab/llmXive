# Data Model: Evaluating the Impact of Code Complexity on LLM Code Understanding

## Entity Definitions

### 1. CodeSnippet
Represents a single Python function extracted from CodeSearchNet.
- `snippet_id`: Unique string (hash of code + index).
- `source_code`: Raw string content of the function.
- `ground_truth`: Dictionary containing expected outputs (e.g., `{"docstring": "...", "bug_injected": false, "injected_bug_type": null}`).
- `complexity_metrics`: Dictionary containing `cyclomatic`, `halstead_volume`, `maintainability_index`.
- `status`: String ("valid", "syntax_error", "context_exceeded").
- `complexity_tier`: String ("Low", "Medium", "High") based on stratified sampling logic.

### 2. InferenceResult
Represents the output of a single LLM run for a specific task.
- `result_id`: Unique string.
- `snippet_id`: Foreign key to CodeSnippet.
- `task_type`: Enum ("summarization", "bug_detection", "code_completion").
- `model_output`: String generated by the LLM.
- `reasoning`: String (if model provides reasoning).
- `execution_status`: Enum ("success", "failed", "timeout", "error", "context_exceeded").
- `is_valid_response`: Boolean (True if output is non-empty and not an error).

### 3. MetricRecord
Aggregated performance metrics for a single snippet-task pair.
- `metric_id`: Unique string.
- `snippet_id`: Foreign key.
- `task_type`: Enum.
- `bleu_4`: Float (0.0 - 1.0).
- `rouge_l`: Float (0.0 - 1.0).
- `execution_pass_rate`: Float (0.0 or 1.0 for binary tasks).
- `bug_detected`: Boolean (for bug detection task; True if injected bug found).
- `substring_match_score`: Float (for bug detection; 0.0-1.0).

### 4. BinningMetadata
Records the boundaries used for stratification and analysis.
- `metric_name`: String (e.g., "cyclomatic_complexity").
- `low_threshold`: Float (33rd percentile value).
- `high_threshold`: Float (67th percentile value).
- `low_percentile`: Integer (33).
- `high_percentile`: Integer (67).
- `distribution_check`: String ("passed", "failed").

### 5. AnalysisReport
Final statistical output.
- `report_id`: Unique string.
- `correlation_matrix`: JSON object mapping (metric, task) → (rho, p_value).
- `glm_coefficients`: JSON object mapping (metric, task) → (coef, se, p_value).
- `binning_strategy`: JSON object defining tertile boundaries (from `BinningMetadata`).
- `hypothesis_status`: Enum ("supported", "rejected", "inconclusive").
- `vif_check`: JSON object with `max_vif` and `pca_applied` status.
- `denominator`: Integer (total valid snippets).
- `numerator`: Integer (successful responses).

## Data Flow

1.  **Input**: Raw Parquet files from CodeSearchNet.
2.  **Processing**:
    -   `01_data_acquisition.py` → `data/raw/codesearchnet_raw.parquet`
    -   `02_complexity_annotation.py` → `data/processed/annotated_snippets.csv` (includes `complexity_metrics`, `complexity_tier`)
    -   `03_inference_pipeline.py` → `results/inference_logs.jsonl`
    -   `04_metric_calculation.py` → `results/metrics.csv`
    -   `05_statistical_analysis.py` → `results/analysis_report.json`, `results/binning_metadata.json`
    -   `06_report_generation.py` → `results/final_report.md`

## Storage Constraints

-   **Raw Data**: ~500 MB (compressed parquet).
-   **Processed Data**: ~100 MB (CSV).
-   **Results**: ~50 MB (JSONL/CSV).
-   **Total Disk**: < 1 GB (well within 14 GB limit).
