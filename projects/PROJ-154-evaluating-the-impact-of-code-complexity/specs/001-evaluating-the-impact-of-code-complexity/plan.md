# Implementation Plan: Evaluating the Impact of Code Complexity on LLM Code Understanding

## Summary

This project investigates the relationship between code complexity metrics (Cyclomatic Complexity, Halstead Volume, Maintainability Index) and the performance of Large Language Models (LLMs) on **Code Summarization** tasks. 

**Scope Change Note**: The original specification included "Bug Detection" and "Code Completion" tasks. However, these tasks require synthetic ground truths (injected bugs, truncated code) that introduce severe construct validity threats, measuring trivial error detection or memorization rather than genuine "understanding" of complex code. To ensure scientific rigor and validity, this plan restricts the scope to **Code Summarization**, which utilizes the native `docstring` ground truth available in the CodeSearchNet dataset.

The technical approach involves: (1) downloading the CodeSearchNet Python subset, (2) computing complexity metrics using the `radon` library, (3) executing inference with a CPU-tractable, -bit quantized `Phi-3-mini-4k-instruct` model on a **stratified** dataset, and (4) performing statistical analysis (Spearman correlation, GLM with VIF check) to test the hypothesis of a negative relationship.

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

1.  **I. Reproducibility**: The plan mandates pinned versions in `requirements.txt`, random seed fixation in `code/`, and deterministic `radon` execution. External datasets are fetched from canonical HuggingFace URLs.
2.  **II. Verified Accuracy**: All dataset citations (CodeSearchNet) map strictly to the verified URLs provided in the spec. No fabricated URLs are used.
3.  **III. Data Hygiene**: Raw data is downloaded to `data/raw/` and checksummed. Derived complexity metrics are stored in `data/processed/` without modifying raw files.
4.  **IV. Single Source of Truth**: All statistical claims in the final report will trace back to specific rows in the `results/analysis.csv` generated by the pipeline.
5.  **V. Versioning Discipline**: Artifacts (datasets, models, scripts) will be tracked via content hashes in the project state file.
6.  **VI. Computational Determinism**: The plan explicitly requires recording the **exact `radon` version and configuration flags** in `data/processed/metadata.json` to ensure complexity metrics remain consistent across runs, preventing drift in the independent variable.
7.  **VII. Statistical Rigor**: The plan mandates:
    -   **Binning Strategy**: Explicitly defining **Low/Medium/High tertiles** for complexity stratification before analysis.
    -   **Multicollinearity Check**: Performing a Variance Inflation Factor (VIF) check. **Decision Rule**: If VIF > 5 for any predictor, the analysis will fall back to Principal Component Analysis (PCA) of the complexity metrics to ensure stable coefficient estimates.
    -   **Significance Threshold**: Pre-defining α = 0.05.
    -   **Contract Validation**: The output of the Analysis phase (Phase 3) MUST be validated against the `contracts/analysis_schema.yaml` contract before any report is generated.
