## Unresolved concerns

- The plan explicitly states 'Causal Inference: Not applicable' and frames the work as 'System Validation' (pass/fail). However, the spec title 'Reproduce & validate: OmniRetrieval' and the use of standard benchmarks (BEIR, Spider, LC-QuAD) imply an intent to validate the *performance* claims of the original paper (e.g., accuracy, F1, execution correctness). A binary 'does it run?' test cannot validate a retrieval system's effectiveness. The methodology conflates 'operational integrity' with 'scientific validation'.
- Success Criteria SC-002 ('Correct Dispatch') relies on 'manual inspection of execution logs'. This is not a reproducible or scalable validation method for a research pipeline. The plan lacks an automated metric to verify that the 'unified' dispatch logic actually preserves structural affordances (e.g., comparing generated SQL/SPARQL against gold standards) rather than just checking log strings.
- The sample size (50 queries per dataset) is justified for 'system validation' but is statistically insufficient to validate the *performance* claims of a retrieval system on heterogeneous benchmarks. If the goal is to claim the system 'works' for retrieval, the methodology must define a metric (e.g., Execution Accuracy for SQL, nDCG for Text) and a threshold, not just 'zero crashes'.
- The plan cites `datasets.load_dataset('lc-quad')` as the primary source for the SPARQL benchmark. The standard LC-QuAD benchmark (1000+ queries) is typically hosted as `lc-quad` or `lc-quad-2` on HuggingFace, but the specific dataset ID `lc-quad` often points to a deprecated or incomplete version, or requires a specific split (e.g., `lc-quad-2`). More critically, the plan assumes the dataset contains `gold_sparql` and `graph_path` fields directly accessible via the loader. LC-QuAD datasets on HF often require post-processing to extract the RDF graph files (Turtle/N-Triples) from the dataset's metadata or separate download links, which contradicts the 'streaming' and 'CPU-only' constraints if the full graph must be materialized. The plan must verify the exact HF dataset ID and confirm the `graph_path` field is directly available or define the extraction logic to avoid runtime failures.
- The plan relies on `datasets.load_dataset('spider')` for the SQL engine. The Spider dataset on HuggingFace is a complex multi-file dataset (train, dev, test, databases). The plan assumes a simple loader exists that provides `db_id` and `gold_sql` alongside the database files. However, the Spider dataset on HF often requires manual handling of the database zip files and schema mapping. The plan's assumption that `datasets.load_dataset('spider')` will directly yield `db_path` (path to SQLite) is likely incorrect; the loader usually returns the schema and queries, requiring the user to download and unzip the database files separately. This gap in the data loading strategy risks violating the 'graceful degradation' and 'no external dependencies' constraints if the database files are not automatically fetched and unpacked.
- Phase 0 Action 1 states: 'Investigate the "NO verified source URL" status...'. While the plan correctly identifies the need to use HuggingFace, it should explicitly cite the specific HuggingFace dataset IDs (e.g., `beir`, `spider`, `lc-quad-2`) in the plan itself rather than deferring this investigation to the implementation phase. The plan should assert the *intended* source IDs to allow for a feasibility check now, rather than treating the dataset selection as an open research question during implementation.
- The 'Constitution Check' section lists generic principles (Reproducibility, Data Integrity) but fails to explicitly map to the per-project `constitution.md` (FR-030) or cite specific constitutional principles (e.g., Principle I: SSoT, Principle V: Real-call testing). The plan must demonstrate explicit compliance with the project's specific constitutional constraints, not just general research best practices.
- Phase 0 states: 'Investigate the "NO verified source found" status...'. This contradicts the `research.md` which already resolved this by identifying HuggingFace `datasets.load_dataset` as the canonical source. The plan should reflect that this research is *complete* (as per the 'Research Complete' header) and move to defining the specific HF dataset IDs, rather than treating it as an open investigation.
- FR-004 (Syntax Validation) is mapped to 'Phase 2 (engine pre-checks)', but the `data-model.md` and `contracts/output.schema.yaml` do not define a specific schema field for 'validation_status' or 'syntax_error_details' in the output. The plan implies a validation step, but the data model lacks the structure to record the *result* of this validation (pass/fail) in the final artifact, creating a gap between the functional requirement and the output contract.
- The plan validates 'Dispatch Correctness' (SC-002) via 'manual inspection of execution logs' (spec.md:SC-002). This is a subjective, non-reproducible validation method for a core architectural claim. A scientific validation of a routing system requires an automated, deterministic oracle (e.g., asserting that a query labeled 'SQL' in the dataset metadata *must* trigger the SQL engine, and asserting failure if it does not). Manual inspection cannot be automated in CI and introduces observer bias.
- The plan explicitly states 'No external LLM API keys (mock/local only)' and uses a 'mock LLM client' (spec.md:US-1). However, the 'OmniRetrieval' paper likely claims performance improvements derived from a unified generation/retrieval loop. Validating the *system's operational integrity* (does it run?) is distinct from validating the *paper's scientific claims* (does it work better?). The plan conflates these, risking a 'null result' where the system runs perfectly but fails to reproduce the paper's actual performance claims because the core generative component is mocked or replaced with a lightweight model (MiniLM) that lacks the necessary reasoning capabilities.
- The plan uses `all-MiniLM-L6-v2` for the Text Retrieval Engine. This is a bi-encoder trained for semantic similarity, not a generative model capable of complex reasoning or multi-hop retrieval often required in heterogeneous benchmarks. If the 'OmniRetrieval' framework relies on a generative component to unify the retrieval logic (as implied by 'Unified Retrieval'), replacing it with a static bi-encoder fundamentally alters the system architecture being tested, making the validation of the 'unified' claim invalid.
