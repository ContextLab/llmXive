# Assessing Energy Consumption of LLM Inference for Code Completion

**Field**: computer science  

## Research question

How does the energy consumption per generated token differ among open‑source code‑completion models (e.g., GPT‑2‑small, CodeBERT, StarCoder‑base) when evaluated on the HumanEval benchmark?

## Motivation

Deploying large language models for code completion is becoming routine, yet the hidden environmental cost of inference is rarely quantified. Understanding the energy‑to‑token trade‑off can guide developers toward more sustainable tooling choices and help the community benchmark models not only on accuracy but also on ecological impact.

## Related work

- [Compute and Energy Consumption Trends in Deep Learning Inference (2021)](https://arxiv.org/abs/2109.05472) — Provides baseline methods for measuring inference energy and discusses scaling trends that inform our instrumentation approach.  
- [Position: LLM Inference Should Be Evaluated as Energy-to-Token Production (2026)](https://arxiv.org/abs/2605.11733) — Argues for energy‑per‑token as a primary metric for LLM deployment, directly motivating our choice of evaluation metric.  

## Expected results

We anticipate a monotonic increase in energy‑per‑token with model size, while larger models may achieve higher correctness rates on HumanEval. The key finding will be the quantified trade‑off curve (energy per token vs. % of correct solutions). Statistical confirmation will rely on ANOVA/linear regression showing significant differences between models (p < 0.05).

## Methodology sketch

- **Data acquisition**: `wget` the HumanEval benchmark dataset from its GitHub repository.  
- **Model collection**: Use `git clone` / `huggingface_hub` to download GPT‑2‑small, CodeBERT, and StarCoder‑base from Hugging Face.  
- **Instrumentation**: Install the `codecarbon` Python library and configure it to log CPU power draw, runtime, and energy (kWh) for each inference run on the GitHub Actions runner.  
- **Inference runs**: For each HumanEval problem, generate code completions with each model using a deterministic temperature (0.0) and a fixed max‑token budget. Record the number of tokens emitted.  
- **Metric computation**: Compute (a) energy per generated token, (b) energy per correctly solved problem (using the HumanEval pass/fail criteria), and (c) latency per token.  
- **Statistical analysis**: Perform one‑way ANOVA to test differences in energy‑per‑token across models; follow with post‑hoc Tukey HSD. Fit a linear regression of energy‑per‑token vs. model parameter count to assess scaling.  
- **Visualization**: Produce bar plots of energy‑per‑token, scatter plots of energy vs. accuracy, and a trade‑off curve overlaying all models.  
- **Reproducibility**: Script the entire pipeline (`bash` + `python`) so that a single `./run.sh` on a fresh GitHub Actions runner reproduces all numbers and figures.  
- **Reporting**: Summarize results, discuss sustainability implications, and draft practical guidelines for energy‑aware model selection.
