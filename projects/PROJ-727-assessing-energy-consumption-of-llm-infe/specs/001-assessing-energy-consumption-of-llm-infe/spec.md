# Specification: Assessing Energy Consumption of LLM Inference for Code Completion

## 1. Introduction

This project aims to quantify the energy consumption of Large Language Models (LLMs) during code completion inference tasks. We will measure energy usage (kWh), runtime, and tokens generated for specific models on the HumanEval dataset.

## 2. Functional Requirements

### FR-001: Model Selection and Feasibility
**Requirement**: The system must evaluate GPT-2-small, CodeBERT, and StarCoder models.
**Feasibility Constraint & Amendment**: Due to RAM constraints on the target execution environment (CPU-only, limited memory), the original specification of "StarCoder-base" (approx. 15B parameters) is infeasible.
**Amendment**: The model "StarCoder-base" is explicitly replaced with **"StarCoder-1B"** (approx. 1.3B parameters) for all inference tasks. This substitution is authorized to ensure the pipeline can complete without Out-of-Memory (OOM) errors while maintaining the comparative analysis of model size vs. energy efficiency.

### FR-002: Energy Measurement
The system must use `codecarbon` to measure energy consumption (kWh) and power (W) during inference.

### FR-003: Token Counting
The system must count the number of tokens generated by each model for every problem in the dataset.

### FR-004: Evaluation
The system must evaluate generated code against the HumanEval test suite and record a pass/fail status.

### FR-005: Statistical Analysis
The system must perform Repeated-Measures ANOVA, Tukey HSD, and linear regression on the collected data.

### FR-006: Visualization
The system must generate bar plots and scatter plots visualizing energy vs. performance metrics.

### FR-009: Edge Case Handling
The system must handle cases where calibration fails or zero tokens are generated by recording `null` values appropriately.

### FR-010: Calibration
The system must include a CPU-bound calibration loop to validate `codecarbon` power draw detection.

### FR-011: Data Aggregation
The system must filter out rows with `null` energy or zero tokens before aggregation.

### FR-012: Sensitivity Analysis
The system must perform a sensitivity analysis by perturbing energy values by ±10% and re-running ANOVA.

## 3. User Stories

### US-1: Quantify Energy-to-Token Metrics
**As a** researcher,
**I want** to execute inference for GPT-2-small, CodeBERT, and StarCoder-1B on HumanEval using CPU,
**So that** I can log energy (kWh), runtime, and tokens generated.

**Acceptance Criteria**:
1. Models are loaded sequentially and unloaded to free RAM.
2. `codecarbon` logs are generated for all three models.
3. Results are written to `data/processed/energy_results_raw.csv`.
4. The substitution of StarCoder-1B for StarCoder-base is applied.

### US-2: Statistical Analysis
**As a** data scientist,
**I want** to perform Repeated-Measures ANOVA, Tukey HSD, and regression,
**So that** I can determine if model size significantly impacts energy efficiency.

### US-3: Sustainability Trade-off Visualizations
**As a** stakeholder,
**I want** to see bar plots and scatter plots of energy vs. accuracy,
**So that** I can understand the trade-offs between model performance and environmental impact.

## 4. Data Model

- **Raw Data**: `data/raw/human_eval_data.jsonl` (HumanEval dataset)
- **Intermediate Data**: `data/processed/energy_results_raw.csv`
- **Aggregated Data**: `data/processed/energy_results_aggregated.csv`
- **Statistics**: `data/processed/stats_report.csv`
- **Sensitivity**: `data/processed/sensitivity_delta.csv`
- **Visualizations**: `data/processed/energy_bar.png`, `data/processed/tradeoff_scatter.png`

## 5. Constraints

- **Hardware**: CPU-only execution environment.
- **Memory**: Limited RAM; StarCoder-1B is the largest model permitted.
- **Dependencies**: No GPU quantization or 8-bit loading is permitted.
- **Data Integrity**: All analysis must use real HumanEval data and real `codecarbon` logs. No synthetic data.
