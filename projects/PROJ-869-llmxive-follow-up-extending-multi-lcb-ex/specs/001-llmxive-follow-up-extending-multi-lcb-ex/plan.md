# Implementation Plan: llmXive follow-up: extending "Multi-LCB: Extending LiveCodeBench to Multiple Programming Languages"

**Branch**: `001-llmxive-multilingual-logic-transfer` | **Date**: 2026-07-13 | **Spec**: `specs/001-llmxive-multilingual-logic-transfer/spec.md`
**Input**: Feature specification from `specs/001-llmxive-multilingual-logic-transfer/spec.md`

## Summary

This feature implements a rigorous evaluation pipeline to test the "Logic Anchor" hypothesis: that providing a partial logic trace (initial steps of a ground-truth Python solution) as a few-shot anchor improves the ability of LLMs to generate correct code in target languages (e.g., Rust, Kotlin) where they previously failed. The system will: (1) extract logic anchors from verified Python solutions in the Multi-LCB dataset, (2) filter a task set where models fail blindly but succeed in Python, (3) **re-verify the blind baseline** on the final filtered set to ensure paired data integrity, (4) run paired "blind" vs. "guided" inference on a CPU-constrained runner, (5) execute generated code in a sandboxed environment with strict timeouts, and (6) perform paired statistical analysis (McNemar's + Bootstrap) with dynamic power adjustment to handle low-baseline scenarios.

## Technical Context

**Language/Version**: Python 3.11  
**Primary Dependencies**: `datasets` (Hugging Face), `transformers` (CPU-optimized), `torch` (CPU), `scikit-learn`, `ast` (stdlib), `subprocess` (stdlib), `pytest`  
**Storage**: Local ephemeral storage (`data/` for cached datasets, `results/` for logs); no persistent database.  
**Testing**: `pytest` (unit tests for anchor extraction, error categorization; integration tests for sandbox execution).  
**Target Platform**: Linux (GitHub Actions free-tier: multiple vCPUs, 7GB RAM, 14GB disk).  
**Project Type**: Research pipeline / CLI tool.  
**Performance Goals**: Process a **target of 200 tasks** (blind + guided) within 6 hours; sandbox timeout per test case ≤ 10s. *Note: If the filtered set is < 200 or the baseline Pass@1 is too low for valid power, the study reports the limitation and adjusts the target N dynamically.*  
**Constraints**: CPU-only execution (no local GPU); strict memory footprint (<7GB RAM); no external data gateways requiring credentials.  
**Scale/Scope**: A set of algorithmic tasks (stratified by difficulty/topic); Two conditions (blind, guided); Several languages (Python, Rust, Kotlin, Go).

## Complexity Tracking

> **Fill ONLY if Constitution Check has violations that must be justified**

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| **Strict 10s Timeout** | Prevents infinite loops from exhausting the 6h CI budget. | No timeout would risk runner crash or incomplete batch processing. |
| **AST-based Anchor Extraction** | Ensures "Partial Logic Trace" is algorithmic, not syntactic. | Simple string slicing fails to capture distinct algorithmic steps (e.g., loop entry vs. body). |
| **Paired Statistical Test** | Required by Constitution Principle VII to control for task difficulty variance. | Unpaired t-test would be invalid due to non-independent task sets. |
| **Dynamic Power Adjustment** | Required to handle the risk of a near-zero baseline in the filtered set. | Fixed N=200 would force an underpowered study or invalid test if baseline is 0.0. |
| **Re-verified Baseline** | Required to ensure the "Blind" state in the final test matches the "Blind" state in the filter. | Using the filter aggregate as the "Blind" state risks pairing errors if stochasticity drifts. |

## Implementation Phases & Tasks

### Phase 0: Feasibility & Setup
- **T001**: Verify dataset accessibility and checksums.
- **T002**: Setup virtual environment and install dependencies.
- **T003**: **Model Feasibility Gate**: Run a single inference step to measure throughput/memory. If > 30s or OOM, trigger GPU escape hatch or downgrade model. **(Pre-requisite for T019)**.

### Phase 1: Data Preparation & Filtering
- **T010**: Load Multi-LCB dataset via streaming.
- **T011**: Extract ground-truth Python solutions.
- **T012**: **Initial Pool Selection**: Select tasks where model previously failed in target language but succeeded in Python (verified against ground truth).
- **T016**: **Stochasticity Filter**: Re-run "blind" condition **3 times** on candidate tasks.
    - **Constraint**: All runs MUST use a fixed random seed and fixed temperature (a predefined constant) to ensure reproducibility.
    - **Verification**: Log seed/temperature for each run. Count failures. Retain only tasks failing ≥ 2/3 runs.
- **T017**: **Attrition Handling**: If filtered set < 200, sample replacements from the next pool, maintaining stratification (Topic ≤ 30%, Difficulty balanced).
- **T015**: **Finalize Dataset**: Save `data/processed/filtered_tasks.json` with SHA-256 checksum. **(Pre-requisite for T019, T039, T022)**.
- **T018**: **Extract Anchors**: Parse Python solutions to extract first algorithmic steps as **language-agnostic pseudo-code**.

### Phase 2: Execution & Evaluation
- **T019**: **Load Model**: Load the quantized model for inference (after T003 success).
- **T039**: **Re-verify Baseline**: Re-run the "blind" condition on the **final filtered set** (T015) to establish the empirical baseline for the paired test. **(Pre-requisite for T022)**.
- **T022**: **Run Guided Condition**: Generate code for the *same* 200 tasks with the Partial Logic Trace anchor.
- **T023**: **Execute & Classify**: Run sandbox execution.
    - **Logic Transfer Failure Logic**: Explicitly verify if generated code implements the 3 anchor steps (keyword/control-flow matching).
    - **Categorize**: Syntax, Library, Runtime, Logic Transfer Failure.
- **T024**: **Stability Monitoring**: Track runner crashes/hangs to calculate Sandbox Stability Rate (SC-005).

### Phase 3: Analysis & Reporting
- **T025**: **Statistical Analysis**:
    - Perform McNemar's test on paired Pass/Fail data (T039 vs T022).
    - Perform Bootstrap Resampling to account for selection bias (scientific soundness).
    - Calculate "Completion of Remaining Steps" metric (non-trivial metric).
- **T026**: **Generate Report**: Output `statistical_report.json` with p-values, error distribution, and stability rate.

### Phase 4: Power & Bias Mitigation (Dynamic)
- **T040**: **Power Check**: If the observed baseline Pass@1 (from T039) is < 0.05 or the sample size is < 50, flag the study as "Underpowered" and adjust the statistical test (e.g., switch to exact binomial or report descriptive statistics only).
- **T041**: **Report Limitations**: Explicitly document the power limitation in the final report if T040 triggers.

## Task Ordering Rationale
- **T003 (Feasibility)** precedes **T019 (Load Model)** to prevent resource exhaustion.
- **T016 (Filter)** and **T017 (Attrition)** precede **T039 (Re-verify Baseline)** and **T022 (Guided Run)** to ensure the correct dataset is used.
- **T039 (Re-verify Baseline)** is now explicitly placed *after* filtering and *before* the Guided run to ensure the "Blind" state for the paired test is derived from the exact same execution context as the "Guided" state, eliminating the risk of stochasticity drift.
- **T025 (Analysis)** depends on the paired results from T039 and T022.
- **T040 (Power Check)** runs after T039 to determine if the study can proceed with McNemar's or must switch to a fallback.
