# Evaluating the Impact of Code Generation Models on Code Review Efficiency  

**Field**: computer science  

## Research question  

Do code snippets generated by large language models (LLMs) require more reviewer effort—measured by review time, number of comments, and perceived difficulty—than comparable human‑written snippets for the same programming task?  

## Motivation  

LLMs now produce functional code that can be inserted directly into software projects, yet little is known about how such automatically generated code affects downstream development activities. Code review is a critical gate‑keeping step; if LLM‑generated code is systematically harder to review, teams may need new tooling or guidelines. Quantifying this impact fills a gap between code generation performance metrics and real‑world development productivity.  

## Related work  

- [Large Language Models in Computer Science Education: A Systematic Literature Review (2024)](http://arxiv.org/abs/2410.16349v1) — surveys LLM applications in CS education, highlighting concerns about code understandability that are relevant to review effort.  
- [Execution-based Evaluation for Data Science Code Generation (2022)](http://arxiv.org/abs/2211.09374v1) — proposes execution‑based metrics for code generation quality, showing the need for downstream task‑specific evaluations.  
- [DeepCodeSeek: Real-Time API Retrieval for Context-Aware Code Generation (2025)](http://arxiv.org/abs/2509.25716v1) — introduces a retrieval‑augmented generation system that improves functional correctness, but does not assess review implications.  
- [EVOR: Evolving Retrieval for Code Generation (2024)](http://arxiv.org/abs/2402.12317v2) — demonstrates state‑of‑the‑art code generation pipelines; its benchmarks focus on BLEU‑style scores rather than human‑centric metrics.  
- [JaCoText: A Pretrained Model for Java Code-Text Generation (2023)](http://arxiv.org/abs/2303.12869v1) — provides a strong baseline for Java code generation, useful as a source model in our experiments.  
- [ReCode: Robustness Evaluation of Code Generation Models (2022)](http://arxiv.org/abs/2212.10264v1) — studies brittleness of generated code, motivating the need to examine downstream review robustness.  

## Expected results  

We anticipate that LLM‑generated snippets will exhibit higher cyclomatic complexity and lower readability scores, leading to statistically significant increases in (1) median review time (measured via timestamps in real PR data) and (2) number of reviewer comments, after controlling for lines of code and task difficulty. A mixed‑effects regression confirming these effects (p < 0.05) would support the hypothesis; a null finding would suggest current models produce review‑friendly code.  

## Methodology sketch  

1. **Dataset acquisition**  
   - Download the *Gerrit* Chromium code‑review dataset (publicly available via the *GHTorrent* MySQL dump) containing PR timestamps, comment counts, and changed files.  
   - Filter to Java and Python PRs that modify ≤ 30 LOC and include a concise natural‑language description in the PR title.  

2. **Task definition extraction**  
   - Parse each filtered PR to extract a short problem statement (e.g., the PR title) and the human‑written code change (the “gold” snippet).  

3. **Code generation**  
   - Use the open‑source *StarCoder* 1B‑parameter model (available on HuggingFace) via the `transformers` library on CPU to generate an alternative implementation for each problem statement.  
   - Prompt format: “Write a function in `<language>` that `<title>`”.  

4. **Feature computation**  
   - For both human and generated snippets compute static code metrics with *radon* (cyclomatic complexity, maintainability index) and *pylint/Checkstyle* scores.  
   - Record lines of code, token count, and presence of API calls.  

5. **Effort proxy modelling**  
   - Train a linear mixed‑effects model on the human‑written data:  
     `review_time ~ cyclomatic_complexity + loc + language + (1|project)`  
     `comment_count ~ cyclomatic_complexity + loc + language + (1|project)`  
   - Validate predictive power (R²) on a held‑out subset.  

6. **Impact estimation**  
   - Apply the fitted models to the generated snippets’ metric values to predict expected review time and comment count.  
   - Compare predicted effort for generated vs. human code using paired statistical tests (Wilcoxon signed‑rank).  

7. **Robustness checks**  
   - Repeat the analysis with a second generation model (*CodeGen‑350M*) to assess model‑specific effects.  
   - Conduct sensitivity analysis varying the LOC filter (≤ 15 LOC vs. ≤ 30 LOC).  

8. **Reproducibility package**  
   - All scripts, model checkpoints, and processed CSVs will be hosted in a public GitHub repository.  
   - The entire pipeline (download → generate → compute metrics → fit models) is designed to run on a GitHub Actions runner within 5 hours (≈ 2 GB RAM, ≤ 2 CPU cores).
