# Research: llmXive follow-up: extending "Qwen-Image-Flash: Beyond Objective Design"

## 1. Research Question & Hypothesis

**Primary Question**: Does training a small language model on a *Low‑Entropy* (coherent, structured) synthetic dataset of logical problems result in significantly faster convergence and higher generalization accuracy compared to *High‑Entropy* and *Target‑Specific* datasets, under strict CPU resource constraints?

**Hypothesis (H1)**:
1. The Low‑Entropy student will reach the loss‑threshold convergence point (loss ≤ 0.1) **in fewer epochs** than the High‑Entropy and Target‑Specific students (pairwise t‑tests, p < 0.05).
2. The Low‑Entropy student will achieve **higher final accuracy** on a held‑out **Generalization Set** than the other two models (one‑way ANOVA, p < 0.05).
3. Because entropy levels are assigned by a rule‑based generator, the effect can be framed as **causal within the synthetic domain** (FR‑006).

**Null Hypothesis (H0)**: No statistically significant difference in convergence speed or final accuracy between the three entropy conditions.

## 2. Dataset Strategy

The study uses **entirely synthetic data** generated by a custom rule‑based engine. No external real‑world datasets are required.

### 2.1 Synthetic Data Generation
* **Source**: `code/generators/logic_generator.py`
* **Mechanism**: Generates propositional‑logic and arithmetic word problems.
  * **High‑Entropy**: Randomly shuffled premises, non‑repetitive operators.
  * **Low‑Entropy**: Structured, repetitive templates (e.g., chained modus‑ponens).
  * **Target‑Specific**: Narrow reasoning styles (e.g., arithmetic chains only).
* **Validation**:
  * Per‑sample entropy scores are computed; a two‑sample t‑test confirms `mean(high) > mean(low)` (p < 0.05) before training.
  * **Structure Type**: Each problem is assigned a `structure_type` (e.g., "modus_ponens", "arithmetic_chain") to ensure the Generalization Set is stratified by structure, not just entropy.
* **Volume**:
 * **Training**: N ≥ 3,000 ([deferred] per subset, each marked `set_type: train`).
  * **Generalization Set**: N ≥ 500, **stratified** to contain equal numbers of High, Low, and Target problems, and **structurally distinct** from training instances (marked `set_type: test_generalization`).

### 2.2 Power Analysis (A Priori)
* **Minimum Detectable Effect Size**: Cohen's d = 0.5 (medium).
* **Sample Size**: N = 1,000 per group (High, Low, Target).
* **Calculated Power**: With N=1,000 per group, a two‑tailed t‑test achieves **≈ 0.86 power** at α = 0.05.  
* The ANOVA with three groups similarly exceeds 0.80 power for the same effect size.  
* **Conclusion**: The chosen sample sizes satisfy FR‑001 and mitigate Type II risk; no "lack of power" admission is required.

## 3. Methodology & Statistical Rigor

### 3.1 Teacher‑Student Distillation
* **Teacher**: Generates 10‑step CoT traces for every training problem.
* **Student**: < 100 M‑parameter transformer (DistilBERT‑base‑uncased).  
* **Loss**: Token‑level KL‑divergence between teacher and student probability distributions.  
  * **Alignment Mechanism**: The student generates 2-3 step traces. To compute KL-divergence, the student's output probabilities are aligned to the teacher's token sequence. Padding and masking are applied to ensure both distributions are defined over the same vocabulary and sequence length, avoiding the category error of mismatched sequence lengths (addresses scientific_soundness-a1104208).  
* **Training Constraints**: CPU‑only, no mixed‑precision, early stopping when validation loss ≤ 0.1 (threshold‑based convergence).  

### 3.2 Trace‑Consistency Validation (FR‑009)
* For each generated teacher trace, **trace entropy** (Shannon entropy of the token‑level distribution) is computed.  
* A trace is deemed **consistent** if its entropy direction matches the problem's `entropy_level` (high ↔ high, low ↔ low).  
* Inconsistent samples are filtered out; only `trace_consistent = true` records are fed to the distillation loop.

### 3.3 Evaluation
* Each student model is evaluated on the **Generalization Set** (500 + samples).  
* **Metric Refinement**: Convergence is defined as the epoch where the model reaches [deferred] of its final accuracy, not just loss plateau, to distinguish "coherence" from trivial "easy optimization" (addresses scientific_soundness-2a9797e0).
* Metrics recorded: final accuracy, convergence epoch, total runtime, peak RAM.

### 3.4 Statistical Analysis Plan
1. **Convergence Speed**: Pairwise t‑tests on convergence epochs (Low vs High, Low vs Target).  
2. **Final Accuracy**: One‑way ANOVA across the three models.  
3. **Multiple‑Comparison Correction**: **Bonferroni correction** applied to **all** p‑values (ANOVA and each pairwise t‑test) (FR‑007).  
4. **Effect Sizes**: Report Cohen's d for t‑tests and η² for ANOVA.  
5. **Causal Framing**: The final report will explicitly label the findings as causal *within the synthetic domain* (FR‑006).  

### 3.5 Edge‑Case Handling
* **Non‑Convergence**: Runs that never reach loss ≤ 0.1 within the epoch limit are logged with `status: failed_non_converge` and assigned a worst‑case epoch count for analysis.  
* **Timeout**: If total wall‑clock time exceeds 5.5 h, the pipeline aborts with exit code 1 (FR‑003, edge case).  
* **Logical Contradictions**: Generated problems are passed through a SAT solver; unsolvable items are discarded before teacher trace generation (addresses US‑1 edge case).
* **Memorization Risk**: The Generalization Set is constructed to be **structurally distinct** from training data (different logical templates) while maintaining the same entropy distribution, ensuring accuracy measures generalization, not memorization (addresses scientific_soundness-ce9de027).

## 4. Compute Feasibility (Free‑Tier CI)

* **Hardware**: 2 CPU cores, 7 GB RAM, ≈ 14 GB disk.  
* **Student Model**: ≈ 45 M parameters (DistilBERT).  
* **Batch Size**: 16 → peak RAM ≈ 5.8 GB.  
* **Epoch Limit**: 20 with early stopping; expected ≤ 2 h per distillation run.  
* **Total Wall‑Clock**: ≤ 4.5 h (three runs) + ≤ 0.5 h for generation/validation + ≤ 0.5 h for analysis → **< 6 h**.  

## 5. Decision Rationale

* **Synthetic Data**: Only a rule‑based generator can enforce strict entropy control (VI).  
* **CPU‑Only**: Directly tests the "edge‑device" claim (VII).  
* **KL‑Divergence Alignment**: Guarantees a mathematically sound distillation loss via token-level alignment and masking (addresses scientific‑soundness‑a1104208).  
* **Statistical Rigor**: Power analysis, Bonferroni correction, and balanced Generalization Set prevent Type I/II errors and construct validity threats (methodology‑4223fb8c, methodology‑49bc97b7).
