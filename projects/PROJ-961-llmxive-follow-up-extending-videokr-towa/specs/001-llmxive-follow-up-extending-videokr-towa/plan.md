# Implementation Plan: llmXive follow-up: extending "VideoKR: Towards Knowledge- and Reasoning-Intensive Video Understandin"

**Branch**: `001-video-reasoning-threshold` | **Date**: 2026-07-13 | **Spec**: `specs/001-llmxive-follow-up-extending-videokr-towa/spec.md`
**Input**: Feature specification from `/specs/001-llmxive-follow-up-extending-videokr-towa/spec.md`

## Summary

This project investigates whether structural complexity in video QA questions (defined by entity hops in a ground-truth knowledge graph) exhibits a non-linear "reasoning cliff" threshold effect on model accuracy. The implementation will ingest the VideoKR-SFT dataset and its associated knowledge graph, programmatically annotate each question with its shortest-path hop count (using entity linking), and perform statistical analysis (Permutation Test for change-point detection) to detect non-linear degradation. The analysis is constrained to run on a CPU-only GitHub Actions free-tier runner (multiple cores, limited RAM) via a two-stage sampling strategy (pilot -> oversampled) to ensure sufficient power for rare high-hop cases.

## Constitution Check

*GATE: Must pass before Phase 0 research.*

1.  **I. Reproducibility**: Plan ensures all random seeds are pinned in `code/` and datasets are fetched from canonical sources (verified URLs).
2.  **II. Verified Accuracy**: All dataset references in `research.md` strictly adhere to the provided "Verified datasets" block. A strict "Data Availability Gate" halts the pipeline if no verified source exists, preventing fabrication.
3.  **III. Data Hygiene**: Plan mandates checksumming of raw data and writing derived data to new files (`data/processed/`).
4.  **IV. Single Source of Truth**: All statistics in the final report will be generated by scripts, not hand-typed.
5.  **V. Versioning Discipline**: Artifacts will carry content hashes. The plan includes a specific `code/utils/versioning.py` script that writes SHA-256 hashes of `data/` artifacts to `state/projects/PROJ-961-llmxive-follow-up-extending-videokr-towa.yaml`.
6.  **VI. Structural Annotation Integrity**: The annotation script will strictly use the ground-truth graph for hop counting, employing entity linking to map question text to graph nodes, independent of model rationales.
7.  **VII. Threshold Sensitivity Analysis**: The plan explicitly includes a sensitivity analysis phase (sweeping thresholds 2, 3, 4 hops) and a Permutation Test for change-point detection.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| Permutation Test for Change-Point | Required by FR-004 to detect non-linear "cliffs" when the knot location is estimated via grid search. Standard LRT is invalid here. | Simple LRT with Bonferroni correction yields inflated Type I errors when the knot is data-driven. |
| Entity Linking (Fuzzy/Embedding) | Required by FR-001 to map natural language question entities to graph nodes. | Direct string matching fails on complex queries; assuming graph nodes exist without mapping leads to high "unresolvable" rates and bias. |
| Two-Stage Sampling (Pilot -> Oversample) | Required to ensure N >= 50 for rare 3+ hop bins. | Simple random sampling risks empty or underpowered bins for high-hop questions. |
| Removal of GAM | GAMs are statistically invalid for low-cardinality discrete ordinal variables (1-5 hops). | GAMs assume smoothness which does not exist in discrete hop counts; they risk overfitting. |
