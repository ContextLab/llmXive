# Evaluating the Impact of Code Generation on Code Review Burden

**Field**: Computer Science

## Research question

Does code generated by large language models require more or less reviewer effort compared to human-written code, as measured by review time, comment density, and issue severity in pull requests?

## Motivation

As LLM-powered coding assistants become ubiquitous in software development, understanding their downstream impact on team workflows is critical for adoption decisions. If LLM-generated code shifts review burden rather than reducing it, teams may need to adjust staffing or review processes. This gap in empirical evidence prevents informed integration of LLMs into development pipelines.

## Literature gap analysis

### What we searched

We queried Semantic Scholar / arXiv / OpenAlex using the following search terms: "LLM code review burden", "AI-generated code review effort", "Copilot pull request metrics", and "code generation downstream impact". The searches returned sparse results, with most papers focusing on education applications or generation quality benchmarks rather than review workflow outcomes.

### What is known

- [SIMCOPILOT: Evaluating Large Language Models for Copilot-Style Code Generation (2025)](http://arxiv.org/abs/2505.21514v1) — Establishes benchmarks for evaluating LLM code generation quality in interactive copilot scenarios, but does not measure downstream review workload.

### What is NOT known

No published work has measured whether LLM-generated code in real pull requests requires different reviewer effort compared to human-written code. Specifically, there are no empirical studies quantifying review time, comment density, or issue resolution patterns for LLM-sourced versus human-sourced code in production repositories.

### Why this gap matters

Software engineering teams need evidence-based guidance on whether LLM tools reduce or shift review burden. Filling this gap would enable better resource allocation, review policy design, and ROI assessment for LLM adoption in development workflows.

### How this project addresses the gap

Our methodology extracts pull request metadata from public GitHub repositories, classifies code as LLM-generated versus human-written using provenance signals (e.g., commit messages, file headers, or tool-specific markers), and compares review metrics across these groups using statistical tests. This produces the first empirical evidence on LLM-generated code's impact on review workload.

## Expected results

We expect to find that LLM-generated code receives higher comment density per line of code, suggesting increased reviewer scrutiny for correctness and style. The statistical test (two-sample t-test or Mann-Whitney U test on comment counts) will determine if the difference is significant at α=0.05. A null result (no difference) would indicate review processes are equally effective for both code sources, while a positive result would support adjusting review policies for LLM-sourced code.

## Methodology sketch

- **Data acquisition**: Download public GitHub repositories with active pull requests using the GitHub REST API (authenticated requests; rate-limited to stay within 5,000 requests/hour). Target repositories: popular open-source projects with LLM copilot integration signals (e.g., commits mentioning "Copilot", "GPT", "Codeium").
- **Code provenance classification**: For each pull request, classify code changes as LLM-generated if (a) commit message contains LLM tool keywords, (b) file header contains LLM attribution, or (c) diff patterns match known LLM generation signatures (e.g., bulk insertions of boilerplate). Human-written code serves as the control group.
- **Review metric extraction**: For each classified pull request, extract: (a) review duration (time from PR creation to merge), (b) comment count, (c) comment-to-code ratio, (d) number of review cycles, (e) issue severity from labels (bug, enhancement, etc.).
- **Data preprocessing**: Filter to pull requests with ≥5 lines of changed code and ≥1 review comment to ensure sufficient signal. Remove outliers using interquartile range (IQR) method (values beyond Q3+1.5×IQR or Q1-1.5×IQR).
- **Statistical analysis**: Perform two-sample tests (Mann-Whitney U for non-normal distributions, t-test for normal) comparing review metrics between LLM-generated and human-written code groups. Compute effect sizes (Cohen's d) to assess practical significance.
- **Validation**: Cross-check LLM classification by sampling 50 pull requests per group and manually verifying provenance; report inter-rater agreement (Cohen's κ).
- **Reproducibility**: Save all data extraction scripts, classification heuristics, and analysis notebooks to the repository; document API rate-limiting strategy for replication.
