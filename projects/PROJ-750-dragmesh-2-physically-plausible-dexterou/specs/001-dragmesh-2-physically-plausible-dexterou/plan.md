# Implementation Plan: DragMesh-2: Physically Plausible Dexterous Hand-Object Interaction with Articulated Objects

**Branch**: `750-dragmesh-2-reproduction` | **Date**: 2026-06-21 | **Spec**: `specs/750-dragmesh-2/spec.md`
**Input**: Feature specification from `/specs/750-dragmesh-2/spec.md`

## Summary

This plan reproduces and validates the "DragMesh-2" paper's core claim: that the PICA (Physically Informed Contact-Aware) agent demonstrates superior robustness in dexterous hand-object interactions under varying contact-load (damping) conditions compared to baseline trajectory tracking. The technical approach involves initializing a CPU-only physics simulation (MuJoCo), training the PICA agent on a single GAPartNet articulated object with domain randomization, evaluating across a spectrum of damping parameters, and generating specific "work-energy" consistency traces. All execution is constrained to the GitHub Actions free-tier (limited CPU, limited RAM, 6h limit).

## Technical Context

**Language/Version**: Python 3.9+  
**Primary Dependencies**: PyTorch (CPU wheel), MuJoCo, Gymnasium, Hydra (config), GAPartNet assets (submodule), Pandas, Matplotlib.  
**Storage**: Local filesystem (for checkpoints, logs, artifacts).  
**Testing**: `pytest` for unit tests; integration tests via CI script execution.  
**Target Platform**: Linux (GitHub Actions Runner).  
**Project Type**: Research reproduction / Simulation pipeline.  
**Performance Goals**: Training epoch < 6 hours; Peak RAM < 7 GB; No GPU usage.  
**Constraints**: Must run on CPU-only; must handle physics singularities (resets); must log force/velocity traces for work-energy verification.  
**Scale/Scope**: Single articulated object (e.g., drawer/door); 7 damping levels; ~50k-100k training steps (sampled) to fit CI time.

> Domain-specific empirical specifics (exact counts, dataset sizes, measured quantities) are deferred to the research/implementation phase.

## Design Principles

*Guiding principles derived from the project specification (FR-030) to ensure alignment:*

1.  **Reproducibility**: The plan explicitly mandates CPU-only execution and version-pinned dependencies to ensure the code runs on the specified CI environment without hardware dependencies.
2.  **Scientific Rigor**: The plan includes a dedicated phase for "Work-Energy Verification" (FR-006, SC-003) to address the specific reviewer request for force-balance diagrams and numerical traces. It explicitly accounts for all energy terms (kinetic, potential, dissipation) to ensure the verification is testable.
3.  **Resource Feasibility**: The plan explicitly restricts the scope to a single object and sampled training steps to guarantee completion within the 6-hour CI limit and 7 GB RAM constraint.
4.  **Data Integrity**: The plan includes a validation step (FR-001) to verify GAPartNet asset integrity (URDFs/Meshes) before simulation, preventing silent failures.
5.  **Robustness**: The evaluation strategy (FR-004) explicitly tests across a range of damping conditions, directly addressing the paper's claim of robustness.

**Structure Decision**: The structure separates the vendored research code (`external/DragMesh-2`) from the local adaptation layer (`src/`). This ensures the original code remains untouched while allowing the implementation of CPU-specific wrappers and the new "work-energy" logging required by the reviewer. The `src/validation` directory is dedicated to the FR-006 requirement. Contracts are defined in `data-model.md` and generated as artifacts in Phase 2.
