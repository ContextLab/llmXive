# Feature Specification: Neuro‑Symbolic Learning Networks: Bridging Neural and Symbolic Reasoning in Education

**Feature Branch**: `[PROJ-559-neuro-symbolic]`  
**Created**: 2026‑06‑24  
**Status**: Draft  
**Input**: User description: “Implement a lightweight neuro‑symbolic explanation framework, generate neural‑only, symbolic‑only, and neuro‑symbolic explanations for mathematics/logic problems, simulate student responses, and compare reasoning accuracy, response time, and self‑reported comprehension across the three conditions.”

## User Scenarios & Testing *(mandatory)*

### User Story 1 – Generate and Deliver Explanations (Priority: P1)

A researcher wants to obtain three types of explanations (neural‑only, symbolic‑only, neuro‑symbolic) for each problem in the ASSISTments/Khan Academy dataset and present one of them to a simulated student.

**Why this priority**: Without reliable generation and delivery of explanations the entire comparative study cannot proceed.

**Independent Test**: Run the explanation generator on a single problem and verify that three distinct explanation files are produced and displayed in the simulated‑student UI.

**Acceptance Scenarios**:

1. **Given** a problem ID from the dataset, **When** the system is invoked with the “neural‑only” mode, **Then** it outputs a natural‑language explanation generated by the fine‑tuned LLM and stores it as `explanation_neural.txt`.
2. **Given** the same problem ID, **When** the system is invoked with the “neuro‑symbolic” mode, **Then** it outputs a hybrid explanation that combines the LLM narrative with a formal symbolic trace and stores it as `explanation_neuro_symbolic.txt`.

### User Story 2 – Simulate Student Interaction (Priority: P2)

A researcher needs to collect simulated student responses (answer correctness, response time, and self‑reported comprehension) for each explanation condition.

**Why this priority**: The core dependent variable (student performance) is derived from this interaction.

**Independent Test**: Execute the simulation on one problem‑explanation pair and verify that answer correctness, elapsed time, and a 1‑5 Likert rating are logged.

**Acceptance Scenarios**:

1. **Given** an explanation displayed to the BKT‑based student simulator, **When** the simulator submits an answer, **Then** the system records a binary correctness flag, the time elapsed (in seconds, rounded to the nearest 0.1 s), and a 1‑5 Likert rating for perceived comprehension.

### User Story 3 – Run Comparative Analysis (Priority: P3)

A researcher wants to run a mixed‑effects regression across all collected data to compare the three explanation conditions on reasoning accuracy, response time, and comprehension.

**Why this priority**: This story delivers the empirical result that addresses the research question.

**Independent Test**: Trigger the analysis script on a pre‑populated CSV of ≥ 1 000 records and verify that it produces a regression table and effect‑size estimates for each pairwise condition comparison.

**Acceptance Scenarios**:

1. **Given** a CSV containing student‑simulation logs for all three conditions, **When** the analysis pipeline is executed, **Then** it outputs (a) a mixed‑effects model summary, (b) Cohen’s d effect sizes with [deferred] confidence intervals for each pairwise comparison, and (c) a concise results markdown file.

### User Story 4 – CI Resource Limits (Priority: P2)

A CI engineer must ensure that the entire pipeline runs within the free‑tier GitHub Actions runner resources.

**Why this priority**: FR‑008 (CPU/memory limits) currently has no story anchor; this story provides the necessary context.

**Independent Test**: Run the full CI workflow on a fresh runner and verify that CPU usage remains within a modest multi‑core range and memory stays within a reasonable bound.

**Acceptance Scenarios**:

1. **Given** the CI job executing the full pipeline, **When** resource monitoring is enabled, **Then** the peak CPU usage is ≤ 2 cores and peak memory usage is ≤ 7 GB.

### User Story 5 – Human Pilot Validation (Priority: P3)

A researcher collects a small set of real‑student interactions to calibrate the BKT simulator and verify that simulated outcomes are not biased by explanation condition.

**Why this priority**: Addresses scientific‑soundness concerns about circular validation and reliance on simulation alone.

**Independent Test**: Collect data from ≥ 50 human participants across the three explanation conditions, compare simulator predictions to human outcomes, and confirm that calibration metrics (e.g., RMSE) meet a predefined threshold (≤ 0.15).

**Acceptance Scenarios**:

1. **Given** a human pilot dataset, **When** the calibration routine runs, **Then** it produces calibration statistics showing no systematic bias of the simulator with respect to explanation condition.

### User Story 6 – Handle Dataset Download Failures (Priority: P2)

A CI engineer needs the pipeline to fail gracefully if the public ASSISTments or Khan Academy datasets cannot be downloaded within the CI timeout.

**Why this priority**: Guarantees robustness of the workflow and directly anchors FR‑007.

**Independent Test**: Simulate a network timeout during dataset download and verify that the CI job aborts with a clear error message indicating which dataset failed and the timeout duration.

**Acceptance Scenarios**:

1. **Given** a forced download failure, **When** the CI job reaches the download step, **Then** it exits with exit code 1 and logs “ERROR: Failed to download [dataset name] within 300 seconds – aborting pipeline.”

### User Story 7 – Integrate Real‑Student Data into Final Analysis (Priority: P2)

A researcher must incorporate real‑student interaction data (≥ 200 participants) into the final comparative analysis to ensure scientific validity.

**Why this priority**: Addresses the scientific‑soundness concern that simulation alone cannot answer the research question about real learners.

**Independent Test**: Append a CSV of ≥ 200 human interaction records to the simulated logs and rerun the analysis pipeline; verify that the regression model includes a “data_source” fixed effect and that effect‑size estimates are reported for the combined dataset.

**Acceptance Scenarios**:

1. **Given** a merged dataset of simulated + real interactions, **When** the analysis script runs, **Then** the output includes a term for `data_source` (simulated vs. real) and shows that primary condition effects remain significant (p < 0.05) in the combined analysis.

### Edge Cases

- What happens when the explanation generation model fails to produce output within the allocated CI time limit?
- How does the system handle missing or corrupt problem records in the source dataset?
- What if the simulated student model returns a non‑numeric comprehension rating?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST ingest problem statements and meta‑data from the public ASSISTments and Khan Academy datasets (downloaded automatically by the CI workflow).
- **FR-002**: System MUST generate **three** distinct explanation artifacts per problem: `neural`, `symbolic`, and `neuro‑symbolic`.
- **FR-003**: System MUST present a selected explanation to the BKT‑based student simulator via a deterministic API call.
- **FR-004**: System MUST record for each interaction (simulated **or** human) the following fields: (a) answer correctness (binary), (b) response time (seconds, precision 0.1 s), and (c) self‑reported comprehension on a 5‑point Likert scale.
- **FR-005**: System MUST aggregate logs into a single CSV file with columns `problem_id`, `condition`, `correct`, `rt_seconds`, `comprehension_rating`.
- **FR-006**: System MUST execute a mixed‑effects regression (using `lme4` or equivalent) that includes fixed effects for **explanation condition**, **prior‑knowledge score**, and **problem difficulty**, and compute Cohen’s d effect sizes with [deferred] confidence intervals for each pairwise condition comparison.
- **FR-007**: System MUST fail gracefully with a clear error message if any external dataset cannot be downloaded within the CI timeout (see User Story 6 for anchoring).
- **FR-008**: System MUST limit total CPU usage to ≤ 2 cores and memory to ≤ 7 GB to stay within the free‑tier GitHub Actions runner constraints.
- **FR-009**: System MUST simulate at least **2 000** students per explanation condition to achieve stable statistical estimates (≈ 6 000 total interactions).
- **FR-010**: System MUST calibrate the BKT‑based student simulator against a human pilot dataset of ≥ 50 participants, ensuring that simulator predictions are not conditioned on explanation type. Calibration is considered successful when (a) the RMSE difference across explanation conditions is ≤ 0.02 **and** (b) a two‑sample t‑test on RMSE per condition yields p ≥ 0.10.
- **FR-011**: System MUST incorporate real‑student interaction data from ≥ 200 participants into the final comparative analysis, and the analysis must include a `data_source` fixed effect distinguishing simulated from real records.

### Key Entities *(include if feature involves data)*

- **Problem**: Represents a single mathematics or logic exercise; key attributes are `problem_id`, `prompt_text`, `solution`.
- **Explanation**: Represents an artifact generated for a problem under a specific condition; attributes include `condition` (`neural`, `symbolic`, `neuro‑symbolic`) and `content`.
- **SimulationLog**: Captures a single simulated (or human) student interaction; attributes are `problem_id`, `condition`, `correct`, `rt_seconds`, `comprehension_rating`.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ≥ 95 % of problems in the selected subset have successfully generated all three explanation types without runtime errors.
- **SC-002**: The analysis pipeline completes on a dataset of **≥ 5 000** logged interactions and produces a regression table (no missing coefficients).
- **SC-003**: Reported effect‑size confidence‑interval width for the primary comparison (neuro‑symbolic vs. neural‑only) is ≤ 0.20, indicating sufficient precision for interpretation.
- **SC-004**: The final analysis includes at least **200** real‑student interaction records and demonstrates that the primary condition effects remain statistically significant (p < 0.05) in the combined simulated + real dataset.
- **SC-005**: ≥ 99 % of simulated interaction logs contain all three required fields (correctness, response time, comprehension rating); the distribution of response times covers the 0.5‑s to 30‑s range with no gaps larger than 5 s, and overall correctness rate is recorded for each condition.
- **SC-006**: During CI execution, peak CPU usage does not exceed 2 cores and peak memory usage does not exceed 7 GB; these metrics are captured and reported in the CI job summary.
- **SC-007**: Calibration on the human pilot dataset yields an RMSE ≤ 0.15 and a two‑sample t‑test on RMSE differences across explanation conditions returns p ≥ 0.10, indicating no systematic bias.
- **SC-008**: If a dataset download exceeds 300 seconds, the CI job aborts with exit code 1 and logs the exact message “ERROR: Failed to download [dataset name] within 300 seconds – aborting pipeline.”

## Assumptions

- Public ASSISTments and Khan Academy datasets are freely downloadable and fit within the 2 CPU / 7 GB RAM CI limits.
- A distilled 1 B‑parameter LLM can generate a neural‑only explanation for a problem in ≤ 2 seconds on the free‑tier runner.
- The BKT implementation **accurately approximates real student learning trajectories for the purposes of this comparative study** and will be calibrated against a human pilot dataset (see FR‑010) to mitigate circularity.
- All statistical analyses are performed in R (version ≥ 4.3) or Python (statsmodels ≥ 0.14) available on the CI image.
- No GPU resources are required; inference is performed on CPU only.
- The human pilot validation (≥ 50 participants) will be conducted offline and its results incorporated before the final regression analysis; an additional larger real‑student dataset (≥ 200 participants) will be gathered for the final analysis (see FR‑011).
