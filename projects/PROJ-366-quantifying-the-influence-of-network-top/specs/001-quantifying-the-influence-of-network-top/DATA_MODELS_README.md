# Data Models and Artifacts Specification

This document details the physical layout, file formats, and storage conventions
for all data artifacts generated by the `PROJ-366` pipeline.

## Directory Structure

The project adheres to the following directory structure for data storage:

```text
data/
├── raw/ # Raw input data (e.g., XYZ files from external sources)
│ └── [external_source]/
├── processed/
│ ├── graphs/ # Serialized AtomicGraph objects and stats
│ │ ├── *.pkl # Pickled graph objects
│ │ ├── node_degree_stats.json
│ │ └── excluded_samples.json
│ ├── conductivities/ # ThermalSample objects and validation reports
│ │ ├── *.pkl # Pickled thermal samples
│ │ └── convergence_report.json
│ └── model_outputs/ # GNN outputs, correlation results, and final analysis
│ ├── *.pkl # Pickled model outputs
│ ├── correlation_pearson.json
│ ├── correlation_pearson_corrected.json
│ ├── power_analysis.json
│ └── final_results.json
└── checksums.json # Manifest of all file checksums
```

## Artifact Specifications

### 1. Atomic Graphs (`data/processed/graphs/*.pkl`)

**Format**: Python Pickle (protocol 4)
**Content**: `AtomicGraph` object conforming to `atomic_graph.schema.yaml`.
**Generation**: `code/ingest/graph_builder.py`
**Key Attributes**:
- `nodes`: List of dicts with `atomic_number`, `position`, `properties`.
- `edges`: List of dicts with `source`, `target`, `distance`.
- `metadata`: `cutoff_distance`, `num_atoms`, `num_bonds`, `checksum`.

### 2. Node Degree Statistics (`data/processed/graphs/node_degree_stats.json`)

**Format**: JSON
**Content**: Global statistics of node degrees across all processed graphs.
**Generation**: `code/ingest/node_degree_stats_generator.py`
**Schema**:
```json
{
 "mode": 4,
 "mean": 3.95,
 "std_dev": 1.2,
 "distribution": { "2": 5, "3": 50, "4": 100, "5": 10 },
 "total_nodes": 165,
 "total_graphs": 2
}
```

### 3. Excluded Samples (`data/processed/graphs/excluded_samples.json`)

**Format**: JSON
**Content**: List of sample IDs excluded due to topological defects.
**Generation**: `code/analysis/outlier_detector.py`
**Schema**:
```json
{
 "excluded_ids": ["sample_001", "sample_005"],
 "reason": "Defect ratio > 15%",
 "threshold": 0.15,
 "timestamp": "2023-10-27T10:00:00Z"
}
```

### 4. Thermal Samples (`data/processed/conductivities/*.pkl`)

**Format**: Python Pickle (protocol 4)
**Content**: `ThermalSample` object conforming to `thermal_sample.schema.yaml`.
**Generation**: `code/simulation/thermal_sample_saver.py`
**Key Attributes**:
- `sample_id`: String.
- `conductivity`: Dict with `value`, `uncertainty`, `method`.
- `converged`: Boolean.
- `topological_metrics`: Dict of computed metrics.

### 5. Convergence Report (`data/processed/conductivities/convergence_report.json`)

**Format**: JSON
**Content**: Summary of Green-Kubo simulation convergence status.
**Generation**: `code/simulation/conductivity_validator.py`
**Schema**:
```json
{
 "total_samples": 2,
 "converged_count": 2,
 "failed_count": 0,
 "samples": [
 { "id": "sample_001", "converged": true, "value": 1.2 },
 { "id": "sample_002", "converged": true, "value": 1.1 }
 ]
}
```

### 6. Power Analysis (`data/processed/model_outputs/power_analysis.json`)

**Format**: JSON
**Content**: Statistical power check results.
**Generation**: `code/analysis/power_checker.py`
**Schema**:
```json
{
 "sample_count": 2,
 "status": "INSUFFICIENT_POWER",
 "warning": "N < 10. Results are proof-of-concept only.",
 "threshold": 10
}
```

### 7. Correlation Results (`data/processed/model_outputs/correlation_pearson.json`)

**Format**: JSON
**Content**: Pearson correlation coefficients between features and conductivity.
**Generation**: `code/analysis/pearson_correlation.py`
**Schema**:
```json
{
 "correlations": [
 { "feature": "degree", "r": 0.85, "p_value": 0.01 }
 ],
 "method": "Pearson"
}
```

### 8. Corrected Correlation Results (`data/processed/model_outputs/correlation_pearson_corrected.json`)

**Format**: JSON
**Content**: Pearson results with Bonferroni correction applied.
**Generation**: `code/analysis/correlation_significance.py`
**Schema**:
```json
{
 "correlations": [
 { "feature": "degree", "r": 0.85, "p_value": 0.01, "p_corrected": 0.05, "significant": true }
 ],
 "correction_method": "Bonferroni",
 "alpha": 0.05
}
```

### 9. Final Results (`data/processed/model_outputs/final_results.json`)

**Format**: JSON
**Content**: Aggregated results from LMM and Pearson analyses.
**Generation**: `code/analysis/final_results_aggregator.py`
**Schema**:
```json
{
 "lmm_results": {... },
 "pearson_results": {... },
 "interpretation": "Topological features show significant correlation...",
 "timestamp": "2023-10-27T12:00:00Z"
}
```

### 10. Checksum Manifest (`data/checksums.json`)

**Format**: JSON
**Content**: SHA-256 checksums for all generated artifacts.
**Generation**: `code/analysis/checksum_verifier.py`
**Schema**:
```json
{
 "files": {
 "data/processed/graphs/sample_001.pkl": "abc123...",
 "data/processed/conductivities/sample_001.pkl": "def456..."
 },
 "timestamp": "2023-10-27T12:00:00Z"
}
```

## Data Integrity and Validation

- **Checksums**: All serialized data files (`.pkl`) are accompanied by a SHA-256
 checksum stored in `data/checksums.json`.
- **Validation**: Before any analysis step, `code/analysis/checksum_verifier.py`
 verifies that all input files match their recorded checksums.
- **Schema Validation**: Contract tests in `tests/contract/test_schemas.py`
 validate that data structures match the defined schemas before processing.

## Usage Guidelines

1. **Reading Data**: Use the provided loader functions in `code/ingest/`,
 `code/simulation/`, and `code/analysis/` to load data. Do not directly
 unpickle files without validation.
2. **Writing Data**: Always write data using the provided saver functions,
 which automatically calculate checksums and update the manifest.
3. **External Data**: Raw data from external sources (e.g., Materials Project)
 must be placed in `data/raw/` and processed via `code/ingest/sample_generator.py`.

## Dependencies

- `numpy`: For numerical arrays in graphs.
- `pandas`: For tabular data in JSON/CSV exports.
- `pickle`: For object serialization.
- `json`: For metadata and report files.
- `hashlib`: For checksum generation.

## Future Extensions

- Support for HDF5 format for large-scale graph storage.
- Integration with `dvc` for version control of large datasets.
- Automatic generation of data lineage graphs.
