# Data Acquisition Limitations and Constraints

## Overview

This document outlines the strict limitations, constraints, and procedural requirements for data acquisition within the `llmXive` pipeline for the project "The Impact of Visual Attention Allocation on Recall of Emotionally Valenced Stories" (PROJ-387).

## 1. Real Data Only Policy

**Strict Prohibition on Synthetic Data:**
- No synthetic, simulated, or "placeholder" datasets are permitted for analysis or model training.
- Hard-coded sample rows (e.g., `pd.DataFrame({"col": [1, 2, 3]})`) are strictly forbidden.
- All analysis must be performed on real, empirically measured data obtained from verified public repositories or downloadable sources.

**Consequence of Missing Data:**
- If a real, verified dataset cannot be programmatically accessed or downloaded, the pipeline must **halt immediately** with a `DATA_BLOCKER` status.
- The system must **not** proceed with fallback data or fake values.

## 2. Required Data Variables

The pipeline requires the following specific variables to be present in any loaded dataset. If any are missing, the ingestion phase (US1) will halt:

- **Fixation Duration**: Duration of gaze fixation (ms).
- **Saccade Amplitude**: Angular distance of saccades (degrees).
- **Gaze Distribution**: Spatial distribution metrics of eye gaze.
- **Recall Accuracy**: Quantitative measure of story recall performance.
- **Valence Label**: Human-rated emotional valence (standardized scale).

## 3. Data Quality Thresholds

Before analysis (US2) or visualization (US3) can proceed, the following quality metrics must be validated:

- **Track Loss**: Must be ≤ 5%. If track loss exceeds this threshold, the pipeline halts.
- **Calibration Status**: Eye-tracker must be confirmed as "calibrated". Uncalibrated data results in a `DATA_BLOCKER`.
- **Valence Annotation**: Must use a standardized rating scale. If human-rated metadata is unavailable, the pipeline halts.

## 4. Data Sources and Acquisition

The pipeline is designed to fetch data from specific, real-world sources. As of this implementation:

- **Primary Source**: Public eye-tracking repositories (e.g., specific datasets from OpenNeuro, Zenodo, or similar academic repositories).
- **Acquisition Method**: Data must be fetched via API or direct download URL at runtime.
- **Verification**: A checksum (MD5) is computed for every downloaded file to ensure integrity (Constitution III).

**Note on Availability:**
If the configured public dataset URLs are unreachable or if no verified dataset containing the required variables exists in the specified repositories, the pipeline will log:
`DATA_BLOCKER: No verified datasets found`
and exit with code 1.

## 5. Memory and Hardware Constraints

- **CPU Only**: All data loading and processing must occur on CPU. No GPU/CUDA acceleration is permitted.
- **Memory Limit**: The system must operate within < 7 GB of RAM. Large datasets must be processed using chunking or streaming (see `code/analysis/memory_loader.py`).

## 6. Error Handling and Halting Behavior

The pipeline implements a "Fail Loudly" strategy:

1. **Missing Columns**: Logs `DATA_BLOCKER: Missing required variables` and exits.
2. **Quality Failure**: Logs specific metric failure (e.g., "Track loss > 5%") and exits.
3. **No Data**: Logs `DATA_BLOCKER: No verified datasets found` and exits.
4. **Corrupted Data**: If checksum verification fails, the file is rejected and the pipeline halts.

## 7. Documentation of Data Gaps

Any known gaps in data availability (e.g., specific valence categories missing from available public sets) must be documented here or in the `data/eye-tracking/quality_report.md` generated during US1.
