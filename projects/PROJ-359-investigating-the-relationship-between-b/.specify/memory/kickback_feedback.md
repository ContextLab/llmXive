## Unresolved concerns

- The plan implements a motion exclusion threshold of mean FD > 0.3mm (per spec) but notes a conflict with the Constitution's >3mm. While the spec takes precedence, a 0.3mm threshold is extremely stringent for resting-state fMRI and will likely exclude >90% of the N=85 sample, reducing the final N to <10. This renders the planned power analysis (targeting N=85) invalid and the study underpowered to detect the claimed effect (r=0.30). The methodology cannot answer the research question with the proposed sample size after this exclusion.
- The plan assumes fMRIPrep 23.1.3 can run on a GitHub Actions free-tier runner (2 cores, 7GB RAM) for N=85 participants within 24 hours. fMRIPrep is computationally intensive; running it sequentially on 2 cores for 85 subjects typically exceeds 24 hours even with ample RAM. The plan lacks a concrete strategy (e.g., downsampling, skipping specific fMRIPrep steps, or using a smaller subset) to meet the 24-hour constraint, risking a failure to generate the necessary data for the regression analysis.
- The regression model includes 'baseline cognitive score' as a covariate (FR-005) but the spec defines the outcome as 'baseline working memory performance'. If the 'baseline cognitive score' is the same as the outcome variable, this introduces perfect collinearity or circular reasoning. If it is a different task, the spec does not define which one. The construct validity of the model is compromised by this ambiguity.
- FR-009 mandates a fatal error if participant IDs are missing between rs-fMRI and behavioral data, and requires logging 'ID_VALIDATION: PASS'. The plan's 'Data Flow' and 'Quickstart' sections describe merging data but lack an explicit task or phase dedicated to this strict ID validation and fatal-error handling. The 'Verification' step in quickstart.md checks for the log message but does not define the implementation logic to trigger the fatal error, leaving the requirement unbacked by a specific plan element.
- SC-003 requires an a priori power analysis documenting that N≥85 provides ≥80% power to detect r=0.30, with results in `power_analysis.txt`. The plan mentions 'Power Analysis' in the 'Statistical Methodology' and 'Risks' sections but does not assign a specific phase, task, or output artifact (e.g., `src/analysis/power.py` or `data/results/power_analysis.txt`) to generate this specific document. The 'Quickstart' Step 4 mentions 'power analysis' generically but does not explicitly link to the generation of the required `power_analysis.txt` file as a distinct deliverable.
- SC-004 requires all generated figures to be reproducible (identical hashes) upon re-run with pinned seeds. While the plan mentions 'Random seeds pinned' in the Constitution Check, it does not explicitly define a plan element (e.g., a specific configuration step, a seed-pinning task in `src/analysis/regression.py`, or a verification step in the pipeline) that ensures this reproducibility for the visualization output. The 'Quickstart' does not mention seed handling for the visualization step.
- FR-007 requires logs to be parsable as JSON and contain specific summary statistics (exclusion counts, runtime). The plan's 'Technical Context' mentions 'JSON logging' in `utils/logger.py`, but the 'Data Flow' and 'Quickstart' sections do not explicitly define the structure of the log output or a task to ensure the specific fields (`exclusion_motion`, `exclusion_missing_wm`, `runtime`) are present and formatted as JSON. The 'Verification' step in quickstart checks for a specific string but does not validate the JSON structure required by FR-007.
- The plan assumes the OpenNeuro ds000277 dataset contains a 'validated working-memory task' baseline score. ds000277 (HCP S1200) primarily contains resting-state fMRI and structural data; the specific behavioral phenotypic file containing a 'baseline working memory score' (distinct from the N-back task used in other HCP releases) is not guaranteed to exist in the standard phenotypic TSV. The plan treats the existence of this specific variable as a certainty ('Assumptions' section) rather than verifying its presence in the dataset's actual schema. If the column is missing, the regression (FR-005) will fail, invalidating the research question.
- The plan targets a GitHub Actions free-tier runner (2 cores, 7GB RAM) to run fMRIPrep 23.1.3 on N=85 participants within 24 hours. fMRIPrep is a heavy Docker container requiring significant RAM (often >8GB for full preprocessing) and CPU time. Running it sequentially on a 2-core/7GB runner for 85 subjects is computationally infeasible; a single subject often exceeds the 24h total budget or the 7GB RAM limit, causing OOM kills. The plan's 'sequential processing' mitigation does not resolve the fundamental resource mismatch between fMRIPrep's requirements and the CI runner's constraints.
- Constitution Principle VII mandates exclusion of participants with mean FD > 3mm. The plan implements a 0.3mm threshold (per spec) but only logs a warning. This violates the 'NON-NEGOTIABLE' nature of the Constitution's motion exclusion rule. A plan that knowingly violates a core constitutional principle without a formal amendment or override mechanism is invalid.
- The plan states 'Motion exclusion threshold >0.3mm (per spec) vs >3mm (per constitution)' and chooses the spec value. This creates a direct conflict where the implementation plan explicitly violates Principle VII of the constitution. The plan must either align with the constitution (3mm) or document a formal constitutional amendment process before proceeding.
- The 'Source Code' section lists `src/data/download.py` and `src/data/preprocess.py`, but the 'Contracts' section in the input does not include a schema for the raw download or the fMRIPrep wrapper output, only for `baseline_metrics`, `dataset_schema`, and `regression_output`. The plan should not proceed with a 'WARN' status on a non-negotiable principle; it requires a 'FAIL' or a documented amendment.
- The plan implements a motion exclusion threshold of mean FD > 0.3 mm (Spec FR-002) while the Constitution mandates > 3 mm (Principle VII). In rs-fMRI, a 0.3 mm threshold is extremely stringent and likely to exclude >80% of the HCP-1200 sample, reducing N below the statistical power minimum (N=30) required for the planned regression. This creates a high risk of a 'null result' driven by sample attrition rather than the absence of a biological effect, rendering the study's primary hypothesis untestable on the available data.
- FR-005 specifies a regression model: `WM_Score ~ FP_Strength + DMN_Strength + E_glob + Q + Age + Sex + baseline cognitive score`. The inclusion of 'baseline cognitive score' as a covariate while the outcome is 'baseline working memory performance' (a specific cognitive score) creates a tautological or collinear predictor problem. If 'baseline cognitive score' is a composite including WM, or if it is highly correlated, the model cannot isolate the unique variance of network metrics. If it is a different task, the term is ambiguous. This violates the independence of predictors from the outcome definition.
- The plan proposes fitting a multiple linear regression with 4 network metrics + 3 covariates (Age, Sex, Baseline Score) on a sample size potentially reduced to N < 30 by the 0.3mm motion threshold. This results in a predictor-to-sample ratio approaching 1:3 or worse, leading to severe overfitting and unstable coefficient estimates. The permutation testing (1,000 shuffles) does not mitigate the fundamental lack of degrees of freedom in the linear model itself.
