# Contributing to PROJ-015: Improving Accessibility and Usability of Complex Computer Systems for People with Disabilities

Thank you for your interest in contributing to this research project.
To ensure the integrity and reproducibility of our scientific findings,
please adhere to the following guidelines regarding data usage and pipeline execution.

## 🚨 Critical Data Integrity Policy

**Do not use synthetic data for final research claims.**

This project implements a rigorous scientific pipeline. The integrity of our
conclusions depends on the use of real, measured interaction data from actual
participants or verified external datasets.

### Guidelines for Simulation and Testing

1. **Local Development & CI Validation Only**:
 - The `--simulate` flag in `code/analysis/run_analysis.py` (and related simulator tools)
 is **strictly** for local pipeline verification, debugging, and Continuous Integration
 (CI) logic checks.
 - It generates deterministic, rule-based synthetic data to ensure the statistical
 pipeline (ANOVA, Holm-Bonferroni, etc.) functions correctly when real data is
 temporarily unavailable for testing.

2. **Production Runs**:
 - In any production environment, CI pipeline, or final analysis run, the `--simulate`
 flag must be **disabled**.
 - The pipeline is configured to **fail loudly** (exit code 1) if `data/raw/` is empty
 or contains no valid real session files when `--simulate` is not explicitly allowed
 by the environment variable `CI_SIMULATE=true`.
 - Never commit or present results derived from the simulator as scientific evidence.

3. **Data Sources**:
 - **Real Data**: Must come from the Streamlit app (`code/simulator/app.py`) running
 with human participants, or from verified external datasets loaded into `data/raw/`.
 - **Synthetic Data**: Only generated by `code/simulator/simulator.py` for the purpose
 of validating the analysis code logic.

## Reporting Issues

When reporting bugs, please include:
- The specific command and flags used.
- Whether you are running with real data or simulation.
- Full error logs.

## Code Style

- Follow PEP 8 standards.
- Use `black` for formatting and `ruff` for linting as configured in Phase 1.
- Ensure all new code includes unit tests in `tests/`.

## Research Ethics

- Respect participant privacy. Anonymize all data before sharing.
- Do not upload raw session files containing PII to public repositories.
- Ensure all data collection protocols comply with the approved IRB guidelines.

By contributing to this project, you agree to uphold these standards to maintain
the scientific validity of our research.
