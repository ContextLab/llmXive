## Summary

This project extends the "LiveEdit" real-time video editing framework by replacing its attention-based temporal coherence mechanism with a lightweight, optical flow-based warping module. The primary requirement is to evaluate the trade-off between memory efficiency (removing attention layers) and temporal stability (flickering/artifacts) under varying motion complexities. 

**Critical Methodological Updates**:
1.  **Metric Validity**: Replaced simple consecutive-frame SSIM with **Background Stability Score (BSS)** and **Flow-Normalized SSIM**. BSS isolates background regions (using the edit mask) and compares them against the original background content to distinguish warping errors from valid motion. Flow-Normalized SSIM accounts for motion magnitude to prevent "freezing" artifacts from being scored as stable.
2.  **Threshold Identification**: Replaced K-S test as the primary threshold method with **Piecewise Regression (Change-Point Detection)**. This models error metrics as a function of flow magnitude to identify the specific "kink" where artifacts become significant. K-S test is retained as a secondary check for distribution differences.
3.  **Compute Feasibility**: Reduced sample size from 500 to **50 clips** to ensure completion within the 6-hour GitHub Actions free-tier limit on CPU. A pilot runtime phase is included to validate this constraint.
4.  **Dataset Integrity**: Clarified that the Baseline is the **LiveEdit architecture with temporal attention layers enabled**, and the Flow-Coherence variant is the **same architecture with layers removed**. Generic SD checkpoints are not used as the baseline.

The technical approach involves: (1) establishing a CPU-optimized baseline on a stratified video dataset (N=50), (2) implementing a "Flow-Coherence" module using pre-computed optical flow (RAFT-small/Farneback) to warp latent features, (3) executing both pipelines on a GitHub Actions free-tier runner (CPU-only, ≤6h), and (4) performing statistical boundary analysis (Piecewise Regression) to identify the specific flow-magnitude thresholds where artifact generation becomes significant.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| **Two Model Variants** (Baseline vs. Flow-Coherence) | Required to isolate the effect of the attention layer removal and measure the trade-off. | A single model would not allow for comparative analysis of the proposed hypothesis. |
| **Piecewise Regression** (vs. K-S) | Required to identify specific motion thresholds (Flow-Regime Boundary) as per Constitution Principle VII. K-S test only compares distributions, not thresholds. | Aggregate metrics alone would fail to identify the "point of diminishing returns" or failure modes. |
| **Invalid Flow Handling** (Identity warp) | Required to handle edge cases (occlusion, blur) without crashing, ensuring data integrity for all 50 clips. | Skipping invalid frames would bias the dataset towards "easy" motion, violating the stratification requirement. |
| **Sample Size Reduction** (50 clips) | Required to fit the 6-hour GHA CPU limit. 500 clips is infeasible. | 500 clips would exceed the time limit by orders of magnitude, making the project impossible to complete. |

## Risk Management: Compute Feasibility

The 6-hour limit is the primary constraint.
- **Pilot Phase**: Before full execution, A pilot of a small set of clips will be run to measure actual inference time per clip.
- **Dynamic Adjustment**: If `time_per_clip * 50 > 5.5 hours`, the sample size will be dynamically reduced to `floor(5.5 / time_per_clip)`.
- **Batching**: Clips will be processed one at a time to minimize memory overhead.
- **Checkpointing**: If the job is interrupted, the pipeline will resume from the last completed clip.

## Dataset Strategy (Clarified)

- **Source**: Verified HuggingFace datasets for DAVIS and YouTube-VOS that provide actual video frames (e.g., `j-hartmann/davis` or similar).
- **Loading**: Use `datasets.load_dataset(...)` for verified video datasets. If the verified URL points to a metadata-only Parquet, the plan will use a pre-downloaded subset of video files hosted in the repo's `data/raw/` (with checksums) to ensure reproducibility, citing the original verified URL as the source.
- **Stratification**: Clips are assigned to categories based on **quantitative flow magnitude**:
  - **Static**: Mean flow magnitude < 0.5 px
  - **Slow Rigid**: 0.5 px ≤ Mean flow magnitude < 5.0 px
  - **Fast Non-Rigid**: Mean flow magnitude ≥ 5.0 px
