## User Scenarios & Testing

### User Story 1 - Core Retrieval and Inference Pipeline (Priority: P1)

The researcher must be able to download the MemLens dataset, filter for multi-session reasoning (MSR) and temporal reasoning (TR) tasks, construct three distinct memory representations (Coarse, Medium, Fine), and run a frozen, CPU-optimized LLM to generate answers for each representation type.

**Why this priority**: This is the foundational capability. Without the ability to generate answers using the three different indexing strategies, no comparative analysis of visual fidelity loss is possible. It validates the core hypothesis that indexing granularity impacts performance.

**Independent Test**: The system can be tested by running the pipeline on a small subset of the dataset (e.g., 50 questions) and verifying that three distinct answer sets are generated, one for each memory strategy, without GPU acceleration.

**Acceptance Scenarios**:
1. **Given** the MemLens dataset is available, **When** the system filters for MSR and TR tasks and constructs Coarse, Medium, and Fine memory stores, **Then** the system must successfully generate a complete set of answers for each store type using a CPU-only LLM.
2. **Given** a specific query requiring multi-session reasoning, **When** the retrieval pipeline selects the top-k chunks from the Fine memory store, **Then** the LLM must generate an answer using only the retrieved context and the model's frozen weights.
3. **Given** the pipeline is executed, **When** the process completes, **Then** latency metrics for retrieval and inference must be recorded and written to a JSON file with keys: `strategy`, `retrieval_latency_ms`, `inference_latency_ms`, `peak_ram_mb`.

---

### User Story 2 - Statistical Comparison and Significance Testing (Priority: P2)

The researcher must be able to compare the accuracy of answers generated by the Fine indexing strategy against the Coarse strategy using a paired Wilcoxon signed-rank test to determine if the observed difference in visual fidelity is statistically significant.

**Why this priority**: This addresses the core research question regarding the *impact* of granularity. Simply observing a difference is insufficient; statistical validation is required to claim that the indexing strategy drives performance degradation.

**Independent Test**: The system can be tested by running the analysis on the generated answer sets and verifying that a Wilcoxon signed-rank test is performed, yielding a p-value and effect size.

**Acceptance Scenarios**:
1. **Given** the accuracy scores for the Fine and Coarse strategies, **When** the statistical analysis module runs, **Then** it must perform a Wilcoxon signed-rank test and output the p-value and test statistic.
2. **Given** a p-value < 0.05, **When** the results are reported, **Then** the system must flag the difference as statistically significant with a clear interpretation.
3. **Given** the analysis is complete, **When** the report is generated, **Then** it must include a comparison of the mean accuracy and standard deviation for each strategy.

---

### User Story 3 - Computational Feasibility and Resource Monitoring (Priority: P3)

The researcher must be able to monitor and verify that the entire analysis pipeline (data processing, retrieval, inference, and evaluation) completes within the constraints of a free-tier CPU-only CI runner (≤ 2 cores, ~7 GB RAM, ≤ 6 hours) when run on the full filtered MSR/TR subset.

**Why this priority**: The methodology explicitly requires CPU-tractability. If the pipeline exceeds resource limits, the project cannot be executed or validated. This ensures the research design is practically feasible.

**Independent Test**: The system can be tested by running the full pipeline on the full filtered MSR/TR subset in a simulated environment with resource limits and verifying that it completes without OOM errors or timeouts.

**Acceptance Scenarios**:
1. **Given** the pipeline is running on a CPU-only environment on the full filtered MSR/TR subset, **When** memory usage is monitored, **Then** it must not exceed 7 GB of RAM at any point.
2. **Given** the total runtime on the full filtered MSR/TR subset, **When** the job completes, **Then** the total execution time must be less than 6 hours.
3. **Given** the inference step, **When** the model is loaded, **Then** it must be loaded in a quantized format (e.g., 4-bit) or a small model size that fits within the available CPU memory without requiring CUDA.

### Edge Cases

- **What happens when** the dataset filtering yields fewer than 30 samples for the paired statistical test? (System should flag insufficient power and defer statistical significance claims).
- **How does the system handle** object detection failures in the Fine memory construction (e.g., YOLOv8-Tiny failing to detect objects in a specific image)? (System must log the failure and fall back to a global image embedding for the specific entry).
- **What happens when** the LLM generates a response that is too long for the context window? (System must truncate the response or implement a sliding window strategy, logging the truncation event).

## Requirements

### Functional Requirements

- **FR-001**: System MUST download the MemLens dataset and filter it to include only questions requiring multi-session reasoning (MSR) and temporal reasoning (TR) (See US-1).
- **FR-002**: System MUST construct three distinct memory stores: Coarse (text summaries only), Medium (summaries + global CLIP embeddings), and Fine (object-level captions + bounding boxes) (See US-1).
- **FR-003**: System MUST implement a retrieval pipeline using cosine similarity on a **multimodal embedding space** (fusing text captions and bounding box coordinates) to select the top-k relevant chunks from the Fine memory store, and text embeddings for Coarse/Medium (See US-1).
- **FR-004**: System MUST execute inference using a frozen, CPU-optimized language model (e.g., Phi-3-mini or Llama-3-8B 4-bit) without GPU acceleration (See US-1, US-3).
- **FR-005**: System MUST record retrieval latency, inference latency, and memory usage for each of the three indexing strategies (See US-1, US-3).
- **FR-006**: System MUST perform a paired Wilcoxon signed-rank test to compare the accuracy distributions between the Fine and Coarse strategies (See US-2).
- **FR-008**: System MUST handle object detection failures in the Fine store construction by logging the error and falling back to a global image embedding for the specific entry (See Edge Cases).
- **FR-009**: System MUST validate the visual fidelity of the Fine memory store by calculating Object Detection Recall against ground-truth bounding boxes, requiring a recall score ≥ 0.6 for the store to be considered valid (See US-2, SC-005).

### Key Entities

- **MemoryStore**: Represents one of the three indexing strategies (Coarse, Medium, Fine) containing session data and retrieval indices.
- **Query**: A specific question from the filtered MemLens dataset requiring multi-session or temporal reasoning.
- **AnswerSet**: The collection of generated responses for a specific Query across all three MemoryStores.
- **PerformanceMetrics**: Aggregated data containing accuracy, latency, and resource usage for each strategy.

## Success Criteria

### Measurable Outcomes

> measured against; defer specific empirical values (counts, dataset sizes,

- **SC-001**: The accuracy of the Fine indexing strategy is measured against the accuracy of the Coarse strategy using the ground truth answers from the MemLens benchmark (See FR-006, US-2).
- **SC-002**: The statistical significance of the accuracy difference is measured against the p-value threshold (α = 0.05) derived from the Wilcoxon signed-rank test (See FR-006, US-2).
- **SC-003**: The total computational cost (CPU time and peak RAM) is measured against the free-tier CI runner limits (≤ 6 hours, ≤ 7 GB RAM) on the full filtered MSR/TR subset (See FR-004, FR-005, US-3).
- **SC-004**: The retrieval latency for each strategy is measured against the baseline latency of the Coarse strategy to assess the computational trade-off (See FR-005, US-1).
- **SC-005**: The visual fidelity preservation is measured by a composite metric: (1) Relative improvement in accuracy of the Fine strategy compared to the Coarse strategy, AND (2) Object Detection Recall ≥ 0.6 on the retrieved context (See FR-009, US-1).

## Assumptions

- The MemLens dataset (789 questions) contains sufficient multi-session and temporal reasoning tasks to achieve statistical power (n ≥ 30) after filtering.
- The YOLOv8-Tiny (or equivalent distilled variant) object detector can run on a CPU-only environment within the 6-hour time limit for the entire dataset.
- The frozen LLM (e.g., Phi-3-mini) in 4-bit quantization fits within 7 GB of RAM on a CPU-only runner without requiring CUDA or mixed-precision training.
- The ground truth answers in the MemLens benchmark are accurate and sufficient for evaluating the visual fidelity of the generated responses.
- The "visual fidelity loss" is adequately proxied by the accuracy of the final answer on MSR/TR tasks, **provided** that the Fine store's Object Detection Recall meets the ≥ 0.6 threshold (See FR-009).
- The top-k retrieval parameter is fixed at k=5 for all strategies to ensure a fair comparison of indexing granularity.
