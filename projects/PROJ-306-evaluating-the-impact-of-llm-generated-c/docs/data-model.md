# Data Model Specification

This document defines the schema, file structures, and data flow for the **llmXive** project.

## Directory Structure

```
project_root/
├── code/ # Source code modules
├── data/
│ ├── benchmarks/
│ │ ├── raw/ # Unmodified dataset files (MBPP, HumanEval)
│ │ └── processed/ # Normalized catalog, transformed test suites
│ ├── generated/ # LLM-generated code files
│ ├── coverage_reports/ # JSON reports of coverage results
│ ├── processed/ # Intermediate analysis data (stats, sensitivity)
│ └── outputs/ # Final visualizations and stratified reports
├── docs/ # Documentation (this file, quickstart.md)
├── tests/ # Unit and integration tests
└── requirements.txt # Python dependencies
```

## File Schemas

### 1. Task Catalog (`data/benchmarks/processed/catalog.json`)

A normalized JSON file containing metadata for all benchmark tasks.

**Schema:**
```json
{
 "tasks": [
 {
 "task_id": "string",
 "prompt": "string",
 "human_solution": "string",
 "test_suite_path": "string (relative path to.py test file)",
 "difficulty": "string (easy, medium, hard)",
 "code_patterns": ["list of detected patterns"],
 "dataset_source": "string (mbpp, humaneval)"
 }
 ]
}
```

**Validation Rules:**
- `task_id` must be unique.
- `test_suite_path` must point to an existing file in `data/benchmarks/processed/tests/`.
- `difficulty` must be one of: `easy`, `medium`, `hard`.

### 2. Coverage Report (`data/coverage_reports/{task_id}.json`)

Generated by `code/coverage_runner.py` after executing tests on generated code.

**Schema:**
```json
{
 "task_id": "string",
 "status": "string (success, failed)",
 "line_coverage": "float (0.0 - 1.0) or null",
 "branch_coverage": "float (0.0 - 1.0) or 'N/A' (for HumanEval)",
 "timestamp": "ISO 8601 datetime string",
 "error_message": "string (if status is failed)"
}
```

**Notes:**
- For HumanEval tasks, `branch_coverage` is explicitly set to `"N/A"` as per FR-012.
- If generation or execution fails, `status` is `"failed"` and `error_message` contains the exception details.

### 3. Statistical Summary (`data/processed/stats_summary.csv`)

Output of `code/analyzer.py` containing comparative statistics between LLM and human solutions.

**Columns:**
- `mean_llm`: Average line coverage for LLM-generated code.
- `mean_human`: Average line coverage for human solutions.
- `mean_diff`: Difference (`mean_llm` - `mean_human`).
- `p_value`: P-value from the statistical test.
- `cohen_d`: Effect size (Cohen's d).
- `test_type`: Name of the test used (`t-test` or `Wilcoxon`).

### 4. Corrected P-values (`data/processed/corrected_pvalues.csv`)

Output of `code/analyzer.py` containing p-values adjusted for multiple hypothesis testing.

**Columns:**
- `subgroup`: Name of the subgroup (e.g., `loops`, `conditionals`).
- `raw_p_value`: Original p-value.
- `bonferroni_corrected`: P-value after Bonferroni correction.
- `holm_corrected`: P-value after Holm-Bonferroni correction.

### 5. Sensitivity Report (`data/processed/sensitivity_report.csv`)

Output of `code/sensitivity_analyzer.py` showing how results change across significance thresholds.

**Columns:**
- `threshold`: Significance level (0.01, 0.05, 0.10, etc.).
- `significant_count`: Number of subgroups with p < threshold.
- `effect_direction`: Dominant direction of effect (positive, negative, mixed).

### 6. Stratified Reports (`outputs/stratified_*.csv`)

Grouped analysis by difficulty and code patterns.

**Columns:**
- `pattern`: Code pattern (e.g., `loops`, `conditionals`).
- `difficulty`: Task difficulty level.
- `mean_gap`: Mean coverage gap (LLM - Human).
- `count`: Number of tasks in the group.

### 7. Annotated Dataset (`data/processed/annotated_catalog.json`)

Extended catalog with code pattern annotations.

**Schema:**
```json
{
 "tasks": [
 {
 "task_id": "string",
 "prompt": "string",
 "human_solution": "string",
 "code_patterns": {
 "loops": "boolean",
 "conditionals": "boolean",
 "recursion": "boolean",
 "boundary_cases": "boolean"
 },
...
 }
 ]
}
```

## Data Flow

1. **Ingestion**: Raw datasets (MBPP, HumanEval) are downloaded and saved to `data/benchmarks/raw/`.
2. **Transformation**: Test suites are converted to executable Python files in `data/benchmarks/processed/tests/`.
3. **Cataloging**: A unified `catalog.json` is created with normalized fields.
4. **Generation**: LLMs generate code, saved to `data/generated/`.
5. **Coverage Execution**: Tests are run, results saved to `data/coverage_reports/`.
6. **Analysis**: Statistical tests and stratification are performed, outputs saved to `data/processed/` and `outputs/`.

## Constraints & Rules

- **HumanEval Branch Coverage**: Always marked as `"N/A"` in coverage reports.
- **4-bit Quantization**: Fallback models must use `bitsandbytes` 4-bit quantization (SC-005).
- **VIF Calculation**: Only performed if `--model-method=regression` is specified (FR-013).
- **Exclusion Rate**: Sensitivity analysis thresholds are excluded from family-wise error correction (FR-011).
