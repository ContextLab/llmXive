# Feature Specification: Reproduce & Validate Bidirectional Evolutionary Search (BES)

**Feature Branch**: `637-reproduce-validate-bes`  
**Created**: 2026-06-01  
**Status**: Draft  
**Input**: User description: "Reproduce & validate: Self-Improving Language Models with Bidirectional Evolutionary Search"

## User Scenarios & Testing

### User Story 1 - Execute BES Inference Pipeline on Benchmark Task (Priority: P1)

**User Journey**: A researcher needs to verify that the vendored BES codebase is functional and can execute the core inference loop on a standard benchmark (e.g., `circle_packing`) without manual intervention, producing valid output artifacts.

**Why this priority**: This is the foundational "smoke test" for the entire project. If the code cannot run end-to-end on a CPU-only CI runner, no further validation of the paper's claims is possible. It directly addresses the "must run the computation" requirement noted by the Stephen Wolfram review.

**Independent Test**: The CI job executes the entry point script for the `circle_packing` task with a fixed random seed and a timeout limit. Success is defined by the existence of output files and a non-zero exit code indicating completion (not crash).

**Acceptance Scenarios**:
1. **Given** the BES submodule is cloned and dependencies are installed, **When** the `run_evo.py` script is executed for `circle_packing` with a timeout of 30 minutes, **Then** the process exits with code 0 and generates at least one solution file in `inference/results/circle_packing/`.
2. **Given** the BES submodule is cloned, **When** the script is executed on a CPU-only runner with no GPU devices available, **Then** the system does not crash due to CUDA errors and logs a warning or fallback message regarding hardware.

---

### User Story 2 - Validate Solution Correctness Against Paper Metrics (Priority: P2)

**User Journey**: A researcher needs to confirm that the solutions generated by the BES pipeline are actually correct and satisfy the problem constraints defined in the paper, rather than just running successfully.

**Why this priority**: Running the code is insufficient; the "improvement" claimed by the paper (as highlighted by the Alan Turing review regarding "operational test for improvement") must be demonstrated by the quality of the output. This validates the *utility* of the search.

**Independent Test**: The generated solution artifacts are passed through the `evaluate.py` script. The output score must be compared against the baseline or "pass" threshold defined in the task configuration.

**Acceptance Scenarios**:
1. **Given** a set of generated solution files from User Story 1, **When** the `evaluate.py` script is run, **Then** the script outputs a success metric (e.g., "valid", "score > 0.9") for at least 1 of the 3 generated attempts.
2. **Given** a generated solution that fails the geometric constraints, **When** the evaluator runs, **Then** the system flags the specific constraint violation (e.g., "circle overlap detected") rather than returning a generic error.

---

### User Story 3 - Reproduce Bidirectional Search Dynamics (Priority: P3)

**User Journey**: A researcher needs to verify that the "bidirectional" nature of the search (forward evolution + backward decomposition) is actually occurring during execution, not just a standard greedy search.

**Why this priority**: This addresses the core novelty of the paper. If the backward decomposition is missing or broken, the system is merely a standard evolutionary search, failing the specific claim of the paper.

**Independent Test**: The execution logs are parsed to confirm the presence of both "forward expansion" and "backward decomposition" events within the same run cycle.

**Acceptance Scenarios**:
1. **Given** a completed run of the BES pipeline, **When** the execution logs are inspected, **Then** at least 5 distinct "backward decomposition" events are recorded before the final solution is generated.
2. **Given** a run with a fixed seed, **When** the intermediate state files are inspected, **Then** the goal tree structure contains at least one node with a depth > 1, proving recursive decomposition occurred.

---

### Edge Cases

- **Timeout Handling**: If the search exceeds the 6-hour CI limit or the 30-minute task limit, the system must gracefully terminate and report "Timeout" rather than hanging indefinitely.
- **Resource Exhaustion**: If the memory usage approaches the 7 GB limit during the search tree expansion, the system must either sample/subset the tree or fail gracefully with a clear "OOM" error, preventing a silent crash.
- **Model Availability**: If the configured LLM provider (e.g., local OpenAI compatible endpoint) is unreachable, the system must retry up to 3 times with exponential backoff before failing, rather than crashing on the first connection error.

## Requirements

### Functional Requirements

- **FR-001**: System MUST execute the BES inference pipeline for the `circle_packing` task within a 30-minute timeout window on a CPU-only runner. (See US-1)
- **FR-002**: System MUST generate at least 3 distinct candidate solution files in the designated output directory (`inference/results/circle_packing/`) per run. (See US-1)
- **FR-003**: System MUST validate every generated solution against the problem constraints using the provided `evaluate.py` script. (See US-2)
- **FR-004**: System MUST log explicit markers for "forward expansion" and "backward decomposition" events to allow verification of the bidirectional mechanism. (See US-3)
- **FR-005**: System MUST fail gracefully with a distinct exit code (non-zero) and a clear error message if the LLM provider is unreachable after a configurable number of retry attempts. (See US-1)
- **FR-006**: System MUST NOT attempt to load any GPU-specific libraries (CUDA, bitsandbytes) or require GPU devices; all operations must be compatible with CPU execution. (See US-1)

### Key Entities

- **Search State**: The current configuration of the forward candidate pool and backward goal tree.
- **Candidate Solution**: A generated code or text artifact attempting to solve the benchmark task.
- **Evaluation Metric**: The quantitative score or binary pass/fail status assigned to a candidate by `evaluate.py`.

## Success Criteria

### Measurable Outcomes

> Planning docs state *what* will be measured and the *source/reference* it is measured against; defer specific empirical values to the implementation phase.

- **SC-001**: The time-to-solution for the `circle_packing` task is measured against the 30-minute timeout limit defined in FR-001. (See US-1)
- **SC-002**: The solution validity rate (percentage of candidates passing `evaluate.py`) is measured against the baseline success rate reported in the paper's abstract. (See US-2)
- **SC-003**: The frequency of backward decomposition events is measured against the theoretical expectation that decomposition reduces sample complexity (as per the paper's claim of exponential reduction). (See US-3)
- **SC-004**: The memory footprint of the search process is measured against the RAM limit of the CI runner. (See Assumptions)
- **SC-005**: The reproducibility of the search dynamics (presence of both forward/backward events) is measured against the requirement for ≥5 backward events per run. (See US-3)

## Assumptions

- The vendored `external/BES` repository contains a functional `inference` module that does not require external GPU hardware to run a minimal test case.
- The `circle_packing` task is the most computationally tractable benchmark available in the suite for a CPU-only environment and will be used as the primary validation target.
- The LLM provider configured in `local_openai_config.py` (or default) is accessible from the CI environment; if a local model is required, a small, CPU-tractable model (e.g., a quantized Llama 3 8B via `llama-cpp-python` or similar CPU-optimized backend) is assumed to be available or simulated.
- The "improvement" metric is strictly defined by the `evaluate.py` script's return value, as no external ground truth is available for the open problem solving benchmarks beyond the script's logic.
- The paper's claim of "exponential reduction" in samples is treated as an associational observation for this validation run; causal claims are not asserted without randomized controlled trials.
- The CI runner's 6-hour total job limit allows for a single full run of the `circle_packing` task with a 30-minute timeout per attempt.
