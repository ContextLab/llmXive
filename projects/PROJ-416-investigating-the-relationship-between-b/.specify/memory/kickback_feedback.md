## Unresolved concerns

- The plan mandates a fallback to 'separate univariate models' if Ridge regression fails (FR-005, FR-012). This breaks the causal/associational validity of the analysis. If the goal is to isolate the effect of a specific network metric (e.g., modularity) while controlling for others, univariate models fail to control for confounding by the other metrics. The plan explicitly rejects Ridge as 'methodologically unsound' in the text but retains the Spec's requirement to use it as a primary step, creating a logical contradiction where the analysis cannot validly answer the research question if Ridge is skipped.
- The plan proposes a sample size of '~20 subjects' to fit within a 6-hour window. For a multiple regression analysis (ANCOVA) with 3-5 predictors (Pre-score, Modularity, Global Eff, Local Eff, Covariates), N=20 provides extremely low statistical power (<20% for medium effects) and violates the '10 events per variable' rule of thumb for stable coefficient estimation. The plan acknowledges SC-004 (Power Calculation) but the proposed N is insufficient to detect the effect sizes (f²=0.15) required by the spec, rendering the study likely to yield false negatives regardless of the statistical method used.
- The plan relies on a single 'Primary Candidate' dataset (OpenNeuro via HuggingFace) and states the pipeline 'halts' if clinical scores are missing. This is a fatal design flaw for a research study: if the only available open dataset lacks the specific pre/post anxiety scores required, the entire research question becomes unanswerable. The methodology lacks a contingency for multi-dataset aggregation or a broader search strategy, making the study's success contingent on a single, unverified data source.
- The plan states SC-004 (Power Calculation) will 'flag limitation' if N < required. However, with N=20 and 5+ predictors, the power is so low that the study is effectively underpowered to answer the research question. A 'flag' does not mitigate the methodological invalidity of drawing conclusions from an underpowered design. The plan should either increase N (if feasible) or explicitly reframe the study as 'exploratory' with no hypothesis testing, rather than attempting ANCOVA.
- FR-010 (Sensitivity Analysis) and SC-006 (Threshold Sensitivity) are mapped to 'Phase 4: SensitivityService' but the plan lacks a specific task or phase step defining the concrete sweep parameters (motion: {2.0, 3.0} mm; p-value: {0.01, 0.05, 0.1}) required by the spec. The plan mentions 'sensitivity analysis' generically but does not explicitly commit to the specific discrete sets mandated by FR-010/SC-006, risking an implementation that fails the success criterion.
- FR-011 (Verify pre/post pairs) and SC-001 (Dataset-variable fit) are mapped to 'Phase 1: DataValidator', but the plan does not explicitly define the 'Verified Source Gate' artifact or the specific validation logic (checking for 'pre_treatment_score', 'post_treatment_score', 'anxiety_instrument' in metadata) required to satisfy the 'halt with fatal error' condition. The plan states 'DataValidator checks' but lacks the explicit 'halt' logic description found in the spec's Edge Cases.
- FR-005 and FR-012 (Collinearity handling: VIF > 5 -> Ridge -> Univariate) are mapped to 'Phase 4: StatisticalService', but the plan's 'Computational Strategy' section and 'Risk Management' table do not explicitly describe the fallback chain (VIF check -> Ridge -> Univariate) with the specific thresholds (lambda=1.0, R² < 0.05) mandated by the spec. - Phase 1 includes 'Data Acquisition & Validation'. The 'Research' doc states that if the verified dataset lacks clinical scores, the pipeline halts. The plan should explicitly add a 'Gate' step in Phase 1 that halts execution if the dataset verification fails, ensuring the 'BLOCKED' status mentioned in the kickback comments is respected and not bypassed by a generic 'Download' step.
- The plan defines the outcome variable as 'Post-treatment anxiety score' ($Y_{post}$) and the primary predictor as 'Baseline Severity: Pre-treatment score' ($Y_{pre}$). It then proposes to test the association of network metrics with $Y_{post}$ while controlling for $Y_{pre}$. This is a standard ANCOVA design, BUT the plan also defines 'Treatment Response' in the data model as the outcome. If the plan intends to model 'Response' (defined as $Y_{post} - Y_{pre}$ or $\Delta$) as the outcome, using $Y_{pre}$ as a predictor creates a mathematical artifact (regression to the mean) where the correlation between baseline and change is definitionally negative. The plan must explicitly state whether the outcome is $Y_{post}$ (ANCOVA) or $\Delta$ (Change Score) and ensure the predictor set does not include the component used to define the outcome if using $\Delta$. Currently, the text implies ANCOVA ($Y_{post}$ ~ $Y_{pre}$ + Metrics), which is valid, but the 'Treatment Response' entity definition in data-model.md suggests a change score, creating a potential conceptual conflict in the validation target definition.
- The plan proposes using 'Modularity (Q)', 'Global Efficiency', and 'Local Efficiency' as simultaneous predictors in a multivariate model. In graph theory, these metrics are often definitionally correlated (e.g., high modularity often implies lower global efficiency in sparse networks). The plan acknowledges this with a VIF check and a fallback to Ridge regression. However, Ridge regression shrinks coefficients but does not resolve the fundamental issue of testing 'independent effects' of definitionally linked variables. If the goal is to identify *which* network property drives the response, multicollinearity makes the individual coefficients uninterpretable regardless of regularization. The plan should explicitly state that the analysis will test *sets* of metrics or use dimensionality reduction (PCA) on the metrics before regression, rather than claiming to test independent effects of Q, E_glob, and E_loc simultaneously.
- The plan specifies using 'statsmodels.stats.power.FTestPower' for power analysis with parameters $f^2=0.15$ (medium effect). However, the sample size is restricted to 'first 20 valid subjects' (Plan: Computational Strategy). A post-hoc power calculation on N=20 with 3-4 predictors will almost certainly show <50% power for a medium effect. The plan states: 'If 5 <= N < N_min: Flag limitation'. This is methodologically weak; the study is likely underpowered to detect the claimed effect size. The plan should either justify the N=20 limit as a pilot study (changing the claim from 'detect association' to 'estimate effect size with wide CI') or explicitly state that the analysis is exploratory and not powered for hypothesis testing.
- The plan relies on a single 'OpenNeuro' dataset (via a specific HuggingFace link) containing 'resting-state fMRI' AND 'paired pre/post clinical anxiety scores'. OpenNeuro datasets are typically task-based or resting-state studies of specific cohorts (e.g., 'anxiety patients'). It is highly improbable that a single public OpenNeuro dataset contains *both* resting-state scans AND a longitudinal clinical trial design with pre/post VR therapy scores. Most public fMRI datasets are cross-sectional. If the dataset is cross-sectional, there is no 'post-treatment' score, and the study design (VR therapy response) cannot be executed. The plan treats the existence of such a dataset as a 'Critical Check' that might fail, but the entire research question depends on a dataset type (longitudinal clinical trial with rs-fMRI) that is extremely rare in open repositories. The plan should acknowledge this high risk of 'dataset mismatch' where the available data does not support the longitudinal research question.
