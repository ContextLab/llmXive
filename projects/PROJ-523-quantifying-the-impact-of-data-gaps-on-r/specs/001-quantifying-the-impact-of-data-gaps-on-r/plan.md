# Implementation Plan: Quantifying the Impact of Data Gaps on Reconstructed CMB Maps

**Branch**: `001-cmb-gap-bias-analysis` | **Date**: 2024-01-15 | **Spec**: `specs/001-cmb-gap-bias-analysis/spec.md`
**Input**: Feature specification from `/specs/001-cmb-gap-bias-analysis/spec.md`

## Summary

This feature implements a computational pipeline to quantify systematic biases in cosmological parameter recovery (H₀, Ωₘ, nₛ, τ) caused by data gaps in Cosmic Microwave Background (CMB) maps. The approach involves: (1) generating simulated CMB maps with controlled gap patterns (fraction, distribution, morphology) and a **Null Model** baseline; (2) applying a **Mode-Coupling Correction** to account for inpainting artifacts; (3) applying three gap-filling algorithms (harmonic interpolation, Wiener filtering, iterative harmonic synthesis); (4) computing angular power spectra (Cℓ) via HEALPix; and (5) estimating parameters using a **corrected** Fisher matrix approximation. The pipeline strictly adheres to free-tier CI constraints (≤6h runtime, ≤7GB RAM, CPU-only) by dynamically adjusting the number of realizations based on a pilot runtime check.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| **Dynamic Budget Check (FR-006)** | CI constraints (6h) are tight for 50 realizations × 3 algos × 5 fractions. | A static run of 50 might timeout; a fixed smaller run might lack statistical power. Dynamic check ensures max valid realizations within time. |
| **Mode-Coupling Correction (FR-009)** | Gap-filling introduces non-Gaussian artifacts violating standard Fisher assumptions. | Ignoring this would conflate gap-filling error with likelihood mismatch, invalidating the bias measurement. |
| **Full Factorial LMM (FR-005)** | Bias depends on interactions between Gap Fraction, Algorithm, and Morphology. | Simple ANOVA ignores these interactions, leading to underspecified statistical models. |
| **Null Model Baseline** | To avoid circular validation where bias is a mathematical artifact of gap fraction. | Without a null model, a significant p-value might just reflect the algorithm's deterministic behavior, not a physical effect. |
| **Fisher Matrix Approx (FR-004)** | Full MCMC (CosmoMC) is too slow for 50 realizations on CPU. | Pre-computed grids or Fisher matrices provide valid statistical inference within iteration limits and fit the 6h budget, *provided* the mode-coupling correction is applied. |

## Phased Implementation Plan

### Phase 0: Research & Feasibility (Output: `research.md`)
- **Task 0.1**: Verify dataset sources. Confirm `camb` and `healpy` availability on CPU-only runners.
- **Task 0.2**: Review literature on gap-filling algorithms and mode-coupling corrections.
- **Task 0.3**: Define "Verified datasets" block. Confirm no external CMB maps are needed (simulations are generated).
- **Task 0.4**: Draft `research.md` with dataset strategy (simulated only), algorithm rationale, and statistical power justification (removing circular logic).

### Phase 1: Data Model & Contracts (Output: `data-model.md`, `contracts/*.yaml`)
- **Task 1.1**: Define schemas for `CMBMap`, `GapConfig`, `PowerSpectrum`, `ParameterPosterior`, and **`SensitivityAnalysis`**. Explicitly include `bias_magnitude`, `p_value`, and `leakage_matrix` fields.
- **Task 1.2**: Create `contracts/simulation.schema.yaml` and `contracts/analysis.schema.yaml`.
- **Task 1.3**: Define `quickstart.md` with environment setup and pilot run instructions.

### Phase 2: Core Pipeline Implementation (Output: `code/`)
- **Task 2.1 (FR-001)**: Implement `generate_maps.py`. Create ground-truth maps with CAMB. Apply gap masks (random, clustered, point-source, Galactic). **Write ground-truth parameters to metadata file.** Generate **Null Model** realizations (random gaps uncorrelated with signal).
- **Task 2.2 (FR-002)**: Implement three gap-filling algorithms. Ensure no NaNs in output. Log execution time. **Record algorithm version and config in metadata (Constitution VI).**
- **Task 2.3 (FR-003)**: Implement `power_spectra.py` using `healpy.anafast`. Verify Nside=512, ℓ≤2000.
- **Task 2.4 (FR-004, FR-009)**: Implement `parameter_est.py`.
    - **Step 2.4a**: Calculate **Mode-Coupling Matrix (Leakage Matrix)** from the gap mask.
    - **Step 2.4b**: Apply leakage matrix to theoretical spectrum to correct the Fisher Matrix calculation.
    - **Step 2.4c**: Estimate parameters (H₀, Ωₘ, nₛ, τ) using the corrected Fisher Matrix. Record ground-truth vs. recovered.
- **Task 2.5 (FR-005, FR-007, FR-009)**: Implement bias calculation.
    - **Step 2.5a**: Calculate `bias_magnitude` = |recovered - ground_truth|.
    - **Step 2.5b**: Perform **Full Factorial Linear Mixed Model (LMM)** with interaction terms (Fraction × Algorithm × Morphology) and random effects for realization ID.
    - **Step 2.5c**: Apply multiple-comparison correction (Benjamini-Hochberg).
    - **Step 2.5d**: Execute sensitivity analysis sweep (α ∈ {low, medium, high}, tolerance ∈ {low, medium, high}) and store results in `SensitivityAnalysis` entity.
    - **Step 2.5e**: Compare observed bias trend against the **Null Model** baseline to ensure independence.
- **Task 2.6 (FR-006)**: Implement pilot run logic.
    - **Step 2.6a**: Run pilot (a limited set of realizations) to estimate runtime.
    - **Step 2.6b**: Calculate max N.
    - **Step 2.6c**: **Explicit Reduction Logic**: If N < 30, reduce N_fractions first, then N_algos. **Log the specific configuration changes (original vs. final N_fractions, N_algos) to `run_log.yaml`.**
    - **Step 2.6d**: Ensure ≥30 valid realizations remain.

### Phase 3: Integration & Validation (Output: `tests/`)
- **Task 3.1**: Write contract tests to validate output data against YAML schemas (including `SensitivityAnalysis` and `leakage_matrix`).
- **Task 3.2**: Run full pipeline on a subset (e.g., a small number of realizations) to verify memory and time.
- **Task 3.3**: Execute sensitivity analysis (FR-007) sweep.

### Phase 4: Final Execution & Reporting
- **Task 4.1**: Run full pipeline with dynamic N.
- **Task 4.2**: Generate final bias plots and tables.
- **Task 4.3**: Verify all FRs and SCs are met.

## Risk Mitigation

- **Risk**: Runtime exceeds 6h.
  - **Mitigation**: Pilot run (FR-006) reduces N_fractions or N_algos if pilot time > budget threshold. **Explicit logging of reduction steps.**
- **Risk**: Algorithm convergence failure.
  - **Mitigation**: Catch exceptions, log failure, exclude from analysis (FR-008). Ensure ≥30 valid realizations remain.
- **Risk**: Memory overflow (HEALPix maps).
  - **Mitigation**: Process maps sequentially. Use `numpy` float32 where precision allows. Avoid loading all 50 maps into RAM simultaneously.
- **Risk**: Mode-Coupling Calculation Failure.
  - **Mitigation**: If leakage matrix calculation fails, log error, exclude that realization, and flag in `run_log.yaml`.

## Compute Feasibility Statement

This plan explicitly targets the GitHub Actions free-tier (standard CPU, 7GB RAM, 14GB Disk, 6h).
- **No GPU**: All methods (CAMB, healpy, scipy, statsmodels) are CPU-native.
- **Memory**: Nside=512 maps are ~3M pixels. A large number of maps would exceed RAM if loaded at once. The pipeline processes one realization at a time.
- **Runtime**: The pilot run ensures the total number of realizations is scaled to fit the 6h window.
- **Libraries**: `healpy`, `camb`, `scipy`, `statsmodels` have pre-compiled wheels for CPU. No CUDA dependencies.
