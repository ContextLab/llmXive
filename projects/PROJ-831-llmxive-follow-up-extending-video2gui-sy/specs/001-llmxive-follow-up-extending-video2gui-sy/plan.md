**Input**: Feature specification from `specs/001-tutorial-bias-analysis/spec.md`

## Summary

This project investigates the "tutorial bias" hypothesis: that agents trained exclusively on linear "happy path" GUI trajectories (simulating video-derived data) lack robustness when encountering non-linear error states, compared to agents augmented with synthetic error-recovery data. The plan implements a CPU-tractable benchmarking pipeline to evaluate three agent variants (Baseline, Linear-Only Proxy, Hybrid) on a synthetically generated set of non-linear GUI tasks. The core deliverable is a statistical comparison (McNemar's test) of error-recovery performance, ensuring all computations fit within GitHub Actions free-tier constraints (CPU, 7GB RAM, 6h limit).

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
| :--- | :--- | :--- |
| **Synthetic Simulator** | Real-world GUI data lacks explicit, labeled error states for non-linear recovery. | Using only real-world data would confound "tutorial bias" with general task difficulty. A simulator is required to isolate the variable. |
| **3 Agent Variants** | Must isolate the effect of error-recovery data (Hybrid) vs. linear data (Linear-Only Proxy) vs. generic instructions (Baseline). | Comparing only 2 agents would not distinguish between "video bias" and "general poor performance" or "lack of GUI grounding". |
| **McNemar's Test** | Paired outcomes (same task, different agents) require a test for dependent proportions. | Independent t-tests or Chi-square would be statistically invalid for paired binary data. |
| **CPU-Only Quantization** | Must run on GitHub Actions free tier (no GPU). | Full precision or GPU-dependent models would fail the feasibility constraint (SC-004). |
| **Taxonomy Split** | To prevent data leakage, training and test sets use disjoint error taxonomies. | Using the same taxonomy for training and testing would allow memorization of recovery rules. |
| **Pilot Study** | To ensure runtime feasibility before full execution. | Running the full set of tasks without a pilot risks timeout and zero data. |
| **Dynamic N Adjustment** | To ensure statistical power if latency exceeds limits. | Fixed N would lead to timeout or insufficient power if latency is high. |
