# Evaluating the Impact of Code Generation Models on Code Security: A Fuzzing Study  

**Field**: computer science  

## Research question  

Do code snippets generated by large language models exhibit systematically different (e.g., higher) security vulnerabilities than comparable human‑written code when evaluated under a standardized fuzzing framework?  

## Motivation  

The rapid adoption of LLM‑based code generators promises productivity gains, yet the security posture of automatically produced code is largely unknown. Identifying whether LLM‑generated programs are more prone to exploitable bugs can guide safer deployment practices, inform model training objectives, and shape tooling for developers who rely on AI assistance.

## Expected results  

We expect to observe a statistically significant difference in the number and severity of bugs uncovered by fuzzing between LLM‑generated and human‑written solutions. A higher median crash count or a larger proportion of high‑severity findings in the LLM group would confirm the hypothesis; a non‑significant difference would falsify it. Evidence will be based on quantitative metrics (crash count, unique bug identifiers, CVSS‑like severity) and statistical tests at α = 0.05.  

## Methodology sketch  

- **Dataset selection** – Use a public repository of programming challenge solutions such as the *HumanEval* benchmark (OpenAI) and the *Codeforces* problem set (available via the Codeforces API).  
- **Model generation** – Prompt each challenge to multiple open‑source code‑generation models (e.g., CodeLlama‑7B, StarCoder‑15B) via their HuggingFace inference APIs; record generated source files.  
- **Human baseline** – Extract the reference human solutions provided in the same benchmark for each problem.  
- **Compilation pipeline** – For each snippet, automatically compile with `gcc`/`clang` using deterministic flags (`-O0 -g`) inside a Docker container to ensure reproducibility.  
- **Fuzzing harness** – Wrap each compiled binary with a libFuzzer or AFL++ harness that supplies random inputs to all exposed functions (e.g., via stdin, command‑line arguments, or network sockets).  
- **Fuzzing execution** – Run the harness on GitHub Actions runners for a fixed budget (e.g., 30 minutes per binary, total ≤ 6 h) using the `timeout` utility; collect crash logs and stack traces.  
- **Bug triage** – Parse crash logs with `coredumpctl` and map each unique stack trace to a vulnerability type (buffer overflow, use‑after‑free, etc.). Assign a severity score using a simplified CVSS‑like rubric.  
- **Statistical analysis** – Compare the distributions of crash counts and severity scores between the LLM‑generated and human groups using non‑parametric tests (Mann‑Whitney U, Kolmogorov‑Smirnov) and compute effect sizes (Cliff’s Δ).  
- **Result visualization** – Produce box‑plots and bar charts (via `matplotlib`) summarizing the differences; generate a concise report in Markdown.
