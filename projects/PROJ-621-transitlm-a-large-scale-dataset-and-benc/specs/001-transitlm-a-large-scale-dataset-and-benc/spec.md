# Feature Specification: Map-Free Transit Route Generation with LLMs

**Feature Branch**: `001-map-free-transit-route-generation`  
**Created**: 2026-06-30  
**Status**: Draft  
**Input**: User description: "Can large language models, trained exclusively on textual transit logs and station metadata without explicit geographic coordinates or map topology, learn to generate valid public transit routes that satisfy real-world connectivity constraints?"

## User Scenarios & Testing

### User Story 1 - Map-Free Route Generation & Validation (Priority: P1)

A researcher provides an origin-destination pair and receives a route sequence generated by a fine-tuned LLM, which is then automatically validated against the ground-truth GTFS graph to determine if the path is topologically valid (i.e., every consecutive station pair exists as a direct connection in the real network).

**Why this priority**: This is the core scientific inquiry. Without the ability to generate a candidate route and validate its topological correctness, the research question regarding "implicit spatial learning" cannot be answered. It defines the MVP of the benchmark.

**Independent Test**: Can be fully tested by running the inference pipeline on a held-out test set (containing origin-destination pairs NOT seen in training text) and computing the "Route Exact Match" and "Connectivity Validity" metrics against the deterministic graph-traversal script.

**Acceptance Scenarios**:

1. **Given** a preprocessed GTFS dataset converted to text sequences (with coordinates stripped), **When** the fine-tuned LLM generates a route for a specific origin-destination pair from the **held-out** set, **Then** the system outputs the sequence of station names and transfer points.
2. **Given** the generated sequence, **When** the deterministic graph-traversal script validates it against the ground-truth GTFS graph, **Then** the system returns a binary "Valid/Invalid" flag and a "Exact Match" score.
3. **Given** a zero-shot general-purpose LLM baseline, **When** it attempts the same generation task on the held-out set, **Then** the system records its validity score for comparative analysis.

---

### User Story 2 - Comparative Statistical Analysis (Priority: P2)

A researcher compares the performance of the fine-tuned model against zero-shot baselines to determine if the improvement in route validity is statistically significant, ensuring the result is not due to chance or memorization.

**Why this priority**: Scientific rigor requires statistical evidence. A raw improvement in accuracy is insufficient without a p-value or confidence interval to confirm the fine-tuning actually caused the topological learning on unseen data.

**Independent Test**: Can be tested by aggregating validity scores from both models on the **topologically distinct** held-out set and running a paired t-test or Wilcoxon signed-rank test to produce a p-value.

**Acceptance Scenarios**:

1. **Given** a dataset of N origin-destination pairs (topologically distinct from training) with validity scores for both the fine-tuned and baseline models, **When** the statistical analysis script is executed, **Then** it outputs a p-value indicating the significance of the difference.
2. **Given** a p-value < 0.05, **When** the results are reported, **Then** the conclusion states that the fine-tuned model demonstrates a statistically significant improvement in topological reasoning on unseen connections.

---

### User Story 3 - Dataset Construction & Variable Verification (Priority: P3)

A data engineer constructs the "map-free" dataset by converting GTFS feeds into natural language sequences, ensuring that no geographic coordinates or map topology are inadvertently included in the input text.

**Why this priority**: The validity of the "map-free" claim depends entirely on the input data. If coordinates leak into the text, the experiment fails the core constraint.

**Independent Test**: Can be tested by sampling 100 random input prompts and verifying via regex/scan that no coordinate patterns (e.g., "lat: 40.71, lon: -74.00") or graph adjacency lists appear in the raw text.

**Acceptance Scenarios**:

1. **Given** raw GTFS `stops.txt` and `routes.txt` files, **When** the conversion script processes them into "Take Line X from Station A to Station B" sequences, **Then** the output text contains only station names, line IDs, and sequence order.
2. **Given** the converted dataset, **When** a validation check scans for numeric coordinate patterns, **Then** the check returns zero matches.

---

### Edge Cases

- What happens when the LLM generates a sequence that loops back on itself (infinite loop) or hallucinates a non-existent transfer station? (Handled by the graph-traversal script returning "Invalid". Note: The "map-free" constraint applies to **input** data; the system is designed to **detect** and **reject** invalid **output**).
- How does the system handle origin-destination pairs that are physically disconnected in the GTFS graph (no valid route exists)? (The baseline and fine-tuned models should ideally fail or report "No Route"; validity is 0).
- What happens if the LLM output format deviates from the expected "Station A -> Station B" pattern (e.g., includes conversational filler)? (The parser must strip non-station tokens before validation).

## Requirements

### Functional Requirements

- **FR-001**: System MUST convert GTFS stop and route data into natural language sequences containing only station names and line identifiers, explicitly stripping all geographic coordinates and map topology data (See US-1).
- **FR-002**: System MUST generate route sequences for a held-out set of origin-destination pairs using a fine-tuned LLM variant (quantized ≤ 1.5B parameters or heavily quantized 3B) that fits within 7GB RAM constraints, verified by logging peak RSS memory usage via a monitoring script (See US-1).
- **FR-003**: System MUST validate generated routes against the ground-truth GTFS graph using a deterministic graph-traversal script to compute "Route Exact Match" and "Connectivity Validity" scores, specifically for pairs topologically distinct from the training set (See US-1).
- **FR-004**: System MUST perform a paired statistical test (t-test or Wilcoxon) comparing the validity scores of the fine-tuned model against a zero-shot baseline on the **held-out** set to determine statistical significance, ensuring the test set prevents data leakage of specific adjacency pairs (See US-2).
- **FR-005**: System MUST enforce a "map-free" constraint where no geographic coordinates or explicit adjacency lists are present in the **input prompt context** for the LLM; the system must also detect and flag any **output** containing hallucinated stations or invalid topology as "Invalid" (See US-3).
- **FR-006**: System MUST handle invalid or disconnected origin-destination pairs by returning a validity score of 0, ensuring the metric reflects true topological failure rather than data error (See US-1).
- **FR-007**: System MUST construct a "held-out" test set where the origin-destination pairs and their intermediate connections are **not** present in the training text sequences, to ensure the validation measures generalization to unseen topological connections rather than memorization (See US-1, US-2).

### Key Entities

- **RouteSequence**: An ordered list of station names and line identifiers generated by the LLM, representing a proposed path.
- **GroundTruthGraph**: The canonical graph derived from GTFS data, used as the oracle for validation.
- **ValidityScore**: A binary (Valid/Invalid) or continuous (0.0-1.0) metric representing the degree to which a generated route matches the ground truth.

## Success Criteria

### Measurable Outcomes

- **SC-001**: Route Validity Rate is measured against the ground-truth GTFS graph to determine the percentage of generated paths that are topologically correct (See FR-003, FR-006).
- **SC-002**: Statistical Significance (p-value) is measured against the standard alpha threshold (0.05) to confirm the improvement of the fine-tuned model over the zero-shot baseline (See FR-004).
- **SC-003**: Coordinate Leakage Rate is measured against the input prompt set to ensure the Coordinate Leakage Rate is [deferred] (measured value deferred) (See FR-005).
- **SC-004**: Total inference time for N=100 samples is measured against the 6-hour total job limit to ensure the analysis completes within the free-tier CI constraints (See FR-002).
- **SC-005**: Exact Match Rate is measured against the ground-truth path sequences to quantify the precision of the generated route, not just validity (See FR-003).

## Assumptions

- The "TransitLM" dataset or a verified subset from a persistent DOI (e.g., Zenodo) is available and accessible; if not, a synthetic benchmark will be constructed from OpenStreetMap GTFS feeds as the primary data source.
- The fine-tuned LLM (e.g., Llama-3-1.5B, Qwen2.5-1.5B, or heavily quantized 3B) will be selected to fit within the 7GB RAM limit of the GitHub Actions free-tier runner, acknowledging that larger models (7B+) are methodologically infeasible for SFT and inference within the 6-hour CPU constraint.
- The analysis is observational; findings regarding "learning" are framed as associational improvements in performance on the benchmark task, not causal claims about the model's internal cognitive architecture.
- The GTFS data provided contains all necessary variables (station IDs, line IDs, stop sequences) to reconstruct the ground-truth graph for validation.
- The graph-traversal script runs deterministically and does not require external network access or heavy computational resources beyond standard CPU operations.
- The fine-tuning process (SFT with LoRA) will be scaled to a dataset size and model architecture (≤ 1.5B params) that ensures completion within the 6-hour job limit on a CPU-only runner.
- No GPU acceleration is available; all model training and inference must rely on CPU execution, potentially requiring batch sizes of 1 or 2.
