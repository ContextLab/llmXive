# Research: llmXive follow-up: extending "Qwen-AgentWorld: Language World Models for General Agents"

## 1. Research Question & Hypothesis

**Primary Question**: To what extent do LLM-generated reasoning traces in the Qwen-AgentWorld environment diverge from the deterministic ground-truth physics, and can this divergence be decomposed into "Hallucinations" (active errors) versus "Rule Gaps" (omissions of inferable logic)?

**Hypothesis**: LLMs will exhibit a higher rate of "Hallucination" errors in complex, long-horizon tasks compared to "Rule Gaps," and the divergence rate will increase non-linearly with task step count.

## 3. Methodology

### 3.1 Ground Truth Oracle Construction (FR-001)
- **Input**: Qwen-AgentWorld environment source code (parsed from the repository).
- **Process**: A parser (`oracle/parser.py`) will traverse the code to identify state variables, action definitions, and transition logic.
- **Schema Alignment**: The parsed Oracle will be validated against the `AgentWorldBench` task definitions to ensure the environment logic matches the benchmark tasks.
- **Output**: A deterministic Python function `oracle_step(state, action) -> next_state` that mathematically represents the environment physics.
- **Validation**: Run on N=1,000 random seeds; compare output against the original simulator (if accessible) or manual verification of edge cases.

### 3.2 Rule Extraction (FR-002)
- **Input**: LLM Chain-of-Thought (CoT) traces generated by `Qwen-1.8B-Chat (Int4)` or pre-existing verified traces.
- **Algorithm**: First-Order Logic (FOL) Inductive Logic Programming (ILP) (e.g., using `prolog` libraries or `skope`).
- **Process**:
 1. Parse CoT traces into structured `(state, action, next_state)` tuples.
 2. Apply FOL/ILP to infer *relational* rules of the form `holds(A, B) AND holds(B, C) -> holds(A, C)`.
 3. **Rule Validation**: Cross-check extracted rules against the Ground Truth Oracle.
 - If `Extracted_Rule` == `Oracle_Logic`, mark as "Valid".
 - If `Extracted_Rule` != `Oracle_Logic`, mark as "Extraction Failure" (Excluded from Rule Gap count).
- **Output**: A set of explicit logical rules (Hypothesized Rule Set) with a "Valid" flag.
- **Uncertainty Handling**: Traces with ambiguous or contradictory reasoning are flagged as "Extraction Uncertainty" and excluded from the rule count.

### 3.3 Divergence Quantification (FR-003, FR-004)
- **Input**: Ground Truth Oracle, Validated Rule Set, LLM Traces.
- **Process**:
 1. Execute a standardized set of planning tasks.
 2. For each step, compare:
 - `LLM_Trajectory` vs `Oracle_Trajectory`
 3. **Classification Logic** (Non-Circular):
 - **Match**: LLM == Oracle.
 - **Hallucination**: LLM != Oracle AND Oracle Logic is Inferable (verified by Oracle itself, not the rule).
 - **Rule Gap**: LLM != Oracle AND Oracle Logic is Inferable AND Extracted Rule (Valid) != Oracle.
 - **Extraction Uncertainty**: Ambiguous trace (Excluded from counts).
 - **Coverage Gap (Cold Start)**: Interaction type never seen in training (Excluded from counts).
- **Output**: Divergence Report with counts and rates.

### 3.4 Statistical Analysis (FR-005)
- **Method**: **Paired Permutation Test** (or McNemar's test extension).
- **Procedure**:
 1. Resample *pairs* of interaction classes (e.g., spatial vs. temporal) N=1,000 times (with replacement).
 2. Calculate the *difference* in divergence rates for each resample.
 3. Compute p-value for the difference being non-zero.
- **Threshold**: α ≤ 0.05.
- **Handling Dependencies**: The permutation is performed over *entire trajectories* to preserve Markovian dependencies.

### 3.5 Boundary Condition Analysis (FR-006)
- **Method**: Regression analysis of LLM accuracy vs. Step Count / State Complexity.
- **Output**: Identification of the step count where accuracy drops below a high-performance threshold.

## 4. Compute Feasibility & Rationale

### 4.1 CPU-First Strategy
- **Feasibility**: The proposed workflow (parsing, FOL/ILP induction on traces, paired permutation testing) is computationally light and fully tractable on a 2-core, 7GB RAM CPU runner.
- **Libraries**: `prolog` (CPU), `pandas` (CPU), `datasets` (streaming), `transformers` (CPU-quantized).
- **No GPU Required**: The plan uses a verified CPU-quantized model (Qwen-1.8B-Int4) for trace generation, which fits within the memory constraints.

### 4.2 GPU Escape Hatch (Not Needed)
- This project does not require the GPU escape hatch. The core analysis (Oracle, FOL/ILP, Statistics) remains CPU-based.

## 5. Risks & Mitigations

| Risk | Impact | Mitigation |
|:--- |:--- |:--- |
| **Source Code Unavailable** | Cannot build Oracle (Fatal). | Use the `AgentWorld` repository code directly; if unavailable, fall back to the `AgentWorldBench` specification if it contains formal logic definitions. |
| **Trace Ambiguity** | High "Extraction Uncertainty". | Strictly flag ambiguous traces; report uncertainty rate separately; do not force a rule. |
| **Cold Start Data** | Inability to evaluate certain interaction types. | Report "Coverage Gap (Cold Start)" explicitly; exclude from Hallucination/Rule Gap counts as per spec. |
| **Statistical Power** | Low power for paired test. | Acknowledge limitation if N=500 is insufficient for small effect sizes; report effect size estimates. |
| **FOL/ILP Complexity** | Rule extraction too slow for CPU. | Limit induction depth; use a simplified relational rule miner if full ILP is too slow; report limitation. |
