## Unresolved concerns

- FR-003 mandates reporting findings for *each* threshold configuration in the sensitivity analysis. Tasks T021 and T021b describe generating a single aggregated report file (`output/sensitivity_report.json`) but lack an explicit task to structure or output the per-configuration breakdown required by the spec, risking an untestable implementation of FR-003.
- SC-001 requires 'blinded expert rubric scoring' where stories are stripped of metadata. Task T032 implements the scoring logic but there is no corresponding task to implement the *data preparation* step that strips metadata from generated stories before scoring, leaving the 'blinding' requirement unimplemented.
- SC-004 requires measuring 'Verification Traceability' as a percentage of claims with valid citations. Tasks T033 and T034 run the pipeline and aggregate metrics, but no task explicitly calculates and records this specific percentage metric (SC-004) in the final output, creating a gap between execution and the measurement of the success criterion.
- FR-006 requires the system to detect and report if sample size is < 30. Task T023 (Phase 4) handles this for the Counterfactual Inspector, but there is no explicit task in Phase 3 (Baseline Generation) to perform this check for the *baseline* narrative, risking a crash or invalid baseline generation on small datasets before the Inspector runs.
- SC-002 defines 'Confirmation Bias' as the proportion of counterfactual claims distinct from the baseline (variable pairs not in top 3 correlations). Task T031 implements the calculation but does not explicitly state the logic must *exclude* variable pairs already in the top 3 correlations, risking an incorrect metric calculation.
- T012-T016 (US1 Baseline) depend on T005 (loader) and T006 (processor) which are in Phase 2. However, T013 (narrative generation) implies using an LLM, but the LLM dependencies and configuration (T002, T007) are not explicitly ordered as prerequisites for T013. More critically, T015 (JSON output schema) is listed as a separate task after T013, but the schema definition is a prerequisite for T013 to generate valid output. T015 should precede T013.
- T020 (inspector.py) and T021 (sensitivity analysis) depend on the baseline narrative output (T013/T015) to analyze. The tasks are correctly placed in Phase 4, but T022 (counterfactual query generation) depends on T020/T021 results. T022 is listed after T021, which is correct, but T023 (sample size check) is listed after T022. T023 should logically precede T022 because if n < 30, the query generation (T022) should be skipped or flagged immediately, not generated and then flagged.
- T028 (synthesizer) depends on the outputs of T013 (baseline) and T020/T021/T022 (inspector). The ordering is correct. However, T029 (citation generation) and T029c (sanitization) are listed as separate tasks after T028. T029c (sanitization of causal language) is a prerequisite for the final output of T028. If T028 generates the text, T029c must run on that text. The current order implies T028 generates the final text, then T029c sanitizes it. This is a valid flow, but T029 (citation) and T029c (sanitization) should be grouped as sub-tasks of T028 or T028 should be split into 'generate raw' and 'finalize with citations/sanitization' to ensure the final output is correct. As written, T028 might produce an unsanitized story if T029c is run separately and not integrated.
- T034 (report generation) depends on T031 (bias metrics) and T032 (rubric scoring). T031 and T032 are in Phase 5. T033 (run full pipeline) is in Phase 6. T033 must complete before T034 can aggregate metrics. The order is correct. However, T035 (Verification Traceability) depends on the output of T034 (aggregated metrics) or the individual story outputs. T035 is listed after T034, which is correct. T036 (feasibility check) depends on the execution of T033. The order is correct.
- T010 and T011 are marked [P] (parallel-safe) and placed in Phase 3. However, they are tests for US1. The spec says 'Write these tests FIRST, ensure they FAIL before implementation'. T010/T011 should be ordered before T012-T016. The current order places them before T012, which is correct. But T010/T011 depend on the existence of the code structure (T001a-T001c) and dependencies (T002). This is satisfied. The [P] tag is correct as they are independent test files.

- Task T021b specifies writing to `output/sensitivity_report.json` but fails to define the required schema or structure for the 'per-configuration findings' mandated by FR-003. An implementer cannot deterministically generate the file without knowing the exact keys (e.g., threshold pairs, p-values, correlation coefficients) required for each entry.
- Task T035 requires calculating 'Verification Traceability' (SC-004) but does not specify the input artifact containing the list of counterfactual claims to be audited. The task assumes the existence of a specific JSON structure from the synthesis phase that is not explicitly named or linked, making the calculation logic ambiguous.
- Task T032 ('Implement blinded rubric scoring logic') is not self-contained. It references 'paired t-tests' but omits the critical prerequisite task of implementing the 'metadata stripping' step required by Constitution Principle VII and Spec SC-001. Without a task to sanitize the input stories (removing 'Baseline' vs 'Inspector' tags), the 'blinded' requirement cannot be executed.
- Task T012 ('Implement baseline.py') lacks a concrete deliverable definition for the 'strongest statistically significant relationship'. It does not specify the output schema (e.g., JSON keys for r-value, p-value, variable names) required by the 'Independent Test' in Phase 3, leaving the implementer to guess the data structure.
- Task T021 describes a 'sweeping' of thresholds but does not explicitly mandate the creation of a data structure to hold the intermediate results for *each* configuration before aggregation. This risks the implementer overwriting data, failing the FR-003 requirement to 'report findings for each threshold configuration'.
- Task T021b specifies writing findings to a single aggregated file `output/sensitivity_report.json`. This semantically weakens FR-003 which mandates reporting findings for *each* threshold configuration. A single aggregated file risks losing the per-configuration breakdown required to verify the sensitivity analysis, effectively relaxing the requirement to a summary.
- Task T032 ('Implement blinded rubric scoring logic') only addresses the scoring algorithm. It omits the mandatory data preparation step to strip metadata (source labels) from stories before scoring. Without a specific task to implement the blinding/preparation step, the Constitution Principle VII (Expert Blinding) and SC-001 cannot be satisfied, as the scoring logic alone cannot blind data it receives unblinded.
- Task T034 ('Implement report generation to aggregate metrics') lists SC-001, SC-002, SC-003, SC-004 but does not explicitly include a task to calculate and record the 'Verification Traceability' percentage (SC-004). SC-004 requires a specific metric calculation ('percentage of counterfactual claims with valid citations'). T035 calculates 'Verification Traceability' but T034's description is vague and risks omitting the specific SC-004 calculation logic in the aggregation phase.
- FR-006 requires the system to detect and report if sample size < 30. Task T023 (Phase 4) handles this for the Counterfactual Inspector, but Phase 3 (Baseline Generation) lacks an explicit task to perform this check. If the baseline pipeline runs on a small dataset (<30 rows) without this check, it violates FR-006's requirement to 'detect and report' before proceeding, potentially generating invalid baseline narratives.
- Task T031 ('Add evaluation metrics calculation... to measure Confirmation Bias') does not explicitly state the logic must exclude variable pairs already in the top 3 correlations. SC-002 defines Confirmation Bias specifically as the proportion of claims *distinct* from the baseline (not in top 3). Without this explicit constraint in the task, the implementation may calculate the metric on the wrong set of claims, violating the success criterion definition.
