## Unresolved concerns

- The plan proposes a paired design (LLM vs. Human) but relies on MBPP, which does not natively provide human reference solutions for the specific test inputs in the same format as the LLM generation. The 'research.md' states 'Human reference solutions are implicit,' but without a verified, executable human baseline for the *exact* test cases used, the 'paired' statistical test (Wilcoxon) is invalid. You cannot pair an LLM output against a non-existent or mismatched human baseline. This breaks the core causal/associational comparison.
- The plan assumes N=30 paired problems can be processed within 6 hours on a 2-core CPU runner. Generating 30 solutions with Phi-3-mini (3.8B) on CPU takes ~30-60s *per problem* (180s timeout budget). 30 * 60s = 1800s (30 mins) just for generation. Profiling 30 pairs (60 runs) with 60s timeout each adds another 3600s (1 hour). Total ~1.5 hours, but this leaves zero buffer for model loading, data I/O, and statistical overhead. If the model takes >60s (common on 2-core free runners), the 6-hour budget is exceeded, risking incomplete data collection and invalid power analysis.
- The plan defines 'Total Resource Cost' as `Memory * Time + Failure_Penalty`. However, the failure penalty is defined as `7GB * 60s` (a massive constant). If even one LLM solution fails (syntax/timeout) while the human succeeds, the LLM's cost becomes dominated by the penalty term, rendering the `Memory * Time` component of the successful runs statistically irrelevant. This creates a floor effect where the test essentially measures 'failure rate' rather than 'memory efficiency,' conflating two distinct constructs and potentially inflating effect sizes artificially.
- The plan asserts that HumanEval has 'NO verified source' and forces a selection of MBPP. This is factually incorrect; HumanEval (ds000001 on HuggingFace) is a standard, publicly accessible benchmark. The plan's logic that 'Using HumanEval would violate the Verified Accuracy principle' is a false constraint based on a hallucinated lack of data availability. This forces an arbitrary dataset choice (MBPP) that may not be optimal for the specific memory profiling question (e.g., if HumanEval's test harness is more deterministic for memory isolation). The plan must acknowledge both datasets as valid options and select based on technical fit (e.g., test harness stability) rather than a fabricated citation gap.
- The plan cites a specific URL for MBPP: 'https://huggingface.co/datasets/google-research-datasets/mbpp/resolve/main/full/prompt-00000-of-00001.parquet'. While the dataset exists, the plan assumes this specific file structure (single parquet file with 'prompt' and 'test_list') is sufficient for the 'paired' analysis described. MBPP typically requires the 'test' split for execution, and the 'full' split often contains only prompts. The plan fails to verify if the cited source actually contains the *human reference solutions* (test inputs/outputs) required to run the 'Human' side of the comparison. If the cited URL only provides prompts, the plan cannot execute the 'Human' profiling phase, rendering the methodology infeasible as written.
- Constitution Principle II (Verified Accuracy) is marked 'Pass' in the plan, but the research.md explicitly states that HumanEval has 'NO VERIFIED SOURCE' and the plan must select MBPP. The plan's Constitution Check table does not reflect this critical constraint or the specific dataset selection logic required to satisfy Principle II. This creates a contradiction where the plan claims compliance while the research phase identifies a blocking violation for one of the two options.
- The 'Technical Context' section lists 'HumanEval OR MBPP' as the dataset choice, but the 'research.md' and 'spec.md' (A-012, FR-001) mandate that exactly one dataset be selected based on verified sources, effectively blocking HumanEval. The plan should explicitly state 'MBPP (selected due to verified source availability)' to align with the research findings and avoid ambiguity in the implementation path.
- The 'Project Structure' lists `data/dataset_manifest.yaml` in the root `data/` directory, but the `data-model.md` (Section 1) and `research.md` specify the path as `data/dataset_manifest.yaml` (consistent) but the `quickstart.md` implies a `data/raw/` structure. While paths match, the plan should explicitly reference the `data-model.md` schema for `dataset_manifest.yaml` to ensure the 'Single Source of Truth' (Constitution Principle IV) is maintained across documents.
- The plan defines 'Residual Memory' = Memory / LOC (US-3) and uses it as the dependent variable in regression (FR-007, US-3). However, 'Memory' is the primary outcome being measured. Using a derived metric that is a direct monotone function of the outcome (divided by a predictor) as the dependent variable creates a mathematical coupling (spurious correlation). If Memory is the true signal, regressing (Memory/LOC) against LOC (or features correlated with LOC) will mechanically produce a negative correlation regardless of the underlying physics, invalidating the 'mechanistic insight' claim.
- The 'Total Resource Cost' metric is defined as `Memory_Bytes * Time_Seconds + (Failure_Penalty * 7GB * 60s)`. The failure penalty term is a constant scalar (420 GB*s) applied to failures. This creates a bimodal distribution where the 'failure' group is separated from the 'success' group by a massive, fixed offset. A statistical test (Wilcoxon/Bootstrap) on this composite score will almost certainly detect a difference driven entirely by the failure rate (FR-012) rather than the continuous memory/time efficiency of successful runs. The plan conflates 'reliability' (failure rate) with 'efficiency' (memory/time) in a way that makes the primary test result trivial if failure rates differ.
- The plan proposes a Shapiro-Wilk test on the difference distribution of 'Total Resource Cost' to choose between Wilcoxon and Bootstrap. Given the massive constant penalty added to failures (see FR-011), the difference distribution will be heavily skewed and non-normal by construction. The symmetry check is therefore a foregone conclusion, making the conditional logic (if symmetric -> Wilcoxon) effectively dead code. The analysis should directly use a method robust to the expected bimodal/skewed nature of the data (e.g., permutation test on the raw difference or a two-part model) rather than a conditional switch based on a test that will always fail.
