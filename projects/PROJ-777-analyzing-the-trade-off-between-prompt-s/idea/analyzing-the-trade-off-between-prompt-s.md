# Analyzing the Trade‑off Between Prompt Size and Code Generation Quality

**Field**: computer science

## Research question

How does the length of a natural‑language prompt influence the functional correctness of code generated by open-source code-generation language models, and where do diminishing returns begin?

## Motivation

Prompt engineering is critical for balancing developer productivity with inference costs in large language models (LLMs). While adding context is often assumed to improve performance, excessive verbosity may consume token budgets or introduce noise that degrades generation quality. Clarifying this non-linear trade-off is essential for establishing evidence-based guidelines on prompt design that maximize code correctness per token.

## Literature gap analysis

### What we searched
We queried Semantic Scholar and arXiv for terms including "prompt length impact on code synthesis quality," "effect of token count in prompts on program generation performance," and "prompt verbosity versus generated code correctness." While numerous papers discuss prompt engineering strategies (e.g., Chain-of-Thought) or specific benchmarks, few directly quantify the functional relationship between raw prompt token count and pass@k scores across controlled prompt variations.

### What is known
- [JaCoText: A Pretrained Model for Java Code-Text Generation (2023)](https://arxiv.org/abs/2303.12869) — Demonstrates that transformer-based models achieve high performance in code-text generation, establishing the baseline capability of models to map natural language to code, but does not isolate prompt length as a variable.
- [Context Engineering for Multi-Agent LLM Code Assistants Using Elicit, NotebookLM, ChatGPT, and Claude Code (2025)](https://arxiv.org/abs/2508.08322) — Highlights that LLMs struggle with complex projects due to context limitations, implying that context management is a bottleneck, yet it does not provide a systematic empirical curve of performance versus raw prompt token count for single-task code generation.

### What is NOT known
There is no published work that empirically maps the specific curve of "prompt token count" vs. "pass@1 correctness" while holding prompt *content* (task semantics) constant and varying only the level of detail or verbosity. Existing literature assumes "more context is better" or focuses on specific prompting styles, leaving a gap in understanding the point of diminishing returns where added tokens become noise or budget waste.

### Why this gap matters
Developers and system architects need to know the optimal prompt density to minimize latency and token costs without sacrificing code reliability. Without this data, prompt engineering remains heuristic, potentially leading to inefficient inference pipelines or suboptimal model performance.

### How this project addresses the gap
This project will generate a controlled dataset of prompt variants for standard coding tasks, systematically varying token counts while preserving semantic intent, and measure the resulting pass@k scores to explicitly plot and model the trade-off curve.

## Expected results

We anticipate a non-linear relationship where pass@k scores initially rise with prompt length as essential specifications are added, followed by a plateau or decline as verbosity introduces noise. A statistically significant peak in the performance curve will confirm the existence of an optimal prompt size, whereas a flat or monotonically increasing curve would challenge the assumption of diminishing returns in current open-source models.

## Methodology sketch

- **Dataset selection**: Download the HumanEval benchmark (164 Python programming problems) from the official HuggingFace repository (`openai/human-eval`).
- **Prompt construction**: For each of the 164 problems, generate a series of prompt variants programmatically. Starting from a *Minimal* prompt (signature + docstring), iteratively append semantically neutral elaborations (e.g., restating constraints in different syntactic forms, adding redundant but true examples) to create a gradient of token counts (e.g., 50, 100, 200, 400 tokens) while ensuring the core task definition remains identical.
- **Tokenization**: Use the specific tokenizer of the target model (`Salesforce/codegen-350M-multi`) to record the exact token count for each generated variant.
- **Model inference**: Run inference on a GitHub Actions runner using `Salesforce/codegen-350M-multi` (CPU-compatible, <7GB RAM). For each (problem, variant) pair, generate a single code completion with `max_new_tokens=256` and a fixed random seed for reproducibility. **All results are derived from live model execution; no pre-generated, dummy, or simulated metrics are used.**
- **Correctness evaluation**: Execute the generated code against the original HumanEval unit tests using the standard `evaluate_functional_correctness` script. Calculate the pass@1 score (binary: 1 if all tests pass, 0 otherwise) for each prompt variant. This evaluation target (unit test pass rate) is independent of the prompt inputs.
- **Statistical analysis**:
  - Aggregate results to compute the mean pass@1 rate for each distinct token-count bin.
  - Perform **Cochran's Q test** to determine if there are statistically significant differences in the binary pass rates across the multiple prompt-length conditions (treating each problem as a block).
  - Fit a **Generalized Linear Mixed Model (GLMM)** with a logit link function: `Pass ~ TokenCount + (1 | ProblemID)`. This accounts for the binary nature of the outcome and the repeated measures (clustering) of multiple prompt variants within the same problem, allowing for a robust estimation of the prompt-length effect while controlling for problem difficulty.
- **Visualization**: Plot pass@1 rate (y-axis) against prompt token count (x-axis) with confidence intervals derived from the GLMM to visualize the trade-off curve.
- **Resource constraint check**: Ensure the total inference time for 164 problems × ~5 variants fits within the 6-hour GHA limit; if necessary, subsample to 100 problems or reduce the number of token bins.
