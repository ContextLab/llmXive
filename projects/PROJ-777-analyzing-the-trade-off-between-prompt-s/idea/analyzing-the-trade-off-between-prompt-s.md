# Analyzing the Trade‑off Between Prompt Size and Code Generation Quality

**Field**: computer science

## Research question

How does the length of a natural‑language prompt influence the correctness of code generated by modern code‑generation language models?

## Motivation

Prompt engineering is a key lever for developer productivity when using large language models for code synthesis. Model token limits and inference latency impose practical constraints, yet it is unclear whether providing more context (longer prompts) systematically improves generated code or introduces diminishing returns. Clarifying this trade‑off will guide developers in designing prompts that balance accuracy and efficiency.

## Related work

- [A Validated Prompt Bank for Malicious Code Generation: Separating Executable Weapons from Security Knowledge in 1,554 Consensus-Labeled Prompts (2026)](https://arxiv.org/abs/2605.03179) — Provides a large, curated set of code‑related prompts and shows how prompt wording can affect model behavior, establishing precedent for systematic prompt‑effect studies.  
- [Qiskit HumanEval: An Evaluation Benchmark For Quantum Code Generative Models (2024)](https://arxiv.org/abs/2406.14712) — Extends the HumanEval benchmark to quantum programming, demonstrating the use of HumanEval‑style tasks for measuring code‑generation quality across model families.  
- [TDD Governance for Multi‑Agent Code Generation via Prompt Engineering (2026)](https://arxiv.org/abs/2604.26615) — Investigates prompt‑engineering strategies to improve test‑driven development with code‑generation models, highlighting the impact of prompt content on downstream correctness.

## Expected results

We anticipate a non‑linear relationship: modest increases in prompt length (adding essential specifications) will raise pass@k scores, but overly verbose prompts will consume token budget and potentially degrade performance. Confirmation will come from statistically significant differences in pass@k and latency across the three prompt‑size conditions; a null result (no systematic effect) would also be informative for prompt‑design guidelines.

## Methodology sketch

- **Select benchmark**: Sample 200 problems from the HumanEval suite (publicly available on GitHub).  
- **Construct prompt variants** for each problem:  
  1. *Minimal*: one‑sentence task description.  
  2. *Detailed*: full docstring with function signature and description.  
  3. *Extended*: add explicit input‑output examples and additional specification comments.  
- **Measure prompt length** (token count) for each variant using the model’s tokenizer.  
- **Load models** on the GitHub Actions runner: `Salesforce/codegen-350M-multi` and `bigcode/starcoderbase` from HuggingFace.  
- **Generate code**: For each (model, problem, prompt‑size) triple, request a single completion (`max_new_tokens=256`). Record generation latency and total tokens used.  
- **Evaluate correctness**: Run the generated code against the HumanEval unit tests, compute pass@1 and pass@10 (using the standard estimator).  
- **Statistical analysis**:  
  - Perform repeated‑measures ANOVA to test the effect of prompt length on pass@k within each model.  
  - Fit linear and quadratic regression models of token count vs. pass@k to assess non‑linear trends.  
  - Use paired t‑tests to compare latency and token efficiency across prompt sizes.  
- **Report**: Plot pass@k and latency as functions of prompt length, tabulate optimal prompt size per model, and discuss trade‑offs.
