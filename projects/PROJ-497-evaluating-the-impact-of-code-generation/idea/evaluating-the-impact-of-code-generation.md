# Evaluating the Impact of Code Generation Models on Code Vulnerability Density

**Field**: computer science

## Research question

Do code generated by large language models exhibit higher vulnerability density than human-written code for equivalent programming tasks, and how does this difference vary across model architectures and training data sources?

## Motivation

As LLMs become integrated into software development workflows, understanding their security implications is critical. While code generation capability has been extensively benchmarked, the systematic comparison of vulnerability rates between model-generated and human-authored code remains underexplored. This work addresses that gap to inform security practices for AI-assisted development.

## Related work

- [LLaVul: A Multimodal LLM for Interpretable Vulnerability Reasoning about Source Code (2025)](http://arxiv.org/abs/2509.17337v1) — Provides automated vulnerability detection capabilities that can be applied to generated code samples.
- [ReCode: Robustness Evaluation of Code Generation Models (2022)](http://arxiv.org/abs/2212.10264v1) — Establishes methodology for evaluating code generation model outputs across varying prompt conditions.
- [Multi-lingual Evaluation of Code Generation Models (2022)](http://arxiv.org/abs/2210.14868v3) — Demonstrates benchmark construction for comparing code generation performance across models and languages.
- [SIMCOPILOT: Evaluating Large Language Models for Copilot-Style Code Generation (2025)](http://arxiv.org/abs/2505.21514v1) — Introduces benchmarking framework for interactive code generation scenarios relevant to developer workflows.

## Expected results

We expect to find statistically significant differences in vulnerability density between LLM-generated and human-written code, with model-specific patterns emerging. The measurement will be confirmed by static analysis tool outputs showing higher vulnerability counts per 1000 lines of code for generated samples, with effect sizes quantified through confidence intervals.

## Methodology sketch

- Download open-source code generation models (StarCoder, CodeGen) from HuggingFace Model Hub
- Generate code samples for 50 programming tasks from HumanEval/MBXP benchmark datasets
- Collect corresponding human-written reference solutions from GitHub repositories linked to benchmark datasets
- Run static analysis tools (Bandit for Python, Semgrep for multi-language) on all code samples
- Parse tool outputs to extract vulnerability types and counts per file
- Calculate vulnerability density as vulnerabilities per 1000 lines of code
- Perform two-sample t-tests comparing generated vs. human code vulnerability densities
- Stratify analysis by vulnerability category (SQL injection, XSS, buffer overflow, etc.)
- Generate summary statistics and visualization of vulnerability distributions
- Document reproducibility with versioned tool outputs and random seeds

## Literature gap analysis

### What we searched

Queried Semantic Scholar and arXiv for terms including "code generation vulnerability," "LLM security," "code vulnerability density," and "AI code security evaluation" across 2022-2025 publications. Retrieved 8 papers from the provided literature block.

### What is known

- [LLaVul: A Multimodal LLM for Interpretable Vulnerability Reasoning about Source Code (2025)](http://arxiv.org/abs/2509.17337v1) — Establishes automated vulnerability detection methods applicable to source code analysis.
- [ReCode: Robustness Evaluation of Code Generation Models (2022)](http://arxiv.org/abs/2212.10264v1) — Provides robustness evaluation framework for code generation model outputs.
- [Multi-lingual Evaluation of Code Generation Models (2022)](http://arxiv.org/abs/2210.14868v3) — Demonstrates benchmark construction for cross-model code generation comparison.

### What is NOT known

No published work has systematically measured and compared vulnerability density between LLM-generated code and equivalent human-written code using static analysis tools. Existing studies focus on generation quality, robustness, or vulnerability detection capabilities rather than comparative security analysis.

### Why this gap matters

Software security teams adopting AI-assisted development need evidence-based guidance on whether model-generated code introduces measurable security risks. Filling this gap enables informed decisions about code review policies, security scanning requirements, and model selection for security-sensitive applications.

### How this project addresses the gap

This project's methodology directly measures vulnerability density in both generated and human code using standardized static analysis tools, then performs statistical comparison to quantify the security differential. The stratified analysis by vulnerability type and model architecture provides actionable specificity beyond existing evaluation frameworks.
