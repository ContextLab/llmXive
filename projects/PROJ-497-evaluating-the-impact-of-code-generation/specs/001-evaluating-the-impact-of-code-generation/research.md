# Research: Evaluating the Impact of Code Generation Models on Code Vulnerability Density

## Problem Statement

Do code samples generated by Large Language Models (LLMs) exhibit higher vulnerability density compared to human-written reference solutions? This research aims to quantify the security risk introduced by LLM-generated code by comparing vulnerability counts across distinct models (StarCoder, CodeGen) and benchmarks (HumanEval, MBPP) using robust statistical methods.

## Dataset Strategy

The study relies on two primary benchmark datasets for task prompts and human reference solutions, and open-source models for generation.

| MBPP | google-research-datasets/mbpp | https://huggingface.co/datasets/google-research-datasets/mbpp/resolve/main/full/prompt-00000-of-00001.parquet | Task prompts and human solutions | Verified via HuggingFace `datasets` loader. |

**Dataset Fit Assessment**:
- **HumanEval**: Contains Python coding tasks with test suites. *Fit*: Perfect for generating and validating Python code samples.
- **MBPP**: Contains Python coding tasks. *Fit*: Perfect for cross-benchmark validation.
- **LLM Models**: StarCoder and CodeGen are selected for their availability on HuggingFace and compatibility with CPU inference. *Fit*: Verified via `transformers` library.

**Note**: The "LLM-generated" datasets listed in the verified block (e.g., `hon9kon9ize/38k-zh-yue-translation-llm-generated`) are translation tasks and **do not contain code**. They are explicitly excluded. The study will **generate** its own code samples using the specified models on the HumanEval and MBPP tasks.

## Methodological Rigor

### Statistical Approach
1.  **Unit of Analysis**: **Task**. To ensure a valid comparison, we aggregate LLM samples per task (mean vulnerability count) to match the single human reference solution per task. This creates a balanced design where N = number of tasks.
2.  **Primary Test**: Zero-Inflated Negative Binomial (ZINB) regression.
    *   **Rationale**: Vulnerability counts are count data with excess zeros and overdispersion. ZINB handles both.
    *   **Model**: `vulnerability_count ~ source_type + lines_of_code + (1|benchmark) + (1|task_id)`.
        *   `source_type`: Fixed effect (LLM vs Human).
        *   `lines_of_code`: Covariate to control for code size (density adjustment).
        *   `task_id`: Random effect to control for task difficulty.
    *   **Fallback**: If ZINB fails to converge after 3 attempts, a permutation test on raw counts will be executed (FR-005, FR-015).
3.  **Multiple Comparison Correction**:
    *   **Method**: Benjamini-Hochberg procedure (False Discovery Rate) applied when testing across vulnerability categories (CWEs) (FR-007).
    *   **Threshold**: Family-wise error rate controlled at α ≤ 0.05 (SC-003).
4.  **Effect Size**:
    *   **Metric**: Incidence Rate Ratio (IRR) with 95% Confidence Interval (SC-002).
5.  **Power Analysis**:
    *   **Condition**: If the number of valid tasks (N) < 64, a post-hoc power analysis will be performed (FR-009).
    *   **Threshold**: Power < 0.80 for medium effect size (d=0.5) will flag the dataset as "under-powered".
    *   **Sample Size Note**: The maximum N is determined by the benchmark size (HumanEval: a benchmark comprising multiple programming tasks, MBPP: [generalized task count]). The power analysis is based on the number of *tasks*, not generated samples.

### Validity & Assumptions
- **Observational Nature**: The study is observational. Findings will be framed as associational differences, not causal claims about model security properties.
- **Measurement Validity**: Bandit is a standard static analysis tool. **We acknowledge that detection rates may vary between LLM and Human code.** A robustness check (manual audit of a subset) is included to validate this assumption.
- **Collinearity**: Predictors (model name, benchmark) are distinct. No definitional collinearity exists between source types.
- **Sample Size**: The target N is the number of tasks. We aim for full coverage of the benchmark (a subset of the total tasks). If valid tasks drop below a sufficient threshold, the power analysis will be reported.
- **FPR Adjustment**: The Reference-Validator Agent (FR-012) is used to estimate False Positive Rates. **This is a sensitivity analysis.** The primary statistical test uses raw counts to avoid circular validation. The FPR is reported as a limitation/contextual metric.

## Computational Feasibility

### Resource Constraints
- **Hardware**: GitHub Actions free-tier (standard CPU allocation, ~7 GB RAM, ~14 GB disk).
- **Memory**: Models (StarCoder-3B, CodeGen-2B) will be loaded in default precision (float32) but with `device_map="cpu"` and `torch_dtype=torch.float32`. We will use `max_memory` limits to prevent OOM. If memory exceeds 7GB, we will switch to a smaller model variant (e.g., CodeGen-350M) or reduce samples per task.
- **Time**: Total runtime ≤ 6 hours.
    *   Generation: ~-45 mins per model/benchmark (A large number of samples across 164 tasks is heavy; we will sample tasks if needed to fit time, or optimize batch size).
    *   Analysis: ~-15 mins (Bandit on ~100 files).
    *   Stats: < 5 mins.
- **Disk**: ~14 GB limit. We will process data in chunks and delete intermediate raw generation files after analysis.

### Risk Mitigation
- **OOM Risk**: If model loading fails, fallback to smaller models or quantized CPU inference (if available without CUDA).
- **Convergence Risk**: ZINB convergence is not guaranteed. The permutation test fallback ensures a result is always produced.
- **Data Scarcity**: If <64 valid tasks are generated, the power analysis will be reported, and the study will be marked as "under-powered" rather than forcing invalid conclusions.

## Decision Log

| Decision | Rationale |
|----------|-----------|
| **Unit of Analysis = Task** | Ensures balanced comparison between Human (1 per task) and LLM (aggregated per task). Solves unbalanced N issue. |
| **ZINB with `lines_of_code` covariate** | Controls for code size differences (density adjustment) while using count as the dependent variable. |
| **ZINB over Poisson** | Vulnerability counts exhibit overdispersion and excess zeros. Poisson assumes mean=variance, which is rarely true for security bugs. |
| **Permutation Fallback** | ZINB is computationally intensive and may fail to converge on small datasets. Permutation tests are robust and require fewer assumptions. |
| **HumanEval + MBPP** | Two distinct benchmarks ensure findings are not specific to one task distribution. |
| **StarCoder + CodeGen** | Two distinct architectures (decoder-only vs. encoder-decoder hybrid) provide architectural diversity. |
| **Bandit only** | Bandit is the standard for Python. Adding Semgrep would double analysis time and complexity without significant added value for this scope. |
| **CPU-only** | GPU is unavailable on the target runner. Models must be run in default precision on CPU. |
| **FPR as Sensitivity Analysis** | Avoids circular validation. The primary test uses raw counts; FPR is reported as a limitation. |
