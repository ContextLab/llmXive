# Implementation Plan: Evaluating the Impact of Code Generation Models on Code Vulnerability Density

**Branch**: `001-eval-code-vuln-density` | **Date**: 2026-06-27 | **Spec**: `specs/001-eval-code-vuln-density/spec.md`

## Summary

This feature implements a reproducible pipeline to evaluate the vulnerability density of code generated by Large Language Models (LLMs) compared to human-written reference solutions. The system will download open-source models (StarCoder, CodeGen), generate code samples for tasks from HumanEval and MBPP benchmarks, run static analysis (Bandit) to count vulnerabilities, and perform robust statistical comparisons (ZINB regression with fallback to permutation tests) while adhering to strict resource constraints (CPU-only, ≤7GB RAM) and project constitution principles.

**Critical Design Update**: The unit of analysis is **Task**, not Sample. To ensure a valid comparison, LLM samples generated for a single task are aggregated (mean vulnerability count) to match the single human reference solution per task. This resolves the unbalanced sample size issue and allows for a valid random effect for `task_id`.

## Implementation Phases

### Phase 1: Data Acquisition & Generation
1.  **Download Benchmarks**: Fetch HumanEval and MBPP datasets from verified HuggingFace URLs.
2.  **Generate LLM Code**: For each task in both benchmarks, generate up to 64 samples per model (StarCoder, CodeGen).
    *   *Constraint*: Only samples passing the benchmark's test suite are retained (FR-002).
    *   *Output*: `data/generated/{model}/{benchmark}/{task_id}/samples/`
3.  **Prepare Human Data**: Extract human reference solutions from the benchmark datasets.
    *   *Output*: `data/human/{benchmark}/{task_id}/reference.py`

### Phase 2: Static Analysis & Aggregation
1.  **Run Static Analysis**: Execute Bandit (pinned version) on all valid LLM samples and human references.
    *   *Config*: Load rules from `code/config/bandit_config.yaml` to ensure consistency (Principle VI).
    *   *Error Handling*: Skip files with syntax errors; log and exclude.
2.  **Aggregate Data**: For each task, calculate the **mean vulnerability count** across all valid LLM samples.
    *   *Unit of Analysis*: **Task**. The dataset will have one row per task (Human) and one row per task (LLM Aggregated).
    *   *Output*: `data/processed/aggregated_analysis_dataset.csv` (Columns: `task_id`, `source_type`, `benchmark`, `lines_of_code`, `vulnerability_count`, `is_valid`).

### Phase 3: Statistical Analysis
1.  **Primary Test (ZINB)**: Run Zero-Inflated Negative Binomial regression.
    *   *Model*: `vulnerability_count ~ source_type + lines_of_code + (1|benchmark) + (1|task_id)`.
    *   *Rationale*: `lines_of_code` controls for size (density adjustment); `task_id` random effect controls for task difficulty.
    *   *Fallback*: If ZINB fails to converge after attempts, execute a permutation test on raw counts (FR-005, FR-015).
2.  **FPR Sensitivity Analysis (FR-012)**:
    *   Execute the Reference-Validator Agent on a stratified random sample (n=20 per group).
    *   Calculate group-specific False Positive Rates (FPR).
    *   *Note*: This FPR is reported as a sensitivity metric. **The primary statistical test uses raw counts.** The adjustment formula is not applied to the primary outcome to avoid circular validation.
3.  **Stratified Analysis (FR-006)**:
    *   Group data by CWE category.
    *   **Skip Condition**: If `n < 5` per group for a category, log a warning and skip the test for that category.
    *   **Correction**: Apply Benjamini-Hochberg correction to p-values for multiple comparisons (FR-007).
4.  **Power Analysis & Flagging (FR-009)**:
    *   If the number of valid tasks (N) is < 64, perform a post-hoc power analysis.
    *   **Flagging**: If achieved power < 0.80, set the `under_powered` flag in the final output.

1.  **Cross-Benchmark Comparison (FR-011)**: Explicitly compare results between HumanEval and MBPP datasets.
2.  **Cross-Model Comparison (FR-013)**: Explicitly compare results between StarCoder and CodeGen models.
