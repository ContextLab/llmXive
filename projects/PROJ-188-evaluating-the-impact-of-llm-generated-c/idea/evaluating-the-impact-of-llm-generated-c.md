# Evaluating the Impact of LLM-Generated Code Explanations on Comprehension

**Field**: computer science

## Research question

Do natural‑language explanations of code generated by large language models (LLMs) improve developers’ comprehension speed and accuracy compared with reading the code alone or with existing documentation?

## Motivation

Understanding unfamiliar code is a major bottleneck in software maintenance and learning. While LLMs can produce human‑readable explanations, it is unclear whether these explanations genuinely aid comprehension or inadvertently introduce misconceptions. Answering this question will inform the design of developer tools that integrate LLM‑generated documentation.

## Related work

- [Evaluating Code Generation of LLMs in Advanced Computer Science Problems (2025)](http://arxiv.org/abs/2504.14964v1) — surveys how students use LLMs for programming tasks and notes mixed effects on learning outcomes.  
- [Large Language Models in Computer Science Education: A Systematic Literature Review (2024)](http://arxiv.org/abs/2410.16349v1) — synthesizes prior work on LLMs in CS education, highlighting a gap in rigorous user‑study evidence for code‑explanation efficacy.

## Expected results

We anticipate that participants who view LLM‑generated explanations will answer comprehension questions faster and with higher accuracy than those who view code alone, especially for high‑complexity snippets. A statistically significant interaction between explanation condition and code complexity (p < 0.05) would confirm the hypothesis; a null effect would falsify it.

## Methodology sketch

- **Dataset selection** – Download 30 Python functions of varying cyclomatic complexity from the CodeSearchNet corpus (https://github.com/github/CodeSearchNet).  
- **Explanation generation** – Use the open‑source StarCoder model (bigcode/starcoder, 15 B parameters) via the HuggingFace `transformers` library on the GitHub Actions CPU runner to generate a single natural‑language explanation per function (limit to 200 tokens).  
- **Question design** – For each function, author 3 multiple‑choice comprehension questions (one factual, one inference, one “why” reasoning) and record the correct answer.  
- **Participant recruitment** – Post the study on Prolific (or MTurk) with a compensation plan; each participant is randomly assigned to one of three conditions: (A) code only, (B) code + LLM explanation, (C) code + official docstring (if available).  
- **Data collection** – Record per‑question response accuracy and time‑to‑submit (captured via JavaScript on the survey platform).  
- **Pre‑processing** – Clean logs, remove participants who finish < 30 seconds or have > 80 % missing data.  
- **Statistical analysis** – Fit a mixed‑effects ANOVA with fixed effects *condition* (3 levels) and *complexity* (low, medium, high) and random intercepts for participants and code snippets; follow up with post‑hoc Tukey tests.  
- **Robustness checks** – Repeat the analysis using only the subset of explanations that achieve ≥ 80 % BLEU similarity to the reference docstrings, to assess the impact of explanation quality.  
- **Reproducibility** – All scripts, generated explanations, and analysis notebooks will be version‑controlled and executed on the same GitHub Actions runner; raw data (anonymized) will be deposited in Zenodo with a DOI.
