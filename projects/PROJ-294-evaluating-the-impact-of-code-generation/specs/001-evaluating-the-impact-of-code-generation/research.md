# Research: Evaluating the Impact of Code Generation Models on Code Testability

## Overview

This research investigates structural and functional testability of code generated by the `Salesforce/codegen-350M-mono` model compared to human‑written solutions from the HumanEval benchmark. Structural metrics (cyclomatic complexity, Halstead suite) are complemented by a **dynamic** testability metric—`dynamic_branch_coverage`—measured by executing the code with `coverage.py` in a sandbox. Functional correctness is captured by a binary `pass_rate`. A **paired** non‑parametric design controls for task difficulty, and a suite of statistical analyses quantifies differences.

## Methodology

### 1. Data Acquisition
- Download HumanEval from the verified URL, pin the commit hash, and compute SHA256 stored in `data/metadata.yaml`.
- Randomly sample 50 tasks (seed = 42) and save as `data/raw/subset.parquet`.

### 2. Citation Validation
- Run `validate_citations.py`, which invokes the Reference‑Validator Agent on all external URLs before any processing. Failure aborts the pipeline.

### 3. Code Generation
- Generate **one** solution per task with `codegen-350M-mono` on CPU.  
- Up to three retries on timeout; syntax errors are logged and the sample is excluded.

### 4. Static & Dynamic Analysis
- **Static**: `radon` computes cyclomatic complexity (`cc`) and full Halstead metrics.  
- **Dynamic**: Execute each snippet (human and LLM) against its test harness inside a sandbox while `coverage.py` records **dynamic_branch_coverage** (percentage of branches actually executed).  
- Record `pass_rate` (1.0 if all tests pass, else 0.0).  
- Failures (syntax, parsing, execution) are logged in `results/logs/errors.log` and excluded from the paired analysis.

### 5. Paired Filtering
- Retain only tasks where **both** human and LLM samples succeeded (`generation_status == "success"` and executed without crash).  
- Minimum paired count: n ≥ 40 (SC‑001). If fewer, the run aborts with a clear log entry.

### 6. Statistical Analysis
- **Continuous metrics** (`cyclomatic_complexity`, `halstead_volume`, `dynamic_branch_coverage`): Wilcoxon Signed‑Rank Test (paired, non‑parametric).  
- **Binary metric** (`pass_rate`): Fisher's Exact Test on the 2×2 contingency of pass/fail vs. source type.  
- **Significance**: α = 0.05 (no correction needed as only three pre‑registered metrics).  
- **Effect Size**: Report `r` for Wilcoxon; odds ratio for Fisher's.  
- **Power Analysis**: Post‑hoc calculation using `statsmodels`. With n≈50 the study is powered to detect **large** effects (Cohen's d ≈ 0.8). If achieved power < 0.8 for moderate effects, results are labeled "inconclusive".  
- **Decoupling Analysis**: Partial correlation of `pass_rate` with `source_type` controlling for `cyclomatic_complexity`; logistic regression `pass_rate ~ source_type + cyclomatic_complexity` to assess independent predictive value and address confounding between structural metrics and functional correctness.

### 7. Versioning & Artifact Hashing
- After pipeline completion, `update_state.py` records SHA256 hashes of all generated artifacts in `state/projects/PROJ-294-evaluating-the-impact-of-code-generation.yaml` with a timestamp, satisfying Constitution Principle V.

### 8. Visualization & Reporting
- `visualize.py` creates histograms and box‑plots (PNG) for each continuous metric.  
- `report.py` compiles `results/results_report.md` containing figures, statistical tables, power analysis summary, decoupling results, and a **limitations** section (including the infeasibility of large‑model sensitivity analysis).

## Decision Log

| Decision | Rationale |
|----------|-----------|
| Use `codegen-350M-mono` | Meets CPU/RAM constraints; specified in FR‑002. |
| Paired Wilcoxon + Fisher's Exact | Controls for task difficulty; appropriate for continuous vs. binary data. |
| Exclude CodeLlama‑7B execution | Prevents CI failure; large model exceeds free‑tier RAM. |
| Dynamic `dynamic_branch_coverage` | Provides a true execution‑based testability metric. |
| Post‑hoc power with explicit "inconclusive" flag | Aligns with SC‑006 while acknowledging realistic power limits for moderate effects. |
| Literature‑based sensitivity discussion | Addresses FR‑009/SC‑008 requirement qualitatively when execution is infeasible. |
| Decoupling Analysis | Explicitly tests independence of structural metrics from functional success to avoid confounding. |

## Limitations

- **Model Size**: Results pertain to a model with a moderate parameter count.; findings may not generalize to larger LLMs. |
- **Power**: With a sample size that is potentially insufficient, the study is underpowered for moderate effects.; any non‑significant result will be interpreted cautiously. |
- **Dynamic Coverage**: Requires successful execution; samples that crash are excluded from coverage analysis. |
- **Sensitivity Analysis**: No empirical large‑model runs; a qualitative discussion is provided instead.
