# Feature Specification: OpenThoughts-Agent Data Pipeline Validation & Reproduction

**Feature Branch**: `[780-openthoughts-agent-validation]`  
**Created**: 2024-06-20  
**Status**: Draft  
**Input**: User description: "Reproduce & validate: OpenThoughts-Agent: Data Recipes for Agentic Models"

## User Scenarios & Testing

### User Story 1 - End-to-End Data Pipeline Execution (Priority: P1)

A researcher needs to execute the core data generation and curation pipeline of the OpenThoughts-Agent project to verify that the vendored codebase functions correctly on a standard CI environment without requiring external GPU clusters.

**Why this priority**: This is the foundational step. If the pipeline cannot generate the dataset artifacts (the "recipes") on available CPU resources, no subsequent validation of the paper's claims or model training is possible. It validates the "reproduce" aspect of the project scope.

**Independent Test**: Can be fully tested by running the `data/generate_and_run_sbatch.py` or equivalent entry point with a `--dry-run` or `--local` flag, or by executing a small-scale subset generation script, and confirming the creation of valid JSON/Parquet data files in the `data/` directory.

**Acceptance Scenarios**:

1. **Given** the `OpenThoughts-Agent` submodule is cloned and dependencies are installed, **When** the user executes the primary data generation script with a small task subset (e.g., 10 tasks), **Then** the system must generate at least 5 valid task trajectory files in the `data/generated/` directory without crashing.
2. **Given** a valid environment variable configuration for API keys (or mock keys), **When** the pipeline attempts to fetch task definitions, **Then** it must successfully parse the task structure and output a summary log indicating the number of tasks processed.

---

### User Story 2 - Artifact Verification & Integrity Check (Priority: P2)

A QA engineer needs to verify that the generated artifacts (datasets, logs, configuration files) match the structural expectations defined in the paper's methodology and the codebase's internal schema.

**Why this priority**: Generating files is insufficient; they must be valid. This ensures the data is usable for the downstream (simulated) training phase and prevents "garbage in, garbage out" scenarios in later validation steps.

**Independent Test**: Can be tested by running the `data/bespoke/run_verification.py` script against the generated artifacts and checking for a [deferred] pass rate on schema validation and content integrity checks.

**Acceptance Scenarios**:

1. **Given** a set of generated task trajectories, **When** the verification script runs, **Then** it must report 0 schema violations for the JSON structure (e.g., presence of `input`, `output`, `thought_process` fields).
2. **Given** a generated dataset file, **When** the file is loaded by a standard Python script, **Then** the system must successfully deserialize the data without `JSONDecodeError` or `KeyError` exceptions.

---

### User Story 3 - Reproducibility Report Generation (Priority: P3)

A project stakeholder needs a clear, human-readable report summarizing the reproduction results, comparing the local execution outcomes against the paper's stated claims (e.g., data volume, task diversity).

**Why this priority**: While the code execution is the primary goal, the project's value is in the *reporting* of the validation. This story ensures the project produces the final "deliverable" artifact required to close the research loop.

**Independent Test**: Can be tested by inspecting the `results/reproduction_report.md` file generated by the pipeline, ensuring it contains sections for "Execution Status," "Artifact Counts," and "Discrepancy Analysis."

**Acceptance Scenarios**:

1. **Given** the pipeline has completed execution, **When** the report generation script is invoked, **Then** it must produce a Markdown file containing a table of generated dataset sizes and a status flag (PASS/FAIL) for each validation step.
2. **Given** a discrepancy between local execution time and the paper's reported time, **When** the report is generated, **Then** the report must explicitly flag this discrepancy with a "Needs Clarification" note rather than silently ignoring it.

---

### Edge Cases

- **What happens when API rate limits are hit?** The system must implement a retry mechanism with exponential backoff (max 3 retries) and log a specific warning without crashing the entire pipeline.
- **How does the system handle missing external dependencies?** If a required submodule (e.g., `tasktrove`) is not initialized, the system must fail fast with a clear error message instructing the user to run `git submodule update --init --recursive`.
- **What if the generated dataset is empty?** The verification step must detect a 0-byte or 0-record dataset and fail the validation with a specific "Empty Dataset" error code.

## Requirements

### Functional Requirements

- **FR-001**: System MUST execute the data generation pipeline using the `data/generate_and_run_sbatch.py` entry point with a configurable `--task-limit` parameter to allow small-scale testing. (See US-1)
- **FR-002**: System MUST validate all generated artifacts against the internal JSON schema defined in `data/bespoke/verify.py` to ensure structural integrity. (See US-2)
- **FR-003**: System MUST implement a retry mechanism for external API calls with a maximum of 3 attempts and a 5-second initial backoff interval. (See US-1)
- **FR-004**: System MUST generate a `reproduction_report.md` file summarizing the execution status, artifact counts, and any discrepancies found. (See US-3)
- **FR-005**: System MUST handle missing git submodules by halting execution and printing a specific error message directing the user to initialize submodules. (See US-1)
- **FR-006**: System MUST ensure that no GPU-specific CUDA operations are invoked during the data generation phase, adhering to CPU-only constraints. (See US-1)

### Key Entities

- **Task Trajectory**: A data object representing a single agentic task instance, containing `input`, `thought_process`, and `output` fields.
- **Dataset Manifest**: A metadata file listing all generated task trajectories, their sizes, and validation status.
- **Reproduction Report**: A Markdown document summarizing the validation results and comparing them to the paper's claims.

## Success Criteria

### Measurable Outcomes

> Planning docs state *what* will be measured and the *source/reference* it is measured against; defer specific empirical values to the implementation/research phase.

- **SC-001**: The percentage of generated task trajectories that pass schema validation is measured against the requirement of [deferred] validity (See FR-002).
- **SC-002**: The number of successfully generated dataset files is measured against the `--task-limit` parameter provided in the execution command (See FR-001).
- **SC-003**: The time taken to generate a small-scale dataset (e.g., 10 tasks) is measured against a baseline of < 10 minutes on a standard CPU runner (See FR-001).
- **SC-004**: The presence of a `reproduction_report.md` file is measured against the requirement that the report must exist and contain non-empty content (See FR-004).

## Assumptions

- **Assumption about environment**: The CI runner has sufficient disk space (~14 GB) and RAM (~7 GB) to hold the generated dataset and Python environment for small-scale runs.
- **Assumption about data access**: The paper's claim of "fully open data" implies that the necessary API keys or public endpoints for task generation are accessible without paid tier restrictions for the validation scope.
- **Assumption about scope boundaries**: The project scope is limited to *data generation and validation*; full model training (fine-tuning Qwen3-32B) is out of scope due to GPU requirements and is only referenced for claim comparison.
- **Assumption about dependencies**: The `OpenThoughts-Agent` submodule is up-to-date and compatible with the current Python version (3.10+) in the CI environment.
- **Assumption about network**: The CI environment has outbound internet access to fetch any required public datasets or API responses during the generation phase.
