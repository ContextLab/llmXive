# Specification: Evaluating Code Smell Frequency in Human vs. LLM-Generated Code

**Project ID**: PROJ-514
**Version**: 2.1
**Status**: Active (Updated per Plan v3.0 & Methodology F30244be)

## 1. Introduction

This study investigates the frequency of specific code smells in human-written code versus code generated by Large Language Models (LLMs). The goal is to determine if there is a statistically significant association between the source of the code (Human vs. LLM) and the presence of code smells, while controlling for repository-level variance.

### 1.1 Objectives
- Collect a balanced dataset of human-written and LLM-generated code samples.
- Analyze four specific code smell categories: Long Method, Duplicated Code, Feature Envy, Long Parameter List.
- Apply a Blocked Permutation Test to compare distributions, stratified by repository.
- Report results with appropriate associational language, avoiding causal claims.

## 2. Functional Requirements

**FR-001: Human Sample Collection**
The system shall collect **150 human-written code samples** (3 per repository × A set of repositories) from public GitHub repositories meeting specific criteria (≥100 stars, ≥5 years history).
*Deviation*: Original spec required ≥1000; reduced to 150 per Balanced Blocked Design in `plan.md` to ensure repository-level matching and statistical power within CI constraints.

**FR-002: LLM Sample Generation**
The system shall generate **150 LLM-generated code samples** (3 per task × 50 tasks derived from the same issues used for human samples).
*Deviation*: Original spec required ≥50; increased to 150 to match human sample size for balanced design.

**FR-003: Data Integrity**
All samples must be validated for syntax correctness (Python/Java) before analysis. Samples failing validation must be logged and excluded from the final statistical set.

**FR-004: Static Analysis**
The system shall execute PMD CLI with custom rulesets to detect the four target code smells. Analysis must handle timeouts and memory limits gracefully.

**FR-005: Tool Validity**
A validity check shall be performed on a reference set. If the false positive rate exceeds a predefined acceptable threshold, the tool configuration is flagged as invalid, and the analysis pipeline halts.

**FR-006: Statistical Analysis**
The system shall perform a **Blocked Permutation Test** (stratified by repository) to compare smell frequencies. Bonferroni correction shall be applied for the family-wise error rate (4 tests, α ≤ 0.05).

**FR-007: [REJECTED] Causal Inference**
The original requirement to establish causal links between LLM usage and code smell reduction/increase is **REJECTED**.
*Rationale*: Replaced by "Balanced Blocked Design" and observational language. The study design is observational; causal claims are not supported without controlled experimental manipulation of the code generation process itself.

**FR-008: Reporting**
The final report must include statistical tables with corrected p-values, effect sizes, sensitivity analysis on thresholds, and visualizations (box plots).

## 3. Non-Functional Requirements

**NFR-001: Reproducibility**
All data collection and analysis steps must be deterministic given a fixed seed and input state.

**NFR-002: Performance**
The pipeline must complete within standard CI limits with parallel execution of independent tasks.

## 4. Statistical Methodology & Design

### 4.1 Design Overview
The study employs a **Balanced Blocked Design**. Repositories serve as blocks to control for project-specific coding standards and complexity.
- **Total Samples**: 300 (150 Human, 150 LLM).
- **Matching**: A representative number of samples per repository per source type.

### 4.2 Hypothesis Testing
For each of the code smell categories, we test:
- $H_0$: The distribution of smell counts is identical between Human and LLM samples within each repository block.
- $H_1$: The distributions differ.

**Test Method**: **Blocked Permutation Test**.
Unlike parametric tests (t-test, Mann-Whitney U) which assume normality or specific distributions, the permutation test is non-parametric and robust to zero-inflated data (common in code smell counts). It calculates the exact distribution of the test statistic under the null hypothesis by permuting labels within blocks.

**Correction**: Bonferroni correction applied to the 4 p-values to control Family-Wise Error Rate (FWER).

## 5. User Stories

### US-1: Data Collection
As a researcher, I want to collect 150 human and 150 LLM code samples from 50 repositories so that I have a balanced dataset for comparison.
**Acceptance Criteria**:
- `data/raw/human_samples` contains 150 valid samples.
- `data/raw/llm_samples` contains 150 valid samples.
- `data/raw/manifest.csv` links samples to repository IDs and commit SHAs.

### US-2: Static Analysis
As an analyst, I want to run PMD on all samples to extract smell counts so that I can quantify code quality.
**Acceptance Criteria**:
- `data/processed/smell_metrics.csv` contains counts for all 4 smells.
- Tool validity check passes (<5% false positive rate).

### US-3: Statistical Comparison
As a researcher, I want to compare the smell distributions using a Blocked Permutation Test so that I can determine if source type is associated with smell frequency.
**Acceptance Criteria**:
- **Scenario 1**: The analysis pipeline executes a **Blocked Permutation Test** (stratified by repository) instead of standard parametric tests.
- **Scenario 2**: The final report includes Bonferroni-corrected p-values and effect sizes.
- **Scenario 3**: The report explicitly states the study is observational and uses associational language.
- *Reference*: Methodology details in `methodology-f30244be`.

## 6. Data Model

### 6.1 CodeSample
- `sample_id`: Unique identifier
- `source_type`: "human" or "llm"
- `repository_id`: GitHub repo identifier
- `issue_id`: Original issue/PR ID
- `task_id`: Task identifier
- `language`: "python" or "java"
- `file_path`: Relative path in repo
- `function_name`: Target function name
- `is_fresh_commit`: Boolean (True if function added in the commit)

### 6.2 SmellMetric
- `sample_id`: Foreign key to CodeSample
- `smell_type`: "LongMethod", "DuplicatedCode", "FeatureEnvy", "LongParameterList"
- `count`: Integer count of violations
- `threshold_used`: Threshold applied (if applicable)
- `continuous_metric_value`: e.g., Cyclomatic Complexity

### 6.3 StatResult
- `smell_type`: Category tested
- `p_value`: Raw p-value from permutation test
- `effect_size`: Cohen's d or equivalent
- `confidence_interval`: 95% CI
- `correction_method`: "Bonferroni"
- `test_method_used`: "Blocked Permutation Test"
