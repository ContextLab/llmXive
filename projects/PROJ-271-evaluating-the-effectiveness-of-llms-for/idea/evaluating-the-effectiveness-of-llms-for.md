# Evaluating the Effectiveness of LLMs for Detecting Code Smells

**Field**: Computer Science

## Research question

Can quantized large language models (LLMs) running on CPU-only hardware identify semantic code smells in Python functions with comparable accuracy to rule-based static analysis tools?

## Motivation

Static analysis tools (e.g., Pylint) are fast but often miss semantic smells like "Long Method" or "Feature Envy" that require context understanding. While LLMs offer semantic reasoning, their computational cost is high. This project addresses the gap by evaluating if lightweight, CPU-deployable LLMs can provide a viable trade-off between accuracy and resource usage for automated code quality assessment.

## Related work

- [Beyond Strict Rules: Assessing the Effectiveness of Large Language Models for Code Smell Detection (2026)](http://arxiv.org/abs/2601.09873v1) — Provides a baseline assessment of LLMs for general code smell detection, establishing the current state of the art.
- [Evaluating LLMs Effectiveness in Detecting and Correcting Test Smells: An Empirical Study (2025)](http://arxiv.org/abs/2506.07594v1) — Focuses specifically on test code smells, offering a narrower scope that complements our focus on production Python code.
- [Specification and Detection of LLM Code Smells (2025)](http://arxiv.org/abs/2512.18020v1) — Discusses smells introduced by LLM integration, distinguishing between smells in code written by humans vs. smells in code generated by LLMs.
- [Investigating The Smells of LLM Generated Code (2025)](http://arxiv.org/abs/2510.03029v1) — Analyzes the quality of code generated by LLMs, relevant for understanding if LLMs can detect the same patterns they might inadvertently introduce.

## Expected results

We expect LLMs to achieve higher recall on semantic smells (e.g., Long Method) compared to static analyzers, but lower precision due to false positives. We anticipate an F1-score improvement of at least 10% on semantic categories while maintaining under 7GB RAM usage during inference.

## Methodology sketch

- **Data Acquisition**: Download the Python subset of the `codeparrot/github-code` dataset from HuggingFace Datasets (`https://huggingface.co/datasets/codeparrot/github-code`) using the `datasets` library; filter to 500 random functions to fit within 6-hour runtime.
- **Ground Truth Generation**: Run Pylint and SonarQube (community edition) on the dataset to generate baseline labels for known smells (e.g., `too-many-locals`, `duplicate-code`).
- **Model Setup**: Load CodeLlama-7B-Instruct-GGUF (4-bit quantization) using `llama-cpp-python` to ensure memory footprint remains under 7GB RAM on CPU-only runners.
- **Inference**: Prompt the model with each code snippet and a specific instruction to list detected smells; parse the output using regex to extract smell categories.
- **Comparison**: Align LLM outputs with static analyzer labels; calculate Precision, Recall, and F1-score for each smell category.
- **Statistical Analysis**: Perform McNemar's test to determine if the difference in detection rates between the LLM and static analyzers is statistically significant.
- **Resource Monitoring**: Log peak RAM usage and total inference time per batch to verify compliance with the 2 CPU / 7GB RAM / 6h constraints.
