# Evaluating the Effectiveness of LLMs for Detecting Code Smells

**Field**: Computer Science  

## Research question

What structural and semantic features of Python code determine whether code smells are detectable through rule‑based static analysis versus semantic understanding, and how do these detection modes complement each other in comprehensive code quality assessment?

## Motivation

Rule‑based static analyzers (e.g., Pylint, SonarQube) excel at spotting syntactic or metric‑based smells but often miss semantic issues such as “Long Method” or “Feature Envy” that require contextual reasoning. Large language models (LLMs) can perform semantic reasoning, yet their CPU‑only, quantized variants remain under‑explored for this task. Understanding which code characteristics favour one detection mode over the other will guide tool‑chain design and resource allocation for automated code quality assessment.

## Related work

- [Beyond Strict Rules: Assessing the Effectiveness of Large Language Models for Code Smell Detection (2026)](https://arxiv.org/abs/2601.09873) — Provides a broad benchmark of LLMs versus traditional detectors, establishing that LLMs can identify smells missed by rule‑based tools.  
- [Evaluating LLMs Effectiveness in Detecting and Correcting Test Smells: An Empirical Study (2025)](https://arxiv.org/abs/2506.07594) — Shows LLMs can recognise test‑specific smells, demonstrating the feasibility of semantic smell detection in code.  
- [Parichayana: An Eclipse Plugin for Detecting Exception Handling Anti‑Patterns and Code Smells in Java Programs (2016)](https://arxiv.org/abs/1701.00108) — Describes a rule‑based static analysis approach for Java, useful as a methodological precedent for constructing comparable Python static‑analysis baselines.

## Expected results

We anticipate that structural features (e.g., cyclomatic complexity, LOC) will predict high recall for rule‑based tools, whereas semantic features (e.g., variable naming coherence, cross‑function dependency patterns) will correlate with LLM‑only detections. Overlap analysis should reveal a non‑trivial set of smells uniquely identified by each method, supporting the claim that the two modes are complementary. Statistical significance will be assessed with McNemar’s test (α = 0.05) on paired detection outcomes.

## Methodology sketch

- **Dataset acquisition**: Retrieve the Python subset of the `codeparrot/github-code` dataset from HuggingFace (`https://huggingface.co/datasets/codeparrot/github-code`). Randomly sample 800 functions (≈ 2 MB) to keep total runtime < 6 h.
- **Static‑analysis labeling**: Run Pylint (default rule set) and SonarQube Community Edition on each function; record detected smell categories as the *rule‑based* label set.
- **Feature extraction**: For every function compute (a) structural metrics (LOC, cyclomatic complexity, nesting depth) using `radon`; (b) semantic embeddings via `sentence‑transformers/all-MiniLM-L6-v2` applied to the source code string.
- **LLM inference**: Load CodeLlama‑7B‑Instruct‑GGUF (4‑bit quantized) with `llama‑cpp‑python` on CPU‑only runner. Prompt each function with: “Identify any code smells present in the following Python function and list their names.” Parse model output into a set of smell categories.
- **Detection‑mode comparison**: For each smell category compute (i) detection by static analysis, (ii) detection by LLM, (iii) union and intersection. Identify features that statistically differentiate (a) smells detected only by static analysis, (b) only by LLM, (c) by both.
- **Statistical analysis**:  
  - Fit logistic regression models predicting detection mode (static vs. LLM) from structural and semantic features.  
  - Apply McNemar’s test on paired binary outcomes per smell to test whether detection rates differ significantly.  
  - Use permutation testing (10 000 permutations) to assess the significance of feature importance differences.
- **Resource profiling**: Record peak RAM, CPU utilization, and total inference time per batch (≤ 50 functions) to verify compliance with the 7 GB RAM / 2‑CPU limit.
- **Reproducibility**: All scripts, environment files, and random seeds will be committed; results will be saved as CSV/JSON for downstream analysis.
