# Measuring the Carbon Footprint of LLM‑Assisted Code Generation

**Field**: computer science  

## Research question

How does the carbon‑footprint per line of code generated by large‑language‑model‑assisted coding compare to that of human‑written code for the same programming tasks?

## Motivation

Sustainable AI is increasingly important, yet the environmental impact of using LLMs for software development is poorly quantified. Quantifying emissions per unit of produced code will reveal whether LLM assistance reduces overall energy use (by speeding development) or adds hidden costs (through inference energy). This knowledge can guide developers, tool builders, and policy makers toward greener software‑engineering practices.

## Related work

- [Comparative Analysis of Carbon Footprint in Manual vs. LLM-Assisted Code Development (2025)](https://arxiv.org/abs/2505.04521) — Directly measures CO₂‑eq emissions for LLM‑assisted versus manual coding, providing baseline human‑task time data and a methodological template for carbon‑footprint comparison.  
- [Quantifying the Carbon Emissions of Machine Learning (2019)](https://arxiv.org/abs/1910.09700) — Introduces the CodeCarbon library and a framework for converting energy consumption into CO₂‑equivalents, which we will adopt for measuring inference energy.  
- [Compute and Energy Consumption Trends in Deep Learning Inference (2021)](https://arxiv.org/abs/2109.05472) — Analyzes inference‑stage energy trends across model sizes, informing our choice of a lightweight open‑source model (GPT‑2‑medium) that fits GitHub Actions CPU constraints.

## Expected results

We anticipate one of two outcomes: (1) LLM‑assisted code generation yields lower emissions per line of code than manual coding (suggesting a sustainability benefit), or (2) it yields higher emissions per line (indicating a hidden environmental cost). A statistically significant difference (p < 0.05) in paired comparisons across prompts will confirm the direction of the effect.

## Methodology sketch

- **Select benchmark**: Download the CodeXGLUE Python code‑generation dataset (public URL from HuggingFace Datasets).  
- **Define task subset**: Randomly sample 200 prompts to keep the total runtime < 6 h on GitHub Actions CPU runners.  
- **Run LLM inference**: Use the open‑source GPT‑2‑medium model on the CPU tier; for each prompt generate code with deterministic sampling.  
- **Instrument energy use**: Wrap the inference loop with the CodeCarbon library (as described in *Quantifying the Carbon Emissions of Machine Learning*). Record total energy (kWh) and convert to CO₂‑eq using the default regional factor.  
- **Gather human baseline**: Extract per‑prompt human development time from the 2025 comparative analysis paper (which reports average minutes spent by developers on the same benchmark).  
- **Convert human time to emissions**: Multiply recorded minutes by an average CPU power draw (≈ 15 W for a typical laptop) and the same CO₂‑eq conversion factor to obtain human‑coded emissions.  
- **Normalize**: Compute emissions per line of code (LOC) for both LLM‑generated and human‑written solutions.  
- **Statistical analysis**: Perform a paired‑samples t‑test (or Wilcoxon signed‑rank test if normality fails) comparing LLM vs. human emissions per LOC across the 200 prompts.  
- **Robustness checks**: Repeat the experiment with a second lightweight model (DistilGPT‑2) to verify that results are not model‑specific.  
- **Reporting**: Produce a concise markdown report with summary statistics, effect size, confidence intervals, and visualizations (boxplots of emissions per LOC).
