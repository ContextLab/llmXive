## Summary

This project investigates whether manipulating the predictability of sequential stimuli alters participants' duration estimates and if larger prediction errors (surprisal) correlate with overestimation of elapsed time. The technical approach involves downloading existing time perception datasets from OpenML and HuggingFace, computing surprisal metrics via a first-order Markov model where missing, and fitting linear mixed-effects models (LMM) with duration estimate as the outcome and surprisal as the fixed effect. The implementation is constrained to CPU-only execution (GitHub Actions free tier: 2 CPU, 7 GB RAM) and must complete within 6 hours.

**CRITICAL BLOCKER**: This project is currently blocked by a **Data Mismatch**. The "Verified datasets" block provided in the user message contains only NLP, Twitter, and Robotics data, none of which contain the required outcome variable ("duration estimates") or the specific experimental design ("sequential stimuli with predictability manipulations"). The pipeline includes a **Gate 0** that will halt execution if no valid dataset is found. The project cannot proceed to analysis until a valid time-perception dataset is added to the verified list.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| **LMM vs. Simple Regression** | Requires modeling participant-level variance (random intercepts) to handle repeated measures within subjects. | Simple regression ignores within-subject correlation, violating statistical assumptions for repeated measures. |
| **Surprisal Computation** | Explicit predictability metrics are often missing in raw datasets; must be algorithmically derived. | Excluding datasets without explicit metrics would severely limit sample size and power. |
| **Chunked Processing** | Datasets may exceed 7 GB RAM; streaming aggregation is required. | Loading full datasets into memory would cause OOM errors on the free-tier runner. |
| **Pingouin for MDE** | `statsmodels` LMM does not natively calculate Minimum Detectable Effect (MDE) easily. | Using `pingouin` ensures accurate power analysis without complex manual derivation. |

## Implementation Phases

### Gate 0: Data Availability & Validity Check
**Goal**: Ensure the project has a valid dataset before proceeding.
1.  **Action**: `download.py` attempts to fetch datasets from the "Verified datasets" list.
2.  **Check**: For each dataset, verify the presence of:
    *   `duration_estimate` (outcome)
    *   `stimulus_sequence` (predictor source)
    *   `participant_id` (random effect)
3.  **Blocker**: If **NO** dataset in the verified list contains these variables, the pipeline **HALTS**.
    *   **Output**: `data/README.md` is updated with "CRITICAL BLOCKER: No valid time-perception dataset found in verified list."
    *   **Status**: Project remains in `planned` state; no further phases run.
4.  **Fallback**: If a dataset is found, proceed to Phase 1. If not, the project is blocked until a valid dataset is added to the verified list.

### Phase 1: Data Preprocessing
**Goal**: Standardize data and compute surprisal.
1.  **Action**: `preprocess.py` loads valid datasets.
2.  **Surprisal Calculation**:
    *   If `surprisal` is missing, compute using a first-order Markov model on `stimulus_sequence`.
    *   **Validity Check**: Only compute if the dataset has a known experimental design (e.g., oddball). If the sequence is unstructured (e.g., tweets), mark as "Invalid for Prediction Error Proxy" and exclude.
3.  **Output**: `data/processed/standardized.csv`.

### Phase 2: Statistical Analysis
**Goal**: Fit models and test hypotheses.
*Note: This phase only runs if Gate 0 passes.*

#### Phase 2a: Model Fitting
1.  **Action**: `analysis.py` fits LMM: `Duration ~ Surprisal + Sequence_Length + Modality + (1 | Participant_ID)`.
2.  **Convergence Check**: Record convergence status. If failed, simplify random effects and log.
3.  **Metric**: Track convergence rate (SC-002).

#### Phase 2b: Hypothesis Testing & Correction
1.  **Action**: Apply Bonferroni or Benjamini-Hochberg correction to p-values if >1 test is run (FR-005).
2.  **Output**: Adjusted p-values logged to `analysis/results.json`.
3.  **Metric**: Verify Family-Wise Error Rate (SC-003).

#### Phase 2c: Effect Size Computation
1.  **Action**: Calculate Cohen's d (or standardized beta) for fixed effects (FR-006).
2.  **Output**: Effect sizes with 95% CI logged to `analysis/results.json`.

#### Phase 2d: Convergence & Quality Metrics
1.  **Action**: Calculate the percentage of models that converged without simplification.
2.  **Metric**: Compare against SC-002 (≥90%).

#### Phase 2e: FWER Verification
1.  **Action**: Verify that the final reported p-values satisfy α≤0.05.
2.  **Metric**: Log `fwer_control_status` to output schema.

#### Phase 2f: Bootstrap Resampling (FR-009)
1.  **Action**: Run bootstrap resampling using `joblib.Parallel(n_jobs=2)` to estimate confidence intervals robustly.
2.  **Constraint**: Ensure total runtime ≤6 hours.

### Phase 3: Visualization & Reporting
1.  **Action**: `visualize.py` generates forest plots and residual diagnostics.
2.  **Output**: `figures/` directory with 300+ DPI plots.
