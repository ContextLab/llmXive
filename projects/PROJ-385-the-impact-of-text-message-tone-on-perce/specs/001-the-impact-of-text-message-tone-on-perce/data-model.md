# Data Model: The Impact of Text Message Tone on Perceived Emotional Support

## 1. Overview

This document defines the data structures used throughout the project lifecycle, from stimulus generation to final analysis results. The model adheres to the **Stimulus-Response Separation** principle (Constitution Principle VII), keeping objective features distinct from subjective ratings until the analysis phase.

## 2. Entity Definitions

### 2.1 Stimulus
Represents a unique text message variant generated by the factorial design.

-   **stimulus_id**: Unique identifier (e.g., `S-001`).
-   **base_scenario**: The core message text (e.g., "I had a rough day").
-   **emoji_level**: Integer (0, 1, 2).
-   **punctuation_type**: String ("standard", "exaggerated").
-   **length_category**: String ("concise", "elaborate").
-   **generated_text**: The final string presented to participants.

### 2.2 Participant
Represents a simulated human rater.

-   **participant_id**: Unique identifier (e.g., `P-001`).
-   **demographic_age**: Integer (18-35).
-   **demographic_gender**: String (optional, for stratification if needed).

### 2.3 Rating
The atomic unit of data, linking a participant, a stimulus, and a context.

-   **rating_id**: Unique identifier.
-   **participant_id**: FK to Participant.
-   **stimulus_id**: FK to Stimulus.
-   **relationship_context**: String ("friend", "acquaintance").
-   **rating_score**: Integer (1-7).
-   **timestamp**: ISO 8601 string (simulated).

### 2.4 AnalysisResult
The output of the statistical models.

-   **model_id**: Identifier for the specific run (e.g., `lmm_primary`, `sensitivity_A`).
-   **term**: The fixed effect term (e.g., "relationship", "cue_intensity", "interaction").
-   **estimate**: Float (coefficient).
-   **std_err**: Float.
-   **p_value**: Float.
-   **ci_lower**: Float.
-   **ci_upper**: Float.

## 3. Data Flow

1.  **Generation**: `01_generate_stimuli.py` reads a configuration file and outputs `data/raw/stimuli.csv`.
2.  **Simulation**: `02_simulate_ratings.py` reads `stimuli.csv`, generates participants, and outputs `data/raw/ratings.csv`.
3.  **Cleaning**: `03_clean_data.py` reads `ratings.csv`, applies exclusion rules (straight-lining), and outputs `data/processed/clean_ratings.csv`.
4.  **Analysis**: `04_run_lmm.py` and `05_sensitivity_analysis.py` read `clean_ratings.csv` and output `data/processed/analysis_results.json`.

## 4. Constraints & Validation

-   **Rating Score**: Must be integer 1-7.
-   **Relationship Context**: Must be one of `['friend', 'acquaintance']`.
-   **Stimulus ID**: Must exist in `stimuli.csv`.
-   **Participant ID**: Must be unique per rating set.
-   **No PII**: No names, emails, or real IDs stored.

## 5. Schema References

Detailed schema definitions are provided in the `contracts/` directory:
-   `stimulus.schema.yaml`: Validates generated stimuli.
-   `rating.schema.yaml`: Validates raw and cleaned rating data.
-   `analysis_result.schema.yaml`: Validates statistical output.
