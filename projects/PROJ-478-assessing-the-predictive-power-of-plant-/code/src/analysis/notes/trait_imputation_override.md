# Trait Imputation Strategy: Spec-Plan Divergence Note

## Overview

This document formally records the divergence between the project's **Feature Specification** (Spec) and the **Implementation Plan** regarding the handling of functional trait data for held-out species during Leave-One-Species-Out (LOSO) cross-validation.

## Specification Requirement (FR-004)

The original Feature Specification (Section: User Story 2, FR-004) mandates:
> "Use **known trait values** for the held-out species when evaluating the trait-augmented model."

This approach assumes that ground-truth trait data is available for the test species, ensuring that any improvement in model performance (AUC/TSS) is attributable to the *predictive power of the traits themselves* rather than the accuracy of trait imputation.

## Plan Override (Trait Imputation Strategy)

The Implementation Plan introduces an alternative strategy to address real-world scenarios where trait data may be missing for the target species:
> "Predict traits for the held-out species using a climate-niche model trained on N-1 species, then evaluate the SDM using these **imputed trait values**."

This approach simulates a more realistic predictive workflow where traits are not known a priori but must be inferred from environmental suitability.

## Implementation Details

The codebase (`src/modeling/loso_cv.py`) implements **both** strategies to allow for comparative analysis:

1. **Spec-Compliant Path (Default)**:
 - Uses `known_trait_values` from the `data/processed/traits.csv` (or similar source) for the held-out species.
 - This is the primary path for validating the *theoretical* predictive power of traits.
 - Flagged in logs as `protocol: spec_compliant`.

2. **Plan-Override Path (Optional)**:
 - Trains an intermediate model (Climate → Traits) on the N-1 training species.
 - Predicts traits for the held-out species based on their climate envelope.
 - Feeds these *predicted* traits into the SDM (Climate + Traits).
 - Flagged in logs as `protocol: trait_imputation`.

## Scientific Validity & Justification

The Plan override is justified for the following reasons:
- **Realism**: In practical conservation applications, trait data is often missing for rare or poorly studied species. The imputation strategy tests the robustness of the SDM under data-scarce conditions.
- **Error Propagation Analysis**: Comparing the results of the Spec path vs. the Plan path allows us to quantify the "imputation penalty"—how much predictive power is lost due to uncertainty in trait estimation.
- **Completeness**: It ensures the pipeline can generate predictions even when the trait database is incomplete, increasing the utility of the tool.

## Reporting Requirements

All final reports generated by `src/analysis/report_generator.py` must:
- Explicitly state which protocol (Spec vs. Plan) was used for the reported metrics.
- If the Plan override was used, include a disclaimer that results include the error margin of trait imputation.
- Provide a side-by-side comparison of AUC/TSS differences between the two protocols if both were executed.

While FR-004 defines the baseline for scientific rigor, the Trait Imputation strategy extends the project's applicability to real-world data gaps. The coexistence of both paths in the codebase ensures that the project satisfies both the strict requirements of the specification and the practical needs outlined in the implementation plan.
