\section{Related Work}

Tool-use agent benchmarks vary along several axes. Early benchmarks evaluate single-turn tool selection and argument grounding over API collections \citep{toolllm24qin, bfcl_25_patil, apibank23Li, ToolQA23Zhuang, mcpverse25lei}. Recent work involves multi-turn interactions in persistent environments, such as web navigation and code repositories \citep{webarena_24_zhou, swebench_24_Jimenez}. 
Our focus is on conversational benchmarks that pair an agent with a simulated user over a mutable environment, and evaluate the resulting world state after the full interaction \citep{taubench_24_yao, tau2barres, toolsandbox25lu}. While valuable, existing agent benchmarks rely heavily on manually authored tasks, a process that is costly, error-prone, and difficult to scale as agent capabilities advance \citep{saber2025cuadron, bestpractices2025Zhu, yehudai2026agenticclearautomatingmultilevel}.

Synthetic generation methods reduce annotation cost. Early approaches bootstrap instructions from seed sets \citep{selfinstruct23wang, unnaturalinstructions23honovich, wizardlm24xu}, while tool-use extensions generate gold tool calls through insertion filtering \citep{toolformer23shick}, synthesis from tool descriptions \citep{toolllm24qin, toolalpaca23tang}, and post-hoc verification \citep{apigen24liu}. Recent pipelines scale to multi-turn using task blueprints, simulated environments, multi-agent role-playing, and trajectory feedback \citep{apigenmt25prabhakar, toucan25xu, closetheloop25li, implicitexperience26xu, toolforge25chen, LAM25Hoang}. Closest to our approach are ToolGrad \citep{toolgrad25zhou} and Trajectory2Task \citep{trajectory2task}, which synthesize tasks from trajectories. ToolGrad iteratively builds tool chains by proposing, executing, and selecting the best API call at each step, extending the sequence until completion before instantiating the user instruction. Trajectory2Task starts from an existing trajectory, progressively replaces tool calls, and evolves user instructions toward challenging intents. Both primarily employ reverse synthesis to increase the plausibility of generating a valid task and to produce training data. Our motivation is different: we start from tool sequences to explicitly explore and cover the space of tool combinations, and our focus is on constructing better agent benchmarks rather than training data.

We define coverage in terms of gold tool sequences, adopting diversity measures from the NLP literature \citep{raven2023mccoy, evaluatingnovelty24merrill, standardizingdiversity24shaib, evaluatingdiversity21tevet} over tools rather than words. Existing benchmarks typically summarize diversity by domain, tool, or schema counts, and characterize difficulty through capability tags \citep{toolllm24qin, mcpverse25lei, toolsandbox25lu, tripbench26Shen}. Such summaries indicate which tools are available, but not whether tasks exercise diverse procedural patterns of tool use. In contrast, sequence-level coverage allows us to measure whether a benchmark explores a broad range of agent behaviors rather than repeatedly testing similar patterns.

% Tool-use benchmarks evaluate planning, tool retrieval, tool selection, argument grounding, and execution over external APIs or environments. Early and large-scale benchmarks such as API-Bank, ToolQA, ToolLLM, and the Berkeley Function Calling Leaderboard study these abilities across increasingly broad tool collections~\citep{apibank23Li,ToolQA23Zhuang,toolllm24qin,bfcl_25_patil}. Stateful conversational benchmarks move closer to our setting: $\tau$-BENCH pairs an agent with a simulated user and grades the final environment state~\citep{taubench_24_yao}, ToolSandbox adds implicit state dependencies and milestone-based grading~\citep{toolsandbox25lu}, and $\tau^2$-BENCH introduces dual-control interactions in which both the user and agent can affect shared state~\citep{tau2barres}. Recent benchmarks further stress executable tool coverage, long-horizon interaction, feasibility changes, task-oriented dialogue, policy graphs, and step-level process diagnostics to complement the final-state evaluation~\citep{mcpverse25lei, tripbench26Shen,userbench25qian, atod26zhang, intelagent25levi, agentprocessbench26fan}. 

% These benchmarks collectively move evaluation toward realistic conversational tool use, but most still rely on fixed, description-first task sets:  scenarios are written first, while the required tool sequence and induced final state are recovered or inspected afterward. Consequently, validity, coverage over gold tool sequences, and controlled difficulty are usually audited post hoc rather than optimized during construction. The action-first pipline in TASTE is designed to change this: it uses tool sequences as the construction substrate through which validity, representative coverage, and difficulty can be explicitly targeted by sampling, medoid selection, and task evolution.

ToolGrad constructs valid tool chains before reverse-engineering user prompts, while Trajectory2Task starts from executable trajectories and derives realistic tasks with ambiguous, changing, or infeasible intents~\citep{toolgrad25zhou, trajectory2task}. We share the insight that actions can be a better starting point than free-form scenarios. However, our goal is benchmark construction rather than training-data generation or robustness studies: generated instances must be plausible and executable, and must provide controlled coverage over gold tool sequences and controlled difficulty. TASTE therefore samples tool sequences for coverage, selects medoids for representativeness, and adversarially evolves tasks while preserving validity.

% A benchmark score is useful only if it measures the intended capability rather than artifacts of task wording, simulator behavior, tool implementation, or reward design. General benchmark-design work therefore argues for explicit construct definitions, justified task and metric choices, uncertainty reporting, reproducibility, and benchmark maintenance \citep{everything21raji,utility20enthayarajh,whatwillittake21bowman,ecbd24liu,helm23liang,betterbench24reuel}. These concerns are amplified for tool-use agents, where each task couples a user instruction, mutable environment state, tool specification, user simulator, and automatic evaluator. Recent audits show that such failures are concrete: agentic benchmarks can contain invalid or underspecified tasks, reward bugs, and mutating-action errors that disproportionately affect final success~\citep{bestpractices2025Zhu, saber2025cuadron}. Beyond validity, difficulty and coverage are often under-specified. Existing benchmarks commonly report coarse difficulty splits, task length, number of turns, capability tags, domain counts, tool counts, or schema counts~\citep{toolsandbox25lu,tripbench26Shen,toolllm24qin,mcpverse25lei}, but these summaries do not reveal whether a benchmark exercises representative procedural tool-use patterns. Synthetic benchmarks add further risks, including generator-specific artifacts, leakage from seed tasks or public benchmarks, unrealistic task distributions, and lexical rather than procedural diversity~\citep{underthesurface24das,leakage24xu,diversityofsyntheticdata24chen}. Prior studies similarly caution that synthetic evaluation data can preserve useful signals in simple settings while failing to replace real data reliably in more complex ones~\citep{syntheticbenchmark24maheshwari, agorabench25kim}. We address these issues operationally for conversational tool use: validity requires tasks to be verifiable and correct; coverage is measured over gold tool sequences; and difficulty is controlled along structural axes, such as sequence length and write-action ratio, and interaction axes, such as ambiguity, decoy entities, and user cooperation.
