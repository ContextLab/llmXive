# Analyzing the Prevalence of Unmaintained Dependencies in Popular NPM Packages

**Field**: computer science

## Research question

To what extent does the age of unmaintained dependencies in popular NPM packages correlate with exposure to unpatched security vulnerabilities in the JavaScript software supply chain?

## Motivation

Software supply chain attacks have become a critical threat vector, with unmaintained ("zombie") dependencies often containing unpatched vulnerabilities. Understanding the relationship between dependency maintenance status and security risk is essential for prioritizing automated dependency-update tools and informing developer risk-assessment practices.

## Related work

- [Insecure Ingredients? Exploring Dependency Update Patterns of Bundled JavaScript Packages on the Web (2025)](https://arxiv.org/abs/2512.15447) — Establishes methodology for analyzing JavaScript package dependency update patterns, providing a precedent for measuring maintenance status through release history.
- [Extracting Insights from the Topology of the JavaScript Package Ecosystem (2017)](https://arxiv.org/abs/1710.00446) — Documents the scale and structure of the NPM ecosystem, establishing baseline metrics for dependency-tree analysis that this project extends to maintenance-status assessment.

## Expected results

We expect to find a positive correlation between dependency age (time since last release) and vulnerability exposure (number of unpatched CVEs), with older unmaintained packages showing significantly higher vulnerability density. A null result (no correlation) would suggest that vulnerability databases capture security issues independently of maintenance activity, which would be equally informative for risk-assessment models.

## Methodology sketch

- Query the npm registry API to retrieve the top 1,000 most-downloaded packages by weekly download count.
- For each package, recursively extract direct and transitive dependencies using the npm `ls` command or registry metadata.
- For each unique dependency, query the GitHub API (via repository metadata) to obtain the timestamp of the last commit and the last release tag.
- Query the npm audit API and npm security advisory database to retrieve known vulnerability counts per dependency.
- Compute "dependency age" as days since last release and "vulnerability density" as vulnerabilities per dependency.
- Apply Pearson correlation coefficient to test the relationship between dependency age and vulnerability count across the sample.
- Stratify analysis by package category (e.g., framework, utility, build tool) using keywords from package metadata.
- Generate visualizations: scatter plot of age vs. vulnerability count, histogram of unmaintained dependency percentages by package category.
