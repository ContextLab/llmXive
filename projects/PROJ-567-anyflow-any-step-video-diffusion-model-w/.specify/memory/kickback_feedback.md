## Unresolved concerns

- The plan admits to dropping VBench metrics if they require GPU. VBench's core value is its multi-dimensional quality assessment. If the CPU fallback only measures 'frame count' or 'brightness' (as suggested in Risk Mitigation), the construct validity of 'video quality' is lost. The plan must define a specific, CPU-tractable subset of metrics that still meaningfully approximates the paper's claims, rather than a generic 'subset'.
- The mitigation for 'Inference time > 6 hours' accepts a 'partial reproduction' (valid artifact, not full fidelity) as success. This contradicts the research goal of 'validating the paper's claims.' If the method cannot be run at the parameters required to produce the claimed results, the research question (does this method work as claimed?) cannot be answered by this plan.
- The 'Model Checkpoint' source is listed as 'To be downloaded... (URL to be confirmed)'. While the plan acknowledges this is pending, the 'Verified Datasets' block should not list a 'Pending Verification' item as a source. The plan must either cite a specific, reachable URL for the smallest available model (even if it's the 14B one with a note on quantization) or explicitly state that the model source is unknown and the project cannot proceed without it. Relying on a 'to be confirmed' URL for a critical resource in a feasibility plan is a gap.
- The 'Constitution Check' section lists gates (Reproducibility, Feasibility, etc.) but does not explicitly map them to the specific principles in the per-project `constitution.md` (e.g., 'Principle I: SSoT', 'Principle V: Real-Call Testing'). The plan asserts compliance but lacks the required traceability to the constitutional text to verify no violations (e.g., silent fallbacks) exist.
- Phase 2 mentions creating 'validation.py' to run VBench. The `contracts/evaluation_report.schema.yaml` defines the output structure, but the plan does not explicitly state that the output of `validation.py` will be validated against this specific contract file. Explicitly linking the implementation step to the contract file ensures the 'Plan ↔ contracts' check is satisfied.
- SC-004 claims to measure 'Reproduction fidelity' by comparing generated VBench scores against 'paper baseline values'. However, the plan explicitly restricts execution to a 1.3B model (or smallest available) and low resolution/steps (Phase 0/1), while the paper's baselines are almost certainly derived from the 14B model at high resolution. Comparing a 1.3B/low-res output to a 14B/high-res baseline is a category error; the result is guaranteed to be negative not due to method failure, but due to architectural mismatch. The validation target (paper baseline) is not independent of the model scale, which is a primary predictor of the metric. This renders the 'fidelity' claim scientifically invalid.
- The plan assumes a 1.3B model variant exists for 'AnyFlow' (Wan2.1). If the paper only releases a 14B model, the 'smallest available' fallback implies running a 14B model with extreme quantization or resolution reduction. If the model architecture itself (e.g., number of layers) is fixed at 14B, running it on CPU with reduced resolution does not change the model's parameter count or the 'flow map' logic being tested. The plan conflates 'computational feasibility' with 'scientific reproduction'. If the 1.3B model does not exist, the plan fails to reproduce the *method* (AnyFlow) and instead reproduces a *downscaled version* of a different model, making the comparison to the paper's claims (which assume the full model) tautological or meaningless.
