# Research: $π$-Bench: Evaluating Proactive Personal Assistant Agents in Long-Horizon Workflows

## Dataset Strategy

The evaluation relies on a **synthetic/mock dataset** generated by the vendored `Pi-Bench` codebase, not an external public repository. The "dataset" consists of `task.yaml` and `profile.yaml` files defining 5 personas (Financier, Law Trainee, Marketer, Pharmacist, and one additional) and ~20 tasks each.

| Dataset Component | Source / Loader | Variables Available | Fit for Purpose |
| :--- | :--- | :--- | :--- |
| **Persona Profiles** | `external/Pi-Bench/profiles/` | Skills, hidden intents, context | **High**: Directly maps to `FR-001`. |
| **Task Definitions** | `external/Pi-Bench/tasks/` | Explicit instructions, hidden intents, expected outcomes, **Canonical Baseline Paths** | **High**: Directly maps to `FR-002` (proactivity) and `FR-005` (novelty). |
| **Mock Environment** | `src/mocks/environment.py` | Simulated API responses (Gmail, Amazon, etc.) | **High**: Ensures deterministic execution for `FR-007` (error handling). |
| **Model Configs** | `src/config/models/` | Model names, CPU-only flags | **High**: Enables `FR-003` (CPU execution). |

**Dataset Variable Fit**:
The spec requires evaluation of "hidden intents" and "proactivity." The `task.yaml` files explicitly define these hidden intents (e.g., "User wants to save money but didn't say it"). The mock environment provides the necessary state to verify if the agent acted on these intents.
*   **Predictors**: Persona type, task complexity, hidden intent presence.
*   **Outcomes**: `task_completion_status`, `proactivity_score`, `novelty_index`.
*   **Covariates**: Model backend (MiniMax, Claude, DeepSeek), latency.

**No external dataset URL is needed or fabricated.** The "dataset" is the codebase itself, which is version-controlled and vendored. This avoids the risk of missing variables or unreachable URLs.

## Methodological Rigor

### Statistical Rigor & Metric Definitions

The plan addresses the Turing-simulated reviewer's concern: *"Is the agent planning, or merely echoing?"* by introducing a **Sequence Novelty Index (SNI)** and revising the Proactivity Score definition.

1.  **Proactivity Score (`proactivity_score`)**:
    *   **Method**: Comparison of the agent's action sequence against a **Set of Valid Explicit-Only Baselines**.
    *   **Definition**: `Score = 1 - (min(Levenshtein_Distance(Actual, Baseline_i)) / Max_Distance)` * `Intent_Coverage_Factor`, where `Baseline_i` iterates over all valid canonical paths for the task.
    *   **Rationale**: This prevents penalizing valid alternative execution orders. If the agent follows *any* valid explicit path, the score is high. Proactivity is only rewarded if the agent deviates from *all* explicit baselines to address a hidden intent.
    *   **Reference**: Derived from `FR-002` and `SC-001`.

2.  **Sequence Novelty Index (`novelty_index`)**:
    *   **Method**: Calculation of deviation from a **Population of Known Retrieval Templates**.
    *   **Formula**: `SNI = 1 - (n-gram_Overlap(Trajectory, Template_Library) / Max_Overlap)`.
    *   **Rationale**:
        *   **Retrieval/Echoing**: If the agent's trajectory matches a known template (high n-gram overlap), SNI is low (near 0).
        *   **Planning/Novelty**: If the agent constructs a sequence of actions not found in the template library (low overlap), SNI is high (near 1).
        *   **Correction**: This avoids the tautology of calculating entropy on a single trajectory. It requires a reference distribution (the template library) to establish "novelty."
    *   **Threshold**: A threshold (e.g., `0.2`) will be used to flag "retrieval" vs. "planning" behaviors (`SC-003`).
    *   **Reference**: `FR-005` (Novelty Check) and `US-2`.

3.  **Statistical Mitigation (Low Sample Size)**:
    *   **Acknowledgement**: The dataset size is fixed by the `Pi-Bench` benchmark, resulting in small cell sizes per model/persona.
    *   **Mitigation**: We will **not** rely on parametric tests (t-tests, ANOVA) or Bonferroni corrections, which are invalid at this N. Instead, we will use **Non-Parametric Bootstrapping** (10,000 resamples) to estimate 95% confidence intervals for the difference in mean `proactivity_score` between models.
    *   **Reporting**: Results will be reported as "Model A is associated with a higher proactivity score (Mean Diff: X, 95% CI: [L, U])" rather than claiming statistical significance (p < 0.05).
    *   **Reference**: `SC-002` (Comparative analysis).

4.  **Causal Inference Assumptions**:
    *   **Observational**: The evaluation is observational (we observe agent behavior on tasks). We **do not** claim causal effects (e.g., "Model X *causes* higher proactivity"). Claims will be framed as **associational** (e.g., "Model X is *associated* with higher proactivity scores").
    *   **Reference**: `FR-006` (Comparative analysis).

5.  **Measurement Validity**:
    *   The `proactivity_score` is validated against the paper's definition (anticipation of unstated needs). The `novelty_index` is a novel metric introduced to address the specific methodological objection regarding retrieval vs. planning.
    *   **Reference**: `US-2` (Validate Proactivity Metrics).

### Computational Feasibility

*   **Hardware**: GitHub Actions free-tier (2 CPU, 7GB RAM).
*   **Strategy**:
    *   **No GPU**: All models will be loaded with `device="cpu"`.
    *   **No Quantization**: Avoid `load_in_8bit` or `bitsandbytes` which require CUDA. Use default precision (float32) or float16 if supported by CPU torch, but default to float32 for stability.
    *   **Sampling**: If the full 100-task run exceeds 6 hours, the plan will implement a **stratified sampling** strategy (e.g., 5 tasks per persona per model) to ensure completion within the time limit while maintaining statistical validity.
    *   **Memory**: Data will be processed in batches. The mock environment will not load large datasets into memory.
    *   **Reference**: `FR-003`, `SC-004`, `SC-005`.

## Decision / Rationale

| Decision | Rationale |
| :--- | :--- |
| **Use Vendored `Pi-Bench` Codebase** | The spec requires reproduction of existing code. Modifying it is avoided unless necessary for CPU enforcement or metric addition. |
| **Introduce Sequence Novelty Index (SNI)** | Directly addresses the Turing-simulated reviewer's critique that "proactivity" might be complex retrieval, using a population-based comparison rather than single-trajectory entropy. |
| **Set-Based Proactivity Score** | Prevents false negatives by comparing against a set of valid baselines rather than a single arbitrary path. |
| **CPU-Only Execution** | Mandatory for "fit the box" constraint. GPU dependencies would cause CI failure. |
| **Mock Environment** | Ensures deterministic, reproducible results without external API dependencies (Gmail, Amazon). |
| **Bootstrapping Strategy** | Mitigates the low power of small sample sizes (N~100) by avoiding parametric p-values and using confidence intervals. |
| **Associational Claims Only** | The study is observational; causal claims would be methodologically invalid. |

## Constitution Check

*   **Reproducibility**: The plan uses the vendored codebase and defines exact metrics.
*   **Compute Feasibility**: CPU-only, sampled data, 6-hour limit.
*   **Methodological Rigor**: Addresses retrieval vs. planning, low sample size (via bootstrapping), and causal assumptions.
*   **Data Integrity**: Mock environment ensures deterministic inputs.
