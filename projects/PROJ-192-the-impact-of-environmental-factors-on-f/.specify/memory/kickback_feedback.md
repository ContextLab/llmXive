## Unresolved concerns

- The plan states it will 'exclude any study missing a required variable', but doesn't detail *how* this exclusion impacts downstream analyses.  Specifically, will the analysis be performed on the remaining studies, or will the entire project be halted? The plan needs to explicitly state the consequence of failing the dataset availability check. This impacts causal validity – if analyses proceed with incomplete datasets, conclusions about the importance of missing variables are impossible.
- The plan states 'fallback to global median if MICE fails'. While pragmatic, this is a potentially strong assumption. The justification for using the global median *instead* of, for example, excluding the variable or using a more sophisticated imputation method, needs to be provided. This impacts construct validity – the imputed values may not accurately reflect the true underlying values, leading to biased results.
- The plan states 'Skip strata with insufficient samples, logging an ERROR'. While logging an error is good practice, the threshold of '≥ 10 samples' is arbitrary and needs justification. A power analysis, even a rough estimate, should be included to support this sample size threshold. Without justification, the statistical power of the biome-specific analyses is questionable.
- FR-001 (ingest data) is addressed in Phase 1 and Phase 2 of the plan, but the plan does not explicitly state how the system handles the case where a required variable is missing from a dataset, beyond logging a warning. The spec explicitly states the system MUST exclude the dataset. This is a missing requirement in the plan.

- The plan mentions the 'Dataset-Variable Fit' edge case, but the implementation in Phase 1 does not explicitly state that the dataset will be *excluded* if a required variable is missing, only that it will be logged. This is a discrepancy between the spec and the plan.
- The plan explicitly acknowledges a 'fatal data-fit gap' regarding ITS dataset availability, yet proceeds with the plan as if a dataset *will* be available. This is a critical flaw. The entire analysis hinges on having a suitable ITS dataset, and the plan needs to be revised to reflect a contingency plan if a verified dataset cannot be obtained. The current plan assumes a dataset will materialize, which is not a valid assumption given the stated lack of verified sources.
- The 'Dataset Strategy' section states 'The workflow will exclude this step until a verified source is supplied'. However, the plan.md still details steps that *require* this dataset. This is a contradiction. The plan needs to be adjusted to reflect the potential for the analysis to be significantly curtailed or impossible if a suitable dataset is not found.
- The 'Dataset Strategy' section lists example datasets (HuggingFace) for RDA and MICE imputation. These are *not* ITS datasets and are only for testing. This is misleading and should be clarified to avoid confusion. The section should explicitly state these are for testing purposes only and do not address the core data availability issue.
- Phase 1 states 'Validate presence of required columns (`pH`, `N`, `P`, `K`, `temperature`, `moisture`, `biome`)'. The data-model.md schema defines 'nitrogen', 'phosphorus', and 'potassium', but the plan uses 'N', 'P', and 'K'. This is a minor inconsistency that should be resolved for clarity.
- Phase 5 mentions 'VIF > 5' as a threshold for flagging collinearity. This value isn't explicitly defined in the spec or data-model. While reasonable, it should be explicitly stated in the spec for full traceability.
- The 'Constitution Check' table lists all principles, which is good. However, it lacks specific examples of *how* each principle is checked within the plan. Adding a brief example for each (e.g., 'Principle I: Verified by checksums of all data files') would strengthen the demonstration of constitutional compliance.
- Phase 10 mentions 'sampling_report.csv' but this file isn't listed in the 'Project Structure' section. Clarify where this report is saved or remove the reference if it's only a temporary log file.
- The plan states 'Apply MICE (via `miceforest`) per study; fallback to global median if MICE fails'. Using the global median as a fallback *after* MICE failure introduces a bias. MICE attempts to preserve multivariate relationships; falling back to a single global median ignores all study-specific information and will artificially reduce variance. This is a scientifically problematic approach. A better fallback would be to exclude samples with missing data after MICE fails, or to use a more sophisticated imputation method that considers study-level information.
- The plan states 'Apply family-wise error correction (Bonferroni) because multiple predictors are tested.' While correcting for multiple comparisons is good practice, Bonferroni is a very conservative correction. Given the potential for ecological correlations between predictors, a less conservative method (e.g., Benjamini-Hochberg FDR control) might be more appropriate to avoid increasing the risk of Type II errors. The choice of correction method should be justified.
