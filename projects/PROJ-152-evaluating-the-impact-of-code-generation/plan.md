# Project Plan: Evaluating the Impact of Code Generation Models on Code Security

## Summary

This project evaluates the security implications of code generated by large language models (LLMs) by comparing three state-of-the-art models: StarCoder-Base (7B), CodeGen (2B), and GPT-NeoX (1.3B).

**Amended Scope (N=30 Prompts):**
Due to strict resource constraints (GitHub Actions environment limited to ~7GB RAM and a 6-hour execution window), the original scope has been amended from a larger dataset to a focused, statistically viable subset:
- **Total Prompts (N=30):**
 - 10 prompts from the CodeXGLUE security subset.
 - 20 handcrafted prompts covering database access, HTML rendering, authentication, and injection scenarios.
- **Total Snippets Generated (N=90):** Each prompt is processed by all 3 models (30 prompts × 3 models = 90 snippets).

This scope ensures the pipeline can complete within the allocated compute budget while maintaining sufficient data density for statistical analysis (Kruskal-Wallis, ZINB).

## Objectives

1. Generate code snippets using three distinct LLMs under identical prompt conditions.
2. Analyze generated code for security vulnerabilities using static analysis tools (Bandit, Semgrep, CodeQL).
3. Compute vulnerability density (V/100LOC) and severity metrics mapped to NIST standards.
4. Perform statistical comparisons to determine if model choice significantly impacts security outcomes.
5. Validate findings through sensitivity analysis and calibration against expert labels.

## Methodology

### Data Collection
- **Source A (CodeXGLUE):** Download and filter for security-related prompts (SQL, XSS, auth). Select top 10 by relevance.
- **Source B (Handcrafted):** Create 20 domain-specific prompts targeting common web vulnerabilities.
- **Consolidation:** Merge into a single manifest with source attribution and checksums.

### Generation
- Load models using 4-bit quantization (CPU-compatible) to fit memory constraints.
- Apply a 120-second timeout per generation to prevent hanging.
- Log all failures to `data/failures.log`.

### Analysis
- Run Bandit (Python), Semgrep (multi-language), and CodeQL (Java/JS) on generated snippets.
- Map raw scanner findings to NIST-based ordinal severity ranks.
- Calculate False Positive Rates (FPR) via manual calibration on a stratified subset (10 snippets/model).

### Statistical Evaluation
- **Primary Test:** Kruskal-Wallis H-test on V/100LOC across models.
- **Post-Hoc:** Dunn's test with Bonferroni correction if significant.
- **Robustness:** Zero-Inflated Negative Binomial (ZINB) regression if zero-vulnerability rate ≥ 20%.
- **Sensitivity:** Sweep severity thresholds to assess risk categorization stability.

## Resource Constraints & Justification

- **Memory:** ~7GB RAM limit on CI runners necessitates 4-bit quantization and small batch sizes.
- **Time:** 6-hour limit requires efficient model loading/unloading and strict timeouts.
- **Dataset Size:** N=30 prompts is the maximum feasible size to generate, analyze, and calibrate within these constraints without exceeding time or memory limits.

## Deliverables

2. **Calibration:** `data/calibration/fpr_results.csv`.
3. **Results:** `data/results/descriptive_stats.csv`, `data/results/statistical_summary.csv`, `data/results/sensitivity_analysis.csv`.
4. **Visualizations:** Box-plots of vulnerability density, heat-maps of CWE distribution.
5. **Reports:** `data/results/run_summary.csv` and final research summary.

## Risks & Mitigations

- **Risk:** Model generation timeout or OOM.
 - **Mitigation:** Strict 120s timeout, 4-bit quantization, and CPU-only execution.
- **Risk:** Scanner false positives skewing results.
 - **Mitigation:** Manual calibration step (T022) to estimate FPR; FPR used for reporting only, not algorithmic correction of counts.
- **Risk:** Insufficient data for statistical power.
 - **Mitigation:** N=90 snippets provides adequate density for non-parametric tests; ZINB handles zero-inflation.
