### Dataset-Variable Fit Justification
The raw CodeXGLUE `codexglue_code_to_text` dataset is primarily designed for general code-to-text tasks and often lacks the specific semantic context required to trigger web-application security vulnerabilities (e.g., SQL injection, XSS). To address this, the plan implements a **filtering and augmentation strategy**:
1.  **Filtering**: The raw dataset is programmatically filtered for prompts containing keywords indicative of security-sensitive operations (e.g., "SQL", "auth", "XSS", "injection", "sanitize", "password").
2.  **Augmentation**: The filtered set is supplemented with a set of handcrafted prompts specifically designed to target web-security patterns.
3.  **Result**: This hybrid approach ensures that the predictor (prompt) contains sufficient context to elicit the outcome (vulnerability findings), satisfying the dataset-variable fit requirement.

## Statistical Methodology & Rigor

### 1. Metric Definition
- **Vulnerability Density (V/100LOC)**: `(Number of Findings / Lines of Code) * 100`. Controls for snippet length.
  - **Correction**: Raw counts are adjusted by subtracting the expected False Positive Rate (FPR) derived from the manual audit (Task 1.3). `Corrected_Count = Raw_Count - (Raw_Count * FPR)`.
- **Mean Ordinal Severity**: Mapped a scale from low to critical based on NIST NVD mappings from raw scanner labels.
  - *Validity Note*: This is a **proxy for scanner agreement** rather than absolute CVSS score. Absolute CVSS equivalence is not claimed (FR-004b).
  - *Validation*: Inter-Rater Reliability (Kappa) is calculated against manual expert review of a stratified subset of snippets per model. If Kappa < 0.6, the mapping is refined. The metric is explicitly labeled "Scanner Severity Proxy" in all reports.

### 2. Hypothesis Testing
- **Omnibus Test**: Kruskal-Wallis H-test (non-parametric ANOVA) to detect differences in FPR-corrected V/100LOC distributions across the three models.
  - *Rationale*: Vulnerability counts are typically non-normal and skewed.
- **Post-Hoc**: If $p < 0.05$, Dunn's test with **Bonferroni correction** is applied.
  - *Correction*: $\alpha_{adj} = 0.05 / 3 \approx 0.0167$ for 3 pairwise comparisons.
- **Robustness Check (Zero-Inflation)**: **ZINB regression is pre-registered as the default model** for count data, regardless of the observed zero-inflation percentage. This avoids data-dependent model selection (p-hacking risk).
  - *Offset*: Log(Lines of Code) to control for snippet length.
  - *Rationale*: Standard models underestimate variance in zero-heavy data.

### 3. Sensitivity Analysis
- **High-Severity Cutoff**: Sweep thresholds at ordinal ranks {3, 4, 5} to report the proportion of "high risk" snippets. This addresses FR-009.

### 4. Causal & Confounding Controls
- **Associational Framing**: Results are framed as "Model Identity correlates with Vulnerability Density" (not causal), as prompts are not randomly assigned to models in a randomized trial (Assumption 10).
- **Collinearity**: Prompt categories are treated as descriptive strata. No claims of independent effects between correlated categories are made (Assumption 12).
- **Length Control**: Snippet length (LOC) is included as an offset/covariate in all models (FR-005c).
- **Quantization Control**: **All three models (7B, 2B, 1.3B) are loaded with 4-bit quantization** to control for the quantization confounder. This ensures the comparison is between model architectures, not quantization artifacts.

### 5. Manual Calibration (Ground Truth)
- A stratified subset of 10 snippets per model (N=30 total) is manually reviewed by two independent experts.
- Experts label vulnerabilities as "True Positive", "False Positive", or "Unclear".
- **False Positive Rate (FPR)** is calculated per scanner/model pair: `FPR = False Positives / (True Positives + False Positives)`.
- This FPR is used to correct the raw vulnerability counts in the main analysis (Task 1.5).
- Inter-Rater Reliability (Cohen's Kappa) is calculated to validate the expert consensus.

## Compute Feasibility & Constraints

- **Hardware**: GitHub Actions Free Tier (CPU, ample RAM).
- **Memory Management**:
  - Models loaded sequentially (not concurrently) to stay within 7 GB RAM.
  - **All models use 4-bit quantization** to fit and control for confounders.
  - **Scanner Resource Budget**: CodeQL is heavy. We enforce a configurable timeout per scan and run scanners sequentially (Bandit -> Semgrep -> CodeQL) to prevent RAM exhaustion.
- **Time Budget**:
  - 30 prompts × 3 models = 90 generations.
  - **Estimated Runtime**:

## Decision Log

| Decision | Rationale |
|----------|-----------|
| **Use Kruskal-Wallis over ANOVA** | Vulnerability data is count-based, skewed, and non-normal. Non-parametric test is robust. |
| **Use Bonferroni over Holm** | FR-005 explicitly mandates Bonferroni for simplicity and strict family-wise error control. |
| **4-bit Quantization (All Models)** | Required to fit 7B model on 7 GB RAM CPU runner. Applied to ALL models to control for quantization confounder. |
| **Sequential Model Loading** | Prevents RAM exhaustion. Models are loaded, used for 30 prompts, then unloaded before next model. |
| **ZINB as Default** | Pre-registered to avoid p-hacking. Zero-inflation is expected in security scans. |
| **Manual Calibration** | Required to validate scanner outputs and establish ground truth for "security" construct. |
| **Reduced Prompt Count (30)** | Required to fit 6h CI budget on 7GB RAM with CodeQL. A larger volume would cause OOM/timeout. |
| **Prompt Filtering/Augmentation** | Raw CodeXGLUE lacks specific security context; filtering and handcrafted augmentation ensure dataset-variable fit for the research question. |
| **FPR Correction** | Required to mitigate "Scanner Noise" and improve construct validity (scientific_soundness-b56981d8). |
