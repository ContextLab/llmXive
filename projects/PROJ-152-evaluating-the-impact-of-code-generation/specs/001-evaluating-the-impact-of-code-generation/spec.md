# Specification: Evaluating the Impact of Code Generation Models on Code Security

## 1. Introduction

### 1.1 Purpose
This document defines the requirements for evaluating the security implications of code generated by Large Language Models (LLMs). The study compares three open-weight models (StarCoder-Base, CodeGen, GPT-NeoX) across a standardized set of security-focused prompts.

### 1.2 Scope
The project generates code snippets from security-oriented prompts, runs static analysis tools (Bandit, Semgrep, CodeQL), and performs statistical analysis on vulnerability density and severity.

## 2. Functional Requirements

### 2.1 Data Collection (FR-001)
- The system shall retrieve prompts from the CodeXGLUE dataset.
- The system shall supplement with handcrafted prompts targeting specific web security categories (SQLi, XSS, Auth, Injection).

### 2.2 Prompt Generation Scope (FR-002) - **AMENDED**
- **Original Requirement**: The system shall process a large corpus of prompts (N > 100) to ensure statistical power.
- **Amended Requirement**: Due to resource constraints (GitHub Actions runner limits: 7GB RAM, 6-hour execution window), the scope is reduced to **N=30 prompts** total:
 - **10 prompts** from the CodeXGLUE security subset (selected for high relevance).
 - **20 handcrafted prompts** (5 each for: Database Access, HTML Rendering, Authentication, Injection).
- **Output**: This results in **N=90 code snippets** (30 prompts × 3 models).
- **Justification**: Full-scale generation exceeds the available compute budget. The N=30 subset allows for a pilot study that fits within the 6h/7GB constraint while maintaining diverse category coverage.

### 2.3 Code Generation (FR-003)
- The system shall generate code using StarCoder-Base (7B), CodeGen (2B), and GPT-NeoX (1.3B).
- Generation must use 4-bit quantization to fit within memory limits.
- Generation timeout is set to 120 seconds per snippet.

### 2.4 Vulnerability Analysis (FR-004)
- The system shall run Bandit (Python), Semgrep (multi-language), and CodeQL (Java/JS) on generated snippets.
- Findings must be mapped to CWE IDs and severity levels.

### 2.5 Metrics Calculation (FR-005)
- Calculate Vulnerabilities per 100 Lines of Code (V/100LOC).
- Calculate mean severity scores.

## 3. Non-Functional Requirements

### 3.1 Performance Constraints (SC-005) - **AMENDED**
- **Original Constraint**: The system must complete generation and analysis within 24 hours on a standard GPU cluster.
- **Amended Constraint**: The entire pipeline (generation + analysis) must complete within **6 hours** on a **CPU-only environment** with **7GB RAM** (GitHub Actions default runner).
- **Implication**: Model loading must use 4-bit quantization. Batch sizes must be 1. The prompt count is limited to 30 to satisfy this constraint.

### 3.2 Reproducibility
- All random seeds must be pinned.
- All artifacts must be versioned with checksums in `state.yaml`.

## 4. Data Model

- **Prompts**: JSON manifest with source attribution.
- **Snippets**: CSV with code, model, prompt_id.
- **Findings**: CSV with scanner output, mapped severity.
- **Results**: CSV with statistical test outputs.

## 5. Limitations

- The study is limited to 3 open-weight models.
- The prompt set is small (N=30), limiting generalizability but sufficient for pilot validation.
- Static analysis tools may produce false positives; a manual calibration step (N=30 snippets) is included to estimate False Positive Rates (FPR) for reporting purposes only.
- No dynamic analysis is performed.
