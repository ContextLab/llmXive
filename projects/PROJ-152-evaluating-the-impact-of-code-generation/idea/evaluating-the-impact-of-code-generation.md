# Evaluating the Impact of Code Generation Models on Code Security  

**Field**: computer science  

## Research question  

How does the prevalence and severity of common security vulnerabilities (e.g., SQL injection, cross‑site scripting, insecure deserialization) differ across code generated by contemporary large‑language‑model code generators when answering identical natural‑language prompts for typical web‑application tasks?  

## Motivation  

Automated code generation promises faster development, yet insecure code can propagate bugs at scale. Existing evaluations focus on functional correctness, leaving a gap in systematic, quantitative evidence about security risks. Understanding these risks will guide safer model design, prompting, and downstream static‑analysis integration.  

## Related work  

- **Security Degradation in Iterative AI Code Generation – A Systematic Analysis of the Paradox (2025)** – <http://arxiv.org/abs/2506.11022v2> — Directly studies how security flaws evolve in LLM‑generated code across iterative generations.  
- **EVOR: Evolving Retrieval for Code Generation (2024)** – <http://arxiv.org/abs/2402.12317v2> — Introduces retrieval‑augmented pipelines that could affect vulnerability patterns in generated snippets.  
- **JaCoText: A Pretrained Model for Java Code‑Text Generation (2023)** – <http://arxiv.org/abs/2303.12869v1> — Presents a Java‑focused generation model; useful as a baseline for language‑specific security analysis.  
- **Non‑Parametric Estimation of a Multivariate Probability Density (1969)** – <https://doi.org/10.1137/1114019> — Methodological reference for density estimation; not directly related to code security but can inform statistical modeling of vulnerability scores.  
- **A survey on Image Data Augmentation for Deep Learning (2019)** – <https://doi.org/10.1186/s40537-019-0197-0> — General survey of augmentation techniques; included for completeness though unrelated to the core topic.  

## Expected results  

- Quantitative vulnerability profiles (vulnerabilities per 100 lines of code, mean CVSS severity) for each model.  
- Statistically significant differences (p < 0.05) in vulnerability density across models, revealed by non‑parametric tests (Kruskal‑Wallis).  
- Identification of specific prompt categories (e.g., database access, HTML rendering) that consistently trigger higher‑risk code, suggesting targeted mitigation strategies.  

## Methodology sketch  

1. **Select models** – download and run three publicly available code generators that fit within a 7 GB RAM limit:  
   - StarCoder‑Base (7B) – <https://huggingface.co/bigcode/starcoderbase>  
   - CodeGen‑2B (2B) – <https://huggingface.co/Salesforce/codegen-2b>  
   - GPT‑Neo‑X‑1.3B (open‑source) – <https://huggingface.co/EleutherAI/gpt-neox-1.3B>  

2. **Construct prompt set** – combine:  
   - 200 prompts from the CodeXGLUE “Code Generation” benchmark (<https://github.com/microsoft/CodeXGLUE/tree/main/Code-Generation>)  
   - 50 handcrafted prompts that request typical web‑app functions (e.g., “Create a PHP function that inserts user input into a MySQL query”).  

3. **Generate code** – for each model, run inference on all prompts using the `transformers` library (batch size = 1, max tokens = 256). Store outputs in a CSV (model, prompt, generated code).  

4. **Static analysis** – pipe each snippet through three open‑source scanners:  
   - Bandit (Python) – <https://github.com/PyCQA/bandit>  
   - Semgrep – <https://github.com/semgrep/semgrep> (use the “security‑best‑practices” rule set)  
   - CodeQL (Java/JavaScript) – <https://github.com/github/codeql>  

   Record number of findings, CWE identifiers, and CVSS‑like severity scores.  

5. **Optional dynamic testing** – for compiled languages (C/C++, Go) compile the generated code and run a lightweight fuzzing harness using AFL++ (<https://github.com/AFLplusplus/AFLplusplus>) for 15 minutes per snippet; log crashes and sanitizer warnings.  

6. **Normalize metrics** – compute vulnerabilities per 100 lines of code (V/100LOC) and average severity per model.  

7. **Statistical comparison** – apply the Kruskal‑Wallis test to V/100LOC across models; if significant, follow with Dunn’s post‑hoc test with Bonferroni correction.  

8. **Visualization & reporting** – generate box‑plots of vulnerability density, heat‑maps of CWE distribution per prompt category, and a summary table of statistical results (Python `matplotlib`/`seaborn`).  

All steps rely on publicly downloadable resources and fit within a ≤6‑hour GitHub Actions job (data ≈ 250 MB, model loading ≈ 4 GB, analysis ≈ 2 GB).
