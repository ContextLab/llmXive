# Research: Assessing the Trade-offs Between Static and Dynamic Analysis for LLM-Generated Code

## Overview

This research investigates the trade-offs between static analysis (targeting security vulnerabilities) and dynamic analysis (targeting functional correctness) for code generated by Large Language Models (LLMs). The study leverages established code generation benchmarks to evaluate detection rates, pass rates, and computational costs.

**Critical Methodological Note**:
- The study acknowledges that **ground truth for security vulnerabilities is NOT available**. Therefore, **Precision, Recall, and F1-score cannot be calculated** for static analysis. The static metric is redefined as **Issue Detection Rate** (raw count of issues found per snippet).
- The study acknowledges that static analysis (security) and dynamic analysis (correctness) measure **distinct constructs**. Therefore, **McNemar's test is inappropriate** for comparing them. The study uses **Spearman's Rank Correlation** or **Chi-squared Test of Independence** to assess the association between static issue density and dynamic pass/fail status.

**Dataset Fit Assessment**:
- **HumanEval**: Contains code and unit tests. Matches requirement for functional correctness oracle. **No security ground truth** available (assumption in spec).
- **HumanEval-X (Java/JS)**: Contains code and unit tests. Ensures coverage of ≥3 languages (Python, JS, Java) with paired oracles.
- **CodeXGLUE**: Contains code and unit tests. Used to supplement language coverage if HumanEval-X is insufficient.
- **TheStack (BigCode)**: Contains code but **no unit tests**. Used for **Static-Only** analysis to satisfy FR-001's mandate to include BigCode. Excluded from comparative statistical tests (which require paired dynamic oracles).

**Sampling Strategy**:
- Stratified random sampling by language (Python, JavaScript, Java) to ensure ≥3 languages (SC-003).
- Target sample size: ≥500 snippets (FR-001).
- **Stratification Logic**:
 - If a language stratum has n < 30, that stratum is **excluded** from the stratified statistical test and reported as a limitation.
 - The study will proceed with available languages (minimum 1) and report the limitation.

## Methodology

### 1. Data Ingestion (FR-001)
- Download datasets from verified HuggingFace URLs.
- Verify checksums; abort on corruption.
- Parse snippets; extract code and test cases.
- Filter for Python, JavaScript, Java.
- **Stratification Validation**: Count samples per language. If n < 30, flag for exclusion.
- **BigCode Handling**: TheStack samples are marked `static_only` and excluded from dynamic analysis.

### 2. Static Analysis (FR-002)
- **Primary Tools**: CodeQL (for security queries), SonarQube (for code smells/vulnerabilities).
- **Fallback**: PyLint (Python), ESLint (JavaScript) if primary tools fail.
- **Execution**: Run on each snippet; log results in `analysis_log.schema.yaml` format.
- **Timeout**: 15 minutes per snippet (aggregate limit).
- **Resource Limit**: Abort if CPU > 2 cores or RAM > 7 GB.
- **Output**: **Issue Detection Rate** (issues per snippet).

### 3. Dynamic Analysis (FR-003)
- **Oracle**: Unit tests from HumanEval/HumanEval-X/CodeXGLUE (pass/fail).
- **Execution**: Run tests via `pytest` (Python) or equivalent for JS/Java.
- **Handling Missing Tests**: Mark as `untestable_dynamic`; **exclude from comparative metrics**.
- **Timeout**: 30 minutes per snippet.
- **Output**: **Pass Rate** (pass/fail status).

### 4. Aggregation & Statistics (FR-004, FR-005, FR-007, FR-008 - Revised)
- **Metrics**:
 - Static: **Issue Detection Rate** (count of issues / total snippets).
 - Dynamic: **Pass Rate** (count of passes / total snippets with oracle).
 - **No Precision/Recall/F1** (ground truth unavailable).
- **Statistical Test**:
 - **Spearman's Rank Correlation** or **Chi-squared Test of Independence** between static issue count and dynamic pass/fail status.
 - **No McNemar's Test** (inappropriate for distinct constructs).
 - Stratify by language if n ≥ 30 per stratum.
- **Sensitivity Analysis**: Sweep α ∈ {0.01, 0.05, 0.1} (FR-007).
- **Multiplicity Adjustment**: Apply Bonferroni correction to **independent statistical tests** (e.g., across languages), not derived metrics (FR-008).

## Statistical Rigor & Limitations

- **Multiple Comparisons**: Bonferroni correction applied to independent statistical tests (e.g., testing correlation across languages).
- **Sample Size**: ≥500 snippets targeted; power analysis deferred to implementation. If n < 30 per language, stratification is skipped and limitation reported.
- **Causal Claims**: Observational study; claims framed as associational. No randomization; confounding variables (e.g., language, code complexity) noted.
- **Measurement Validity**:
 - HumanEval/HumanEval-X tests are validated for functional correctness.
 - Static tools (CodeQL/SonarQube) are industry-standard but may have false positives/negatives.
- **Collinearity**: Static and dynamic analysis are independent modalities; no definitional overlap.
- **Ground Truth Limitation**: Security ground truth is unavailable. Static analysis results are reported as "Issue Detection Rate" (raw count), not Precision/Recall.

## Compute Feasibility

- **Hardware**: GitHub Actions Free Tier (2 CPU, 7 GB RAM, 14 GB disk, no GPU).
- **Methods**:
 - Static analysis: CodeQL CLI (CPU-only), SonarQube (local scanner).
 - Dynamic analysis: `pytest` (CPU-only).
 - Statistics: `scipy.stats` (lightweight).
- **Optimization**:
 - Sample data to fit RAM; process snippets sequentially or in small batches.
 - Timeout enforcement to prevent runaway processes.
 - No GPU/CUDA dependencies; no large model inference.

## Decisions & Rationale

| Decision | Rationale |
|----------|-----------|
| **Use HumanEval + HumanEval-X + CodeXGLUE** | Only verified sources with code + tests for ≥3 languages. |
| **TheStack for Static-Only** | Satisfies FR-001 mandate for BigCode; excluded from comparative tests due to lack of oracles. |
| **Fallback static analyzers** | Ensures pipeline robustness if CodeQL/SonarQube fail (FR-002). |
| **Exclude `untestable_dynamic` snippets** | Prevents bias in functional correctness metrics; aligns with spec. |
| **Bonferroni correction on independent tests** | Conservative control of Type I error for multiple tests (FR-008). |
| **Spearman/Chi-squared instead of McNemar** | Appropriate for assessing association between distinct constructs (security vs. correctness). |
| **No Java dataset URL fabrication** | Use HumanEval-X/CodeXGLUE; if insufficient, report limitation. |

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| **Insufficient Java samples** | Use all available; if n < 30, exclude from stratified test and report limitation. |
| **Static tool crashes** | Retry up to 3 times; fallback to PyLint/ESLint; log `tool_failure`. |
| **Timeout during dynamic execution** | Mark as `timeout`; exclude from runtime metrics. |
| **Resource exhaustion** | Enforce CPU/RAM limits; abort with error code. |
| **No security ground truth** | Use "Issue Detection Rate" instead of Precision/Recall; frame findings as associational. |
| **BigCode lack of tests** | Use for static-only reporting; exclude from comparative statistical test. |
| **Spec contradiction** | The spec mandates Precision/Recall and McNemar's test, which are scientifically invalid for this study. The plan explicitly rejects these and uses valid alternatives. **Spec update required.** |
