# Assessing the Trade-offs Between Static and Dynamic Analysis for LLM-Generated Code

**Field**: computer science

## Research question

How do static analysis tools and dynamic testing compare in detecting security vulnerabilities and functional errors in code generated by large language models, and what is the optimal combination of both approaches for efficient quality assurance?

## Motivation

LLM-generated code is increasingly used in production environments, yet traditional quality assurance methods may not be optimally effective for this novel code source. Understanding the relative strengths and limitations of static versus dynamic analysis for LLM output can guide practitioners in allocating limited testing resources and designing more robust code review pipelines for AI-assisted development.

## Related work

- [Evaluating the Effectiveness of Static Analysis Tools on AI-Generated Code](https://arxiv.org/abs/2305.14782) — Chen et al. (2023) find significant gaps in vulnerability detection when static tools are applied to LLM-generated code compared to human-written code.
- [Dynamic Testing Strategies for LLM-Generated Software](https://dl.acm.org/doi/10.1145/3589334.3645678) — Smith et al. (2024) propose novel dynamic testing frameworks specifically designed for evaluating code output from language models.
- [CodeBERT and Beyond: Static Analysis for Pre-trained Code Models](https://arxiv.org/abs/2302.09847) — Feng et al. (2023) investigate how pre-trained code models affect traditional static analysis pipelines and suggest hybrid approaches.
- [Security Vulnerabilities in LLM-Generated Code: A Systematic Review](https://ieeexplore.ieee.org/document/10234567) — Garcia et al. (2024) catalog common vulnerability patterns and detection challenges in LLM-generated code.
- [Comparative Analysis of Static vs Dynamic Testing for Software Reliability](https://link.springer.com/article/10.1007/s10664-022-10234-5) — Anderson et al. (2022) provide baseline metrics for defect detection rates across different analysis paradigms.
- [HumanEval and Beyond: Benchmarking Code Generation Quality](https://arxiv.org/abs/2304.07890) — Chen et al. (2023) extend HumanEval with additional test suites for evaluating functional correctness across multiple languages.
- [Automated Test Generation for Code Produced by Language Models](https://arxiv.org/abs/2401.03456) — Williams et al. (2024) present an automated framework for generating test cases targeting LLM code weaknesses.
- [The Cost of Quality: Trade-offs in Code Analysis Approaches](https://dl.acm.org/doi/10.1145/3567890.3567901) — Davis et al. (2023) measure time, computational resources, and defect detection effectiveness across analysis methods.

## Expected results

We expect to find that static analysis detects more style and simple security issues quickly, while dynamic testing reveals more functional errors and runtime vulnerabilities. The combined approach should achieve 20-30% higher defect detection rates than either method alone, with diminishing returns beyond a certain test coverage threshold. Statistical significance will be assessed using McNemar's test on paired detection outcomes across code samples.

## Methodology sketch

- **Data acquisition**: Download LLM-generated code from HumanEval (https://github.com/openai/human-eval) and CodeXGLUE (https://arxiv.org/abs/2102.04664), plus additional samples from BigCode (https://huggingface.co/bigcode); total ~500 code snippets across Python, JavaScript, and Java.
- **Static analysis pipeline**: Run CodeQL (https://codeql.github.com/) and SonarQube open-source scanner on all code samples; capture vulnerability detections and false positive rates using Defects4J (https://github.com/rjust/defects4j) as ground truth where applicable.
- **Dynamic analysis pipeline**: Execute existing unit tests from HumanEval benchmarks; for code without tests, use automated test generation via the framework from Williams et al. (2024) to create minimal test suites.
- **Ground truth validation**: Manually verify a stratified random sample of 50 code snippets (10% of dataset) by expert review to establish true vulnerability/error counts.
- **Detection rate comparison**: Calculate precision, recall, and F1-score for static vs dynamic analysis; use McNemar's test (α=0.05) to assess statistical significance of differences.
- **Cost analysis**: Measure wall-clock time and CPU/memory usage for each analysis method; compute cost-effectiveness ratio (defects detected per minute of analysis).
- **Hybrid approach evaluation**: Test sequential application (static first, then dynamic on flagged code) to identify optimal workflow; compare against parallel and single-method baselines.
- **Visualization**: Generate ROC curves and cost-benefit plots using matplotlib; produce summary tables for each programming language subset.
- **Reproducibility package**: Create Dockerfile and shell scripts to reproduce all analysis steps; publish code and processed data on Zenodo with DOI.
