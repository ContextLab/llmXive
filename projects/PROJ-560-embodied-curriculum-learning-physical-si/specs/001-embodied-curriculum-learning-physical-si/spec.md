# Feature Specification: Embodied Curriculum Learning: Physical Simulation for Abstract Concept Teaching

**Feature Branch**: `001-embodied-curriculum-learning`  
**Created**: 2026-06-02  
**Status**: Draft  
**Input**: User description: "Does embodied training through physics-based virtual manipulation improve transfer to abstract mathematical reasoning more than static visual instruction for adult learners?"

## User Scenarios & Testing

### User Story 1 - Data Processing and Synthetic Data Generation (Priority: P1)

The system must provide a deterministic, CPU-tractable environment that operates in two modes: (1) **Secondary Analysis Mode**, which loads and processes existing public educational datasets containing pre-test, post-test, and `instruction_type` metadata; and (2) **Synthetic Data Generation Mode**, which creates a controlled dataset with known ground truths for validating the analysis pipeline. The system does not "render" educational content for the primary research question but processes existing data or generates synthetic data for validation.

**Why this priority**: Without a functional, reproducible environment that can either process existing labeled data or generate synthetic data for validation, no analysis can occur. This is the minimum viable product (MVP) required to execute the research methodology and validate the statistical pipeline.

**Independent Test**: The system can be tested by either (a) loading a sample of the OpenML math reasoning dataset (if it contains `instruction_type`) and outputting a processed CSV file with gain scores, or (b) invoking the synthetic data generator to create a labeled dataset and outputting a valid CSV file with known ground truths, all without requiring external network calls or GPU resources.

**Acceptance Scenarios**:

1. **Given** a CSV file containing pre-test and post-test scores with an `instruction_type` column (values: "embodied" or "static"), **When** the simulation module is initialized with the `--mode=secondary_analysis` flag, **Then** the system outputs a processed CSV file where the `instruction_type` column is preserved and no runtime errors occur on a constrained RAM limit.
2. **Given** a configuration file requesting synthetic data generation, **When** the simulation module is initialized with the `--mode=synthetic` flag, **Then** the system outputs a valid CSV file with the `instruction_type` column, pre/post scores, and the same number of rows as requested, matching the `DatasetRecord` schema.
3. **Given** a dataset row with missing pre-test scores, **When** the processing pipeline runs, **Then** that row is excluded from the gain score calculation and logged in a `skipped_records.log` file with the reason "missing_pre_test".

### User Story 2 - Statistical Comparison and Inference (Priority: P2)

The system must perform independent samples t-tests comparing the mean gain scores between the "embodied" and "static" conditions (identified by the `instruction_type` column), explicitly framing results as associational (since this is a secondary analysis of existing data) and applying corrections for multiple comparisons if multiple concepts are tested. The system also supports testing on synthetic data generated by US-1 for pipeline validation.

**Why this priority**: This delivers the primary research answer: whether the embodied condition shows a statistically significant advantage. It directly addresses the research question.

**Independent Test**: The system can be tested by running the analysis script on a synthetic dataset (generated by US-1) where the "embodied" group has a known mean gain of 10.0 and the "static" group has a known mean gain of 5.0, verifying that the output reports a t-statistic and p-value consistent with these inputs.

**Acceptance Scenarios**:

1. **Given** a processed dataset with ≥ 30 participants in both "embodied" and "static" groups, **When** the analysis script is executed, **Then** the output JSON contains a `t_statistic`, `p_value`, and `effect_size_cohen_d` for the comparison of mean gain scores.
2. **Given** a dataset where the "embodied" and "static" groups are tested on N distinct mathematical concepts, **When** the analysis runs, **Then** the output includes a `corrected_p_value` using the Bonferroni method (α_adj = 0.05 / N) and flags the result as "associational" in the `inference_framing` field.
3. **Given** a dataset with unequal variances between groups (Levene's test p < 0.05), **When** the analysis runs, **Then** the system automatically switches to Welch's t-test and reports the adjusted degrees of freedom in the output.

### User Story 3 - Sensitivity Analysis for Thresholds (Priority: P3)

The system must execute a sensitivity analysis sweeping the inclusion threshold for "valid learning gains" (e.g., minimum pre-post difference) across a defined set (0.01, 0.05, 0.10) to demonstrate the robustness of the headline effect size, but only if the dataset size (N) ≥ 30.

**Why this priority**: This addresses the methodological soundness requirement for threshold justification. It ensures that the research conclusion is not an artifact of an arbitrary cutoff choice.

**Independent Test**: The system can be tested by running the analysis with the `--sweep` flag on a dataset with N ≥ 30 and verifying that the output contains a table of effect sizes corresponding to each threshold value.

**Acceptance Scenarios**:

1. **Given** a processed dataset with N ≥ 30, **When** the analysis is run with `--sweep_thresholds=0.01,0.05,0.10`, **Then** the output JSON contains a `sensitivity_analysis` array where each entry includes the `threshold`, `n_participants_retained`, and `effect_size_cohen_d`.
2. **Given** a dataset where the headline effect size is 0.6 at threshold 0.05, **When** the sweep is run, **Then** the output reports the effect size at threshold 0.01 and 0.10, allowing a user to verify if the effect holds across the range.
3. **Given** a scenario where the effect size drops below 0.2 at any threshold in the sweep, **When** the analysis completes, **Then** the `summary_report` includes a `robustness_warning` flag set to `true`.
4. **Given** a dataset with N < 30, **When** the analysis is run with `--sweep`, **Then** the system skips the sensitivity analysis and flags the result as "insufficient data for robustness check".

### Edge Cases

- **Dataset Variable Fit**: What happens if the public dataset lacks a specific covariate (e.g., prior math anxiety) required for a robust model? The system must detect missing columns and output a warning in the results log. If the critical `instruction_type` variable is missing, the system must first attempt to load public data; if `instruction_type` is absent, it must invoke the Synthetic Data Generator (FR-009) to create a labeled dataset for validation, rather than failing silently or imputing arbitrarily.
- **Small Sample Power**: How does the system handle a condition with N < 10? The system must calculate the achieved power for the observed effect size and flag the result as "underpowered" in the output, preventing over-interpretation of null results.
- **Collinearity**: If the dataset includes predictors that are definitionally related (e.g., "time spent" and "total interactions" where one is a function of the other), the system must detect high correlation (|r| > 0.8) and report a collinearity diagnostic rather than running a multivariate regression that implies independent effects.

## Requirements

### Functional Requirements

- **FR-001**: System MUST load and parse public educational datasets (CSV/JSON) containing pre-test, post-test, and `instruction_type` metadata to calculate gain scores (See US-1).
- **FR-002**: System MUST execute independent samples t-tests (Student's or Welch's) comparing mean gain scores between "embodied" and "static" conditions (identified by `instruction_type`), reporting t-statistic, p-value, and Cohen's d (See US-2).
- **FR-003**: System MUST frame all statistical findings as "associational" in the output report, explicitly stating that no causal inference is drawn due to the observational nature of the secondary data analysis (See US-2).
- **FR-004**: System MUST perform a Bonferroni correction for multiple comparisons when >1 hypothesis test is conducted across different mathematical concepts, adjusting the significance threshold α dynamically as α_adj = 0.05 / N_concepts (See US-2).
- **FR-005**: System MUST execute a sensitivity analysis sweeping the gain score inclusion threshold over the set {0.01, 0.05, 0.10} and report the resulting effect sizes and participant counts for each, but ONLY if the dataset size (N) ≥ 30; otherwise, it MUST skip the sweep and flag "insufficient data for robustness check" (See US-3).
- **FR-006**: System MUST detect and report high collinearity (|r| > 0.8) between any predictor variables in the dataset, preventing the interpretation of independent effects for such variables (See US-2).
- **FR-007**: System MUST calculate and report the achieved statistical power for the observed effect size given the sample size, flagging results as "underpowered" if power < 0.80 (See US-2).
- **FR-008**: System MUST validate that all required variables (pre-test, post-test, `instruction_type`) exist in the input dataset before proceeding; if `instruction_type` is missing in public data, the system MUST invoke the Synthetic Data Generator (FR-009) to create a labeled dataset for validation, failing with a clear error message only if synthetic generation also fails (See US-1).
- **FR-009**: System MUST generate synthetic datasets with known ground truths (pre/post scores, `instruction_type`) for the purpose of validating the analysis pipeline, including configurable mean differences and sample sizes (See US-1).

### Key Entities

- **DatasetRecord**: Represents a single participant's data, containing attributes: `pre_test_score`, `post_test_score`, `instruction_type` (values: "embodied" or "static"), and optional `covariates`.
- **AnalysisResult**: Represents the outcome of a statistical test, containing attributes: `t_statistic`, `p_value`, `effect_size_cohen_d`, `confidence_interval`, and `inference_framing`.
- **SensitivitySweep**: Represents the output of the threshold analysis, containing attributes: `threshold_value`, `n_participants`, `effect_size_cohen_d`, and `robustness_flag`.

## Success Criteria

- **SC-001**: The analysis pipeline successfully processes a sample of the OpenML math reasoning dataset (n≥30 per condition) or a synthetic dataset and outputs a valid JSON report conforming to the `AnalysisResult` schema within 10 minutes on a 2-core CPU (See FR-001, FR-002).
- **SC-002**: The output report explicitly labels all findings as "associational" and includes a Bonferroni-corrected p-value (α_adj = 0.05 / N_concepts) when multiple concepts are tested, ensuring methodological framing is correct (See FR-003, FR-004).
- **SC-003**: The sensitivity analysis section of the report includes results for thresholds 0.01, 0.05, and 0.10 (if N ≥ 30), demonstrating how the effect size varies across these values, and flags `robustness_warning` if the effect size drops below 0.2 (See FR-005).
- **SC-004**: The system detects and reports collinearity diagnostics for any predictor pairs with correlation |r| > 0.8, preventing misinterpretation of independent effects (See FR-006).
- **SC-005**: The power analysis calculation is included in the report, correctly flagging any condition with achieved power < 0.80 as "underpowered" (See FR-007).

## Assumptions

- **Computational Constraints**: The analysis is assumed to run entirely on CPU within the standard GitHub Actions time limit, utilizing lightweight Python libraries (pandas, scipy, statsmodels) without GPU acceleration or large model inference.
- **Variable Definitions**: It is assumed that "gain score" is defined as `post_test_score - pre_test_score` and that a positive gain indicates learning improvement.
- **Inference Framing**: It is assumed that the available datasets represent observational or quasi-experimental data where random assignment to "embodied" vs. "static" is not guaranteed, necessitating associational framing.
- **Threshold Justification**: The sensitivity thresholds {0.01, 0.05, 0.10} are assumed to be sufficient to demonstrate robustness; if the effect disappears at 0.01, it is considered unstable.
- **Sample Size**: It is assumed that the combined sample size from available public datasets will be ≥ 60 (30 per condition) to allow for meaningful statistical testing, though power may be limited.

## Risks

- **Dataset Availability**: It is a risk that public educational datasets (e.g., OpenML, UCI) may lack the necessary variables (specifically `instruction_type`) to reconstruct the "embodied" vs. "static" comparison. If this occurs, the system will fallback to synthetic data generation (FR-009) for pipeline validation, but the primary research question (secondary analysis of existing data) cannot be answered without pre-labeled data.
