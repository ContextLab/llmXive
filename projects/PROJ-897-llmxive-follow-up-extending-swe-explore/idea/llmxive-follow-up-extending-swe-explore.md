# llmXive follow-up: extending "SWE-Explore: Benchmarking How Coding Agents Explore Repositories"

## Summary of the prior work
SWE-Explore introduces a benchmark to isolate and evaluate the repository exploration capabilities of coding agents by asking them to rank relevant code regions under a fixed line budget, using ground truth derived from successful agent trajectories. The study demonstrates that agentic explorers significantly outperform classical retrieval methods and that metrics like line-level coverage and ranking efficiency strongly correlate with downstream bug-fix success. However, the current benchmark relies on a selection bias toward solvable issues and treats exploration as a static, one-shot retrieval task without modeling the iterative nature of real-world debugging.

## Proposed extension
**Research Question:** Does an iterative, feedback-driven exploration strategy (where the agent refines its search based on partial context execution or error signals) yield higher line-level coverage and repair success on *unsolvable* or *ambiguous* issues compared to the static, one-shot exploration evaluated in SWE-Explore?

This matters because the original benchmark explicitly excludes instances where current agents fail, yet real-world debugging often requires navigating ambiguous or difficult problems where initial context retrieval is insufficient. By shifting the focus to iterative exploration on the "hard tail" of the distribution, we can determine if the bottleneck is the initial retrieval quality or the agent's ability to dynamically adjust its exploration path, a capability not captured by the current static benchmark.

## Methodology sketch
**Data:** Reuse the 848 issues from SWE-Explore but specifically curate a subset of "hard" instances (e.g., those with low initial coverage scores or those where the original benchmark reported low resolution rates) and augment them with a synthetic set of ambiguous issues generated by mutating valid code to introduce non-deterministic behaviors.

**Procedure:** Implement a CPU-tractable evaluation harness where agents perform a multi-turn exploration loop: (1) propose a set of code regions, (2) simulate a lightweight execution or static analysis check on the returned context to generate a "feedback signal" (e.g., "missing dependency," "ambiguous variable"), and (3) update the query for the next turn until the line budget is exhausted. Compare the final coverage and ranking metrics against the original one-shot SWE-Explore baselines using the same ground truth derivation method.

**Expected Result:** We hypothesize that iterative agents will show a statistically significant improvement in line-level coverage and downstream repair potential on the "hard" subset compared to one-shot agents, particularly in cases where the initial issue description is vague, suggesting that dynamic feedback loops are the missing link for tackling complex repository exploration tasks.
