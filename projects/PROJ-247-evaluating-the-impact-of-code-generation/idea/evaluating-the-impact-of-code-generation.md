# Evaluating the Impact of Code Generation on Long-Term Code Maintainability

**Field**: computer science

## Research question

Does code generated by large language models exhibit different long-term maintainability characteristics compared to human-written code, as measured by code churn, bug fix latency, and structural complexity metrics over a six-month observation window?

## Motivation

As LLM-based code generation becomes ubiquitous in software development, understanding its impact on code quality is critical for engineering teams making adoption decisions. Current benchmarks focus on correctness and functionality at generation time, but there is a gap in empirical evidence about how LLM-generated code behaves under real-world maintenance conditions over months of evolution.

## Related work

- [SIMCOPILOT: Evaluating Large Language Models for Copilot-Style Code Generation (2025)](http://arxiv.org/abs/2505.21514v1) — Provides a benchmark framework for evaluating LLM copilot-style code generation, establishing metrics for code quality assessment.
- [Reinterpretation and Long-Term Preservation of Data and Code (2022)](http://arxiv.org/abs/2209.08054v1) — Discusses principles for long-term preservation and reinterpretation of code artifacts, relevant to maintainability measurement methodology.

## Expected results

We expect to find measurable differences in maintainability metrics between LLM-generated and human-written code, with LLM code showing either higher code churn or longer bug fix times. Statistical significance will be assessed using t-tests or Mann-Whitney U tests on matched code samples, with effect sizes reported to quantify practical significance.

## Methodology sketch

- Download 500+ public Python/JavaScript repositories from GitHub using the GitHub REST API (rate-limited, batched over multiple workflow runs if needed)
- Use an LLM code detection tool (e.g., CodeBERT-based classifier) to flag potentially LLM-generated code blocks in commit history
- Extract code complexity metrics (cyclomatic complexity, lines of code, nesting depth) using `radon` or `pylint` static analysis tools
- Track bug fix commits and time-to-fix intervals from GitHub Issues and Pull Request data for each flagged code block
- Compute code churn metrics (change frequency, edit distance) using `git log` parsing with `pygit2` library
- Match LLM-generated code blocks with comparable human-written blocks (same function type, similar LOC) for controlled comparison
- Perform statistical tests (Mann-Whitney U for non-normal distributions) comparing maintainability metrics between groups
- Generate visualization artifacts (box plots, density plots) using `matplotlib` and save as PNG files
- All data processing runs in ≤30-minute chunks to fit GHA 6h job limit; intermediate results cached via workflow artifacts
