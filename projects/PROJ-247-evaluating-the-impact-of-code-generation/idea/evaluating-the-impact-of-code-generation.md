# Evaluating the Impact of Code Generation on Long-Term Code Maintainability

**Field**: Computer Science

## Research question

Does code generated by large language models exhibit different long-term maintainability characteristics compared to human-written code, as measured by code churn, bug fix latency, and structural complexity metrics over a six-month observation window, when controlling for code complexity and project size at generation time?

## Motivation

As LLM-based code generation becomes ubiquitous in software development, understanding its impact on code quality is critical for engineering teams making adoption decisions. Current benchmarks focus on correctness and functionality at generation time, but there is a gap in empirical evidence about how LLM-generated code behaves under real-world maintenance conditions over months of evolution, specifically when accounting for initial complexity biases.

## Related work

- [AI-Generated Smells: An Analysis of Code and Architecture in LLM and Agent-Driven Development (2026)](https://arxiv.org/abs/2605.02741) — Establishes that LLM-generated code often contains specific architectural smells overlooked by functional correctness benchmarks.
- [SoHist: A Tool for Managing Technical Debt through Retro Perspective Code Analysis (2023)](https://arxiv.org/abs/2304.14464) — Provides methodology for tracking technical debt progression, relevant to measuring maintainability decay over time.
- [PHP code smells in web apps: survival and anomalies (2020)](https://arxiv.org/abs/2101.00090) — Demonstrates how code smells persist in production, supporting the hypothesis that initial generation quality impacts long-term evolution.

## Expected results

We expect to find measurable differences in maintainability metrics between LLM-generated and human-written code, potentially showing higher code churn or longer bug fix times for LLM code due to subtle architectural smells. Statistical significance will be assessed using Mann-Whitney U tests on matched code samples, with effect sizes reported to quantify practical significance after controlling for initial complexity.

## Methodology sketch

- Identify 50-100 active Python/JavaScript repositories via GitHub Search API (`https://api.github.com/search/repositories?q=topic:llm-generated+OR+topic:copilot`) to ensure a dataset of LLM-assisted projects.
- For each repository, clone the repository history (shallow clone with depth=100) to reduce storage overhead within the 7GB RAM limit.
- Use a pre-trained CodeBERT classifier (hosted locally or via lightweight ONNX runtime) to tag code blocks as LLM-generated vs. human-written in the initial commit window.
- Extract static complexity metrics (cyclomatic complexity, nesting depth) using `radon` for all tagged blocks at the time of generation.
- Track maintenance metrics (code churn, bug fix latency) for a six-month window following the initial generation using `git log` and GitHub Issues API.
- Match LLM-generated blocks with human-written blocks from the same repository (same function type, similar LOC) to control for project-specific variables.
- Perform statistical tests (Mann-Whitney U for non-normal distributions) comparing maintainability metrics between groups while regressing for initial complexity.
- Generate visualization artifacts (box plots, density plots) using `matplotlib` and save as PNG files.
- Cache intermediate analysis results via workflow artifacts to stay within the 6h GHA job limit; split processing into 30-minute chunks per repository batch.
