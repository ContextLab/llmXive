# Research: Evaluating the Impact of Code Generation on Long-Term Code Maintainability

## Research Question
Does code generated by large language models exhibit different long-term maintainability characteristics compared to human-written code, as measured by code churn, bug fix latency, and structural complexity metrics over a six-month observation window, when controlling for code complexity and project size at generation time?

## Dataset Strategy

**Primary Source**: GitHub API (Programmatic Fetch)
The study requires longitudinal data (commit history, issue tracking) which is not available in static text datasets. We will fetch data directly from GitHub repositories matching specific criteria.

**Selection Criteria**:
- **Languages**: Python, JavaScript.
- **Activity**: ≥ 1 commit in last 90 days, ≥ 5 stars.
- **Tags**: `topic:llm-generated`, `topic:copilot`, or keywords "LLM generated code", "Copilot generated".
- **Target**: 50 repositories.

**Addressing the "Missing Producer" Concern**:
The previous iteration failed to extract repo-level covariates (stars, age, total LOC) required for matching.
**Correction**: `01_curation.py` will explicitly extract and store `repo_stars`, `repo_created_at`, `repo_updated_at`, and **`repo_total_loc`** (total lines of code in the repository) for every repository before cloning. These fields will be joined with block-level metrics in `03_matching.py` to form the complete propensity score model.

**Dataset Limitations**:
- **Missing Variables**: If a repository lacks issue tracking or commit messages with "Fixes #N", those blocks will be excluded from latency analysis (FR-004).
- **API Limits**: Rate limiting may restrict the number of repos; the pipeline will log exclusions due to API limits.
- **Selection Bias**: Repositories using specific tags may not be representative of general LLM usage.

**Verified Datasets Note**:
The "Verified datasets" list provided in the prompt (e.g., `flan-t5-onnx`, `llm-generated-texts`) contains static text or metadata only. They **do not** contain the required longitudinal git history, issue links, or repository context. Therefore, they are **not** used as the primary data source. The project relies on the GitHub API as the sole data source for the analysis.

## Methodology

### 1. Data Curation & Tagging (FR-001, FR-002, FR-003)
- **Repo Search**: Query GitHub API for active repos.
- **Cloning**: Shallow clone (depth=100) to save bandwidth.
- **Metadata Extraction**: Extract `repo_stars`, `repo_age`, and **`repo_total_loc`** (total lines of code in the repo).
- **Block Extraction**: Parse files to extract functions/classes.
- **Classification**: Use `transformers` + `onnxruntime` with `microsoft/codebert-base` to classify blocks.
  - **Threshold**: Confidence ≥ 0.8.
  - **Exclusion**: Blocks < 0.8 confidence are dropped (logged).
  - **Circularity Note**: The classifier uses patterns (style, comments) that may correlate with maintainability. The "LLM" label is a proxy. Ground truth verification (FR-007) is critical to quantify this risk.
- **Static Metrics**: Use `radon` to calculate Cyclomatic Complexity and Nesting Depth.

### 2. Propensity Score Matching (FR-008)
- **Covariates**:
  - *Block-level*: Cyclomatic Complexity, Nesting Depth, Lines of Code (LOC).
  - *Repo-level*: Star count, Repository Age (days since creation), **Repository Total LOC** (project size).
- **Method**: 1:1 Nearest Neighbor matching *within the same repository* (to control for team/context heterogeneity).
- **Filter**: Repos with < 5 LLM and < 5 Human blocks after tagging are excluded.
- **Limitation**: Matching controls for observed covariates but cannot fully control for unobserved developer skill or style differences.

### 3. Longitudinal Metric Extraction (FR-004)
- **Window**: 6 months post-generation.
- **Definition of Introduction**: The first commit in the *cloned* history where the block appears. (Acknowledged limitation: may not be true generation time).
- **Code Churn**: Sum of lines added/deleted in the window (excluding initial commit).
- **Bug Fix Latency**: Time difference between commit containing "Fixes #N" and the closing of issue #N.
  - *Note*: Only commits with explicit issue links are counted. Pairs without links are excluded from latency analysis but kept for churn.
  - *Validity Note*: "Latency" measures issue resolution speed for tracked bugs. It does not control for issue severity and may be biased by issue complexity.

### 4. Statistical Analysis (FR-005, FR-006)
- **Test**: Wilcoxon Signed-Rank Test (non-parametric, paired). *Note: This is the correct test for matched pairs; the Constitution's mention of Mann-Whitney U is an error for this design.*
- **Correction**: Benjamini-Hochberg (FDR) for multiple comparisons (Churn vs. Latency).
- **Effect Size**: Cohen's d.
- **Visualization**: Box plots and density plots via `matplotlib` (CPU-only).
- **Interpretation**: Results are framed as "associational differences" between blocks with LLM-like patterns and human-like patterns.

### 5. Ground Truth Verification (FR-007)
- Randomly select ≥ 10 blocks for manual labeling.
- Calculate Precision/Recall of the classifier.
- **Success Criteria**: Precision/Recall ≥ 0.85 (SC-006).
- **Role**: Quantifies misclassification risk to inform the interpretation of the main analysis.

## Statistical Rigor & Assumptions

- **Observational Nature**: The study is observational. No causal claims are made; results are framed as "associational differences".
- **Multiple Comparisons**: Benjamini-Hochberg correction applied to control FDR.
- **Power Analysis**: Post-hoc power analysis performed on final matched pair count (SC-001). Target: ≥ 0.80 power to detect Cohen's d ≥ 0.5.
- **Collinearity**: Repo-level (stars, total LOC) and block-level (complexity) metrics are distinct. However, style and origin are confounded; matching on complexity may not fully isolate the origin effect.
- **Measurement Validity**: CodeBERT is used as a proxy for LLM origin. Ground truth verification (FR-007) quantifies misclassification risk. Latency is a proxy for issue resolution speed, not general maintainability.
- **Confounding**: Within-repo matching controls for team context, but individual developer skill remains a potential confounder.
