# Implementation Plan: llmXive follow-up: extending "Orca: The World is in Your Mind"

**Branch**: `001-llmxive-orca-counterfactual` | **Date**: 2026-07-10 | **Spec**: `spec.md`
**Input**: Feature specification from `specs/001-llmxive-orca-counterfactual/spec.md`

## Summary

This project implements a computational study to test whether the "conscious" linguistic scaffolding in the Orca world model encodes *associational advantages* for physical interactions better than raw pixel data. The approach involves: (1) curating a subset of the Orca dataset (or a verified synthetic fallback) for physical interactions; (2) extracting frozen latent vectors and applying counterfactual edits via vector arithmetic/masking; (3) generating ground-truth counterfactual outcomes by rendering synthetic scenarios in a physics engine (avoiding inverse graphics); (4) training lightweight `DecisionTreeClassifier` and `LinearProbe` models on CPU; and (5) performing statistical comparisons (Counterfactual Sensitivity Score) against a pixel-based baseline. The entire pipeline is constrained to run on a GitHub Actions free-tier runner (vCPU, limited RAM, no GPU).

**Critical Methodological Shift**: The study no longer claims "causal priors" but tests "associational advantage of counterfactual edits." The main model (N=450) is trained on *original* video labels. The counterfactual analysis is strictly limited to a verified subset (N=50) where ground truth is generated by simulation.

| :--- | :--- | :--- |

## Computational Feasibility & Methodological Rigor

### Compute Feasibility
- **Hardware**: Targeting GitHub Actions `ubuntu-latest` (multiple vCPU, substantial RAM).
- **Strategy**:
  - **Latent Extraction**: Use the `datasets` library to stream the Parquet file. The Orca model will be loaded in `torch.float32` (default) on CPU. To prevent OOM, video frames will be downsampled (e.g., 64x64) and processed in small batches (dynamic batch size adjustment as per Edge Cases).
  - **Model Training**: `DecisionTreeClassifier` and `LinearProbe` (Logistic Regression) are extremely lightweight on CPU. Training on a small number of samples will take seconds to minutes.
  - **Physics Simulation**: `mujoco` will be used with simplified scenes (single object physics) to minimize CPU load. Simulation IDs are generated deterministically by `physics_verify.py`.
  - **Memory Guardrails**: Scripts will include a `psutil` check to reduce batch size if RAM usage > 6GB.

### Methodological Rigor
- **Multiple Comparisons**: Since multiple tests are run (Latent vs. Pixel, Decision Tree vs. Linear Probe, Full vs. Unconscious), a Bonferroni correction or Holm-Bonferroni method will be applied to the p-values to control family-wise error rate.
- **Power Analysis**: With N=500, a paired t-test has high power (>0.9) to detect medium effect sizes (Cohen's d > 0.3). The plan acknowledges that if the effect size is very small, the study may be underpowered, but this is the maximum feasible sample size for the CPU constraint.
- **Causal Assumptions**: The study is observational regarding the *dataset* but experimental regarding the *latent manipulation*. Claims will be framed as "associational" regarding the dataset but "causal" regarding the *intervention* (vector arithmetic) *if* the physics engine validation (FR-010) holds.
- **Collinearity**: If "conscious" and "unconscious" vectors are definitionally related (e.g., one is a subset of the other), independent effects will not be claimed. Instead, the difference will be reported as the "scaffolding contribution."
- **Counterfactual Sensitivity**: The primary metric is the "Counterfactual Sensitivity Score," measuring the correlation between vector edit magnitude and prediction change, avoiding the invalid comparison of accuracy on different targets.

## Phase Order
1. **Phase 0: Reference-Validator Gate & Audit Logging**: Verify dataset URLs (Principle II) and initialize audit logs (FR-008).
2. **Phase 1: Data Curation & Disentanglement Probe**: Download/filter dataset (or synthetic fallback) and verify existence of distinct latent components.
3. **Phase 2: Latent Extraction & Counterfactual Injection**: Extract frozen latents and apply counterfactual edits (FR-002).
4. **Phase 3: Ground Truth Generation (Synthetic)**: Generate ground truth labels for the N=50 subset by rendering scenarios in `mujoco` (FR-009).
5. **Phase 4: Vector Arithmetic Validation**: Validate the edit mechanism on the N=50 subset using a hold-out set (FR-010, SC-008).
6. **Phase 5: Model Training**: Train Latent, Pixel, and Ablation models on original labels (N=450) and counterfactual labels (N=50) (FR-003, FR-004, FR-006, FR-007).
7. **Phase 6: Statistical Analysis**: Calculate Counterfactual Sensitivity Score and report metrics (FR-005, SC-007).
8. **Phase 7: Versioning & State Update**: Update state file with content hashes and timestamps (Principle V).
