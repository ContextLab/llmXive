# Data Model: Quantifying the Impact of Data Artifacts on Planetary Nebula Morphology

## Overview

This document defines the data structures, schemas, and file formats used throughout the project. It ensures that data generated by the synthetic pipeline, processed metrics, and validation results are consistent, typed, and verifiable against the contracts defined in `contracts/`.

## Entity Definitions

### 1. ImageSet
A collection of FITS images representing planetary nebulae, either synthetic or real.
*   **Attributes**:
    *   `id`: Unique identifier (UUID).
    *   `source`: "synthetic" or "hst_mast".
    *   `filename`: Path to the FITS file.
    *   `metadata`: Dictionary containing `exposure_time`, `filter`, `pixel_scale`, `wcs`.
    *   `ground_truth`: (Only for synthetic) Dictionary with `true_ellipticity`, `true_asymmetry`, `true_position_angle`.

### 2. ArtifactConfig
Configuration for an artifact injection event.
*   **Attributes**:
    *   `noise_sigma`: Float, relative to median signal.
    *   `saturation_fraction`: Float, fraction of brightest pixels clipped (0.0 to 1.0).
    *   `psf_fwhm`: Float (optional), PSF blur size.
    *   `seed`: Integer, random seed for reproducibility.

### 3. MorphologyMetrics
Calculated metrics for a single image.
*   **Attributes**:
    *   `image_id`: Reference to ImageSet.
    *   `ellipticity`: Float (0.0 to 1.0).
    *   `asymmetry`: Float (0.0 to 1.0).
    *   `position_angle`: Float (degrees).
    *   `bias_ellipticity`: Float (measured - true).
    *   `bias_asymmetry`: Float (measured - true).

### 4. CalibrationModel
A fitted regression model describing the relationship between artifact intensity and bias.
*   **Attributes**:
    *   `metric`: "ellipticity" or "asymmetry".
    *   `artifact_type`: "noise" or "saturation".
    *   `model_type`: "linear", "polynomial", "spline".
    *   `coefficients`: List of floats.
    *   `r_squared`: Float.
    *   `p_value`: Float (adjusted for multiple comparisons).

### 5. VerificationLog
A log file recording verification results, seeds, and library versions for reproducibility.
*   **Attributes**:
    *   `timestamp`: ISO 8601 timestamp.
    *   `citations`: List of verified citation objects (source, status, link).
    *   `seeds`: Dictionary of random seeds used in the run.
    *   `library_versions`: Dictionary of package versions.
    *   `checksums`: Dictionary of file checksums.

## File Formats

### FITS (Flexible Image Transport System)
*   **Usage**: Storage of raw and synthetic images.
*   **Structure**:
    *   Primary HDU: Image data (float32).
    *   Extensions: `METADATA` (keywords), `HISTORY` (processing steps).
*   **Standards**: Follows IVOA FITS standard.

### CSV/Parquet
*   **Usage**: Storage of aggregated metrics and regression results.
*   **Structure**: Tabular data with columns matching `MorphologyMetrics` and `CalibrationModel`.
*   **Encoding**: UTF-8.

### JSON
*   **Usage**: Configuration files (`config.yaml`), artifact parameters.
*   **Structure**: Hierarchical key-value pairs.

### Text (Log)
*   **Usage**: `logs/verification.log`.
*   **Structure**: Line-delimited JSON or structured text containing verification results, seeds, and checksums.

## Data Flow

1.  **Generation**: `synthetic/generator.py` creates `ImageSet` (FITS) with `ground_truth`.
2.  **Injection**: `synthetic/artifacts.py` applies `ArtifactConfig` to create modified FITS.
3.  **Measurement**: `metrics/ellipticity.py` and `metrics/asymmetry.py` compute `MorphologyMetrics`.
4.  **Aggregation**: Metrics are written to `data/processed/metrics.csv`.
5.  **Analysis**: `analysis/regression.py` fits `CalibrationModel` and saves to `data/processed/models.json`.
6.  **Validation**: `analysis/validation.py` applies models and writes residual stats to `data/processed/validation_results.csv`.
7.  **Logging**: `io/writer.py` generates `logs/verification.log` with seeds, versions, and checksums.

## Constraints

*   **Immutability**: Raw FITS files in `data/raw` and `data/synthetic` are never modified.
*   **Checksums**: All FITS and CSV files in `data/` must have a corresponding `.sha256` checksum file.
*   **Metadata**: Every FITS file must contain `HISTORY` entries detailing the software version and random seed used.
*   **Verification**: `logs/verification.log` must be generated and stored for every run to satisfy Constitution Principle II.
