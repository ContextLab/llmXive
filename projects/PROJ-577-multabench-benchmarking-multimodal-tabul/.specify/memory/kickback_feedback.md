## Unresolved concerns

- Tasks T004-T009 (Foundational utilities: registry verification, device/memory utils, config schema, error handling) have no corresponding User Story or Functional Requirement in spec.md. The spec assumes the vendored code is 'functionally complete' (Assumptions) and only defines US1-US3 for execution/validation. These tasks imply creating new infrastructure not mandated by the spec, risking scope creep or implementation of non-existent features.
- Task T023 specifies dataset 'MUL_IMAGE_CIFAR10' for the subset run. The spec.md (US2 Acceptance Scenario 1) explicitly mandates 'MUL_IMAGE_CBIS_DDSM'. This is a missing match for the required FR/SC element, as the task implements a different dataset than the one required for validation.
- Task T027 requires asserting 'requires_grad' on backbone weights. The spec.md (US3, SC-004) only defines 'tuned' vs 'frozen' as a high-level comparison. It does not mandate a specific implementation mechanism (like `requires_grad` flags) which may not exist in the vendored code. There is no FR/SC requiring this specific internal assertion, making the task unanchored to the spec's requirements.
- Task T039 mandates 'Security hardening (validate all external URLs)'. The spec.md (Edge Cases, FR-004) only requires graceful skipping on download failure. There is no FR/SC requiring proactive URL validation or security hardening. This task has no matching spec element.
- Tasks T007 and T023 reference creating 'config_subset.yaml' with specific dataset IDs. While a config file is implied by the plan, the specific IDs in T023 ('MUL_IMAGE_CIFAR10') contradict the spec's required ID ('MUL_IMAGE_CBIS_DDSM'). This is a stale/incorrect tag reference to a non-existent or incorrect spec requirement.
- Foundational tasks T004-T009 define new utility modules (device_utils, memory_profiler, error_handler) and config schemas that are consumed by User Story tasks (T020, T021, T022). However, the plan.md explicitly states 'No new source code structure is created' and relies on the 'existing structure of the vendored external/MulTaBench repository'. This creates a semantic dependency violation: the tasks assume the creation of artifacts (new files) that the plan forbids, breaking the producer-consumer flow. The tasks must be reordered or redefined to either modify existing vendored files or the plan must be updated to allow new structure.
- Task T023 (Producer) creates `config_subset.yaml` with dataset `MUL_IMAGE_CIFAR10`. However, the spec.md (US2 Acceptance Scenario 1) mandates `MUL_IMAGE_CBIS_DDSM`. This is a semantic data-flow violation: the consumer tasks (T024-T027) expect to run on the spec-defined dataset, but the producer task generates a config for a different dataset. The artifact produced by T023 does not satisfy the input requirements of the downstream consumer tasks.
- Task T027 (Consumer) asserts `requires_grad` status on backbone weights. This task depends on the existence of a specific 'tuned' training loop implementation in the vendored code. The spec.md and plan.md do not guarantee this implementation exists (they only define the *goal* of comparing tuned vs. frozen). If the vendored code lacks this specific hook, T027 has no producer to consume, creating a broken dependency chain. The task order assumes an artifact (the hook) that may not be produced by the foundational phase.
- User Story 1 tasks (T012-T016) are listed after the 'Foundational' phase, which is correct. However, T010 and T011 (Tests) are marked [P] (parallel) with T012-T016. While tests can be written in parallel, the 'Independent Test' requirement for US1 states tests must 'FAIL before implementation'. If T010/T011 are executed in parallel with T012-T016, they might pass prematurely if the implementation is already running, or fail to catch the 'fail-first' state. The ordering should explicitly sequence 'Write Tests' -> 'Verify Fail' -> 'Implement' to respect the TDD flow implied by the spec.
