# Implementation Plan: The Impact of Subtle Linguistic Cues on Perceived Authenticity in AI Chatbots

**Branch**: `001-linguistic-cues-authenticity` | **Date**: 2026-06-28 | **Spec**: `specs/001-linguistic-cues-authenticity/spec.md`
**Input**: Feature specification from `specs/001-linguistic-cues-authenticity/spec.md`

## Summary

This project investigates whether subtle linguistic variations—specifically first-person pronoun frequency, hedging language, and emotional valence—predict human ratings of perceived authenticity in AI chatbot conversations. The technical approach involves extracting these features from a corpus of chatbot dialogues using a fixed lexicon and VADER sentiment analysis, then performing robust statistical analysis (correlations, multiple linear regression with VIF checks, and non-linear term testing) to determine predictive power while controlling for conversation length and topic.

**Critical Methodological Pivot**: Unlike initial drafts, this plan explicitly rejects synthetic authenticity labels for the primary analysis due to construct validity concerns. The primary analysis relies on **human-annotated authenticity scores** generated via a dedicated annotation protocol. The verified public datasets (AIC, AICC) are used *only* for extracting text-based features (independent variables). The dependent variable (authenticity) is generated by human raters on a stratified subset of these conversations, with sample size determined by a prerequisite power analysis.

The implementation is strictly constrained to CPU-only execution on standard CI runners, utilizing `scikit-learn`, `statsmodels`, and `nltk`.

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

1.  **I. Reproducibility**: The plan mandates pinned versions in `requirements.txt` and random seed setting in all scripts. External datasets will be fetched from verified HuggingFace URLs only.
2.  **II. Verified Accuracy**: All citations in `research.md` will be restricted to the provided "Verified datasets" block. The analysis will explicitly frame results as associational, avoiding causal claims.
3.  **III. Data Hygiene**: The plan requires raw data checksums before processing. Derivations (feature extraction) will produce new files in `data/` without modifying raw inputs.
4.  **IV. Single Source of Truth**: The `data-model.md` defines the schema for extracted features. The `contracts/` schemas will enforce this structure. The final report will only contain statistics derived from these schema-compliant files.
5.  **V. Versioning Discipline**: **Implementation**: The plan defines a `state.yaml` file in `state/projects/` that tracks content hashes of all artifacts in `data/` and `code/`. A `checksum.py` script computes SHA-256 hashes for every file in `data/` and updates `state.yaml`. The `Advancement-Evaluator` checks these hashes to invalidate stale records.
6.  **VI. Linguistic Feature Extraction Transparency**: The fixed hedge lexicon and VADER configuration will be explicitly documented in the code and `research.md`.
7.  **VII. Human Rating Documentation**: **Implementation**: The plan mandates a `data/raw/rater_metadata.json` file storing the rating scale, instructions given to raters, and inter-rater reliability metrics (Cohen's Kappa). No aggregated scores are published without this metadata.

## Implementation Phases

### Phase -1: Power Analysis (Prerequisite Gate)
*   **Goal**: Determine required sample size (N) for the regression model to achieve power ≥ 0.8 (FR-011).
*   **Input**: Assumed effect size (f²) from literature or pilot.
*   **Action**: Run `src/analysis/models.py --step power_analysis`.
*   **Output**: `data/results/power_analysis_results.yaml` containing required N.
*   **Gate**: If required N > annotation budget, project is flagged as underpowered. Proceed only if N is feasible.
*   **FR Mapping**: FR-011.

### Phase 0: Lexicon Validation (FR-010)
*   **Goal**: Verify the precision of the fixed hedge lexicon against human judgment.
*   **Action**:
    1.  Select a random sample of conversation turns from the raw dataset.
    2.  Present to human raters with instructions: "Mark a word as a 'hedge' if it expresses uncertainty or lack of commitment."
    3.  Calculate Precision = (Lexicon Matches ∩ Human Matches) / Lexicon Matches.
*   **Output**: `data/results/lexicon_validation_results.yaml`.
*   **Gate**: If Precision < 0.8, flag dataset for manual review or refine lexicon definition.
*   **FR Mapping**: FR-010.

### Phase 1: Data Ingestion & Validation
*   **Goal**: Validate input datasets for required columns (FR-006).
*   **Action**:
    1.  Load raw Parquet files.
    2.  Check for `text_content` column. **Error** if missing.
    3.  Check for `authenticity_score` column. **Warning** if missing (triggers annotation protocol).
*   **Output**: Validated raw data file.
*   **FR Mapping**: FR-006.

### Phase 2: Feature Extraction
*   **Goal**: Extract linguistic metrics (FR-001, FR-008).
*   **Action**: Run `src/extraction/extract_features.py`.
*   **Output**: `data/processed/features.csv` (LinguisticMetrics).
*   **FR Mapping**: FR-001, FR-008.

### Phase 3: Human Annotation (If needed)
*   **Goal**: Generate `authenticity_score` for the analysis subset.
*   **Action**:
    1.  Sample N conversations (from Phase -1).
    2.  Human raters provide scores on Likert-type scales.
    3.  Store raw ratings and metadata in `data/raw/rater_metadata.json`.
    4.  Calculate inter-rater reliability (Cohen's Kappa).
*   **Output**: `data/processed/annotated_features.csv`.
*   **FR Mapping**: FR-007 (Sample size check), Constitution Principle VII.

### Phase 4: Statistical Analysis
*   **Goal**: Compute correlations, regression, VIF, and non-linear terms (FR-002, FR-003, FR-009).
*   **Action**: Run `src/analysis/models.py --step analyze`.
*   **Steps**:
    1.  Calculate VIF. Exclude collinear variables (VIF > 5) or log warning.
    2.  Run Multiple Linear Regression (controls: length, turn count).
    3.  Test non-linear terms (quadratic, interactions).
    4.  Apply Benjamini-Hochberg correction (FR-004).
*   **Output**: `data/results/stats_results.csv`.
*   **FR Mapping**: FR-002, FR-003, FR-004, FR-009.

### Phase 5: Sensitivity Analysis (SC-003)
*   **Goal**: Test robustness of findings via leave-one-out lexicon sweep.
*   **Action**: Iterate through the hedge lexicon, removing one word at a time, and re-run Phase 4 regression.
*   **Output**: `data/results/sensitivity_analysis.csv`.
*   **SC Mapping**: SC-003.

### Phase 6: Visualization & Reporting
*   **Goal**: Generate plots and summary report (FR-005).
*   **Action**: Run `src/analysis/visualize.py`.
*   **Output**: `data/results/plots/*.png`, `data/results/report.md`.
*   **FR Mapping**: FR-005.

### Phase 7: Versioning & State Update
*   **Goal**: Update `state.yaml` with content hashes (Constitution Principle V).
*   **Action**: Run `src/checksum.py`.
*   **Output**: Updated `state/projects/PROJ-316-...yaml`.
