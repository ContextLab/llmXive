# Research: Kairos: A Native World Model Stack for Physical AI

## Research Question
How does the integration of hybrid multi-scale temporal memory mechanisms within a native world model stack influence the long-horizon planning fidelity and sample efficiency of embodied agents operating in open-world, safety-critical environments?

*Scope Note: Due to the lack of verified sources for Open X-Embodiment, the "open-world" scope is bounded to the task diversity present in the BridgeData V2 Kitchen subset. The study explicitly acknowledges this as a scope reduction from the original spec. If BridgeData V2 is unavailable, the study is aborted.*

## Dataset Strategy

The project requires a dataset containing real-world robotics manipulation trajectories (state, action, reward) to train and evaluate world models.

| Requirement | Status | Source / Note |
| :--- | :--- | :--- |
| **Open X-Embodiment** | **Excluded** | No verified source URL found in the project's verified dataset list. Excluded to prevent reliance on unverified data. |
| **BridgeData V2** | **Verified Source** | **Primary Source**. The project will use the `bridge_data` dataset via the `ucimlrepo` or verified HuggingFace loader. Specifically, the "kitchen" subset will be used as it contains diverse multi-step manipulation tasks (cups, bottles, doors) suitable for testing long-horizon planning. |
| **Synthetic Fallback** | **REMOVED** | **No synthetic fallback**. If BridgeData V2 is unavailable, the study is **aborted**. Training on synthetic data generated by the same simulator used for evaluation creates a circular validation loop, invalidating the "open-world" and "sample efficiency" claims. |

**Data Preprocessing**:
- **Stratified Sampling**: The BridgeData V2 dataset will be split into [deferred] training, [deferred] test, stratified by task type (e.g., "move_cup", "open_door") to ensure representation of diverse dynamics.
- **Normalization**: State and action vectors will be normalized to [0, 1] or Z-score based on training set statistics.
- **Sequence Construction**: Trajectories will be chunked into fixed-length sequences (e.g., T=50) for model input.

**Data Diversity Validation**:
- Before training, the system will compute the variance of state/action vectors in the stratified subset.
- The report will include a "Data Diversity Score" comparing the subset's variance to the full dataset (if metadata available) or listing the specific task types covered.
- The "open-world" claim is explicitly bounded to the **BridgeData V2 Kitchen** task distribution.

## Methodology

### 1. Baseline Model (Transformer)
- **Architecture**: Standard Encoder-Decoder Transformer (or Decoder-only) with self-attention.
- **Input**: Current state $s_t$, Action $a_t$.
- **Output**: Predicted next state $\hat{s}_{t+1}$.
- **Loss**: Mean Squared Error (MSE) between $\hat{s}_{t+1}$ and ground truth $s_{t+1}$.
- **Training**: 50 epochs (or 10k steps) on CPU. *Note: Convergence is not expected; sample efficiency is measured by the slope of the learning curve (Initial Learning Slope).*

### 2. Kairos Model (Hybrid Multi-Scale Memory)
- **Architecture**: Baseline Transformer with a **GatedDeltaNet** (or equivalent open-source linear attention/memory) module injected to handle long-range dependencies.
- **Mechanism**: The memory module maintains a compressed state of past observations to improve prediction over long horizons (N > 50 steps).
- **Training**: Identical hyperparameters to baseline.
- **Sparsity**: Optional sparsity constraints applied to the memory attention mask to test latency trade-offs.

### 3. Evaluation Protocol
- **Environment**: MuJoCo with Franka Panda arm.
- **Task**: Multi-step "pick and place" with obstacles (mapped to BridgeData V2 task semantics).
- **Metrics**:
  - **Long-Horizon Success Rate**: % of tasks completed within N steps.
  - **Prediction Error (MSE)**: Average squared error of state prediction.
  - **Physics Consistency Score (PCS)**: 
    - *For Simulated Data*: % of trajectories with zero **Kinetic Energy Deviation** (>5% threshold) or **Momentum Conservation Error** (>5% threshold).
    - *For Real-World Data (BridgeData V2)*: **Physical Plausibility Score**. Defined as % of trajectories with zero **Velocity Jumps** (>5% threshold) or **State Discontinuities** (teleportation). *Note: Strict momentum conservation is uncomputable for real-world data due to unknown mass/inertia. Heuristics are used instead.*
  - **Latency**: p95 inference time per step. *Note: A fixed total loop budget is split: approximately half for MuJoCo simulation and half for model inference.*
  - **Resource Usage**: Peak RAM.
- **Statistical Test**: **Bootstrap Confidence Interval** (primary) for the difference in success rate/MSE across 5 seeds, with Wilcoxon signed-rank as a secondary check.

## Statistical Rigor & Limitations

- **Multiple Comparisons**: Since only two models are compared (Baseline vs. Kairos) on a single primary metric (Success Rate), family-wise error correction is not strictly required, but Bonferroni correction will be noted if multiple secondary metrics are tested.
- **Power Analysis**: With N=5 seeds, statistical power is low for parametric tests. The **Bootstrap CI** method is used to estimate the distribution of the difference without assuming normality, providing a more robust effect size estimate. Wilcoxon is used as a secondary non-parametric check.
- **Power Limitation**: The study explicitly acknowledges that with N=5, the power to detect small effects is low. A non-significant result (p >= 0.05) does not prove the null hypothesis, only that the effect was not detectable under these constraints.
- **Causal Inference**: This is an observational comparison of two architectures on the same data. Claims will be framed as "performance improvement" rather than "causal mechanism of memory" without further ablation studies.
- **Collinearity**: Predictors (state dimensions) may be correlated; the model architecture (Transformer) handles this via attention, but independent effects of specific memory scales will be reported descriptively.
- **Dataset Limitation**: The "open-world" claim is bounded to the tasks present in BridgeData V2 Kitchen. Generalization to unseen domains (e.g., navigation) is not tested.

## Compute Feasibility Plan

- **Hardware Target**: GitHub Actions Free Runner (multi-core CPU, 7 GB RAM).
- **Model Size**: Small Transformer (e.g., 4 layers, 256 hidden dim) to fit in RAM.
- **Dataset Size**: Stratified subset of ~10k steps from BridgeData V2 to ensure <6h runtime.
- **Libraries**: `torch` (CPU wheel), `mujoco` (CPU backend). No CUDA, no 4-bit/8-bit quantization.
- **Thread Pinning**: `torch.set_num_threads(1)` is enforced to strictly limit CPU usage to 1 active thread.
- **Latency Budget**: The total loop is split: ~50% for simulation, ~50% for inference. If the simulation step exceeds 100ms, the inference budget is reduced to maintain the total <200ms, or the run is **flagged as "Fail Real-Time"** (but data is still recorded).
