## User Scenarios & Testing

### User Story 1 - Generate Teacher Routing Ground Truth (Priority: P1)

The researcher MUST be able to generate a synthetic dataset of `(prompt_embedding, noise_level, routing_label, velocity_vector)` tuples by running the pre-trained DanceOPD teacher model on a sample of ImageNet-1K and LAION-400M prompts, capturing the exact expert field selection (routing label) made by the dynamic policy at each step.

**Why this priority**: This is the foundational step; without the ground truth routing labels generated by the complex teacher model, no static approximation (decision tree) can be trained or evaluated. The entire hypothesis test relies on the fidelity of this dataset.

**Independent Test**: The system can be tested by verifying that the output dataset contains at least 1,000 image samples, that the `routing_label` column contains valid expert identifiers matching the teacher's known architecture, and that the `velocity_vector` dimensions are consistent with the model's output layer.

**Acceptance Scenarios**:

1. **Given** a valid pre-trained DanceOPD model weights file and a subset of image prompts, **When** the data generation pipeline is executed, **Then** a CSV/Parquet file is produced containing ≥ 1,000 rows with columns `prompt_embedding`, `noise_level`, `routing_label`, and `velocity_vector`.
2. **Given** the generated dataset, **When** the `routing_label` values are inspected, **Then** they match the set of expert field IDs defined in the DanceOPD configuration (e.g., `expert_text_to_image`, `expert_editing`), with no null values.

---

### User Story 2 - Train and Evaluate Static Decision Trees (Priority: P2)

The researcher MUST be able to train a series of Decision Tree classifiers with varying `max_depth` across a range from shallow to deep configurations on the generated teacher dataset to approximate the routing labels, and compute the "Routing Consistency" (classification accuracy) for each depth.

**Why this priority**: This directly addresses the research question of whether static, low-complexity structures can capture the teacher's logic. It allows for the empirical mapping of the relationship between tree complexity and routing fidelity.

**Independent Test**: The system can be tested by training a single tree with a fixed depth, saving the model, and verifying that it achieves a specific, reproducible accuracy score on a held-out test split, distinct from the teacher's performance.

**Acceptance Scenarios**:

1. **Given** the teacher-generated dataset split into a majority portion for training and a minority portion for testing, **When** a Decision Tree with `max_depth=5` is trained, **Then** the model is saved to disk and reports a validation accuracy on the test set.
2. **Given** a loop of `max_depth` values ranging from shallow to deep configurations, **When** the training pipeline runs, **Then** a results table is produced showing the `max_depth` and corresponding `routing_accuracy` for each model.

---

### User Story 3 - Quantify Fidelity Degradation and Statistical Significance (Priority: P3)

The researcher MUST be able to correlate the `max_depth` of the static trees with the resulting image quality (FID, CLIP Score) when using the tree's predicted routing to drive a simplified Euler integrator, and perform a bootstrap test on FID distributions and a paired t-test on per-sample CLIP scores to determine if the performance drop at low depths is statistically significant compared to the teacher baseline.

**Why this priority**: This connects the theoretical approximation (routing accuracy) to the practical outcome (image fidelity), providing the final evidence for the "sharp threshold" hypothesis mentioned in the expected results.

**Independent Test**: The system can be tested by running the inference pipeline with the `max_depth=5` model and the teacher model on the same [deferred] image samples, calculating the FID and CLIP Score for both, and verifying that the statistical tests return valid p-values indicating significance (or non-significance) as per the data.

**Acceptance Scenarios**:

1. **Given** the trained Decision Tree models and the teacher baseline, **When** the inference pipeline generates images for a held-out test set of image samples using the tree-predicted routing, **Then** the system calculates the Fréchet Inception Distance (FID) and CLIP Score for each model.
2. **Given** the FID scores for the teacher and the `max_depth=5` tree, **When** a bootstrap hypothesis test is performed on the FID distributions and a paired t-test on per-sample CLIP scores, **Then** the system outputs p-values and conclusion statements regarding the statistical significance of the fidelity drop.

### Edge Cases

- **What happens when** the teacher model encounters a prompt that triggers an undefined expert routing path?
  - *Handling*: The pipeline must log the event and exclude the sample from the training set to prevent label noise, or assign a default "fallback" label if the architecture supports it.
- **How does the system handle** memory exhaustion when loading the full ImageNet-1K validation set into RAM for feature extraction?
  - *Handling*: The system must process data in batches and stream results to disk rather than loading the entire dataset into memory at once.
- **What happens when** the Decision Tree achieves [deferred] accuracy on the training set but fails on the test set (overfitting)?
  - *Handling*: The evaluation metrics must report both training and test accuracy, and the analysis must focus on the test set performance to ensure generalizability.

## Requirements

### Functional Requirements

- **FR-001**: System MUST generate a synthetic dataset of `(prompt_embedding, noise_level, routing_label, velocity_vector)` tuples by executing the pre-trained DanceOPD teacher model on a sampled subset of ImageNet-1K and LAION-400M prompts (See US-1).
- **FR-002**: System MUST train multiple Decision Tree classifiers with `max_depth` values ranging from shallow to deep configurations. using scikit-learn on CPU-only hardware (See US-2).
- **FR-003**: System MUST compute "Routing Consistency" (classification accuracy) for each trained tree against the teacher's ground truth labels on a held-out test set (See US-2).
- **FR-004**: System MUST execute a CPU-only inference loop where the trained tree predicts the expert field, re-runs the selected expert field to generate the velocity vector, and a simple Euler integrator samples the image using this newly generated vector (See US-3).
- **FR-005**: System MUST calculate image quality metrics (FID and CLIP Score) for images generated using the static tree routing and compare them against the teacher baseline (See US-3).
- **FR-006**: System MUST perform a bootstrap hypothesis test on the FID distributions and a paired t-test on per-sample CLIP scores to determine if the performance drop in image quality at low `max_depth` values is statistically significant compared to the teacher baseline (See US-3).

### Key Entities

- **TeacherRoutingDataset**: A structured dataset containing input embeddings, noise levels, ground truth routing labels, and velocity vectors derived from the DanceOPD teacher.
- **DecisionTreeModel**: A scikit-learn `DecisionTreeClassifier` instance configured with a specific `max_depth` to approximate the teacher's routing logic.
- **InferenceResult**: A record containing the generated image path, the routing path used (tree vs. teacher), the generated_velocity_vector (to verify re-inference), and the associated quality metrics (FID, CLIP Score).

## Success Criteria

### Measurable Outcomes

- **SC-001**: Routing Consistency (classification accuracy of the tree vs. teacher) is measured against the teacher's ground truth routing labels on the held-out test set (See US-2).

- **SC-003**: Statistical significance of the fidelity drop is measured against a significance threshold (α = 0.05) using a bootstrap test on FID distributions and a paired t-test on per-sample CLIP scores (See US-3).
- **SC-004**: The relationship between `max_depth` and routing accuracy is measured against the theoretical expectation of a saturation point where accuracy plateaus (See US-2).
- **SC-005**: Computational feasibility is measured against the constraint of running the Decision Tree training and the inference of [deferred] image samples on a CPU-only environment with ≤7 GB RAM and ≤6 hours total runtime, with a stop-early condition if the time limit is exceeded (See FR-002, FR-004).

## Assumptions

- **Assumption about data/environment**: The pre-trained DanceOPD teacher model weights are publicly available and can be loaded into memory on a CPU-only runner (potentially requiring quantization or a smaller subset of the model if the full weights exceed RAM limits, though the methodology specifies CPU inference for the teacher data generation).
- **Assumption about scope boundaries**: The study focuses exclusively on the *routing logic* compressibility; the fidelity of the underlying expert fields themselves is assumed to be constant and is not re-trained or optimized.
- **Assumption about target users**: The primary user is a researcher evaluating theoretical compressibility; the output is a dataset and analysis report, not a deployable edge application.
- **Assumption about dataset-variable fit**: The ImageNet-1K and LAION-400M subsets contain sufficient diversity in prompts to trigger the full range of routing decisions available in the DanceOPD teacher model.
- **Assumption about methodological constraints**: The teacher data generation step (running the full model) is assumed to be performed on a separate GPU resource or via a pre-computed dataset if the free-tier CPU runner cannot execute the teacher inference within the 6-hour limit; the 6-hour CPU constraint applies strictly to the Decision Tree training and the inference of [deferred] image samples. If the 19-tree training loop exceeds 6 hours, the pipeline MUST stop early and report the results for completed depths.
- **Assumption about statistical power**: The sample size of the held-out test set ([deferred] image samples) is assumed to be sufficient to detect a medium effect size in the statistical tests with ≥ 0.8 power.
