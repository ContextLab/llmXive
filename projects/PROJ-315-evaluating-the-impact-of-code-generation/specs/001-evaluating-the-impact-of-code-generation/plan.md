# Implementation Plan: Evaluating the Impact of Code Generation on Code Review Quality

**Branch**: `001-evaluating-llm-code-review-impact` | **Date**: 2025-01-15 | **Spec**: `specs/001-evaluating-llm-code-review-impact/spec.md`

## Summary

This project implements a computational pipeline to evaluate whether code generated by Large Language Models (LLMs) receives different code review feedback compared to human-written code. The approach involves acquiring a public GitHub Pull Request (PR) dataset, classifying PRs as "LLM-generated" or "human-written" using commit message heuristics (keyword matching), computing code complexity and bug density metrics, and performing statistical comparisons.

**Key Methodological Adjustments**:
1. **Heuristic Robustness**: We acknowledge the lack of ground-truth LLM labels. The plan uses **Simulation-Extrapolation (SIMEX)** to adjust effect sizes for misclassification error, validated by a manual audit.
2. **Causal Structure**: Code complexity is treated as a **mediator** (LLM -> Complexity -> Review) rather than a confounder. We **do not** use PSM to control for complexity to avoid over-control bias. PSM is used only for observed confounders (e.g., repository size, language) if available.
3. **Survival Analysis**: Merge latency is analyzed using **Cox Proportional Hazards** models to handle censored data (unmerged PRs), replacing Mann-Whitney U for this specific metric.
4. **Diagnostic Reporting**: The sentiment-bug density correlation (FR-016) is treated as a diagnostic flag. If r > 0.5, the result is reported with a caveat rather than silently excluded.

The study is strictly observational; all findings are framed as **associational**. The implementation is constrained to run on a CPU-only GitHub Actions runner (2 cores, 7 GB RAM, ≤6 hours) with a dataset limit of [deferred] PRs.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|:--- |:--- |:--- |
| **Simulation-Extrapolation (SIMEX)** | Required to correct effect sizes for the noisy treatment label (heuristic-based). | Standard regression assumes perfect classification; without SIMEX, bias is uncorrected. |
| **Cox Proportional Hazards** | Required for Merge Latency (time-to-event) to handle censored data (unmerged PRs). | Mann-Whitney U ignores censoring, leading to biased estimates of merge speed. |
| **Manual Audit Heuristic** | Required by FR-015 to validate the keyword-based classification (≥70% accuracy). | Automated validation alone cannot detect semantic misclassification; human-in-the-loop audit is necessary for ground truth (used for SIMEX calibration). |
| **Exact Matching Fallback** | Required if PSM fails due to computational constraints or lack of common support. | PSM can be unstable with small samples or high dimensionality; exact matching on bins is robust. |

## Implementation Phases

### Phase 0: Data Acquisition & Variable Fit Check
1. **Download**: Fetch `prs-v2-sample` from HuggingFace.
2. **Verify Schema**: Check for `diff`, `review_comments`, `created_at`, `merged_at`, **`commit_message`**.
 - *Halt Condition*: If `commit_message` is missing, stop with "Data Missing: commit_message".
3. **Checksum**: Record SHA-256 of raw file.

### Phase 1: Preprocessing & Metric Computation
1. **Clean**: Filter out PRs with missing `diff` or `merged_at` (if not censored analysis) or mark as censored.
2. **Metrics**: Compute `loc`, `cyclomatic_complexity`, `bug_density` using `radon`/`lizard`.
3. **Sentiment**: Compute `sentiment_score` using `textblob`.

### Phase 2: Labeling & Validation (The Audit)
1. **Heuristic**: Apply keyword matching (≥2 distinct keywords) to classify `label`.
2. **Audit Sample**: Randomly sample 50 PRs per group (`LLM`, `Human`).
3. **Manual Review**: (Simulated/Pre-run) The script generates a report of these 50+50 PRs.
 - *Note*: In CI, this step assumes the heuristic is pre-validated. If the heuristic accuracy (from a pre-computed ground truth or manual check) is <70%, the pipeline halts with "Insufficient Heuristic Accuracy".
4. **SIMEX Calibration**: Use the audit results to estimate the misclassification matrix for SIMEX.

### Phase 3: Propensity Score Matching (PSM)
1. **Covariates**: Use `repo_size`, `language`, `author_experience` (if available). **Do NOT use code complexity** (it is a mediator).
2. **Match**: Perform PSM. Check Balance (VIF < 5).
3. **Fallback**: If VIF > 5 or match rate < 10%, switch to **Exact Matching on Discrete Bins** (LOC/Complexity binned into 5 levels).
4. **Output**: `data/processed/matched_sample.csv`.

### Phase 3.5: Confounding Check (FR-016)
1. Compute correlation between `sentiment_score` and `bug_density`.
2. If `r > 0.5`, set `confounder_flag = True`.
3. **Action**: Do **not** exclude sentiment. Instead, include it in the final report with a "Potential Confounder" warning.

### Phase 4: Sensitivity Analysis (FR-007)
1. Iterate thresholds: `{1, 2, 3}`.
2. For each threshold: Re-classify, Re-match (PSM/Fallback), Re-test.
3. Save results to `reports/sensitivity_results.csv`.
4. Generate plot: Classification rate vs. Threshold.

### Phase 5: Statistical Testing
1. **Comment Count**: Mann-Whitney U (on matched sample).
2. **Merge Latency**: Cox Proportional Hazards (on matched sample).
3. **Sentiment**: Mann-Whitney U (with confounder flag).
4. **Correction**: Benjamini-Hochberg for multiple comparisons.
5. **Correction**: Apply SIMEX to effect sizes using the audit matrix.

### Phase 6: Visualization & Reporting
1. Generate Boxplots (Comment Count, Sentiment).
2. Generate Survival Curve (Merge Latency).
3. Generate Sensitivity Plot.
4. Compile HTML report with "Associational Only" disclaimer.

## Risk Mitigation

| Risk | Mitigation Strategy |
|:--- |:--- |
| **Dataset Variable Mismatch** | Pre-flight check in `data/download.py` to verify schema. If missing critical fields, halt with error. |
| **Insufficient LLM Samples** | If <500 LLM-labeled PRs found, expand keyword list. If still <500, report "Insufficient Power" and stop. |
| **Heuristic Accuracy <70%** | Manual audit (FR-015) triggers halt. Heuristics are refined or study is aborted. |
| **Collinearity (VIF > 5)** | Switch to Exact Matching on Discrete Bins. |
| **Runtime Exceeds 6h** | Implement sampling (e.g., process every 2nd PR) if full dataset is too large. |
| **Censoring in Merge Latency** | Use Cox PH model instead of Mann-Whitney U. |
| **Measurement Error** | Apply SIMEX correction to effect sizes. |
