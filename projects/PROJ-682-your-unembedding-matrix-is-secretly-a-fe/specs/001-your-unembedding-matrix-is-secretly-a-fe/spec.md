# Feature Specification: Reproduce & Validate EmbedFilter: Your UnEmbedding Matrix is Secretly a Feature Lens

**Feature Branch**: `682-reproduce-embfilter`  
**Created**: 2024-05-21  
**Status**: Draft  
**Input**: User description: "Reproduce & validate: Your UnEmbedding Matrix is Secretly a Feature Lens for Text Embeddings (Paper: arXiv 2606.07502, Code: CentreChen/EmbFilter)"

## User Scenarios & Testing

### User Story 1 - Execute End-to-End Reproduction Pipeline (Priority: P1)

A researcher needs to run the vendored `EmbFilter` codebase on a standard CPU-only CI runner to confirm the implementation executes without errors and produces the primary artifacts (embedding matrices, evaluation logs) described in the paper's "quickstart" workflow.

**Why this priority**: This is the foundational requirement. Without a successful, crash-free execution of the provided code, no validation or comparison against the paper's claims is possible. It validates the "Reproduction" status of the project.

**Independent Test**: The CI job runs the specific entry point script (e.g., `run4llama_echo.py`) with a small, pre-defined sample dataset. The test passes if the script exits with code 0 and generates the expected output files (e.g., `.pt` embedding files, `.json` logs) in the designated output directory.

**Acceptance Scenarios**:
1. **Given** the `EmbFilter` submodule is cloned and dependencies are installed, **When** the `run4llama_echo.py` script is invoked with a minimal test configuration (e.g., `--max_samples 10`), **Then** the process completes within 15 minutes on a 2-CPU runner and generates at least one non-empty embedding artifact file.
2. **Given** a valid LLM model path is provided, **When** the script attempts to load the model and apply the `EmbFilter` transformation, **Then** the system logs the specific dimensions of the input and output embeddings, confirming the linear transformation was applied without dimension mismatch errors.
3. **Given** the execution environment has no GPU access, **When** the script initializes the model, **Then** it explicitly loads the model onto the CPU device (verified by log output) and does not attempt to allocate CUDA memory or load quantization layers requiring `bitsandbytes`.

---

### User Story 2 - Validate Performance Against Paper Claims (Priority: P2)

A researcher needs to compare the reproduction results (zero-shot downstream performance) generated by the script against the quantitative claims in the paper (e.g., improvement in MTEB scores or reduced dimensionality performance) to verify the "Feature Lens" hypothesis.

**Why this priority**: This validates the scientific contribution. It moves beyond "code runs" to "code works as claimed." It confirms that the `EmbFilter` mechanism actually improves embeddings as described in the abstract.

**Independent Test**: The system runs the evaluation suite (`eval.py`) on a standard benchmark subset (e.g., a representative sample from a subset of MTEB or a specific task listed in `utils/tasks.py`). and outputs a JSON report containing the performance metric (e.g., Spearman correlation). This value is compared against a baseline (raw embeddings) and the paper's reported delta.

**Acceptance Scenarios**:
1. **Given** the `EmbFilter` transformation is active, **When** the evaluation script runs on the `STS` (Semantic Textual Similarity) task subset, **Then** the output report records a Spearman correlation score that is strictly greater than the baseline (unfiltered) score for the same model on the same subset, demonstrating the "enhancement" claim.
2. **Given** the dimensionality reduction feature is enabled (e.g., `--dim 128`), **When** the script evaluates the reduced embeddings, **Then** the reported performance degradation (if any) is logged as a percentage, allowing verification of the claim that quality is "fully preserved" or degraded within a negligible margin (e.g., < 2% drop).
3. **Given** the baseline and filtered results are computed, **When** the system generates the comparison summary, **Then** it explicitly flags the result as "Reproduction Successful" if the filtered score meets or exceeds the paper's reported improvement trend for the specific backbone (Llama/Mistral/Qwen) used.

### User Story 3 - Document Methodological Constraints & Sensitivity (Priority: P3)

A researcher needs to confirm that the reproduction adheres to the paper's methodological constraints (e.g., no randomization, observational analysis) and that any thresholds used (e.g., frequency cutoffs) are explicitly logged and justified, ensuring the study is methodologically sound.

**Why this priority**: This ensures the scientific validity of the reproduction. It prevents the project from claiming causal effects where only associations exist and ensures that any "magic numbers" (thresholds) are transparent and defensible.

**Independent Test**: The system parses the execution logs and configuration files to verify that the analysis is framed as "associational" and that any threshold parameters (e.g., `freq_threshold`) are set to explicit values with a recorded justification or sensitivity check in the logs.

**Acceptance Scenarios**:
1. **Given** the evaluation results are generated, **When** the system reviews the output report, **Then** the report explicitly states "Observational Study" or "Associational Analysis" and avoids causal language (e.g., "causes improvement") in its summary, framing findings as correlations.
2. **Given** a frequency threshold is applied to filter tokens, **When** the script runs, **Then** it logs the specific threshold value used (e.g., `threshold=0.05`) and includes a note in the output metadata citing the paper's rationale (e.g., "Based on top-k frequent token analysis in Section 3.2 of arXiv 2606.07502").
3. **Given** the analysis involves multiple downstream tasks, **When** the results are aggregated, **Then** the system logs a warning if no multiple-comparison correction (e.g., Bonferroni) is applied, or explicitly notes that the results are presented as per-task uncorrected p-values/correlations to maintain transparency.

### Edge Cases

- **What happens when the LLM model weights are missing?** The system MUST fail fast with a clear error message indicating the missing model path, rather than hanging or attempting to download a model that exceeds the available disk limit.
- **How does the system handle models not listed in `modeling_*.py`?** The system MUST reject the configuration with a "Model architecture not supported" error, as the vendored code only supports Llama, Mistral, and Qwen2 variants.
- **What if the dataset is too large for 7GB RAM?** The system MUST detect the dataset size and automatically sample a subset (e.g., `--max_samples 100`) or fail with a "Memory constraint exceeded" error, ensuring the CI job does not OOM (Out Of Memory).

## Requirements

### Functional Requirements

- **FR-001**: System MUST execute the vendored `EmbFilter` scripts (e.g., `run4llama_echo.py`) on a CPU-only environment without requiring GPU/CUDA drivers (See US-1)
- **FR-002**: System MUST apply the linear `EmbFilter` transformation to text embeddings derived from LLMs, explicitly filtering the subspace associated with high-frequency tokens (See US-1)
- **FR-003**: System MUST evaluate the filtered embeddings against at least one standard downstream task (e.g., STS, summarization) to generate performance metrics (See US-2)
- **FR-004**: System MUST output a structured report (JSON or CSV) containing the baseline score, filtered score, and the calculated improvement delta (See US-2)
- **FR-005**: System MUST explicitly log the frequency threshold and dimensionality reduction parameters used during the transformation, including the rationale for their selection (See US-3)
- **FR-006**: System MUST detect and prevent the use of any quantization methods (e.g., 8-bit, 4-bit) that require CUDA or `bitsandbytes` to ensure CPU compatibility (See US-1)
- **FR-007**: System MUST frame all performance comparisons as associational observations rather than causal claims in the generated report (See US-3)

### Key Entities

- **Embedding Matrix**: The numerical representation of text tokens after passing through the LLM's unembedding layer, subject to the `EmbFilter` transformation.
- **Frequency Threshold**: A scalar value determining which tokens are considered "high-frequency" and thus filtered out; this is a critical parameter for the transformation.
- **Downstream Task**: A specific evaluation benchmark (e.g., MTEB subset) used to measure the quality of the embeddings before and after filtering.
- **Performance Delta**: The difference in metric scores (e.g., Spearman correlation) between the baseline (unfiltered) and filtered embeddings.
