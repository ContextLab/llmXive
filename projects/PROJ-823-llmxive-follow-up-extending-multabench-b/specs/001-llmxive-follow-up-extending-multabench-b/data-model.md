# Data Model: llmXive Follow-up: Extending MulTaBench

## Overview

This document defines the data structures, schemas, and propagation rules for the llmXive pipeline extending MulTaBench. It serves as the central reference for data flow, artifact generation, and validation across all user stories (US1, US2, US3).

## Core Concepts

### Run ID (`run_id`)

The `run_id` is a unique, deterministic identifier for a specific execution of the pipeline or a specific stage of the pipeline. It ensures reproducibility and traceability of all artifacts generated during a run.

- **Format**: `<project_id>_<timestamp>_<seed>_<hash>`
 - `project_id`: Fixed identifier for the project (e.g., `PROJ-823`)
 - `timestamp`: ISO 8601 formatted timestamp of the run start (e.g., `20231027T100000`)
 - `seed`: The random seed used for the run (e.g., `42`)
 - `hash`: A short hash of the configuration parameters to ensure uniqueness for different configurations
- **Propagation**:
 - `run_id` is generated at the entry point of the pipeline (e.g., `run_baseline.py`, `run_conditioned.py`).
 - It is passed to all downstream components (generators, trainers, analyzers).
 - All output artifacts (Parquet files, JSON reports, CSV summaries) MUST include `run_id` as a column or metadata field.
 - `run_id` is used to link related artifacts across different stages (e.g., embeddings → projections → correlation analysis).

## Data Schemas

The following contract files define the strict schemas for core data artifacts. All data generation tasks MUST adhere to these schemas.

### Frozen Embeddings Schema

**Contract File**: `contracts/frozen_embedding.schema.yaml`

Defines the structure for frozen embeddings generated in User Story 1 (US1).

**Key Fields**:
- `run_id`: Identifier of the run that generated the embedding.
- `dataset_id`: Identifier of the source MulTaBench dataset.
- `sample_id`: Unique identifier for the sample within the dataset.
- `modality`: Type of data (e.g., `image`, `text`).
- `embedding_vector`: Array of floats representing the embedding.
- `model_name`: Name of the model used (e.g., `clip-vit-b/32`, `sentence-bert-base`).

**Usage**:
- Generated by: `code/pipelines/run_baseline.py` (T015)
- Consumed by: `code/pipelines/run_conditioned.py` (T025)

### Tabular Metadata Schema

**Contract File**: `contracts/tabular_metadata.schema.yaml`

Defines the structure for tabular metadata statistics computed in User Story 2 (US2).

**Key Fields**:
- `dataset_id`: Identifier of the source MulTaBench dataset.
- `feature_name`: Name of the tabular feature.
- `cardinality`: Number of unique values (for categorical features).
- `missingness`: Proportion of missing values (0.0 to 1.0).
- `sparsity`: Proportion of zero values (for numerical features).
- `variance`: Statistical variance of the feature (for numerical features).
- `feature_type`: Type of feature (`categorical`, `numerical`).

**Usage**:
- Generated by: `code/analysis/metadata_stats.py` (T024)
- Consumed by: `code/pipelines/run_conditioned.py` (T025)

## Data Flow

### Phase 1: Embedding Generation (US1)

1. **Input**: Raw MulTaBench datasets (images, text, tabular).
2. **Process**:
 - Load datasets using `code/data_loader.py`.
 - Generate frozen embeddings using CLIP ViT-B/32 and Sentence-BERT.
 - Apply `run_id` to all generated embeddings.
3. **Output**: `data/processed/embeddings_{run_id}.parquet` (Schema: `frozen_embedding.schema.yaml`).

### Phase 2: Projection Training (US2)

1. **Input**:
 - Frozen embeddings from US1.
 - Tabular metadata from `code/analysis/metadata_stats.py`.
2. **Process**:
 - Train lightweight projection modules (MLP/Attention) conditioned on tabular features.
 - Freeze backbone weights; train only projection layer.
 - Apply `run_id` to all training artifacts.
3. **Output**:
 - `data/artifacts/metrics_conditioned_{run_id}.json`
 - Trained projection weights.

### Phase 3: Correlation Analysis (US3)

1. **Input**:
 - Aggregated baseline metrics from US1 (`data/artifacts/frozen_baseline_aggregated_{run_id}.json`).
 - Conditioned performance metrics from US2 (`data/artifacts/metrics_conditioned_{run_id}.json`).
 - GPU-Tuned baselines (from MulTaBench paper).
2. **Process**:
 - Calculate "Recovery Ratio".
 - Perform Pearson correlation and statistical significance testing.
 - Apply Benjamini-Hochberg correction.
3. **Output**: `data/artifacts/correlation_report_{run_id}.json`

## Artifact Naming Convention

All output artifacts MUST follow the naming convention:
- `{artifact_type}_{run_id}.{ext}` or `{artifact_type}_summary.{ext}`

Examples:
- `embeddings_run_20231027T100000_42_a1b2c3d4.parquet`
- `metrics_conditioned_run_20231027T100000_42_a1b2c3d4.json`
- `correlation_report_run_20231027T100000_42_a1b2c3d4.json`

## Validation Rules

1. **Schema Compliance**: All generated Parquet/JSON files MUST validate against their respective contract schemas.
2. **Run ID Consistency**: All artifacts in a single pipeline run MUST share the same `run_id`.
3. **Completeness**: All tasks MUST process **ALL available datasets** as per project requirements.
4. **Determinism**: Re-running the pipeline with the same `run_id` and configuration MUST produce identical outputs.
