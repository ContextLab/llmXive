# Data Model: llmXive Follow-up: Extending MulTaBench

## 1. Overview

This document defines the data structures, schemas, and flow for the `llmXive` follow-up project. All data artifacts must adhere to the schemas defined in `contracts/`.

## 2. Entities

### 2.1 FrozenEmbedding
Vector representation of unstructured data (image or text) generated by a frozen encoder.

*   **Source**: `code/embeddings/generator.py`
*   **Storage**: `data/processed/embeddings/{dataset_id}_embeddings.parquet`
*   **Attributes**:
    *   `run_id`: String (Global unique identifier for the pipeline run).
    *   `dataset_id`: String identifier for the dataset.
    *   `sample_id`: Unique identifier for the sample within the dataset.
    *   `modality`: Enum ("image", "text").
    *   `vector`: List of floats (embedding dimension, e.g., 768 for CLIP ViT-B/32).
    *   `model_version`: String (e.g., "clip-vit-base-patch32").
    *   `generated_at`: ISO 8601 timestamp.
    *   `seed`: Integer (The random seed used for this specific generation run).

### 2.2 TabularMetadata
Statistical summary of structured features for a dataset.

*   **Source**: `code/analysis/metadata_stats.py`
*   **Storage**: `data/processed/metadata/{dataset_id}_metadata.json`
*   **Attributes**:
    *   `dataset_id`: String identifier.
    *   `cardinality`: Integer (number of unique values in categorical features).
    *   `missingness_rate`: Float (proportion of missing values, 0.0 to 1.0).
    *   `sparsity`: Float (proportion of zero values in sparse features).
    *   `variance`: Float (average variance of numerical features).
    *   `feature_count`: Integer (total number of tabular features).

### 2.3 EvaluationMetrics
Performance metrics for different pipeline stages.

*   **Source**: `code/analysis/correlation.py`
*   **Storage**: `data/artifacts/metrics.csv`
*   **Attributes**:
    *   `run_id`: String (Links back to the specific `FrozenEmbedding` rows).
    *   `dataset_id`: String.
    *   `task_type`: Enum ("classification", "regression").
    *   `metric_type`: Enum ("AUC", "RMSE", "MAE").
    *   `frozen_baseline`: Float (Mean of 5 seeds, re-computed in this pipeline).
    *   `frozen_baseline_variance`: Float (Variance of the 5 seeds).
    *   `cpu_conditioned`: Float (performance of projection model).
    *   `gpu_tuned_baseline`: Float (from paper/supplementary).
    *   `recovery_ratio`: Float (calculated).
    *   `metadata_stats`: JSON object (snapshot of metadata used).

## 3. Data Flow

1.  **Ingestion**: Raw MulTaBench data (`data/raw/`) is validated and checksummed.
2.  **Embedding**: `run_baseline.py` generates `FrozenEmbedding` artifacts with a global `run_id` and specific `seed`.
3.  **Metadata**: `metadata_stats.py` generates `TabularMetadata` artifacts.
4.  **Training**: `run_conditioned.py` trains the projection layer, producing `cpu_conditioned` metrics.
5.  **Analysis**: `run_analysis.py` aggregates metrics (averaging over 5 seeds for P_frozen), computes `recovery_ratio`, and generates the correlation table.
6.  **State Update**: `update_state.py` hashes artifacts and updates `state/projects/...yaml`.

## 4. Constraints

*   **Determinism**: Primary run uses `random_seed=42`. Sensitivity analysis uses seeds `[42, 123, 456, 789, 101]`.
*   **Immutability**: Raw data in `data/raw/` is never modified.
*   **Validation**: All generated files must pass the schema validation defined in `contracts/`.
*   **Traceability**: Every `EvaluationMetrics` row must contain a `run_id` that matches the `run_id` in the corresponding `FrozenEmbedding` rows.
