# Evaluating the Impact of Code Generation on Code Review Time

**Field**: computer science

## Research question

Does code generated by large language models (LLMs) require more or less time for effective code review compared to human-written code, and what factors contribute to any observed difference in review duration?

## Motivation

While LLM-powered code generation promises to accelerate development workflows, the downstream impact on code review efficiency remains unclear. Understanding whether LLM-generated code introduces unfamiliar patterns, subtle errors, or reviewer uncertainty that extends review timelines is critical for assessing the true productivity gains of these tools in software engineering practice.

## Related work

- [Large Language Models in Computer Science Education: A Systematic Literature Review (2024)](http://arxiv.org/abs/2410.16349v1) — Provides context on LLM adoption trends and their growing role in programming tasks.
- [SIMCOPILOT: Evaluating Large Language Models for Copilot-Style Code Generation (2025)](http://arxiv.org/abs/2505.21514v1) — Introduces a benchmark for evaluating interactive coding assistants, relevant for understanding LLM code quality.
- [Using StackOverflow content to assist in code review (2018)](http://arxiv.org/abs/1803.05689v1) — Demonstrates prior work on leveraging public data sources to support code review processes.
- [EVOR: Evolving Retrieval for Code Generation (2024)](http://arxiv.org/abs/2402.12317v2) — Explores retrieval-augmented generation techniques that may influence the characteristics of generated code.

## Expected results

We expect to find that LLM-generated code either requires longer review times due to unfamiliar patterns or shorter times if reviewers recognize standard LLM output. The key measurement will be the difference in median review duration (first review response time and total approval time) between LLM-generated and human-written pull requests, with statistical significance assessed at p < 0.05.

## Methodology sketch

- **Data acquisition**: Download pull request metadata from GitHub using the GitHub REST API (`api.github.com/repos/{owner}/{repo}/pulls`) for repositories with ≥1000 stars that have adopted Copilot or similar tools (identified via commit messages mentioning "copilot", "llm", or "generated").
- **LLM code identification**: Use commit message patterns and code style heuristics (e.g., consistent formatting, lack of inline comments) to classify code changes as likely LLM-generated vs. human-written; validate classification on a 50-sample manual review subset.
- **Review timeline extraction**: For each pull request, extract timestamps for: (1) PR creation, (2) first review comment, (3) last review comment, (4) merge/close. Calculate `first_review_time` and `total_review_time` in minutes.
- **Dataset construction**: Build a CSV with 2,000 pull requests (1,000 LLM, 1,000 human) including fields: repo, PR number, author, code_lines_changed, review_time_first, review_time_total, reviewer_count, iteration_count.
- **Statistical analysis**: Perform Mann-Whitney U test to compare median review times between LLM and human code groups (non-parametric due to likely skewed distribution).
- **Confounding variable control**: Include code size (lines changed), reviewer experience (contributor count), and repository activity level as covariates in a robust regression model.
- **Visualization**: Generate boxplots comparing review time distributions and a scatter plot of code size vs. review time with regression lines for each group.
- **Feasibility check**: All steps fit within 6-hour GitHub Actions job; API calls limited to 5,000/hour free tier; dataset stored as CSV (≤100 MB); analysis done in Python with pandas/scipy.
