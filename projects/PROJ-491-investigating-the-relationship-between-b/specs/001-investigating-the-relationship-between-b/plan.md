# Implementation Plan: Investigating the Relationship Between Brain Network Dynamics and Anticipatory Reward Processing

**Branch**: `001-gene-regulation` | **Date**: 2026-06-29 | **Spec**: `specs/001-gene-regulation/spec.md`
**Input**: Feature specification from `/specs/001-gene-regulation/spec.md`

## Summary

This project implements a computational neuroscience pipeline to investigate the **associational relationship** between resting-state brain network flexibility (derived from dynamic functional connectivity) and anticipatory reward processing (ventral striatum activation) using Human Connectome Project (HCP) data from OpenNeuro. The system downloads a subsample of subjects., extracts BOLD time series using the Power atlas, computes flexibility scores via sliding-window K-means clustering (normalized for scan length), and performs correlation analysis with permutation testing. The implementation is strictly constrained to run on a GitHub Actions free-tier runner with limited CPU and RAM resources and no GPU access..

**Testing**: `pytest` (unit tests for math/logic, integration tests for pipeline flow).  
**Target Platform**: Linux (GitHub Actions free-tier runner).  
**Project Type**: Computational research pipeline / CLI tool.  
**Performance Goals**: Complete full pipeline (A cohort of participants) within 6 hours; peak RAM < 7 GB.  
**Constraints**: No GPU usage; no deep learning libraries (PyTorch/TensorFlow) unless CPU-only and strictly necessary (avoided here); sliding window analysis must be memory-efficient (streaming or chunked if needed, though 50 subjects fits in RAM if handled carefully).  
**Scale/Scope**: 50 subjects; A substantial number of time points per subject (resting state); dynamic states.

## Phase Plan

### Phase 0: Research & Feasibility
- **Goal**: Confirm dataset availability, variable fit, and construct validity.
- **FR-001, FR-008, SC-004**: Verify OpenNeuro contains both resting-state and task-fMRI for A cohort of subjects with distinct session IDs.
- **FR-002, Assumption**: Confirm Power 264 atlas coordinates and Ventral Striatum ROI definition are compatible with HCP MNI152NLin2009cAsym space used in OpenNeuro. **ACTION**: Verify coordinate alignment; if mismatch, plan resampling.
- **FR-003, FR-009, Assumption**: Validate that sliding window K-means (K=4, seed=42) is computationally feasible for 50 subjects on CPU cores within 6 hours. **ACTION**: Confirm TR of downloaded data (expected a latency on the order of one second) to validate TR window is physiologically appropriate.
- **FR-005, FR-006**: Verify `scipy`/`numpy` can handle A sufficient number of permutations efficiently without GPU.

### Phase 1: Data Model & Contracts
- **FR-001, FR-002**: Define schema for raw NIfTI metadata and extracted CSV time series.
- **FR-003a**: Define schema for dynamic state sequences (`states.schema.yaml`).
- **FR-005, FR-006**: Define schema for correlation results (r, p, CI).
- **FR-007**: Define schema for final report structure (ensuring "associational" phrasing).

### Phase 2: Implementation
- **FR-001**: Implement `data_ingestion.py` with checksumming, session validation, and TR verification.
- **FR-002**: Implement `preprocessing.py` for BOLD extraction (Power 264 + VS).
- **FR-003, FR-003a, FR-004**: Implement `connectivity.py` for sliding window and flexibility (normalized for scan length).
- **FR-005, FR-006, FR-009**: Implement `analysis.py` for correlation, permutation, and sensitivity analysis.
- **FR-007**: Implement `reporting.py` to generate the final markdown report.
- **Edge Cases**: Implement logic to skip corrupted subjects, fail if <50 valid subjects, and handle zero-variance flexibility scores.
- **Constitution V**: Implement `state_manager.py` to compute content hashes for all generated artifacts and update `state/` YAML file.

### Phase 3: Testing & Validation
- **FR-001, FR-010**: Verify failure on insufficient data.
- **FR-002, FR-003**: Verify determinism (seed=42) and memory constraints.
- **FR-007**: Verify report content (no "causal", includes "associational").
- **SC-001, SC-002, SC-003**: Run full pipeline and measure resources and robustness.
- **Constitution V**: Verify that all artifacts in `data/processed/` and `paper/` are hashed and recorded in `state/`.
