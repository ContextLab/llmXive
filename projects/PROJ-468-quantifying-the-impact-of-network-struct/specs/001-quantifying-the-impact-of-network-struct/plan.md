# Implementation Plan: Quantifying the Impact of Network Structure on Energy Dissipation in Driven Granular Materials

**Branch**: `001-network-dissipation` | **Date**: 2026-06-27 | **Spec**: `spec.md`

## Summary

This project implements a computational pipeline to quantify the statistical association between contact network topology (coordination number, clustering coefficient, force heterogeneity) and energy dissipation rates in driven granular materials. The system ingests Yade-DEM simulation outputs, extracts time-series graph metrics, and performs autocorrelation-corrected regression (GLS/Newey-West) with robustness checks (Quantile Regression, GAM) to test hypotheses. The pipeline is designed to run entirely on free-tier GitHub Actions CPU runners (≤7GB RAM, ≤6h) by enforcing strict memory caps, automatic subsampling, and CPU-tractable statistical methods.

**Critical Note on Data Validity**: This pipeline distinguishes between **Code Validation** (testing logic on synthetic data) and **Scientific Validation** (testing hypotheses on real DEM data). If no verified real DEM dataset is available, the pipeline will produce a "Code Validated" report, but the scientific conclusions regarding network-dissipation relationships will be flagged as "Pending Real Data".

## Phase Breakdown

### Phase 0: Research & Data Strategy
- **Goal**: Confirm dataset availability and variable fit. **Gated Step**: Run `Reference-Validator` on all citations.
- **Action**: Verify that the candidate HuggingFace datasets contain `particle_positions`, `contact_forces`, and `energy` fields.
- **Decision Logic**:
  - If real data exists: Proceed to scientific validation.
  - If NO real data exists: **Proceed with Synthetic Data for CODE VALIDATION ONLY**. Mark "Scientific Validation" as BLOCKED in the final report. Do NOT proceed with a "simplified custom solver" that lacks physics; use a minimal synthetic generator only to test code paths (parsing, graph construction) and explicitly flag that results are not scientifically valid.
- **Output**: `research.md` with dataset strategy table and "Code vs. Science" status.

### Phase 1: Data Model & Contracts
- **Goal**: Define strict input/output schemas.
- **Action**: Create YAML schemas for the extracted time-series data (including `metadata` header) and the final regression results (including `driving_amplitude` control variable and ANOVA results).
- **Output**: `data-model.md`, `contracts/*.schema.yaml`.

### Phase 2: Implementation (Delegated to Implementer Agent)
- **Goal**: Code the pipeline.
- **Action**: Implement `extract.py`, `analyze.py`, `viz.py` adhering to the schemas.
- **Key Tasks**:
  - **FR-003 Edge Cases**: Implement the specific zero-work fallback logic: `if Work_Input == 0: Dissipation = |ΔKE + ΔPE|`. Implement normalization only for non-zero work cases.
 - **FR-008 Memory**: Implement `linear extrapolation` estimation on the first [deferred] of the file as per spec.
  - **FR-009 Metrics**: Implement `degree_distribution_entropy` (Topology-Only) and `force_chain_percolation` (Force-Only) to distinguish structural connectivity from force transmission.
  - **FR-005b Control**: Explicitly include `driving_amplitude` as a control variable in the GLS model.
  - **Metadata**: Extract simulation metadata from raw files and populate the `metadata` header in the processed dataset, verifying against `dataset.schema.yaml`.
  - **Robustness**: Implement GAM check for non-linearity and Quantile Regression for heavy-tailed errors.

### Phase 3: Validation & Reporting
- **Goal**: Generate PDF report and validate results.
- **Action**: Run pipeline on test data; generate `results/report.pdf`.
- **Gated Step**: Run `Reference-Validator` on all citations in the report.
- **SC-003 Check**: Explicitly check the ANOVA p-value against the standard significance threshold. If p > 0.05, record "Slopes Consistent" in the report. If p ≤ 0.05, record "Slopes Differ".
- **Output**: Final PDF with scatter plots, heatmaps, and statistical summaries, including a "Data Validity" section distinguishing Code vs. Science results.

## Risk Mitigation

- **Risk**: Dataset lacks required variables (e.g., contact forces).
  - **Mitigation**: `research.md` explicitly states the variable mismatch. If no real data exists, the project proceeds with "Code Validation Only" and flags "Scientific Validation" as blocked.
- **Risk**: OOM on large files.
  - **Mitigation**: FR-008 logic triggers subsampling immediately if estimated RAM > 6GB using linear extrapolation.
- **Risk**: Non-stationary time series.
  - **Mitigation**: FR-010 mandates ADF test; if non-stationary, data is segmented into fixed-length windows.
- **Risk**: Circular/Non-Gaussian inference.
  - **Mitigation**: Use "Absolute Dissipation" as primary outcome. Use Quantile Regression and GAM for robustness.
