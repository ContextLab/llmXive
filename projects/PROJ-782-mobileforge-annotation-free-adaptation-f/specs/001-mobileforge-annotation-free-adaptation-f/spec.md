# Feature Specification: MobileForge: Annotation-Free Adaptation for Mobile GUI Agents with Hierarchical Feedback-Guided Policy Optimization

**Feature Branch**: `782-mobileforge-reproduction`  
**Created**: 2024-05-22  
**Status**: Draft  
**Input**: User description: "Reproduce & validate: MobileForge: Annotation-Free Adaptation for Mobile GUI Agents with Hierarchical Feedback-Guided Policy Optimization"

## User Scenarios & Testing

### User Story 1 - Execute Core Reproduction Pipeline (Priority: P1)

**Description**: The researcher MUST be able to execute the vendored MobileForge codebase end-to-end on a CPU-only CI runner to generate initial rollout artifacts and evaluation logs without requiring GPU acceleration or external Android device emulation that exceeds resource limits.

**Why this priority**: This is the foundational step. Without a successful execution of the pipeline on the target hardware constraints (CPU-only, ~7GB RAM), no validation of the paper's claims is possible. It verifies the code's viability in the constrained environment.

**Independent Test**: Can be fully tested by running the `rollout/run.py` and `evaluation/androidworld/run.py` scripts with a subset of tasks and verifying the existence of generated JSON trajectory logs and a summary metrics file.

**Acceptance Scenarios**:
1. **Given** the `MobileForge` submodule is cloned and dependencies are installed, **When** the user runs the entry point script with a `--num-tasks 10` flag on a CPU-only runner, **Then** the system MUST complete the rollout phase within 60 minutes and produce a `trajectories.jsonl` file containing at least 10 valid entries.
2. **Given** the rollout artifacts exist, **When** the user triggers the evaluation script against the `AndroidWorld` registry, **Then** the system MUST produce an `evaluation_results.json` file containing success/failure flags for each task without crashing due to memory limits (>7GB) or timeout (>6h).
3. **Given** the pipeline runs, **When** the process encounters a missing dependency or incompatible library, **Then** the system MUST exit with a non-zero status code and print a clear error message identifying the missing component within 10 seconds.

---

### User Story 2 - Validate Hierarchical Feedback (HiFPO) Data Generation (Priority: P2)

**Description**: The researcher MUST be able to verify that the Hierarchical Feedback-Guided Policy Optimization (HiFPO) module correctly generates step-level feedback and corrective hints from the trajectory data, ensuring the "annotation-free" aspect produces usable training signals.

**Why this priority**: The core novelty of MobileForge is the HiFPO mechanism. Validating that it produces structured feedback (not just pass/fail) is essential to confirming the method's ability to adapt the agent without human labels.

**Independent Test**: Can be tested by inspecting the intermediate data artifacts generated by the `explore/curriculum_generator_refactored_add_aw_few-shot/main.py` script and confirming the presence of `step_feedback` and `hints` fields in the output data structure.

**Acceptance Scenarios**:
1. **Given** a set of completed rollout trajectories, **When** the curriculum generator script processes them, **Then** the output data MUST contain a `feedback` object for at least 80% of the steps where the task was not completed, with fields for `error_type` and `hint_text`.
2. **Given** the feedback data exists, **When** the data is sampled for review, **Then** the `hint_text` field MUST be non-empty and grammatically complete (length ≥ 10 characters) for at least 90% of the annotated steps.
3. **Given** the feedback generation runs, **When** the system encounters a trajectory with a silent failure (no error code), **Then** the system MUST flag this trajectory with a `warning` status rather than silently dropping it or crashing.

---

### User Story 3 - Reproduce Performance Metrics Against Baseline (Priority: P3)

**Description**: The researcher MUST be able to compute and report the Pass@k success rate on the AndroidWorld benchmark for the adapted agent and compare it against the reported baseline (GUI-Owl-1.5-8B) to validate the paper's claim of reaching a target success rate.

**Why this priority**: This is the final validation step. It confirms whether the reproduction matches the scientific claims of the paper, providing the ultimate "truth" for the project.

**Independent Test**: Can be tested by aggregating the results from the evaluation run and calculating the Pass@3 metric, then comparing it to the paper's abstract value ([deferred]).

**Acceptance Scenarios**:
1. **Given** the evaluation results for the adapted agent, **When** the researcher calculates the Pass@3 metric, **Then** the result MUST be reported with a 95% confidence interval calculated via bootstrapping (1000 iterations) over the task set.
2. **Given** the baseline results are loaded from the paper's reference or a cached run, **When** the comparison is made, **Then** the report MUST explicitly state the absolute percentage point difference and the relative improvement percentage.
3. **Given** the computed metric, **When** the result differs from the paper's claim by more than 5 percentage points, **Then** the system MUST flag this as a "Discrepancy Alert" and generate a detailed log of the specific tasks that failed differently than expected.

---

### Edge Cases

- **What happens when the Android emulator (or simulated environment) fails to start due to CI resource constraints?** The system MUST fall back to a "headless" or "mock" representation mode defined in `evaluation/androidworld/env/interface.py` and log a `SIMULATION_MODE` flag, ensuring the pipeline continues without crashing, though results may be marked as "approximate."
- **How does the system handle a trajectory where the agent enters an infinite loop?** The system MUST enforce a hard timeout of 300 seconds per task. If exceeded, the task MUST be terminated, marked as `TIMEOUT`, and the partial state MUST be saved for analysis.
- **What if the `Qwen3-VL-8B` model weights cannot be downloaded due to network restrictions?** The system MUST detect the missing weights at initialization, halt execution with a `MISSING_MODEL_WEIGHTS` error, and provide a clear instruction to manually download the weights to the specified cache directory.

## Requirements

### Functional Requirements

- **FR-001**: The system MUST execute the `rollout/run.py` entry point to generate interaction trajectories for at least 10 distinct tasks from the AndroidWorld registry within a single CI job (See US-1).
- **FR-002**: The system MUST implement a memory-monitoring mechanism that triggers a graceful shutdown if RAM usage exceeds 6.5 GB (leaving 0.5 GB buffer) to prevent OOM crashes on the free-tier runner (See US-1).
- **FR-003**: The system MUST parse the trajectory logs to extract step-level outcomes and generate a structured feedback object containing `error_type`, `step_id`, and `hint_text` for failed steps (See US-2).
- **FR-004**: The system MUST calculate the Pass@3 metric by aggregating the success status of the top 3 generated attempts for each task and outputting a single scalar value with a 95% confidence interval (See US-3).
- **FR-005**: The system MUST validate that every generated feedback hint is non-empty and meets a minimum length threshold of 10 characters before writing it to the training dataset (See US-2).
- **FR-006**: The system MUST enforce a hard timeout of 300 seconds per task execution to prevent infinite loops, marking any task exceeding this limit as `TIMEOUT` (See Edge Cases).
- **FR-007**: The system MUST detect missing model weights (e.g., `Qwen3-VL-8B`) at startup and exit with a specific error code `1` rather than attempting to proceed with a null model (See Edge Cases).

### Key Entities

- **Trajectory**: A sequence of states, actions, and observations generated by the agent interacting with the Android environment. Key attributes: `task_id`, `steps` (list), `final_status`.
- **Feedback Signal**: A structured output derived from a trajectory step, containing `error_type`, `hint_text`, and `step_context`.
- **Evaluation Metric**: A statistical summary of agent performance, specifically `Pass@3` (percentage of tasks solved within 3 attempts) and its confidence interval.

## Success Criteria

### Measurable Outcomes

- **SC-001**: The rollout phase MUST complete for 10 tasks within 60 minutes on a CPU-only runner, measured against the CI job timer (See FR-001, FR-002) (See US-1).
- **SC-002**: The generated training dataset MUST contain valid `hint_text` fields for ≥ 90% of the failed steps, measured by counting non-empty strings against the total number of failed steps (See FR-003, FR-005) (See US-2).
- **SC-003**: The reported Pass@3 metric MUST include a 95% confidence interval derived from 1000 bootstrap iterations, measured by verifying the presence of `lower_bound` and `upper_bound` in the output JSON (See FR-004) (See US-3).
- **SC-004**: The system MUST successfully terminate any task exceeding 300 seconds without crashing the entire pipeline, measured by the presence of `TIMEOUT` status flags in the logs (See FR-006) (See Edge Cases).
- **SC-005**: The system MUST exit with error code `1` and a clear message if model weights are missing, measured by the exit code and error log content (See FR-007) (See Edge Cases).

## Assumptions

- **Assumption about hardware**: The reproduction assumes the free-tier GitHub Actions runner (2 CPU, ~7GB RAM) is sufficient to run the `Qwen3-VL-8B` model in 4-bit or 8-bit quantization mode on CPU, or that a smaller distilled version is used as a fallback, as full-precision 8B models on CPU may exceed the 6-hour job limit.
- **Assumption about data**: The `AndroidWorld` task registry and associated app environments are available within the `external/MobileForge` submodule and do not require external network downloads of large APK files during the CI run.
- **Assumption about environment**: The `MobileGym` environment can be simulated or mocked on a Linux-based CI runner without requiring a physical Android device or a full Android emulator instance, relying on the `adb_utils` and `interface.py` abstractions.
- **Assumption about model weights**: The `Qwen3-VL-8B` weights are either pre-cached in the runner or accessible via a public HuggingFace mirror that does not require authentication tokens during the CI execution.
- **Assumption about inference framing**: Since the reproduction is observational (running existing code), any performance claims are framed as "observed performance on this specific run" and not causal claims about the method's universal superiority.
