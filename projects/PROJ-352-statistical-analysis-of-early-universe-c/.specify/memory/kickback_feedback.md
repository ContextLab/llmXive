## Unresolved concerns

- Phase 4 states: 'Compute area, perimeter, and genus... using healpy’s pixel-based estimators'. healpy does not provide native Minkowski Functional estimators. This plan relies on a non-existent library function, rendering the core measurement invalid. A custom implementation or external library (e.g., `pyMinkowski`) must be specified.
- Phase 5 claims to generate 1,000 simulations in ~2.5h on a 2-CPU runner. Generating 1,000 full-sky Nside=128 Gaussian realizations with beam convolution and noise injection is computationally intensive. The timing estimate appears optimistic and risks violating the 6h constraint (FR-007), potentially forcing a reduction in sample size that undermines statistical power.
- Phase 9 proposes aborting simulations at 500 if RAM > 6.5GB. Reducing N from 1,000 to 500 significantly increases the variance of the covariance matrix estimate used in the Hotelling's T² test. The plan does not account for the resulting loss of statistical power or the potential for Type II errors (false negatives) in the null hypothesis test.
- Phase 4 specifies thresholds 'e.g., 5 equally spaced σ levels'. Minkowski Functionals are most sensitive to non-Gaussianity at specific thresholds (often near 0σ and ±1σ). Arbitrary 'equally spaced' levels may miss the peak sensitivity of the genus curve, reducing the test's ability to detect the specific topological signatures of cosmic strings.
- The plan assumes generating 1,000 Gaussian random-field simulations on a GitHub Actions free-tier runner (2 CPU, 7GB RAM) within the 6-hour window. Generating 1,000 full-sky Nside=128 maps with beam convolution and noise injection via `healpy.synfast` is computationally intensive. Even with vectorization, the I/O overhead of writing 1,000 FITS files (approx. 12MB each) plus the CPU cost of spherical harmonic transforms likely exceeds the 6-hour limit and risks OOM errors before the adaptive fallback triggers. The plan's timing estimate (2.5h for generation) appears optimistic for the free tier without explicit batch-streaming or in-memory-only processing strategies.
- The 'Dataset Strategy' table lists 'Power spectrum' as a dataset to be loaded from 'Planck Legacy Archive – planck_likelihood/ files'. However, `healpy.synfast` requires the full TT, TE, EE power spectra (Cl arrays) as input, not just the likelihood files. The plan does not explicitly cite the specific data product (e.g., 'COM_CMB_IQU-2048_R3.00_full.fits' or a specific Cl file) that contains the necessary Cl_TT, Cl_TE, Cl_EE arrays. Relying on 'likelihood files' may result in missing the required spectral data for simulation generation.
- The 'Source (Verified)' column for the Primary CMB map states 'Planck Legacy Archive (no URL provided in verified block)'. While the upstream resolver guarantees reachability, the plan text itself should explicitly reference the specific dataset accession or URL pattern (e.g., 'COM_CMB_SMICA_2048_R3.00_full.fits') to ensure the implementation script knows exactly which file to fetch, rather than relying on a generic archive reference.
- Phase 9 ('Resource Monitoring & Adaptive Simulation Count') describes a silent fallback mechanism: 'abort further simulations after 500 runs and log a warning'. This violates Constitution Principle II (Verified Accuracy) and Principle V (Versioning Discipline) by allowing the pipeline to complete with a reduced sample size (500 vs 1000) without explicitly failing the build or requiring a manual override. The plan must enforce a hard failure or a distinct 'partial success' state that halts downstream analysis, rather than silently proceeding with reduced statistical power.
- Phase 5 states: 'add white Gaussian noise (σ² = 1.1 µK² arcmin)'. The data-model.md defines the 'GaussianSimulation' entity with 'noise_variance: float = 1.1', but the unit 'µK² arcmin' is dimensionally inconsistent with a variance term in a standard HEALPix simulation context (which typically expects µK²). While likely a typo for 'µK²', the plan must align the unit description exactly with the data model's implied units or the spec's definition to ensure contract validation passes.
- Phase 4 mentions computing functionals 'at a set of thresholds (e.g., 5 equally spaced σ levels)'. The data-model.md defines 'MinkowskiFunctionalResult' with 'thresholds: List[float]', but the spec (US-3) and research.md explicitly define the thresholds as '±0.5σ, ±1σ, 0σ' (5 specific values). The plan should reference these specific values or the exact list from the spec to ensure the 'e.g.' does not lead to implementation drift where a different set of thresholds is chosen.
- Phase 10 lists 'generate a CSV of results' as a deliverable. The contracts/analysis_results.schema.yaml defines a JSON structure for 'hotelling_result.json' and 'observed_minkowski' (array). The plan should explicitly name the output file format (JSON vs CSV) to match the contract schema, as the contract does not define a CSV schema for the final Hotelling result.
- The plan performs a Hotelling's T² test comparing observed Minkowski Functionals (MFs) to the mean of simulations generated from the *same* Planck power spectrum. This is a circular validation. The MFs are functionals of the field; if the simulations perfectly match the observed power spectrum (as required by Phase 5), the MFs of the simulations will, by definition, cluster around the MFs of the observed map (which generated the spectrum). The test will almost certainly fail to reject the null hypothesis regardless of non-Gaussian defects, because the null model is constructed to match the primary statistic (the power spectrum) of the data. The validation target (MF deviation) is not independent of the predictor (power spectrum used to generate sims).
- The plan claims to 'translate the significance into an upper bound on Gμ' if the null is rejected. However, the Hotelling's T² test only detects *any* deviation from Gaussianity. It does not distinguish between cosmic strings, primordial non-Gaussianity (e.g., local f_NL), or systematic errors. A rejection of the null does not imply a detection of topological defects. The mapping from a generic 'non-Gaussian p-value' to a specific physical parameter (Gμ) is scientifically invalid without a specific defect template or a likelihood ratio test comparing a 'Gaussian + Defect' model against a 'Gaussian' model.
- Generating 1,000 Gaussian random fields using `healpy.synfast` with the *observed* Planck power spectrum creates a null model that is conditioned on the data. While standard for checking Gaussianity, the plan fails to account for the fact that the observed power spectrum itself contains cosmic variance. A more rigorous null model for defect constraints would involve generating simulations from the *theoretical* LCDM power spectrum (with cosmic variance) or using a Monte Carlo approach that marginalizes over the power spectrum uncertainty, rather than fixing the spectrum to the observed noisy realization.
- The plan proposes an 'Adaptive Simulation Count' where the simulation count drops from 1,000 to 500 if RAM > 6.5GB. This introduces a data-dependent stopping rule that invalidates the statistical properties of the Hotelling's T² test. The degrees of freedom and the distribution of the test statistic depend on the sample size (N). Changing N based on runtime resource usage (a random variable in this context) biases the p-value calculation. The test assumes a fixed N determined *a priori*.
