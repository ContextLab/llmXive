# Missing‑Data Handling (Extended Documentation)

This document provides a detailed description of how missing data are detected,
recorded, and dealt with throughout the **quantifying-the-complexity-of-knot-diagr**
project.  It complements the shorter overview in
`docs/reproducibility/missing_data_handling.md` and is intended for developers
and reproducibility auditors who need to understand the exact workflow.

## Detection

* **Schema validation** – Each input record is validated against the JSON‑Schema
  definitions found in `specs/010-quantifying-the-complexity-of-knot-diagr`
  (e.g., `dataset.schema.yaml`).  Missing required fields trigger a *validation
  error* that is captured by the `code/data/validator.py` module.
* **Custom flagging** – Certain fields are optional but critical for downstream
  analysis (e.g., `hyperbolic_volume`, `braid_index`).  The `code/analysis`
  sub‑package contains utility functions (`_utils.py`) that explicitly check
  for `None` or empty strings and raise a `MissingDataError`.

## Recording

When missing data are encountered, the system records the event in two places:

1. **Validation flags file** – `data/processed/validation_flags.json` stores a
   mapping from record identifiers to a list of missing‑field codes.  This file
   is generated by `code/analysis/validation_status_generator.py`.
2. **Log entries** – The reproducibility log (`data/logs/reproducibility.log`
   and the human‑readable version in `docs/reproducibility/LOGS_LOCATION.md`)
   contains timestamped entries such as:

   ```
   2026-06-18T14:32:07Z [MISSING_DATA] record_id=K1234 missing_fields=["hyperbolic_volume"]
   ```

## Handling Strategies

The project adopts a *conservative* approach: any analysis that requires a
missing field will **exclude** the corresponding record unless the user opts
in to apply imputation.  The supported strategies are:

| Strategy | Description | Implementation |
|----------|-------------|----------------|
| **Exclude** | Drop the record from the computation. | Implemented in `code/analysis/data_quality.py` via `filter_missing()`.
| **Mean Imputation** | Replace numeric missing values with the column mean. | Available through `code/analysis/_utils.py::impute_mean()`; requires explicit call.
| **Custom Imputation** | User‑provided function or model‑based prediction. | Hooked via the `impute_custom` parameter in `code/analysis/composite_metric_*` modules.

All default pipelines (e.g., `complexity_visualization_runner.py`) use the **Exclude**
strategy to avoid bias.

## Configuring the Pipeline

The handling method can be overridden in the configuration YAML files under
`specs/010-quantifying-the-complexity-of-knot-diagr/contracts/`.  For example, to
enable mean imputation for the `crossing_number` field, add:

```yaml
imputation:
  crossing_number: mean
```

The `code/analysis/composite_metric_extended.py` module reads this configuration
and applies the appropriate function before metric computation.

## Auditing Missing Data

* **Summary report** – Run `python -m code.analysis.data_quality_report` to
  generate `docs/reproducibility/data_quality_report.md`, which includes a table
  of missing‑field frequencies.
* **Interactive map** – The `interactive_knot_map.py` visualizer highlights
  records with missing data in red, allowing users to inspect patterns.

## Future Work

* Implement advanced imputation techniques (e.g., K‑NN, regression‑based) and
  expose them via a plugin system.
* Provide a command‑line flag `--handle-missing=impute` that automatically
  selects the best strategy based on data distribution.
