# Research: llmXive follow-up: extending "AutoResearchClaw"

## 1. Research Question & Hypothesis

**Primary Question**: Can deterministic rules distilled from structural features derived from agent reasoning traces outperform or match a full probabilistic baseline agent in "Time-to-Pivot" and "Success Rate" for specific failure types (syntactic, logical) while operating under severe resource constraints (2 CPU, 7 GB RAM)?

**Hypothesis**:
- **H1**: For **Syntactic Errors** and **Logical Loops**, the distilled rule engine will achieve a significantly lower "Time-to-Pivot" than the baseline agent with no statistically significant difference in "Success Rate".
- **H2**: For **Semantic Ambiguity** and **Missing Context**, the distilled rule engine will have a significantly lower "Success Rate" (higher failure rate) than the baseline, as these require probabilistic context retrieval.
- **H3**: The interaction term (Failure Type * Method) in the mixed-effects logistic regression will be statistically significant (p < 0.05), confirming that the viability of the rule-based method is contingent on the structural nature of the failure.

## 2. Dataset Strategy

### 2.1 Source Verification
The project relies on the **ARC-Bench** dataset. Per the verified datasets list, we will utilize the following sources:
- **Primary Source**: ` (and equivalents for Bengali/Gujarati).
- **Verification Note**: The spec mentions "ARC-Bench 25-topic subset". The verified list contains Indic-language ARC-C datasets. These datasets contain `question`, `answer`, and `reasoning` traces, but **NOT** explicit `error_log` columns.
- **Mitigation Strategy**: We will **NOT** synthesize failure logs. Instead, we will parse the `reasoning` traces to derive structural features. For example, a trace containing "SyntaxError" or "IndentationError" will be annotated as "Syntactic Error". A trace containing "I am stuck in a loop" or repeated steps will be annotated as "Logical Loop". This ensures the data is derived from the source, not generated by the researcher.

### 2.2 Data Loading Strategy
- **Loader**: `datasets.load_dataset("parquet", data_files=...)`
- **Sampling**: To ensure CPU feasibility (7 GB RAM), the dataset will be loaded in chunks. Only the `reasoning`, `answer`, and `topic` columns will be retained.
- **Derivation**: The `01_data_ingestion/parse_reasoning_traces.py` script will scan the `reasoning` field for keywords and patterns (e.g., "Error:", "Exception:", "Timeout", "loop") to generate a `derived_error_log` field. This field serves as the input for annotation.

### 2.3 Dataset Variable Fit Check
- **Required Variables**: `derived_error_log` (parsed from reasoning), `resolution` (ground truth action), `topic` (stratification).
- **Availability**: The verified HuggingFace parquet files contain `reasoning` traces.
- **Gap Analysis**: The verified datasets do **not** explicitly contain a column named `error_log`.
- **Mitigation Strategy**: The `01_data_ingestion` phase will include a transformation step that **derives** `derived_error_log` fields from the `reasoning` traces. This is a necessary adaptation to the available verified data, ensuring the "Failure Type" is an emergent property of the data, not a synthetic injection.

## 3. Methodology

### 3.1 Phase 1: Annotation & Distillation (US-1)
- **Input**: 50 failure cases (derived from reasoning traces).
- **Annotation**: Manual or semi-automated labeling of `derived_error_log` into: `Syntactic Error`, `Logical Loop`, `Semantic Ambiguity`, `Missing Context`, `Unstructured`.
- **Distillation**: Use a CPU-tractable small model (e.g., `TinyLlama-1.1B` or `Llama-3-8B` if RAM permits) to generate "If-Condition-Then-Action" rules.
- **Constraint**: The model must run on CPU. If `Llama-8B` exceeds 7 GB RAM, the implementation will fallback to `TinyLlama-1.1B` or a regex-based heuristic distillation.
- **Output**: `rules_library.json` (validated against `contracts/distilled_rule.schema.yaml`).

### 3.2 Phase 2: Execution & Comparison (US-2)
- **Rule Engine**: A lightweight Python script that parses `derived_error_log`, matches against `rules_library.json` (using regex or exact string match), and executes the `pivot_action`. This runs on the 2-core CI runner.
- **Baseline Execution**: The full "AutoResearchClaw" baseline agent **MUST** be executed on the same tasks. Since it cannot run on the 2-core/7GB CI runner, it will be executed in a parallel workflow on a larger runner (e.g., 4 cores, 16 GB RAM) or locally. The results (logs) will be uploaded to a shared artifact storage and merged with the CI results.
- **Metrics**: `Time-to-Pivot` (seconds), `Success` (binary).
- **Stratification**: By `Failure Type`.
- **Paired Design**: The same `task_id` is used for both the Rule Engine and the Baseline to enable the mixed-effects model.

### 3.3 Phase 3: Statistical Analysis (US-3)
- **Model**: Mixed-effects logistic regression.
- **Formula**: `Success ~ Failure_Type * Method + (1 | Task_ID)`
- **Tools**: `statsmodels` (Python).
- **Power Analysis**: Before running the model, a power analysis will be conducted. If the sample size (100 tasks) is insufficient to detect the interaction term, the analysis will default to a robust logistic regression, and the study will be framed as a "feasibility study" with explicit limitations.
- **Correction**: Bonferroni correction for multiple comparisons if testing multiple interaction terms.
- **Error Taxonomy**: Manual inspection of failed pivots to categorize as "Coverage Gap" vs "Distillation Error".

## 4. Statistical Rigor & Feasibility

- **Multiple Comparisons**: The interaction term test is the primary hypothesis. If post-hoc tests are run per failure type, Bonferroni correction will be applied.
- **Power Analysis**: The sample size (100 tasks) is small. The plan acknowledges this limitation. Power will be calculated post-hoc; if underpowered, the conclusion will be framed as "preliminary evidence" rather than definitive proof.
- **Causal Inference**: This is an observational comparison (Rule Engine vs. Baseline). Claims will be limited to "associational" benefits of the rule engine for specific failure types, not causal claims of superiority in all contexts.
- **Collinearity**: `Failure Type` and `Method` are the primary predictors. No definitional collinearity is expected, but `Failure Type` categories are mutually exclusive by design.
- **Compute Feasibility**:
 - **No GPU**: All steps use CPU.
 - **Memory**: Data is sampled to < 1 GB. The distillation model will be the smallest viable CPU model (likely < 4 GB RAM).
 - **Time**: The pipeline is designed for < 2 hours total runtime to stay well within the 6-hour CI limit. The baseline execution is offloaded to a separate runner to avoid CI timeout.

## 5. Decision Rationale

| Decision | Rationale |
|:--- |:--- |
| **Derive Failure Logs from Reasoning Traces** | The verified datasets lack explicit `error_log` columns. Deriving features from `reasoning` traces ensures the data is empirical and not synthetic, preserving construct validity. |
| **External Baseline Execution** | Running a full multi-agent debate baseline on a 2-core CPU/7GB RAM runner is infeasible. Offloading to a larger runner ensures the comparison is empirical and not simulated, preserving causal validity. |
| **Small CPU Model for Distillation** | Large LLMs cannot run on the target CI. A parameter model in the range of billions (e.g., TinyLlama, Llama-3-8B-INT4 if available as CPU wheel) is the maximum viable option for rule generation. Fallback to TinyLlama is mandatory if RAM is exceeded. |
| **Mixed-Effects Model with Power Check** | Required to account for the nested structure of data (multiple pivots per task). A power check ensures the model is appropriate; if not, a simpler model is used. |
