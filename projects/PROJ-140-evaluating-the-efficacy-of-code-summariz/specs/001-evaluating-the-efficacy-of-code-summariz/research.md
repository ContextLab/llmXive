# Research: Evaluating Code Summarization Techniques for Bug Localization

## Problem Statement

The research aims to determine if code summaries generated by Large Language Models (LLMs) or rule-based extractors improve the speed and accuracy of bug localization compared to a baseline (no summary) for graduate-level software engineers.

## Statistical Analysis Plan

### 1. Accuracy Analysis (Binary Outcome)
- **Metric**: **Top-5 Accuracy** (1 if `ground_truth_line` is within the top 5 selected lines by the participant, else 0).
 - *Rationale*: 'Exact line match' is likely to result in a floor effect (near [deferred] accuracy) for all conditions, leading to sparse discordant pairs (<10) and invalid McNemar's tests. Top-5 accuracy provides sufficient variation for statistical testing.
- **Test**: **McNemar's Test**.
 - *Rationale*: Paired nominal data (same participant performs task under Condition A and Condition B).
 - *Comparisons*: Baseline vs. LLM-Sim, Baseline vs. Rule-based.
 - *Fallback*: If discordant pairs < 10, switch to **Exact Binomial Test** or aggregate at participant level and use **Wilcoxon Signed-Rank Test**.
- **Effect Size**: **Odds Ratio (OR)** with 95% CI via bootstrapping (1,000 resamples, fixed seed).

### 2. Speed Analysis (Continuous Outcome)
- **Metric**: Time-to-decision (milliseconds).
- **Transformation**: `log(time + 1)` to handle right-skewed distribution.
- **Test**: **Linear Mixed-Effects (LME) Models**.
 - *Formula*: `log_time ~ condition + (1 | participant_id) + (1 | task_id)`
 - *Rationale*: Accounts for repeated measures (random intercept for participant) and **task difficulty** (random intercept for task) to avoid pseudoreplication.
 - *Comparisons*: Baseline vs. LLM-Sim, Baseline vs. Rule-based.
 - *Diagnostics*: Shapiro-Wilk test for residual normality. If violated, switch to **Gamma GLMM** or non-parametric alternative.
- **Effect Size**: **Cohen's d** (marginal) with 95% CI via bootstrapping.

### 3. Multiple Comparison Correction
- **Method**: **Holm-Bonferroni** correction.
- **Rationale**: Multiple hypothesis tests are performed (accuracy and speed). This controls the family-wise error rate (FWER) at α=0.05.

### 4. Power & Sample Size Considerations
- **Assumption**: 12 participants, 30 tasks each (360 observations).
- **Power Analysis**: For McNemar's test with 12 participants and 30 tasks, the study has low power to detect small effect sizes (OR < 1.5). The study is explicitly framed as a **pilot** to estimate effect sizes and confidence intervals, rather than to definitively prove a hypothesis.
- **Sensitivity Analysis**: Results will be reported across α levels {0.01, 0.05, 0.10} to assess robustness.

## Compute Feasibility & Methodological Adjustments

### LLM Generation Constraint
The spec requires a Code Llama model with 8-bit quantization.
- **Conflict**: 8-bit quantization requires CUDA/GPU. The CI environment is CPU-only.
- **Resolution**:
 1. **Analysis Phase**: The `run_statistics.py` script will load summaries generated by `code/data_prep/generate_summaries.py`.
 2. **Generation Logic**:
 - **LLM-Sim (Default for CI)**: A deterministic algorithm: "Method: <name>\nParams: <args>\nReturns: <type>\nComment: <first_comment>". This mimics the structure of an LLM summary without requiring inference.
 - **Rule-Based**: Uses `srcML` to extract comments and structure.
 - **Real LLM (Optional)**: A separate script can run real LLM inference if GPU is available, but the CI run uses LLM-Sim to ensure reproducibility and CPU feasibility.
 3. **Fallback**: If LLM generation fails (timeout, empty, or GPU not available), it defaults to the rule-based summary.

### Resource Budget
- **RAM**: ≤7 GB. The dataset (60 methods) is tiny (<1MB). The statistical models (LME) with a limited number of rows are trivial.
- **Time**: ≤6 hours. The analysis will take <5 minutes.
- **Disk**: ≤14 GB. All artifacts fit easily.

## Risks & Mitigations

| Risk | Impact | Mitigation |
|:--- |:--- |:--- |
| **Dataset Mismatch** | Defects4J might not have the specific "prompt" fields needed. | Use `adapt_defects4j` which contains raw source and line numbers. Fallback to other verified datasets if columns missing. |
| **Participant Dropout** | <12 participants complete the study. | Use LME models which handle missing data (unbalanced design) better than repeated-measures ANOVA. |
| **LLM Failure** | LLM generation fails for many methods. | The fallback to rule-based summary ensures every task has a condition. The analysis will flag these as "Fallback" cases. |
| **Outliers** | Participants taking >30 mins per task. | Flag outliers in `data/interaction_logs.csv`. Perform sensitivity analysis excluding participants with ≥2 outliers. |
| **Sparsity in Accuracy** | Top-5 accuracy still too sparse. | Switch to 'Buggy Method Identification' (coarser) or non-parametric tests. |
