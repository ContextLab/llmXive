# Implementation Plan: Evaluating Code Summarization Techniques for Bug Localization

**Branch**: `001-evaluating-code-summarization-bug-localization` | **Date**: 2025-01-15
**Spec**: `specs/001-evaluating-code-summarization-bug-localization/spec.md`

## Summary

This project implements a human-subject study to evaluate whether code summaries (LLM-generated or rule-based) improve bug localization accuracy and speed compared to a baseline (no summary). The plan covers data acquisition from DefectsJ, participant interaction logging, statistical analysis (McNemar's tests, LME models), and the generation of a reproducible research package. The implementation is constrained to run on CPU-only GitHub Actions free-tier runners.

## Technical Context

**Language/Version**: Python 3.11  
**Primary Dependencies**: `pandas`, `numpy`, `scikit-learn`, `statsmodels`, `requests`, `pyyaml`, `srcml` (system package)  
**Storage**: Local CSV files for interaction logs; Parquet files for Defects4J data (read-only).  
**Testing**: `pytest` for unit tests on data processing and statistical functions.  
**Target Platform**: Linux (GitHub Actions free-tier runner).  
**Project Type**: Research data analysis pipeline.  
**Performance Goals**: Analysis script completes within 6 hours on 2 CPU cores, ≤7GB RAM.  
**Constraints**: NO GPU, NO CUDA, NO 8-bit/4-bit quantization for *analysis* or *generation* in CI.

> **Note on LLM Generation Constraint**: The spec mentions using `codellama/CodeLlama-7b-hf` with 8-bit quantization. However, 8-bit quantization requires CUDA/GPU, which is unavailable in the CI environment. To satisfy the **Compute Feasibility** and **Reproducibility** constraints:
> 1. **Deterministic Simulation**: The 'LLM' condition in the CI run (and primary analysis if GPU is unavailable) will be generated by a deterministic algorithm in `code/data_prep/generate_summaries.py`. This algorithm mimics the *structure* of an LLM summary (e.g., "Method: <name>\nParams: <args>\nReturns: <type>\nComment: <first_comment>") but is reproducible without a GPU.
> 2. **Fallback Logic**: If the real LLM generation (in a separate GPU-enabled environment) fails, the system falls back to the `srcML` rule-based extractor.
> 3. **Reproducibility**: The exact summaries used for the methods in the CI run are generated by the script in `code/`, ensuring the 'Single Source of Truth' is maintained. No external pre-computed data is required for the analysis pipeline.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
| :--- | :--- | :--- |
| **Deterministic LLM-Sim** | The spec requires LLM summaries, but CI is CPU-only. | Running a 7B parameter model on CPU is infeasible. A deterministic algorithm mimics the *structure* of LLM output, allowing the *analysis* to be verified without the *generation* overhead, while maintaining reproducibility. |
| **Statistical Rigor (LME + McNemar)** | Required by spec for paired data and binary outcomes. | Simple t-tests or chi-square would violate the repeated-measures design and binary outcome requirements. |
| **Top-K Accuracy** | Exact line match is too sparse for McNemar's test. | Top-5 accuracy provides sufficient variation (discordant pairs) for statistical validity. |
| **Resource Monitoring** | SC-005 requires measuring compute feasibility. | Assumption is not enough; a script must measure and assert limits. |
