## Unresolved concerns

- The plan proposes a 'Paired Wilcoxon signed-rank test' to compare R²/RMSE distributions across halide pairs. However, the data structure (5-fold CV) yields only 5 paired observations per halide pair (one per fold). A Wilcoxon test on N=5 is statistically underpowered to detect any meaningful effect size, rendering the comparative claim invalid regardless of the p-value. The methodology must be revised to either aggregate metrics differently or acknowledge the impossibility of statistical significance testing at this sample size.
- The plan states: 'Compare top 3 features against known chemical determinants... If no overlap, flag chemically implausible.' This introduces a circular validation loop where the model is judged against the very hypothesis it is meant to test. Without an independent ground truth or a blind validation set of known determinants, this 'validation' is merely a consistency check that cannot falsify the model's predictive capability. The construct validity of this 'chemical plausibility' metric is unsound.
- The fallback strategy (FR-011) switches to 'simulated data' if real data is insufficient. While this ensures pipeline execution, the plan fails to define the generative model for the simulation. If the simulation does not explicitly encode the known physics of halide binding (e.g., electrostatics, cavity size), the resulting 'feature importance' and 'selectivity trends' will be artifacts of the random generator, not the chemistry. The methodology for generating the fallback data is missing, making the fallback results scientifically meaningless.
- FR-013 requires validating top features against a 'curated list of known chemical determinants from literature'. The plan.md mentions 'Compare top 3 features against known chemical determinants' in Section 3.5, but does not define the source of this curated list, nor does it specify how the 'chemically implausible' flag is triggered if no overlap is found. This leaves the success criterion (SC-007) unmeasurable as the reference standard is undefined in the plan.
- FR-001 mandates downloading data from NIST and PubChem. The plan.md (Section 2.1) explicitly states 'No verified raw URL found' for NIST and that the implementation 'will attempt to scrape' or use a fallback. However, the plan does not define the specific scraping logic, rate limits, or error handling for the NIST source, nor does it detail the fallback mechanism's trigger conditions beyond a generic 'if primary sources are insufficient'. This creates a gap in the implementation path for a mandatory requirement.
- FR-011 requires switching to a simulated dataset if <50 hosts or missing columns. The plan.md (Section 2.1) mentions a fallback to simulated data, but the spec requires the system to 'reduce the scope to single-halide prediction' and 'log a warning that the original research question (comparative analysis) is unanswerable'. The plan.md does not explicitly describe the logic for reducing the scope to single-halide prediction or the specific warning message format, leaving the requirement partially unbacked.
- SC-007 requires measuring chemical plausibility against a 'curated list of known chemical determinants'. The plan.md (Section 3.5) mentions comparing features but does not specify the 'curated list' content or the threshold for 'overlap' (e.g., top 3 features must match at least 1 from the list). Without this definition in the plan, the success criterion cannot be implemented or tested.
- The plan cites 'NIST Chemistry WebBook' and 'WebBooks-1' as primary sources for binding constants, but explicitly admits the verified list lacks a direct URL for NIST halide binding data and that 'WebBooks-1' is a text dataset, not a structured chemistry table. The plan relies on a 'simulated data fallback' (FR-011) to run the pipeline, but the research question (predicting halide selectivity trends) requires real experimental data to be answerable. Relying on simulated data for the primary comparative analysis renders the research question unanswerable with the cited resources.
- The plan lists 'PubChem' (sagawa/pubchem-10m-canonicalized) and 'SMILES Transformers' as sources for host structures, but correctly notes these lack the specific 'binding_constant' column. The plan fails to identify a verified dataset that actually contains the required (Host SMILES, Halide ID, Binding Constant) tuple. Without a real dataset containing these specific variables, the proposed ML pipeline cannot train on the phenomenon of interest, only on synthetic noise.

- The 'Source Code' structure lists `code/data_ingestion.py` as handling FR-001, FR-010, FR-011. However, FR-011 (Simulated Data Fallback) is a critical branching logic that significantly alters the data flow. The plan does not explicitly describe how the 'Simulated' source flag (defined in data-model.md) is propagated through the pipeline to downstream modules like `model_training.py` or `analysis.py`, creating a potential integration gap.
- The plan states 'Fallback: Simulated data if primary sources are insufficient.' The `data-model.md` defines a `source` enum with 'Simulated' and a constraint (Invariant 3) that a warning flag must be set. The plan does not explicitly map the 'Simulated' data generation step to the specific contract or schema validation that ensures the simulated data adheres to the `dataset.schema.yaml` (e.g., ensuring `host_smiles` pattern validity).
- The plan proposes using a 'Paired Wilcoxon signed-rank test' to compare R²/RMSE distributions across halide pairs. This is a category error: the model is trained on a single dataset containing all halides, and the 'per-halide' metrics are derived from the *same* cross-validation folds (same host splits). The R² values for F⁻, Cl⁻, etc., are not independent samples nor paired observations of the *same* underlying entity in a way that Wilcoxon applies. The 'pairing' is artificial (same fold index), but the variance comes from the model's performance on different chemical subsets, not a paired difference in a single subject. A more valid approach is to compare the *distributions* of residuals or use a bootstrap approach to test if the difference in mean R² is significantly non-zero, rather than a paired test on fold-level aggregates which assumes a specific dependency structure not guaranteed here.
- The plan states: 'Compare top 3 features against known chemical determinants... If no overlap, flag chemically implausible.' This validation step is circular and tautological. The 'known chemical determinants' (e.g., H-bond donors) are likely *already included* as input features (RDKit descriptors) in the model. If the model selects them, it confirms the model learned the input; if it doesn't, it may indicate the model found a non-intuitive pattern or the 'known' list is incomplete. Using the presence of a feature in the input set as the 'ground truth' for validation is not an independent validation of the model's predictive power or chemical insight. It validates that the model can find the features you gave it, not that it predicts binding correctly.
- The plan requires a 'post-hoc power analysis' targeting power ≥0.8 for effect size ≥0.1. In the context of comparing model performance metrics (R²) across halides with a small sample size (N < 50 hosts), post-hoc power analysis is statistically controversial and often uninformative. Power is a function of sample size, effect size, and alpha. If the sample size is fixed and small, calculating power *after* seeing the data (or assuming a specific effect size) does not validate the result; it merely restates the sample size constraint. This analysis adds no scientific value and may mislead readers into thinking the study was 'powered' when it was not. The plan should focus on confidence intervals for the performance differences instead.
