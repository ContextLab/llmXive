# Data Model: Reproduce & validate: TUA-Bench

## Overview

This document defines the data structures used during the reproduction pipeline. The model focuses on the input task definition, the execution result, the independent oracle, and the aggregated report. All data is persisted as JSON or YAML files to ensure reproducibility and easy parsing.

## Entity Definitions

### 1. Task Definition
Represents a single unit of work in the TUA‑Bench.

```yaml
Task:
  task_id: string       # Unique identifier (e.g., "000-count-nuclei")
  family: string        # Task family (e.g., "file-manipulation")
  instruction_path: string # Relative path to instruction.md
  config_path: string   # Relative path to task.toml
  solution_dir: string  # Path to solution/
  reference_dir: string # Path to tests/reference/
  environment:
    dockerfile: string  # Path to Dockerfile
    image_name: string  # Derived or explicit image name
```

### 2. Execution Result
The output of a single task execution, generated by the orchestrator.

```yaml
ExecutionResult:
  task_id: string
  family: string
  status: enum[pass, fail, timeout, oom, error]
  execution_time_seconds: number
  error_message: string? # Nullable; one of "File Missing", "Value Mismatch", "Timeout", "OOM", "Network Timeout", or "Unknown Error"
  verification_score: number? # 0.0 or 1.0 if applicable
  timestamp: string      # ISO 8601
```

### 3. Independent Oracle (Ground Truth)
Hand‑crafted expected output for a task, used to validate the scoring protocol independently of the benchmark’s own reference files.

```yaml
Oracle:
  task_id: string
  expected_output: string   # Exact terminal output or JSON representation
```

### 4. Aggregated Report
The final summary document comparing results to paper claims.

```yaml
ReproductionReport:
  metadata:
    benchmark_version: string
    total_tasks_attempted: integer
    total_tasks_passed: integer
    success_rate: number
    paper_claimed_count: integer
    paper_claimed_families: integer
    actual_families_found: integer
    success_rate_scope: string # "offline_subset_only"
    deviations:
      - description: string
        severity: enum[low, medium, high]
  task_results:
    - ExecutionResult
  family_summary:
    - family: string
      count: integer
      pass_rate: number
```

## Schema Validation

The `contracts/` directory contains YAML schemas for `ExecutionResult` and `ReproductionReport`. These schemas are used by the test harness to validate the JSON reports generated by the pipeline.

## Data Flow

1. **Input**: `external/TUA-Bench/tasks/` (directory structure).  
2. **Processing**: `scripts/run_validation.py` parses directories, builds containers, executes tasks, runs verification (original `verify.py`) against either the original reference or an **independent oracle** (for control cases), and captures errors.  
3. **Output**: `results/<task_id>.json` (ExecutionResult) and `results/summary.json` (ReproductionReport).  
4. **Validation**: The report is compared against the paper’s claims (SC‑003, SC‑005) and explicitly notes the scope limitation of the success‑rate metric.

## Error Reason Mapping (FR‑007)

The orchestrator must capture `stderr` from `verify.py` and map it to the standardized `error_message` using the following regex patterns:

| Pattern | Mapped Reason |
|---------|---------------|
| `FileNotFoundError` | File Missing |
| `ValueError|expected.*got` | Value Mismatch |
| `TimeoutError|Timed out` | Timeout |
| `MemoryError|OOM` | OOM |
| `NetworkError|Connection refused|Network timeout` | Network Timeout |
| *no match* | Unknown Error |

`verify.py` is required to emit one of the above keywords in its error output for the mapping to succeed.
