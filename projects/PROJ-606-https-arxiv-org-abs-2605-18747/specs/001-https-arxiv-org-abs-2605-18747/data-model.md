# Data Model: Reproduce & Validate Code as Agent Harness

## 1. Overview

This document defines the data structures used by the harness to track execution, artifacts, and validation results. The model is designed to be lightweight (JSON/Text) to fit within the 7 GB RAM constraint and to be easily parsable by the validation scripts. All structures are validated against the schemas in `contracts/`.

## 2. Entity Definitions

### 2.1 Execution Log
Captures the state of the subprocess execution.
- **Fields**: `timestamp`, `exit_code`, `stdout_snippet`, `stderr_snippet`, `duration_seconds`, `gpu_detected` (boolean), `resource_warning` (string, optional).

### 2.2 Artifact Manifest
A record of all files generated by the submodule.
- **Fields**: `path`, `size_bytes`, `type` (e.g., `json`, `png`, `log`), `checksum` (MD5).

### 2.3 Validation Mapping
The core link between code output and paper claims.
- **Fields**: `claim_id` (e.g., "C1-Interface"), `claim_text`, `artifact_id`, `evidence` (string describing the match, **must include semantic content**), `status` (supported, partial, unsupported).

### 2.4 Limitation Record
A record of reproducibility barriers.
- **Fields**: `category` (API, Dependency, Hardcoded, Resource), `description`, `sensitivity_recommendation` (string).

## 3. Schema Definitions

### 3.1 Execution Log Schema (Internal)
```json
{
  "timestamp": "ISO8601",
  "exit_code": "integer",
  "stdout_snippet": "string",
  "stderr_snippet": "string",
  "duration_seconds": "float",
  "gpu_detected": "boolean",
  "resource_warning": "string (optional)"
}
```

### 3.2 Artifact Manifest Schema (Internal)
```json
{
  "artifacts": [
    {
      "path": "string",
      "size_bytes": "integer",
      "type": "string",
      "checksum": "string"
    }
  ]
}
```

### 3.3 Validation Report Structure (Output)
The `validation_report.md` will be structured as a Markdown document, but internally represented as a list of mappings:
```json
{
  "mappings": [
    {
      "claim_id": "C1-Interface",
      "claim_text": "Unified interface for agents",
      "artifact_id": "output/interface_config.json",
      "evidence": "Found 'agent_list' key with 3 items (Semantic check passed)",
      "status": "supported"
    }
  ]
}
```

## 4. Data Flow

1.  **Ingest**: Submodule runs -> `stdout`/`stderr` captured -> `Execution Log` created.
2.  **Scan**: `output/` directory scanned -> `Artifact Manifest` created.
3.  **Validate**: Validation script reads `Artifact Manifest` and `paper_abstract.txt` -> **Validates against `contracts/` schemas** -> Generates `Validation Mappings`.
4.  **Output**: Mappings rendered to `validation_report.md`; Limitations rendered to `limitations.md`.
