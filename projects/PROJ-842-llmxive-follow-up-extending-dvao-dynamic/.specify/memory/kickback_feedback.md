## Unresolved concerns

- FR-006 (Spec) and SC-002 (Spec) require a 'one-sample t-test' and 'coincidence' metric. The Plan explicitly revises this to a 'Paired T-Test' and 'Correlation Analysis' to avoid tautology. Task T021 in Phase 2 explicitly states: 'One-sample t-test against theoretical bound is explicitly FORBIDDEN'. However, there is NO task in Phase 2 or Phase 5 that implements the REQUIRED 'Paired T-Test' logic (Heuristic vs. Full-Batch) as the primary validation. T038 (Phase 5) implements the paired test, but T021's prohibition creates a gap where the foundational stats module lacks the primary validation logic until Phase 5, and the 'forbidden' tag in T021 might mislead implementers into thinking the validation logic is entirely absent or contradictory. The Plan's revision must be fully reflected in the task decomposition without leaving 'forbidden' tags on foundational tasks that block the new logic.
- SC-001 requires the derived bound to be verified by a 'peer review checklist'. Task T030 creates the checklist file. However, there is no task defined to actually EXECUTE the peer review process or record the sign-off. Without a task to 'Conduct peer review' and 'Record sign-off', the success criterion SC-001 (which requires the checklist to confirm algebraic consistency) cannot be met or measured by an automated workflow or agent.

However, FR-006 also requires a 'one-sample t-test' (Spec) which the Plan replaces with a 'Paired T-Test'. Task T038 implements the Paired T-Test. The Spec's FR-006 is technically contradictory to the Plan. The tasks correctly follow the Plan, but the lack of a task explicitly mapping to the 'Stability Check' as a distinct success criterion (SC-003) separate from the Paired T-Test (SC-002) creates ambiguity. T039 covers the check, but the traceability to SC-003 is weak in the task description compared to T038's explicit link to the Paired T-Test.
- FR-009 requires sensitivity analysis on noise correlation structure (rho in {0, 0.2, 0.5}). Task T036 extends the MDP generator to support this. However, there is no task in Phase 4 or Phase 5 that explicitly RUNS the simulation with these different rho values and LOGS the results to verify if the scaling law holds.
