## Unresolved concerns

- FR-008 requires halting and logging 'Model Incapability' if pass rate < 1%. Task T018 logs 'Potentially Biased' if difference > 10pp, but no task explicitly implements the 'halt and log Model Incapability' logic for the < 1% threshold.
- The plan mandates computing diversity metrics for ALL generated samples (pre-filter) to mitigate selection bias. While T025 mentions computing metrics for 'ALL' samples, it is placed in Phase 3 (US2) which logically depends on US1 (filtering). There is no explicit task to generate `metrics_all.csv` *before* or *independently* of the filtering logic, risking the 'Survivorship Bias' data gap.
- Task T032 references 'Dunn's test' as a post-hoc for the LMM. This is a methodological mismatch (Dunn's is for Kruskal-Wallis), but more critically, it implies a task exists for a test that the spec (FR-006) requires but the plan/tasks have replaced. The tag [US3] is valid, but the content describes a method not aligned with the implemented LMM or the spec's Kruskal-Wallis requirement.
- User Story 2 (Metrics) tasks (T021-T027) are ordered AFTER User Story 1 (Generation) tasks (T012-T018). However, T025 explicitly requires computing metrics for 'ALL generated samples (pre-filter)' and T026 for 'VALID samples (post-filter)'. The current ordering implies T025 depends on T017 (which writes 'samples.csv' containing only valid samples per T017 description). This creates a data-flow violation: the 'pre-filter' artifact (all samples) is not produced by T017 (which filters), yet T025 needs it. T017 must be split or reordered to ensure 'all samples' are persisted before filtering, or T025 must be moved to run immediately after generation (T014) and before filtering (T015).
- T017 description states: 'write data/processed/samples.csv (task_id, style, sample_id, code, pass_status)'. This implies the CSV contains only passed samples (as per T015 filtering). However, the Plan (Phase 2.5) and Spec (FR-009) require preserving 'ALL generated samples' to compute 'metrics_all.csv' for survivorship bias analysis. T017 as written destroys the 'pre-filter' data required by T025. A task to write 'samples_all.csv' (unfiltered) must precede T015, or T017 must be split into 'write_all' and 'write_valid' steps.
- T031 (LMM) and T032 (Dunn's test) are ordered after T030 (Normalization). However, T034 (Survivorship bias comparison) requires comparing 'Valid' vs 'All Generated' results. The 'All Generated' metrics are produced by T025 (US2), and 'Valid' metrics by T026 (US2). T034 is in US3 (Stats). The ordering is correct (US2 before US3), BUT T034 depends on the *output* of T025 and T026. If T025 is broken (see first concern), T034 cannot function. Additionally, T035 (Reporter) depends on T034. The chain is valid only if T025 is fixed to actually produce the 'all' metrics.
- T010 and T011 are marked [P] (parallel-safe) and placed in Phase 3. However, T010 tests 'prompt loading' which depends on T004 (Prompt templates). T004 is in Phase 2. This is a valid dependency (Phase 2 before Phase 3). However, T011 tests 'generation loop' which depends on T012 (Loader), T013 (Generator), T014 (Loop), T015 (Tester). Placing tests *before* implementation tasks (T012-T018) in the same phase is a TDD pattern, but the [P] tag implies they can run in parallel with T012-T018. This is semantically incorrect: T011 cannot run in parallel with T013/T014 if it tests them. The [P] tag on T011 is misleading; it should be sequential to the implementation or the implementation tasks should be split to allow test-first execution of specific modules.
- Task T031 mandates implementing a Linear Mixed-Effects Model (LMM) using statsmodels. However, the spec (FR-006, SC-001) and plan (Phase 4.3) explicitly mandate a Kruskal-Wallis H-test (or explicitly forbid LMM in favor of KW in the spec). The task description contradicts the functional requirement it is supposed to implement, making it impossible to execute deterministically without resolving the spec/plan conflict first.
- Task T032 requires implementing 'Dunn's test with Bonferroni correction' as a post-hoc for the LMM. Dunn's test is the post-hoc for Kruskal-Wallis, not LMM (which typically uses pairwise t-tests or Tukey HSD). This task is methodologically incoherent and cannot be executed as written because the statistical procedure described does not match the model specified in T031.
- Task T031 implements a Linear Mixed-Effects Model (LMM) and T032 implements Dunn's test. This directly violates FR-006 and US-3 Acceptance Scenario 1, which explicitly mandate a 'Kruskal-Wallis H-test' followed by 'Dunn's post-hoc test'. The task list silently replaces the required statistical methodology with a different one (LMM), weakening the FR-006 constraint.
