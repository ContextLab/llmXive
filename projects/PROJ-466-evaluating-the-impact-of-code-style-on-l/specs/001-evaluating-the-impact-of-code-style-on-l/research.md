# Research: Evaluating the Impact of Code Style on LLM Code Generation Diversity

## Problem Statement

This research investigates whether explicit stylistic constraints in prompts (Neutral, Strict PEP8, Aggressive Minification) modulate the structural diversity of code solutions generated by small-scale LLMs. The hypothesis is that strict constraints reduce the entropy of the latent search space, leading to lower structural diversity (measured by AST edit distance and n-gram entropy) compared to neutral prompts.

## Methodology Overview

1.  **Data Source**: HumanEval benchmark (a set of programming tasks).
2.  **Model**: `Salesforce/codegen-2B-mono` (CPU-optimized, 2B parameters).
3.  **Experimental Design**: Within-subjects design. Each task is solved under three style conditions.
4. **Samples**: **20 samples** per task per style (total [deferred] samples for 164 tasks).
5.  **Metrics**: Pairwise AST edit distance, token n-gram entropy (aggregated at task level).
6.  **Analysis**: Kruskal-Wallis H-test for group differences; Dunn's post-hoc for pairwise comparisons; Sensitivity analysis on significance thresholds.

## Statistical Rigor & Methodological Decisions

### 1. Multiple Comparisons & Family-Wise Error
- **Method**: When the Kruskal-Wallis H-test indicates a significant difference (p < 0.05), we will perform **Dunn's post-hoc test** with **Bonferroni correction**.
- **Rationale**: We are comparing three groups (Neutral, PEP8, Minified). This results in 3 pairwise comparisons. Bonferroni correction adjusts the alpha level to `0.05 / 3 ≈ 0.0167` to control the family-wise error rate.
- **Implementation**: `scikit-posthocs` or a custom implementation of the rank-based method with Bonferroni adjustment.

### 2. Sample Size & Power
- **Constraint**: The HumanEval benchmark comprises a standardized set of programming tasks. We will generate **20 samples** per task per style.
- **Unit of Analysis**: The statistical test is performed on **Task-level aggregates** (mean pairwise distance per task). The unit of analysis is the Task (N=164), ensuring independence of observations.
- **Power Analysis**: With 164 tasks, the power to detect a medium effect size (Cohen's f ≈ 0.35) in a Kruskal-Wallis test with 3 groups is > 80% at α=0.05.
- **Metric Stability**: Generating a sufficient number of samples per task yields a substantial number of pairwise comparisons per task., providing a robust estimate of the diversity distribution for each task, mitigating the variance issues associated with smaller sample sizes (e.g., N=5).
- **Fallback**: If the test yields a non-significant result with high variance, the report will explicitly state "Insufficient Power to Detect Effect" rather than claiming no effect.

### 3. Causal Inference Assumptions
- **Design**: This is a **controlled experiment** where the prompt style is the manipulated independent variable.
- **Claim**: We can infer **causal impact within the model's response distribution** for *valid* solutions. The randomization is implicit in the model's temperature sampling (T=0.7) and the fixed seed (42) ensures reproducibility of the *distribution* of samples.
- **Caveat**: We cannot claim these results generalize to *human* coding behavior or other models without further study. The causal claim is strictly limited to the `Salesforce/codegen-2B-mono` model under the specified temperature.

### 4. Measurement Validity
- **AST Edit Distance**: Validated metric for structural code difference. Implemented via `networkx` on AST graphs.
- **N-gram Entropy**: Validated metric for token-level variability.
- **Functional Correctness**: Validated by the official HumanEval unit tests.

### 5. Predictor Collinearity (FR-017)
- **Issue**: AST distance and N-gram entropy are likely correlated.
- **Action**: FR-017 mandates computing **Spearman rank correlation** (corrected from Pearson due to non-normal distribution).
- **Interpretation**: A high correlation (r > 0.9) is acknowledged as a **tautological consequence** of the data generation (structural similarity implies token similarity). However, to satisfy the functional requirement, the system will **flag** this redundancy and recommend using **AST distance** as the primary metric for the final report, effectively selecting the more robust structural metric.

### 6. Missing Data Mechanism & Survivorship Bias
- **Scenario**: If a style group has a pass rate < 1%, the system halts (T018).
- **Handling**: No imputation is performed. The study reports "No Diversity Data Available" for that group in the primary analysis.
- **Bias Quantification**: A "Survivorship Bias Report" is generated to compare the pass rates of the style groups. If the pass rate difference exceeds 10 percentage points, the analysis is flagged as "Potentially Biased".
- **Implication**: The comparison of diversity is limited to groups that passed the functional correctness threshold. The report will explicitly discuss this limitation as a qualitative finding of "Model Incapability" for that style, rather than attempting to infer diversity from non-functional code.

## Compute Feasibility & Resource Constraints

- **Hardware**: GitHub Actions Free Tier (2 CPU, 7 GB RAM, 14 GB Disk, No GPU).
- **Model**: `codegen-2B-mono` is ~4GB in 16-bit precision. Loading it leaves sufficient space for data and OS.
- **Strategy**:
  1.  **Dynamic Batch Sizing**: Start with batch size 50. Monitor RAM. If >6GB usage, reduce to a small number, then a smaller number, then 1.
  2.  **CPU Inference**: Use `torch.set_num_threads(2)` to limit CPU contention.
  3.  **Data Sampling**: If the full set of tasks exceeds a predefined time limit, the system will process a representative subset. (e.g., 50 tasks) and scale the results, logging the subset size.
  4.  **No Quantization**: Avoid `bitsandbytes` (CUDA dependency). Use standard 16-bit float (float16) if memory permits, otherwise float32 with careful batch control.

## Decision Log

| Decision | Rationale |
| :--- | :--- |
| **Use `codegen-2B-mono`** | Smallest viable CodeGen model that fits in 7GB RAM on CPU while maintaining reasonable code generation quality. |
| **Kruskal-Wallis over ANOVA** | Diversity metrics (AST distance) are not guaranteed to be normally distributed. Non-parametric test is robust. |
| **Dunn's Post-hoc** | Standard non-parametric post-hoc for Kruskal-Wallis. Bonferroni correction ensures rigorous error control. |
| **Pre-filter Metrics** | Required to detect "Survivorship Bias". If style constraints cause code to fail tests, diversity of *valid* code might be artificially low or high. We must compare valid vs. all. |
| **Sensitivity Analysis** | To ensure results are not artifacts of an arbitrary p-value cutoff (e.g., 0.05 vs 0.01). |
| **Spearman Correlation** | Mandated by FR-017. Corrected from Pearson due to non-normal distribution. Used as a redundancy flag to select AST distance if correlation is high. |
| **Sample Size N=20** | Increased from 5 to 20 to provide 190 pairwise comparisons per task, stabilizing the diversity metric distribution and ensuring robust statistical power. |
