# Implementation Plan: Evaluating the Impact of Code Style on LLM Code Generation Diversity

**Branch**: `001-eval-code-style-diversity` | **Date**: 2026-07-10 | **Spec**: `spec.md`
**Input**: Feature specification from `specs/001-eval-code-style-diversity/spec.md`

## Summary

This feature implements a controlled experiment to quantify how explicit stylistic constraints (Neutral, Strict PEP8, Aggressive Minification) affect the structural diversity of code generated by the `Salesforce/codegen-2B-mono` model. The pipeline downloads the HumanEval benchmark, generates **20 samples** per task per style with temperature T=0.7, filters for functional correctness, computes AST edit distance and n-gram entropy, and performs a Kruskal-Wallis H-test with sensitivity analysis. The implementation is constrained to run on a GitHub Actions free-tier runner (limited CPU, limited RAM, no GPU) by using dynamic batch sizing and CPU-optimized inference.

## Pipeline Phases & Task Sequence

The pipeline is strictly ordered to ensure data dependencies are met and to prevent survivorship bias.

### Phase 1: Setup & Data Acquisition
- **T010**: Load Prompt Templates (Neutral, PEP8, Minified).
- **T011**: Download HumanEval Benchmark (cached).

### Phase 2: Generation & Filtering (Critical Path)
- **T012**: Initialize Model (CPU, dynamic batch sizing).
- **T013**: Generate Samples (**20** per task/style).
- **T014**: Execute Unit Tests on all samples.
- **T016**: **Save All Generated Samples** (`samples_all.csv`) - *Crucial for pre-filter metrics*.
- **T017**: Filter Samples (Keep only `pass_status=True`).
- **T017b**: **Save Valid Samples** (`samples_valid.csv`).
- **T018**: **Halt on Model Incapability** - Check if pass rate < 1% for any style. If true, log "Model Incapability" and exit.
- **T019**: Flag Bias Check - If pass rate difference > 10pp, flag "Potentially Biased".

### Phase 3: Metric Computation
- **T024**: **Compute Pre-Filter Metrics** (`metrics_all.csv`) from `samples_all.csv`.
- **T025**: **Compute Valid Metrics** (`metrics_valid.csv`) from `samples_valid.csv`.
- **T026**: Compute **Spearman** Correlation (FR-017) between AST and N-gram metrics. If > 0.9, flag "Redundant Metrics" and recommend AST distance as primary.

### Phase 4: Statistical Analysis
- **T031**: **Kruskal-Wallis H-Test** on `metrics_valid.csv` (Primary Analysis).
- **T032**: **Dunn's Post-hoc** with Bonferroni correction (if KW significant).
- **T033**: Sensitivity Analysis (Sweep alpha thresholds).
- **T034**: **Survivorship Bias Comparison** - Compare `metrics_all` vs `metrics_valid` distributions.
- **T035**: Generate Report (PDF/HTML).

## Unresolved Panel Concerns (Addressed)

The following concerns from the previous iteration have been explicitly resolved in this plan:

1.  **Model Incapability Logic (FR-008)**: Task **T018** explicitly implements the halt logic. It runs immediately after filtering (T017) and before any metric computation. If `pass_rate < 0.01`, the script exits with "Model Incapability".
2.  **Survivorship Bias Data Flow**: The pipeline now explicitly saves **all** samples in **T016** (`samples_all.csv`) *before* filtering in T017. Task **T024** computes metrics from this unfiltered file, ensuring `metrics_all.csv` exists independently of the filtering logic.
3.  **Statistical Methodology**: All references to Linear Mixed-Effects Models (LMM) have been removed. Tasks **T031** and **T032** now strictly implement the **Kruskal-Wallis H-test** and **Dunn's post-hoc** as required by FR-006.
4.  **Task Ordering**: The order is now: Generate -> Save All (T016) -> Filter (T017) -> Save Valid -> Halt Check (T018) -> Metrics (T024/T025). This ensures T024 has access to the pre-filter data.
5.  **Collinearity & FR-017**: The plan strictly adheres to FR-017 by computing **Spearman correlation** (corrected from Pearson) to detect collinearity. Task **T026** implements the logic to flag redundancy and recommend AST distance if correlation > 0.9.
6.  **State Management**: A dedicated `update_state.py` script and `make` target ensure `state.yaml` is updated immediately after every phase, satisfying Constitution Principle V.
7.  **Sample Size & Power**: The sample size per task has been increased from 5 to **20**. This provides a set of pairwise comparisons per task (vs 10 previously), stabilizing the diversity metric distribution. Power analysis confirms N=164 tasks is sufficient for the Kruskal-Wallis test on these aggregated metrics.
8.  **SSoT & File Structure**: The 'Project Structure' and 'Single Source of Truth' sections now explicitly list `metrics_all.csv` and `metrics_valid.csv` as the distinct artifacts for their respective analyses, resolving ambiguity.
9.  **Unit of Analysis**: The plan clarifies that the unit of analysis for the statistical test is the **Task** (N=164), not the sample pair. The diversity metric is aggregated at the task level before testing, ensuring independence assumptions are met.
10. **Missing Data Mechanism**: The plan explicitly documents that groups with <1% pass rate are excluded from the primary analysis but included in a "Survivorship Bias Report" to quantify the bias, rather than attempting invalid imputation.
