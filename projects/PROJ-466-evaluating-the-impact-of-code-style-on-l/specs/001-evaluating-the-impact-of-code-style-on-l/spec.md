# Specification: Evaluating the Impact of Code Style on LLM Code Generation Diversity

## 1. Introduction

This project evaluates how different code style constraints (Neutral, PEP8, Minified) affect the structural diversity of code generated by Large Language Models (LLMs) for the same functional problem. We utilize the HumanEval benchmark to generate multiple solutions per task under varying style prompts and analyze their syntactic variance.

## 2. Objectives

- Quantify structural diversity using AST edit distance and n-gram entropy.
- Determine if specific code styles lead to significantly lower or higher diversity.
- Assess statistical significance of observed differences.
- Generate a comprehensive report with visualizations and recommendations.

## 3. Functional Requirements

### FR-001: Data Source
The system must use the `openai/human-eval` dataset from the Hugging Face `datasets` library as the source of programming problems.

### FR-002: Style Profiles
The system must support three distinct prompt styles:
1. **Neutral**: No specific style constraints.
2. **Strict PEP8**: Enforce Python PEP8 style guidelines.
3. **Minified**: Enforce code minification (no whitespace, short variable names).

### FR-003: Sample Generation
The system must generate **exactly 20 code samples** per task per style profile. This ensures sufficient statistical power for diversity metrics.

### FR-004: Functional Filtering
The system must execute generated code against the original HumanEval unit tests. Only samples passing all unit tests are considered "valid" for diversity analysis.

### FR-005: Diversity Metrics
The system must calculate:
- **AST Edit Distance**: Pairwise structural difference between valid solutions.
- **N-gram Entropy**: Token-level diversity within the solution set.

### FR-006: Statistical Testing
The system must perform a Kruskal-Wallis H-test to determine if diversity distributions differ significantly across the three style groups.

### FR-007: Sensitivity Analysis
The system must perform a sensitivity analysis by sweeping significance thresholds (α) and reporting the stability of the results.

### FR-008: Reporting
The system must generate a final report (PDF/HTML) containing:
- Summary statistics per style group.
- Statistical test results (H-statistic, p-value).
- Post-hoc analysis (Dunn's test with Bonferroni correction).
- Sensitivity analysis plots.
- Survivorship bias comparison.

### FR-009: Data Hygiene
All input and output artifacts must be checksummed (SHA256) to ensure reproducibility.

### FR-010: Error Handling
The system must handle timeouts and memory constraints gracefully, logging errors and skipping problematic tasks without halting the entire pipeline.

### FR-011: Power Analysis
The system must include a power analysis section in the documentation confirming that N=164 tasks is sufficient for the intended statistical power (0.8) at alpha=0.05.

### FR-012: Survivorship Bias
The system must compare diversity metrics calculated on "All Generated" samples versus "Valid" samples to quantify survivorship bias.

### FR-013: Zero Variance Detection
The system must detect and log cases where a style group exhibits zero variance in diversity metrics.

### FR-014: Dynamic Batch Sizing
The generation pipeline must dynamically adjust batch sizes based on available memory to prevent OOM errors.

### FR-015: Timeout Enforcement
The generation pipeline must enforce a configurable timeout per task using a signal-based decorator.

### FR-016: Bias Flagging
If the difference in pass rates between style groups exceeds a substantial threshold, the system must flag "Potentially Biased" in the final report metadata.

### FR-017: Collinearity and Suggestion
The system must compute the **Spearman correlation coefficient** between AST distance and n-gram entropy. If the correlation coefficient (r) exceeds 0.9, the system must **inject a suggestion into the final report** stating: "Suggestion: Use AST Distance only" to indicate metric redundancy.

## 4. Non-Functional Requirements

- **Performance**: The full pipeline must complete within 6 hours on a standard cloud runner.
- **Reproducibility**: All random seeds must be fixed to a deterministic value to ensure experimental reproducibility.
- **Modularity**: Code must be organized into distinct modules for generation, analysis, and reporting.
- **Logging**: Comprehensive logging to `data/logs/` for debugging and audit trails.

## 5. Data Model

- **Task**: ID, Prompt, Solution
- **Sample**: TaskID, Style, Code, PassStatus, ASTMetrics, NgramMetrics
- **Result**: Style, MeanAST, MeanEntropy, StdDev, Count

## 6. Assumptions

- **Power Analysis**: This confirms the full HumanEval benchmark is sufficient for statistical validity.
- The `openai/human-eval` dataset is accessible via the Hugging Face Hub.
- The LLM API or local model has sufficient capacity to generate 20 samples per task without degradation in quality.

## 7. Constraints

- Maximum execution time per task: A bounded, finite duration suitable for the operational context.
- Memory limit: Sufficient RAM (dynamic adjustment required).
- No synthetic data generation; all analysis must be based on real LLM outputs.
