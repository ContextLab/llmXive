# Evaluating the Impact of Code Style on LLM Code Generation Diversity

**Field**: Computer Science

## Research question

How do explicit stylistic constraints in prompts modulate the structural diversity of solutions in the latent search space of generative language models for code?

## Motivation

Current evaluations of code LLMs focus heavily on functional correctness, often ignoring whether stylistic conditioning collapses the model's exploration of alternative algorithmic solutions. Understanding this relationship is critical for designing prompts that balance readability requirements with the need for solution diversity during automated code synthesis or refactoring tasks.

## Literature gap analysis

### What we searched
We queried Semantic Scholar, arXiv, and OpenAlex using terms such as "LLM code generation stylistic variations," "impact of coding conventions on model output diversity," "stylistic prompting for code generation," and "code formatting influence on LLM performance." The search yielded a sparse set of results, with most literature focusing on general benchmarking or retrieval-augmented generation rather than the specific isolating of style constraints as a variable for diversity measurement.

### What is known
- [SIMCOPILOT: Evaluating Large Language Models for Copilot-Style Code Generation (2025)](https://arxiv.org/abs/2505.21514) — Establishes a benchmark for interactive coding assistants but does not vary style constraints as a parameter to measure diversity effects.
- [Vendi-RAG: Adaptively Trading-Off Diversity And Quality Significantly Improves Retrieval Augmented Generation With LLMs (2025)](https://arxiv.org/abs/2502.11228) — Demonstrates the importance of diversity metrics in RAG systems but focuses on retrieval quality rather than the intrinsic structural variance of code generation under stylistic prompts.
- [Open-LLM-Leaderboard: From Multi-choice to Open-style Questions for LLMs Evaluation, Benchmark, and Arena (2024)](https://arxiv.org/abs/2406.07545) — Discusses evaluation paradigms for LLMs but does not address the specific impact of code formatting rules on output entropy.

### What is NOT known
No published work has quantitatively measured how rigid adherence to specific coding standards (e.g., PEP8 vs. minification) alters the structural entropy of valid code solutions generated by small-scale open-weight models. The existing literature lacks an analysis of whether stylistic constraints act as a regularization mechanism that inadvertently limits the discovery of functionally equivalent but structurally distinct algorithms.

### Why this gap matters
Filling this gap is essential for developers and researchers who rely on LLMs for code exploration; if style constraints significantly reduce diversity, current prompting strategies may be limiting the potential for finding more efficient or novel implementations. Conversely, if models maintain diversity, it validates the robustness of current style-guided generation pipelines.

### How this project addresses the gap
This project directly addresses the gap by generating multiple code samples per task under varying style constraints and applying structural metrics (AST edit distance and n-gram entropy) to quantify the reduction in the solution space. The methodology isolates style as the independent variable to determine its causal impact on the latent diversity of the model's output.

## Expected results

We expect strict style constraints to significantly reduce structural diversity compared to neutral prompts, measurable via lower n-gram entropy and reduced AST edit distances. Confirming this would imply that style prompting acts as a regularization mechanism, potentially limiting exploration of alternative algorithms, while a null result would suggest models can maintain diverse solution spaces despite strict formatting.

## Methodology sketch

- **Data acquisition**: Download the HumanEval benchmark subset (publicly available on HuggingFace `openai/human-eval`) containing 164 programming tasks.
- **Model selection**: Load a small open-weight model (e.g., `Salesforce/codegen-350M-mono`) via `transformers` on CPU, ensuring memory usage stays under 7GB by processing in batches.
- **Prompt engineering**: Create three style variants for each task: (1) Neutral (no style instructions), (2) Strict PEP8 (explicit indentation/naming rules), (3) Aggressive Minification (remove whitespace/comments).
- **Generation**: For each task and style variant, generate 5 code samples using temperature sampling (T=0.7) to capture intrinsic model variability.
- **Functional filtering**: Execute generated code against the HumanEval unit tests; discard samples that fail to run to ensure diversity metrics reflect valid solutions only.
- **Metric computation**: Calculate token-level n-gram entropy and pairwise Abstract Syntax Tree (AST) edit distance between the 5 valid samples within each style group to derive a "diversity score."
- **Statistical analysis**: Apply the Kruskal-Wallis H-test to determine if diversity score distributions differ significantly across the three groups; if significant, follow up with Dunn's post-hoc test with Bonferroni correction to specifically test the directional hypothesis that Strict PEP8 has lower diversity than Neutral.
- **Validation independence**: Ensure the diversity metrics (AST distance, entropy) are computed solely from the generated code structures and are independent of the functional correctness labels (pass/fail) used for filtering, preventing circular validation where the "result" is guaranteed by the filtering criteria.
- **Scope management**: Verify that the statistical pipeline runs end-to-end within the 6-hour GitHub Actions limit by processing a subset (e.g., 50 tasks) if the full 164 exceeds time constraints, ensuring the method is scalable.
