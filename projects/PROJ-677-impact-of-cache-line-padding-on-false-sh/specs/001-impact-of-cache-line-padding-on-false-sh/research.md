# Research: Impact of Cache Line Padding on False Sharing in Concurrent Counters

## Research Question

How does cache-line padding of thread-local counters affect memory bus contention and throughput under high-frequency concurrent updates on multi-core processors?

## Background & Theory

### False Sharing
False sharing occurs when multiple threads on different cores modify variables that reside on the same cache line. Although the variables are logically independent, the cache coherence protocol (MESI) treats the entire cache line as a single unit. When one core modifies its variable, the cache line is invalidated on other cores, forcing them to reload the line from memory or another core's cache. This "cache-line bouncing" introduces significant latency and reduces throughput.

### Cache Line Padding
Padding variables to align with the cache line size (typically 64 bytes on modern x86/x64) ensures that each variable resides in a separate cache line. This eliminates false sharing by preventing unrelated variables from sharing a cache line, thereby reducing cache coherence traffic.

### Expected Outcome
We hypothesize that the **padded** variant will demonstrate significantly higher throughput (operations per second) than the **packed** variant as the thread count increases, particularly at 4 and 8 threads where contention is highest. The packed variant is expected to suffer from severe performance degradation due to false sharing.

## Experimental Design

### Variables
- **Independent Variables**:
  - `thread_count`: {1, 2, 4, 8}
  - `configuration`: {packed, padded}
- **Dependent Variable**:
  - `throughput`: Operations per second (calculated as `iteration_count / wall_clock_time_ms * 1000`)
- **Controlled Variables**:
  - `iteration_count`: Fixed at 10⁷ per thread.
  - `compiler_flags`: `-O3 -march=native`.
  - `hardware`: GitHub Actions runner (dynamically detected).

### Procedure
1.  **Memory Layout Verification (Critical)**: Before running the benchmark, compile and execute `verify_layout.cpp`. This utility prints the memory addresses of the counters for both `packed` and `padded` configurations.
    *   **Pass Condition**: `packed` counters must reside on the same 64-byte cache line (address difference < 64 bytes); `padded` counters must reside on distinct cache lines (address difference ≥ 64 bytes).
    *   **Failure**: If the layout does not match expectations (e.g., due to allocator alignment), the experiment halts and reports a configuration error. This ensures the independent variable (padding) maps to the intended physical state.
2.  **Hardware Control**: Set the CPU governor to `performance` and use `taskset` to pin the benchmark threads to specific physical cores to mitigate "noisy neighbor" effects and frequency scaling variance.
3.  **Compilation**: Compile C++ code with `-O3 -march=native`.
4.  **Execution**: Run benchmark for each combination of `thread_count` and `configuration` (4 × 2 = 8 scenarios).
5.  **Replication**: Perform multiple independent runs per scenario to account for system noise. (See Power Analysis below).
6.  **Data Collection**: Record `wall_clock_time_ms` for each run.
7.  **Hardware Detection**: Capture CPU model, core count, and cache line size for each run.

### Statistical Analysis Plan
1.  **Descriptive Statistics**: Calculate mean and standard deviation of throughput for each scenario.
2.  **Inferential Statistics**:
    *   **Primary Test**: Perform a **Two-Way ANOVA** (Configuration × Thread Count) to test for the interaction effect. This determines if the effect of padding *depends* on the thread count, which is the core hypothesis of false sharing.
    *   **Post-Hoc Analysis**: If the interaction is significant, perform two-sample t-tests (α = 0.05) comparing padded vs. packed throughput at each thread count.
    *   **Effect Size**: Compute Cohen's d for each significant comparison.
    *   **Multiplicity Control**: Apply Benjamini-Hochberg FDR correction (q ≤ 0.05) to the set of post-hoc t-tests.
3.  **Visualization**: Generate a line plot of throughput vs. thread count with 95% confidence intervals.

## Power Analysis & Sample Size Justification

The study design specifies 5 runs per condition (n=5).
*   **Justification**: This is a pilot-style benchmark on CI infrastructure where runtime is strictly capped at a reasonable duration.
*   **Risk**: Performance benchmarks on shared infrastructure (CI runners) often exhibit high variance due to "noisy neighbor" effects, CPU frequency scaling, and thermal throttling. A sample size of 5 may be underpowered to detect small-to-medium effects (Cohen's d < 0.5) if the noise floor is high, leading to Type II errors.
*   **Mitigation**:
    1.  **Hardware Control**: Enforcing `performance` governor and core pinning (Step 2 above) reduces variance.
    2.  **Variance Monitoring**: If the coefficient of variation (CV) across the 5 runs exceeds 10%, the script will automatically trigger 5 additional runs (total n=10) for that specific condition.
    3.  **Robustness**: If normality assumptions fail (Shapiro-Wilk test), the analysis will switch to a non-parametric Mann-Whitney U test.
    4.  **Reporting**: If results are null (p > 0.05), the report will explicitly state the power limitation and the observed effect size, avoiding false negatives.

## Assumptions & Limitations

- **Assumption 1**: The GitHub Actions runner has ≥2 CPU cores and sufficient RAM.
- **Assumption 2**: Cache line size is 64 bytes on the target architecture.
- **Assumption 3**: The workload (10⁷ increments) is large enough to measure throughput differences but small enough to complete within 6 hours.
- **Assumption 4**: Atomic instruction overhead is constant across both variants; the observed difference is due to cache-line bouncing.
- **Assumption 5 (Corrected)**: **The study employs a controlled causal intervention.** The independent variable (memory layout) is directly manipulated, and runs are randomly assigned to configurations. Therefore, findings are framed as **causal** (padding *causes* the throughput difference), not merely associational.
- **Limitation 1**: Hardware variability across GitHub Actions runners may introduce noise. This is mitigated by recording hardware specs, running multiple replications, and enforcing CPU governor settings.
- **Limitation 2**: The study does not account for different cache coherence protocols or non-x86 architectures.

## Decision Rationale

- **Why C++?**: C++ provides low-level control over memory layout (`alignas`) and atomic operations (`std::atomic`), essential for demonstrating false sharing.
- **Why 10⁷ iterations?**: This count balances the need for measurable throughput with the 6-hour runtime constraint on CI.
- **Why 5 runs?**: Five replications provide a reasonable estimate of variance for t-tests without excessive runtime, with an automatic escalation mechanism for high-variance conditions.
- **Why Two-Way ANOVA?**: The design is a 2x4 factorial. ANOVA is statistically more powerful than isolated t-tests for detecting the specific interaction effect (does padding help more at higher thread counts?) and controls the family-wise error rate better in this structured design.
- **Why FDR correction?**: With multiple post-hoc comparisons, controlling the False Discovery Rate is standard practice in performance benchmarking to avoid spurious significance.
