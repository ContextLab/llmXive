## Unresolved concerns

- McNemar's test is proposed to compare coverage outcomes of Ordered vs. Shuffled for each trial. However, McNemar's test requires paired binary data where the 'pairs' are distinct subjects. Here, the 'subjects' are the 1,000 trials. The test compares the *proportion* of covered intervals between two conditions. While the trials are paired (same series, two conditions), McNemar's is typically used for 2x2 contingency tables of counts (e.g., how many trials were covered in Ordered but not Shuffled). The plan states 'paired binary data' but the implementation logic needs to ensure the contingency table is constructed correctly (n11, n10, n01, n00) across the 1,000 trials, not per-trial p-values. If the plan intends to run McNemar's on the aggregate counts, this is valid, but the phrasing 'for each trial' is ambiguous and risks a statistical error.
- The plan filters real-world segments for 'phi > 0.3' before running the bootstrap comparison. This introduces a selection bias (collider bias) if the goal is to estimate the *average* coverage drop for the dataset. By only analyzing segments with high autocorrelation, the results will overestimate the impact of ordering on the *entire* dataset. The analysis should either: (a) stratify results by phi bins (e.g., 0.0-0.1, 0.1-0.2, etc.) to show the curve, or (b) report the weighted average coverage across all segments, not just the high-phi subset. Filtering before analysis invalidates the generalizability claim for the 'practitioner' aspect.
- The plan defines the ground truth for synthetic data as 'Theoretical mean = 0'. However, for an AR(1) process $X_t = \phi X_{t-1} + psilon_t$ with $psilon \sim N(0,1)$, the theoretical mean is indeed 0. But for the *real-world* data (UCI), the plan states 'theoretical mean (0)' in the summary but 'sample mean' in the data model description. The spec (FR-003) says 'theoretical mean of the process (0)'. For real data, there is no known 'theoretical mean' of 0. The plan must explicitly state that for real data, the 'ground truth' is the *sample mean of the segment* (or the mean of the full series if stationary), otherwise the coverage calculation for real data is meaningless (checking if 0 is in the CI when the data is centered at 2000W).
- The plan claims 'N=100 per series is small, which may increase variance in phi_hat estimation'. However, for an AR(1) process, estimating phi with N=100 is actually quite precise (standard error ~ 1/sqrt(N) ~ 0.1). The real issue is that N=100 is small for the *bootstrap* to accurately estimate the sampling distribution of the mean if the underlying distribution is heavy-tailed, but for Gaussian AR(1), N=100 is likely sufficient. The concern about 'variance in phi_hat' is a red herring; the primary risk is that with N=100, the *empirical coverage* of the bootstrap CI might be noisy due to the small number of resamples (1000) relative to the number of possible permutations, but 1000 is standard. The justification for N=100 seems weak; a sensitivity analysis on N (e.g., N=50, 100, 200) would be more robust to prove the effect is due to autocorrelation and not small sample size artifacts.
- The plan explicitly states 'NO verified source found' for the UCI Individual Household Electric Power Consumption dataset and proposes a 'load if available, skip if not' fallback. This renders User Story 3 (Real-World Validation) non-deterministic and potentially unexecutable. The research question requires validating synthetic findings on empirical data; if the dataset is missing, the validation step fails entirely. The plan must identify a reachable, verified URL (e.g., via UCI Machine Learning Repository or a verified HuggingFace mirror) before proceeding, rather than deferring to a runtime check that may result in incomplete research.
- Constitution Principle VII mandates a 'two-proportion z-test' for comparing ordered vs. shuffled coverage. The plan (and spec/research) exclusively specifies 'McNemar's test'. This is a direct contradiction in the statistical methodology required by the constitution versus the plan's execution strategy.
- Constitution Principle II (Verified Accuracy) requires citations to be validated against primary sources. The plan states 'Citations for McNemar's test and UCI dataset will be validated', but the research.md explicitly admits the UCI Power Consumption dataset has 'NO verified source found' in the provided list. The plan does not define a fallback mechanism that satisfies the 'Verified Accuracy' gate if the dataset remains unverified, risking a constitutional violation.
- The 'Project Structure' section lists `contracts/` as a Phase 1 output, but the `contracts/coverage_metrics.schema.yaml` file is provided in the inputs. The plan should explicitly reference the existence of this contract or clarify if the schema is considered part of the 'Phase 1 output' that has already been drafted.
- The plan proposes using McNemar's test to compare coverage outcomes of 'Ordered' vs. 'Shuffled' conditions for the *same* time series. This is a circular validation. Shuffling an AR(1) process with $\phi > 0$ destroys the temporal dependence, effectively transforming the data into an i.i.d. sample (or close to it). The standard bootstrap is *designed* to be valid for i.i.d. data. Therefore, the 'Shuffled' condition is definitionally the 'correct' baseline for the bootstrap, while the 'Ordered' condition is the 'incorrect' one. Comparing them via McNemar's test does not validate the *impact of ordering* on the bootstrap's *assumption*; it merely confirms that the bootstrap fails on dependent data and works on independent data. The analysis is tautological: 'Bootstrap fails on dependent data' vs 'Bootstrap works on independent data' is a known theorem, not an empirical result requiring statistical testing. The research question should be about the *magnitude* of the coverage drop (empirical coverage vs. nominal 95%), not a significance test against a shuffled baseline which is guaranteed to succeed.
- The plan validates the bootstrap CI against the 'theoretical mean (0)'. For synthetic AR(1) data $X_t = \phi X_{t-1} + psilon_t$ with $psilon_t \sim N(0,1)$, the theoretical mean is indeed 0. However, the plan states: 'Ground Truth: Theoretical mean = 0'. This is correct for the synthetic generation. But for the 'Real-World' data (UCI), the plan states: 'Ground Truth: ... sample mean for real' (in data-model.md) but 'theoretical mean (0)' in the plan summary. This is a contradiction. For real data, there is no known 'theoretical mean' of 0. The coverage probability for real data *cannot* be calculated against a known ground truth unless the true population mean is known (which it isn't for UCI power consumption). The plan must clarify that for real data, the 'coverage' metric is ill-defined or must be replaced by a different metric (e.g., comparing the width of CIs, or using a block bootstrap as a reference). Calculating 'coverage' against the sample mean is circular (the CI always contains the sample mean by construction for the percentile method if the sample mean is the center, or it's a tautology if checking if the sample mean is in the CI of the sample mean).
- The plan proposes estimating $\hat{\phi}$ for each segment and then filtering for $\hat{\phi} > 0.3$. This introduces a selection bias. By selecting only segments with high estimated autocorrelation, the analysis is guaranteed to find a coverage drop. The 'validation' on real data becomes a self-fulfilling prophecy: 'We selected data where the bootstrap should fail, and we found that it failed.' A sound methodology would analyze the relationship between $\hat{\phi}$ and coverage across *all* segments (including those with low $\hat{\phi}$) to demonstrate the continuous degradation of coverage as a function of $\phi$, rather than a binary pass/fail on a selected subset.
