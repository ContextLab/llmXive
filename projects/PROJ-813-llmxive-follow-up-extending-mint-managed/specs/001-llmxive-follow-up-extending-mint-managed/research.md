# Research: llmXive follow-up: extending "MinT: Managed Infrastructure for Training and Serving Millions of LLMs"

## Problem Statement

Multi-tenant LLM serving faces a "cold-start" bottleneck when switching between LoRA adapters. Current scheduling policies (e.g., FCFS, LRU) ignore the structural similarity between adapters. If Adapter A and Adapter B share significant parameter updates, loading B after A should be faster (or require less memory bandwidth) if the system can exploit this overlap. The "MinT" infrastructure concept suggests managing these resources, but lacks a scheduling policy that explicitly leverages the "LoRA Topology" (the graph of parameter overlaps).

## Research Questions

1. **RQ1**: Can a "LoRA Topology Graph" be reliably constructed from synthetic LoRA adapters with controlled cluster structures?
2. **RQ2**: Does a "Topological Lookahead" scheduling policy, which pre-fetches adapters based on graph proximity, reduce average cold-start latency compared to FCFS and Greedy Frequency baselines?
3. **RQ3**: Is the observed latency reduction statistically significant (p < 0.05) and does it meet the target of ≥15% reduction under realistic memory constraints (7 GB)?

## Methodology

### 1. Data Generation (Synthetic)

**Strategy**: Since no public dataset contains "LoRA adapters with injected correlation" for this specific research, we generate a synthetic dataset.
**Rationale**: This allows precise control over the "overlap variance" (the independent variable) required to test the Topological policy. Real-world LoRA weights are often proprietary or lack the specific correlation structure needed for a controlled experiment.

**Procedure - Adapters**:
- Generate $N=10,000$ synthetic LoRA adapters organized into $K=50$ distinct latent clusters.
- Within each cluster, adapters share a high degree of correlation (high overlap); between clusters, correlation is low.
- Each adapter $A_i$ is a matrix $W_i$ of shape $(d_{out}, d_{in})$ with rank $r \in [1, 256]$.
- **Clustered Generation**: $W_i = \alpha \cdot W_{cluster\_basis} + (1-\alpha) \cdot W_{noise}$. This ensures the Topology Graph has a sparse, clustered structure rather than a uniform clique.
- Compute Pairwise Overlap Matrix $M_{ij} = \frac{\cosine(W_i, W_j) + 1}{2}$.
- **Feasibility**: [deferred] adapters $\times$ [deferred] comparisons = 100M pairs. Using optimized `numpy` vectorization on CPU, this fits in ~8GB RAM (float64) and completes in < 30 mins.

**Procedure - Request Trace**:
- Generate a trace of $N_{req}=1,000,000$ requests.
- **Clustered Markov Chain**: The trace is generated by a Markov process where the probability of requesting adapter $B$ given current adapter $A$ is higher if $A$ and $B$ are in the same cluster (temporal locality). This ensures the trace has the necessary structure for the Topological Lookahead policy to exploit, while the specific sequence is independent of the exact weight overlaps.
- **Independence**: A unique trace is generated for each replication.

### 2. Simulation Environment (MinT Extension)

**Tool**: `simpy` (Discrete Event Simulation).
**Constraints**:
- **Memory Cap**: 7 GB (simulated).
- **Load Time (Delta Loading)**: The core hypothesis is that overlap reduces load time. The simulation models this via a "Delta Loading" mechanism:
  - $T_{load}(B) = T_{base}$ if cache is empty or no high-overlap adapter exists.
  - $T_{load}(B | A) = T_{base} \times (1 - \text{overlap}(A, B) \times \eta)$ if adapter $A$ is in cache, where $\eta$ is an efficiency factor (e.g., 0.8).
  - This models the physical reality that only the *difference* (delta) needs to be transferred if a high-overlap adapter is already loaded.
- **Stochastic Jitter**: Load time includes a $\pm$ small jitter to prevent tautological validation.
- **Eviction**: LRU or Policy-specific (e.g., evict least overlapping).

**Policies**:
1. **FCFS**: First-Come-First-Served (Baseline).
2. **Greedy Frequency**: Load most frequently requested adapters first.
3. **Topological Lookahead**: Use the request trace's Markov transitions to predict the next $k$ requests, then schedule the adapter that maximizes overlap with the current cache state (or pre-fetches the most "central" neighbor in the topology graph).

### 3. Statistical Analysis

**Design**: 30 independent replications.
- **Replication Structure**: In each replication $r$:
  1. Generate a unique request trace $Tr$ (using a unique seed).
  2. Run the simulation for $Tr$ under **FCFS**. Record $L_{FCFS, r}$.
  3. Run the simulation for the **same** $Tr$ under **Topological Lookahead**. Record $L_{Topo, r}$.
- **Test**: **Paired t-test** on the 30 pairs of $(L_{FCFS, r}, L_{Topo, r})$.
- **Rationale**: This design controls for trace variance (since the same trace is used for both policies in a replication) while ensuring independence across replications (since traces differ).
- **Metric**: Average cold-start latency (time from request arrival to adapter load completion).
- **Power**: N=30 replications provides >80% power to detect a medium effect size (Cohen's d ~ 0.5) at $\alpha=0.05$.

## Dataset Strategy

**Source**: Synthetic Generation (No external URL).
**Justification**: The research requires a specific correlation structure (injected overlap variance) that does not exist in public datasets.
**Verified Datasets**: None required. The "dataset" is generated by `code/data/generate_adapters.py` and `code/data/generate_trace.py`.
**Note**: If a real dataset were used, it would need to be verified for the presence of LoRA weight matrices and correlation metadata, which is not currently available in standard repositories (HuggingFace `datasets` usually contains raw models, not pre-computed LoRA overlaps).

## Feasibility & Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| **Memory Overflow (10k x 10k Matrix)** | High (Crash) | Use `float` for the overlap matrix. Process in chunks if needed. |
| **Simulation Runtime > 6h** | High (CI Failure) | Optimize SimPy event loop. Use `numpy` for vectorized load time calculations. If too slow, reduce trace size to a smaller scale (documented trade-off). |
| **Topological Policy Overhead** | Medium | Simplify Markov chain to 1-step transition. Pre-compute graph adjacency. |
| **Statistical Significance Failure** | Low (Research Outcome) | If p > 0.05, report confidence intervals and effect size. Do not force significance. |

## Decision Log

- **Decision**: Use Clustered Synthetic Data.
  - **Rationale**: Control over variables (correlation, rank, sparsity) is critical for the hypothesis. A uniform distribution would fail to create a meaningful topology.
- **Decision**: Delta Loading Mechanism.
  - **Rationale**: To model the physical benefit of overlap (reduced data transfer), not just cache hit rates.
- **Decision**: Paired Statistical Design with Unique Traces.
  - **Rationale**: To isolate the policy effect (paired) while ensuring independence across replications (unique traces).
- **Decision**: CPU-only execution.
  - **Rationale**: GitHub Actions Free Tier constraints. No GPU available.
- **Decision**: 30 Replications.
  - **Rationale**: Standard for simulation studies to ensure statistical robustness without excessive runtime.
- **Decision**: Block-bootstrapping vs. t-test.
  - **Rationale**: Start with paired t-test on replication means. If variance is non-normal, switch to block-bootstrapping.
