## Unresolved concerns

- The plan proposes 10 independent replications of a 10^6 request trace to achieve statistical power. However, the trace generation itself is deterministic per seed. If the trace is generated once per seed and reused across policies, the 'independence' is only across seeds, not within the trace. If the trace is regenerated per replication, the variance in trace generation (hotspot placement) becomes a confound. The plan must explicitly state whether the trace is fixed per seed (paired design) or regenerated (unpaired), as this dictates the validity of the paired t-test/Wilcoxon test.
- FR-009 requires calibrating the cost model against 'published MinT hardware benchmarks'. The spec treats 'MinT' as a hypothetical reference. If no such primary source exists (as implied by 'Verified Datasets: None'), the calibration is impossible. Without a calibrated cost model (I/O bandwidth, memory copy time), the 'latency' metric is an arbitrary unit, not a valid measure of 'cold-start latency' in milliseconds, invalidating the claim of a '15% reduction' in real-world terms.
- The plan assumes 'hotspots' in the synthetic trace will mimic real-world multi-tenant patterns. However, the Topological Lookahead policy relies on 'parameter overlap' (structural similarity), while the trace relies on 'access frequency' (temporal locality). The plan does not define a mechanism to correlate structural similarity with access patterns (e.g., 'users requesting Adapter A are likely to request Adapter B because they share parameters'). Without this correlation, the Topological policy has no signal to exploit, and the hypothesis that it will outperform FCFS is untestable.
- The plan states '10^6 request trace simulation with statistical replications' must complete in 6 hours. A 10k x 10k sparse matrix lookup for every request in a 10^6 trace, repeated 10 times, involves 10^7 graph lookups. In Python/SimPy, this is computationally heavy. The plan lacks a specific algorithmic complexity analysis or a 'warm-up' strategy to ensure the 6-hour limit is not violated, risking a failure to collect the required data for the statistical test.
- Plan lists `code/data_generation/request_trace.py` in the source structure, but `data-model.md` defines the Request Trace entity and `research.md` describes its generation. However, `contracts/request_trace.schema.yaml` defines fields (`is_hotspot`, `cluster_id`) that are not explicitly mentioned in the `plan.md`'s data generation section, creating a minor traceability gap between the plan's description and the contract's strict schema.
- The plan's Constitution Check table lists Principle VI as 'Simulation Determinism & Topological Fidelity' and Principle VII as 'Statistical Rigor'. While the plan claims PASS, the `plan.md` text does not explicitly describe the mechanism for 'block bootstrapping' or the specific 'Shapiro-Wilk' test implementation in the `analysis/statistics.py` section, only referencing them in the summary. The plan should explicitly map these specific statistical methods to the `code/analysis/statistics.py` file to satisfy the 'Single Source of Truth' principle (Principle IV) regarding the methodology.
- The `plan.md` lists `code/simulation/cost_model.py` (FR-009) but does not explicitly detail how the 'calibration against published MinT hardware benchmarks' (required by FR-009 in spec.md) is handled in the plan's narrative. The plan mentions 'cost model' but lacks a specific step or data source reference for the calibration values, creating a gap between the plan's execution steps and the spec's requirement for verified accuracy (Constitution Principle II).
- The `plan.md` lists `contracts/` as a Phase 1 output directory but does not list the specific contract files (e.g., `adapter.schema.yaml`, `simulation-result-schema.schema.yaml`) in the `Project Structure` tree, whereas `data-model.md` and `quickstart.md` imply their existence. The plan should explicitly list the contract files in the directory tree to ensure the 'Plan ↔ contracts' consistency check is fully satisfied.
- The plan defines the 'Topological Lookahead' policy's decision logic using the 'pairwise parameter overlap graph' (derived from adapter weights) but validates its success against 'cold-start latency' and 'eviction counts' derived from a synthetic request trace. The critical flaw is the independence of the trace generation from the topology. If the trace is generated with 'hotspots' (clusters of frequently accessed adapters) that are *not* correlated with the parameter overlap clusters, the Topological policy has no structural advantage to exploit, making the null result (no improvement) a tautology of the data generation, not a failure of the algorithm. Conversely, if the trace generation *implicitly* uses the same clustering logic as the topology (e.g., 'generate requests for adapters in cluster A'), the validation is circular: the policy succeeds because the workload was designed to match the policy's heuristic, not because the heuristic is generally superior. The plan must explicitly decouple the 'topology generation' (weight overlap) from the 'trace generation' (access patterns) and define a mechanism to vary their correlation (e.g., 'correlation coefficient between weight-overlap clusters and access clusters') to ensure the evaluation is non-trivial and the result is empirical.
- The plan proposes using a 'paired t-test' or 'Wilcoxon signed-rank test' on '10 independent replications' of a $10^6$ request trace. However, the unit of replication is the *entire trace*, not individual requests. If the trace generation process (e.g., 'hotspot' placement) is the primary source of variance, 10 replications may be insufficient to achieve statistical power for a 15% reduction if the variance between traces is high. More critically, if the trace is fixed (one seed) and only the simulation randomness varies, the 'replications' are not independent regarding the workload structure, violating the independence assumption of the t-test. The plan must clarify if the '10 replications' involve regenerating the *entire* synthetic dataset (adapters + trace) or just re-running the simulation on the same trace. If the latter, the statistical test is invalid for comparing policies against workload variance. If the former, the computational cost (10x full generation + simulation) must be verified against the 6-hour limit.
- The plan relies on 'synthetic data generation' where adapters have 'randomly masked weight updates' and the trace has 'hotspots'. The validity of the 'Topological Lookahead' hypothesis depends entirely on the assumption that 'parameter overlap' predicts 'access sequence'. In real LoRA serving, this correlation is non-trivial (e.g., similar tasks use similar adapters). The plan fails to define the *mechanism* by which the synthetic trace generates this correlation. If the trace is purely random or based on frequency alone, the Topological policy is mathematically guaranteed to underperform or equal FCFS/Greedy, rendering the experiment a 'straw man' validation. The plan must specify how the synthetic trace generation injects a 'topological bias' (e.g., 'requests for adapter B are more likely if adapter A was just requested, where A and B have high overlap') to make the hypothesis testable.
