\section{Preliminaries}
\label{sec:def}

To support the discussion in the rest of this paper, we first clarify the scope of \emph{agent memory} from a data management perspective. Although recent studies have examined memory from viewpoints such as cognitive taxonomy, agent architecture, and graph-based organization~\cite{zhang2024survey,du2026memory,hu2025memory,wu2026memoryeab,tang2026survey,yang2026graph}, the underlying concept is still often treated primarily as an algorithmic component of the LLM or agent pipeline~\cite{zhang2024survey,du2026memory,hu2025memory}. In contrast, we study agent memory as a standalone data management object and system infrastructure, with explicit attention to how it is represented, stored, retrieved, updated, and maintained under real agent workloads. Under this view, we introduce a set of definitions below.
% This perspective provides the foundation for the formal definitions and system abstractions introduced below.

%The rapid development of agent memory has prompted several surveys to organize the landscape. Early surveys~\cite{zhang2024survey} approach memory primarily from a natural language processing perspective, categorizing memory into formats (e.g., text, embeddings) and basic operations (e.g., reading, writing). More recent works examine memory through the lenses of autonomous agent architectures~\cite{du2026memory,hu2025memory}, unified structural representations~\cite{wu2026memoryeab,tang2026survey}, and graph-based paradigms~\cite{yang2026graph}. However, these surveys largely treat memory as an algorithmic component of the LLM rather than a standalone data management system. They lack a comprehensive, workload-driven evaluation of the underlying storage, routing, and maintenance infrastructure. In this work, we bridge this gap by formalizing agent memory from a data management perspective and benchmarking the critical operational costs and robustness trade-offs that dictate real-world deployment.

\hi{Memory Types.} For an LLM agent, a wide variety of information is produced and may need to be memorized, including dialogue history, tool execution logs, distilled facts, and user preferences~\cite{zhang2024survey}. Following established cognitive frameworks, memory can be broadly organized along two axes~\cite{hu2025memory, tang2026survey}. 
(1) Along the \emph{temporal} axis, {short-term memory} holds the volatile state of an ongoing session, while {long-term memory} persists across sessions. 
(2) Along the \emph{functional} axis, long-term memory is further divided into information such as concrete past events (episodic memory), abstracted factual knowledge (semantic memory)~\cite{hu2025memory, wu2026memoryeab}, reusable action strategies (procedural memory), and user preferences.

\hi{Agent Memory.} We define the \emph{agent memory} $\mathcal{M}$ as the persistent data management object~\cite{packer2023memgpt, wu2026memoryeab} that maintains this cumulative state beyond a single inference step and makes it accessible to the agent during future reasoning and action~\cite{zhang2024survey, hu2025memory}.

\noindent $\bullet$ \underline{(1) Memory Representation and Storage $\mathcal{R}$:} A mapping that defines the logical and physical memory format with a data model of two facets: (a) logical representation, spanning simple primitives (discrete tokens, continuous vectors) to complex topologies (knowledge graphs, trees, and composites); and (b) physical storage, utilizing transient registers, specialized single-engine databases, or multi-engine backends for persistence and indexing.

\noindent $\bullet$ \underline{(2) Memory Extraction $\mathcal{S}$:} A mechanism governing how heterogeneous input streams (e.g., multi-turn dialogues, tool logs) are transformed into logical memory primitives via pipelines such as raw sequence concatenation, schema-free semantic extraction, or schema-constrained structured extraction.

\noindent $\bullet$ \underline{(3) Memory Retrieval and Routing $\mathcal{Q}$:} A function that dynamically identifies relevant memory subsets based on a query context, utilizing specific routing algorithms to traverse indices. Mechanisms span native attention-based retrieval, semantic $K$-nearest neighbor search, topological subgraph traversal, autonomous agentic routing via \llm planning, and multi-stage hybrid execution.

\noindent $\bullet$ \underline{(4) Memory Maintenance $\mathcal{U}$:} Policies governing the dynamic lifecycle of memory entries, decomposed into three sub-operations: \textit{(a) Conflict Resolution and Versioning} handles contradictions via multi-versioning, invalidation, or precedence rules; \textit{(b) Capacity Management} enforces bounded growth through constraint-based hard eviction (e.g., FIFO, token limits) or score-based priority eviction (e.g., temporal decay); and \textit{(c) Semantic Consolidation} utilizes the \llm to merge redundant assertions into dense summaries or execute CRUD operations via tool-calling interfaces.

\hi{Distinction from RAG and Context Engineering.} %Distinguished from related paradigms like Retrieval-Augmented Generation (RAG)~\cite{RAG,khan2026rag} and context engineering~\cite{anthropic2025context}. 
Retrieval-Augmented Generation (RAG)~\cite{RAG,khan2026rag} typically operates as a stateless, read-only retrieval primitive: given a query, it fetches relevant passages from a static corpus to augment a single generation step. Context engineering~\cite{anthropic2025context} is the broader practice of curating the finite LLM context window at each inference turn (e.g., dynamically selecting prompts, tool descriptions, and retrieved facts) to mitigate context rot~\cite{anthropic2025context}. In contrast, an agent memory system (1) is a persistent and updatable infrastructure for managing agent-specific state over time and (2) governs the full long-term memory lifecycle, including memory representation, storage, retrieval, and maintenance, rather than merely packing the current context window.


%(1) encompassing the read path (retrieval) but actively managing the \emph{write path} and \emph{maintenance lifecycle}---persisting new observations, resolving conflicting facts, and enforcing capacity limits across continuous, multi-session agentic execution. In short, RAG is a retrieval mechanism, context engineering optimizes the LLM's transient working memory, while an agent memory system manages the persistent, evolving knowledge base that grounds the agent.


\hi{Distinction from Traditional Database Workloads.} Agent memory workloads differ substantially from conventional database OLTP / OLAP workloads~\cite{packer2023memgpt, liu2026supporting}. First, memory access is often \emph{semantic} rather than purely predicate-based~\cite{kang2025bigvectorbench, caminal2025filtered}. 
Queries are commonly expressed through natural language, partial context, or latent intent, and therefore rely on approximate matching, query rewriting, or \llm-guided retrieval rather than only exact logical predicates over rigid schemas. Second, memory contents evolve under \emph{continuous and potentially conflicting observations}. 
Unlike conventional transactional settings, where updates typically overwrite tuples under a predefined schema and consistency model, agent memory must accommodate uncertain, partial, and sometimes contradictory information collected across time, tools, and environments~\cite{hu2025memory, zheng2025lifelong}. Third, agent memory workloads are \emph{highly heterogeneous} in both access pattern and granularity. 
A single workload may combine long-context synthesis, episodic recall, structured fact lookup, temporal reasoning, and streaming updates. 
As a result, practical systems often require hybrid execution strategies that combine semantic retrieval, structured filtering, and topology-aware traversal within one memory architecture~\cite{wu2026memoryeab, packer2023memgpt}. 
These properties distinguish agent memory from traditional databases, motivating dedicated abstractions and evaluation methodologies.

\subsection{Benchmarking Agent Memory}

\zw{short summary about existing work, illustrating around the category from the holistic method view}

Previous studies have extensively evaluated various aspects of agent memory systems, primarily focusing on their cognitive utility and task-level success~\cite{maharana2024locomo, wu2024longmemeval, memoryagentbench2026, tan2025membench}.  \texttt{LoCoMo}~\cite{maharana2024locomo} evaluates retrieval from long conversational histories by posing open-domain, multi-hop, temporal, and single-hop questions over multi-turn dialogues averaging 9,209 tokens and 304 turns. \texttt{LongMemEval}~\cite{wu2024longmemeval} assesses cross-session question answering and temporal knowledge updates over up to 1.5 million tokens, introducing a dynamic long-context setting where facts may change across sessions. \texttt{MemoryAgentBench}~\cite{memoryagentbench2026} tests four core competencies---accurate retrieval, test-time learning, long-range understanding, and conflict resolution---across 14 datasets with context lengths ranging from 103K to 1.44M tokens. \texttt{MemBench}~\cite{tan2025membench} measures memory capabilities across different abstraction levels (factual versus reflective) and noise conditions, stress-testing systems with up to 100K sessions. \texttt{LongBench}~\cite{bai2023longbench} provides a multi-task evaluation under standard context limits with 503 multiple-choice questions spanning short, medium, and long context buckets. Beyond conversational settings, \texttt{StreamBench}~\cite{wu2024streambench} evaluates continuous improvement of agents over streaming input-feedback sequences, and \texttt{LifelongAgentBench}~\cite{lifelongagentbench2025} measures sequential procedural skill transfer across structurally related database, operating system, and knowledge graph tasks. More recently, \texttt{MemoryArena}~\cite{memoryarena2026} introduces interdependent multi-session agentic tasks spanning web shopping, travel planning, and formal reasoning, while \texttt{AMA-Bench}~\cite{amabench2026} targets long-horizon memory in realistic agent-environment trajectories. Despite the growing number of benchmarks, existing evaluations exhibit several critical limitations when examined from a data systems perspective.

\noindent $\bullet$ \textbf{Saturation under Long-Context Models.} Both LoCoMo and LongMemEval were designed in an era of 32K-token context windows, where the entire premise was that an agent \textit{could not} simply pass all history into a single model call. With modern models supporting million-token context windows, a naive ``dump everything into context'' approach now scores competitively on these benchmarks~\cite{chhikara2025mem0}, rendering them unable to distinguish between a brute-force context stuffer and an architecturally sound memory system.

\noindent $\bullet$ \textbf{Overemphasis on Retrieval over Maintenance.} Current benchmarks predominantly measure whether an agent can extract information from a provided context (e.g., Recall@$K$, exact match, or F1). They largely ignore the \textit{write path} and the complexities of memory maintenance: real-world agents must continuously absorb frequent, potentially contradictory observations, necessitating precise conflict resolution, state mutation, and capacity management (e.g., eviction or semantic consolidation). While MemoryAgentBench and MemBench include conflict resolution and knowledge-update subsets, these remain a small fraction of the overall evaluation and do not measure the \textit{cost} of maintenance operations.

\noindent $\bullet$ \textbf{Neglect of Operational Costs and Scalability.} Most existing evaluations fail to report the operational overhead of memory architectures. Key database performance indicators, such as query latency, index construction time, token consumption per successful query, and storage amplification, are rarely measured. %Consequently, a highly structured but computationally exorbitant system (e.g., a full knowledge graph with 16.69s construction time and 40.09s query latency) may appear to match a lightweight vector store (0.01s construction, 0.59s query latency) on accuracy alone, masking the severe cost--performance trade-offs that govern scalability in production environments.

\noindent $\bullet$ \textbf{Static Contexts vs.\ Dynamic Long-Horizon Evolution.} Benchmarks often evaluate memory over fixed, pre-compiled datasets. They fail to capture the \textit{long-horizon stability} required when memory history grows dynamically over extended periods. As temporal distances increase and the storage footprint scales from thousands to millions of tokens, many append-only memory stores suffer from catastrophic degradation.

\noindent $\bullet$ \textbf{Chatbot-Centric Paradigm vs.\ Agentic Execution.} The majority of existing datasets are built around chatbot use cases, such as conversation history between two speakers, question-answering over past sessions. Modern agents, however, research, plan, execute multi-step tasks, and build knowledge across diverse interactions and tool calls. Memory problems in these agentic workflows are fundamentally different from conversational recall, requiring the memory system to be tightly coupled with the agent's reasoning loop, tool execution, and routing logic. While LifelongAgentBench and MemoryArena begin to address this gap, they do not decompose memory into its constituent data management modules for isolated evaluation.
