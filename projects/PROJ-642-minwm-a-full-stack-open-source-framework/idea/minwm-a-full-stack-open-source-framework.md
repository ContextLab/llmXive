# minWM: A Full-Stack Open-Source Framework for Real‑Time Interactive Video World Models  

**Field**: computer science  

## Research question  

How does the number of diffusion inference steps in a camera‑controllable video diffusion model affect (a) first‑frame latency and (b) the quantitative accuracy of camera‑pose following, measured on an independent pose‑estimation benchmark?  

## Motivation  

Interactive video world models must generate frames fast enough for a user to see immediate feedback, yet still respect user‑specified camera motions. Existing systems trade latency for controllability by using many diffusion steps, but it is unclear whether fewer steps inevitably degrade pose fidelity or whether a more efficient inference schedule can preserve controllability. Clarifying this trade‑off will guide the design of truly real‑time, interactive video generators.  

## Related work  

- [GEM‑4D: Geometry‑Enhanced Video World Models for Robot Manipulation (2026)](https://arxiv.org/abs/2605.22882) — Shows that video world models can generate realistic futures but often lose geometric consistency across time, motivating the need for explicit camera‑pose fidelity metrics.  
- [Sora as a World Model? A Complete Survey on Text‑to‑Video Generation (2024)](https://arxiv.org/abs/2403.05131) — Provides a comprehensive overview of video generation approaches, highlighting latency and controllability as open challenges for interactive applications.  

## Expected results  

We expect to identify a step‑count regime (e.g., 2–4 diffusion steps) that yields ≥30 % reduction in first‑frame latency relative to the full‑step baseline while keeping the mean pose‑error increase below 5 % of the baseline. Confirmation will be the observation of a statistically significant latency improvement *and* a non‑significant change in pose error (p > 0.05). A failure to find such a regime would indicate an inherent trade‑off, informing future architectural designs.  

## Methodology sketch  

- **Data acquisition**  
  - Download the publicly available *SpatialVid* and *OpenVid* camera‑annotated video datasets (≈2 GB total) via their Zenodo URLs.  
- **Model preparation**  
  - Clone the open‑source *minWM* repository and install dependencies from the provided `requirements.txt` (PyTorch ≤ 2.0, `diffusers`, `opencv-python`).  
  - Use the pre‑trained video diffusion backbone (e.g., *Stable Diffusion Video*) supplied in the repo.  
- **Camera‑conditioning integration**  
  - Apply the existing PRoPE camera‑conditioning module to inject per‑frame pose vectors into the diffusion process.  
- **Inference schedule variants**  
  - Run inference on a single NVIDIA T4‑class GPU (the runner’s default GPU) for each of the following step counts: 1, 2, 4, 8, and the full 16‑step baseline.  
  - For each variant, generate 100 short clips (≈2 s) using the same random seed set.  
- **Latency measurement**  
  - Record wall‑clock time from input pose command to the completion of the first generated frame; compute mean ± standard deviation across the 100 runs.  
- **Controllability evaluation (independent)**  
  - Run a pretrained, *external* 6‑DoF pose estimator (e.g., `DeepV2D`) on the generated videos to recover the camera trajectory.  
  - Compute the mean absolute error (MAE) between the commanded pose sequence and the estimated trajectory (units: degrees / meters). This metric is independent of the diffusion steps themselves.  
- **Statistical analysis**  
  - Perform a repeated‑measures ANOVA to test the effect of step count on latency and on pose‑error separately.  
  - Apply post‑hoc paired t‑tests with Bonferroni correction to compare each reduced‑step variant against the full‑step baseline.  
- **Reproducibility**  
  - All scripts, random seeds, and environment files are version‑controlled; the entire pipeline can be executed on a GitHub Actions runner in ≤ 5 h.
