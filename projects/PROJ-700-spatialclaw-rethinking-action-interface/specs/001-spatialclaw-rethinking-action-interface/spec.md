# Feature Specification: Reproduce & Validate SpatialClaw

**Feature Branch**: `700-reproduce-validate-spatialclaw`  
**Created**: 2024-05-21  
**Status**: Draft  
**Input**: User description: "Reproduce & validate: SpatialClaw: Rethinking Action Interface for Agentic Spatial Reasoning"

## User Scenarios & Testing

### User Story 1 - Execute End-to-End Reproduction Pipeline (Priority: P1)

**Journey**: A researcher triggers the reproduction pipeline on the CI runner. The system initializes the environment, loads the vendored `SpatialClaw` code, executes the agent on a subset of the `BLINK` benchmark, and outputs a structured log file confirming the run completed without errors.

**Why this priority**: This is the core "MVP" for a reproduction project. Without a successful end-to-end execution, no validation or comparison can occur. It confirms the code is runnable and the environment is correctly configured.

**Independent Test**: Can be tested by running the entrypoint script with a minimal dataset configuration and verifying the existence of a `results/` directory containing a non-empty JSON report and a `stdout` log showing "Run completed successfully".

**Acceptance Scenarios**:

1. **Given** the `SpatialClaw` submodule is cloned and dependencies are installed, **When** the user executes the entrypoint `spatial_agent/entrypoints/run.py` with `--dataset blink --subset 5`, **Then** the system must produce a `results/blink_subset_5.json` file and exit with code 0.
2. **Given** the environment is ready, **When** the pipeline encounters a missing dataset file, **Then** the system must fail gracefully with a clear error message "Dataset not found: <path>" and exit with code 1, rather than crashing with a stack trace.

---

### User Story 2 - Validate Artifact Integrity & Reproducibility (Priority: P2)

**Journey**: A reviewer inspects the artifacts generated by the P1 run. The system must generate a validation report that compares the generated outputs (e.g., accuracy metrics, intermediate reasoning traces) against the paper's reported baseline values for the specific subset to ensure the code behaves as claimed.

**Why this priority**: Reproduction is only valuable if the results are verifiable against the original claims. This step ensures the code isn't just "running" but is actually performing the spatial reasoning tasks described in the paper.

**Independent Test**: Can be tested by running a validation script that parses the `results/` JSON and asserts that the reported accuracy is within a 5% tolerance of the paper's reported accuracy for the same benchmark subset (or flags a discrepancy if the subset is too small for statistical significance).

**Acceptance Scenarios**:

1. **Given** a completed run with `results/blink_subset_5.json`, **When** the validation script compares the `accuracy` field against the paper's reported `[deferred]` (adjusted for subset size), **Then** the report must flag "PASS" if the value is within expected variance or "WARN: Deviation > 5%" if not.
2. **Given** the run produced intermediate logs, **When** the validation script checks for the presence of `intermediate_steps` in the JSON, **Then** the report must confirm the presence of at least one reasoning step per query, verifying the "code-as-action" interface is active.

---

### User Story 3 - Resource Feasibility Audit (Priority: P3)

**Journey**: A system administrator verifies that the reproduction run adheres to the strict compute constraints (CPU-only, <7GB RAM, <6h). The system must automatically log resource usage (peak RAM, CPU time) and fail the job if it exceeds the free-tier limits.

**Why this priority**: The project explicitly targets a CPU-only CI environment. If the code requires GPU or exceeds memory, the project is dead on arrival. This audit prevents wasted CI cycles.

**Independent Test**: Can be tested by running the pipeline with a resource monitor attached and verifying the final report includes a `resource_usage` block that asserts `peak_ram_mb <= 7000` and `cpu_time_sec <= 21600`.

**Acceptance Scenarios**:

1. **Given** the run is executing on a standard GitHub Actions runner, **When** the process attempts to load a GPU-accelerated model (e.g., via `torch.cuda`), **Then** the system must detect the lack of GPU and raise a `RuntimeError: GPU not available; aborting` immediately.
2. **Given** the run is processing data, **When** the memory usage exceeds 6.5 GB, **Then** the system must log a warning and attempt to reduce the batch size or sample rate, or fail with a clear "Memory Limit Exceeded" error if recovery is impossible.

---

### Edge Cases

- **What happens when** the dataset download fails or is incomplete? The system must retry up to 3 times, then fail with a specific error code indicating a network or storage issue, preventing partial data processing.
- **How does the system handle** a VLM that returns a non-executable code block? The kernel must catch the syntax error, log the specific failure, and attempt a "reflection" step (if configured) or skip the sample and log it as "unanswerable".
- **What happens when** the paper's reported metrics are for a different model version than the one configured in `config/model/`? The validation step must flag a "Model Mismatch" warning rather than silently failing the accuracy check.

## Requirements

### Functional Requirements

- **FR-001**: System MUST execute the vendored `SpatialClaw` entrypoint `spatial_agent/entrypoints/run.py` against a configured subset of the `BLINK` dataset (See US-1).
- **FR-002**: System MUST produce a structured JSON artifact containing the agent's responses, intermediate reasoning steps, and final accuracy score for the subset (See US-2).
- **FR-003**: System MUST validate the generated artifacts against the paper's reported baseline accuracy (59.9% average) with a tolerance of ±5% for the specific subset size (See US-2).
- **FR-004**: System MUST enforce a hard memory limit of a substantial capacity and a CPU time limit of a substantial duration per job, aborting if exceeded (See US-3).
- **FR-005**: System MUST detect and block any attempt to initialize GPU-specific libraries (e.g., `cuda`, `bitsandbytes`) on CPU-only runners, failing fast with an explicit error (See US-3).
- **FR-006**: System MUST log all intermediate Python kernel execution steps and VLM prompts to a human-readable log file for manual inspection (See US-1).

### Key Entities

- **Reproduction Run**: A single execution instance of the `SpatialClaw` agent on a specific dataset subset, characterized by a unique run ID, dataset name, and model configuration.
- **Validation Report**: A structured document generated post-run containing pass/fail status for artifact integrity, accuracy deviation, and resource compliance.
- **Spatial Reasoning Trace**: The sequence of code cells and VLM responses generated by the agent for a single query, stored as part of the run artifact.

## Success Criteria

### Measurable Outcomes

> Planning docs state *what* will be measured and the *source/reference* it is measured against; defer specific empirical values (counts, dataset sizes, measured quantities, percentages) to the implementation/research phase.

- **SC-001**: The percentage of test samples in the subset for which the agent produces a valid, executable code block is measured against the paper's reported "success rate" for the same benchmark (See US-1).
- **SC-002**: The deviation of the agent's accuracy on the subset from the paper's reported [deferred] average is measured against the ±5% tolerance threshold defined in FR-003 (See US-2).
- **SC-003**: The peak memory usage of the reproduction run is measured against a predefined memory limit. defined in FR-004 (See US-3).
- **SC-004**: The total wall-clock time of the run is measured against the predefined execution time limit. defined in FR-004 (See US-3).
- **SC-005**: The presence of "GPU not available" errors when attempting to load CUDA-dependent modules is measured against the requirement for CPU-only execution (See US-3).

## Assumptions

- **Assumption about data**: The `BLINK` dataset and other benchmarks are accessible via the paths defined in `spatial_agent/config/dataset/` or can be downloaded from the public source if the submodule includes a download script.
- **Assumption about environment**: The GitHub Actions runner provides a Python environment with sufficient disk space to install dependencies and cache the dataset..
- **Assumption about model**: The reproduction will use a smaller, CPU-tractable VLM (e.g., `gemma-4-26b` quantized or a smaller variant) as the primary back-end, rather than the large models listed in `config/model/` which may require GPU.
- **Assumption about scope**: The validation focuses on the `BLINK` benchmark subset as a representative sample; full 20-benchmark reproduction is out of scope for the initial P1 run due to time constraints.
- **Assumption about paper claims**: The paper's reported [deferred] accuracy is based on the average across multiple benchmarks; a single benchmark subset may naturally deviate, so the 5% tolerance accounts for statistical variance.
