## Unresolved concerns

- Task T034 (Human Proxy) is listed as a dependency for the analysis in Phase 5, but the task description states it runs `HuggingFaceH4/image-reward` to 'break circular dependency'. However, T030 (Scoring) loads VLM reward models. If T034 is intended to provide the 'ground truth' for FR-008, it must produce an artifact (scores) that T031/T032/T033 consume. Currently, T034 is a standalone task with no explicit output artifact defined that feeds into the 'Generalization Gap' calculation tasks (T031-T033). The data flow is broken: the 'Human Proxy' scores are not shown as an input to the degradation calculation, making the validation of the gap impossible within the defined task order.
- Task T015 (curate_prompts.py) generates the prompt sets. Task T016 (validate_ood.py) validates them. However, T015 is marked `[P]` (parallel-safe) while T016 is not. T016 depends on the output of T015. If T015 and T016 are executed in parallel (as implied by the `[P]` tag on T015 and the lack of explicit sequential ordering in the 'Within Each User Story' section), T016 will fail or read incomplete data. The task order must explicitly enforce T015 -> T016, and T015 should not be marked `[P]` if it is the sole producer for a critical validation step that must complete before the story is considered 'functional'.
- Task T033 (statistical_test.py) performs the Welch's t-test. Task T035 (power_analysis.py) calculates statistical power. The plan (plan.md:Statistical Power) requires a 'Pilot Phase' to determine sample size *before* the full run. T035 is listed as a post-hoc task in Phase 5. The task order implies the full analysis (T030-T033) happens, and *then* power is calculated. This violates the 'Producer before Consumer' logic for the Pilot-to-Target strategy: the power analysis (T035) must run *after* the Pilot (N=20) but *before* the Target (N=500) generation (T021-T023) to determine if N=500 is feasible. The current order forces the full 500-prompt generation before knowing if it's statistically necessary or feasible.
- Task T017 (error handling) and T018 (logging) are listed after T016. While logically sound, T017 mentions 'aborting if > 0.3' which is the logic of T016. T017 should be integrated into T016 or T015 to ensure the abort logic is part of the producer/consumer flow, not a separate post-processing step. As written, T017 implies a separate execution step that might run after T016 has already passed, creating a race condition or redundant check.
