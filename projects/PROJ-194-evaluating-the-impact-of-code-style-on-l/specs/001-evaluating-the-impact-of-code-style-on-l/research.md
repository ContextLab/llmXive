# Research: Evaluating the Impact of Code Style on LLM Code Understanding and Generation

## Research Question
How do specific code style dimensions (formatting, commenting) and semantic obfuscation (naming) impact LLM performance on code completion, bug detection, and summarization tasks?
**Note**: This is an **exploratory pilot study**. Due to CPU constraints, the sample size (N=30 functions) is insufficient for high-power confirmatory inference on interaction effects. The primary goal is to estimate effect sizes and variance components for future powered studies.

**Synthetic Bug Injection Strategy**:
- **Source**: No verified Python bug-detection dataset exists in the source block.
- **Method**: Use `mutmut` or a custom `ast-based` mutation library to inject bugs (e.g., variable swap, operator flip) into the "clean" variants.
- **Ground Truth**: Labels are synthetic: `1` if mutation applied, `0` if clean. This ensures a balanced dataset (FR-008) without relying on external bug datasets.

**Metric Calculation**:
- **ROUGE-L / BLEU**: Calculated algorithmically against the `docstring` field of the CodeSearchNet dataset. **No external dataset is required or used** for metric calculation. The previous citation to `gswamy/pythia...` is removed as it was semantically incorrect.

**Dataset Fit Confirmation**:
- **Predictors**: Style factors (Formatting, Naming, Commenting) are synthetic, generated by the pipeline.
- **Outcomes**: LLM metrics (Exact Match, CodeBLEU, etc.) are calculated from model outputs.
- **Covariates**: Function complexity (token count, lines of code) derived from `code` field.
- **Missing Variables**: The dataset does *not* contain "post-task anxiety" or "rumination" (not applicable to code). It does contain `docstring` which serves as the ground truth for summarization. The plan explicitly preserves `docstring` during the "strip comments" transformation to ensure GT availability (FR-001).

**Decision**: The CodeSearchNet Python subset is the correct and only viable dataset. Bug detection relies on synthetic mutation.

## Methodology & Statistical Rigor

### Experimental Design
- **Design**: 2x2x2 Full Factorial Design **+ 1 Control Condition**.
 - Factor A: Formatting (Black vs. Minified)
 - Factor B: Naming (Meaningful vs. Generic)
 - Factor C: Commenting (Present vs. Stripped)
 - **New Control**: **GN-PC (Generic Naming + Preserved Comments)**. This condition isolates the effect of "semantic loss" from "style opacity".
- **Unit of Analysis**: Single function instance.
- **Replication**: 30 unique functions sampled from CodeSearchNet (Pilot).
- **Confound Control**:
 1. **Semantic Opacity**: "Semantic Opacity" (Generic + Stripped) is compared against **GN-PC** to determine if the drop is due to style or semantic loss.
 2. **Tokenization**: `token_count` is included as a covariate in all models to isolate "pure style" from "length" effects.

### Statistical Analysis Plan
1. **Normality Check**: Shapiro-Wilk test on residuals of the initial linear model.
2. **Model Selection (Per Metric Type)**:
 - **Exact Match (Binary)**: **Generalized Linear Mixed-Effects Model (GLMM) with Binomial distribution**.
 - **CodeBLEU, ROUGE-L (Continuous)**: **Linear Mixed-Effects Model (LMM) with Gaussian distribution**.
 - **Rationale**: A single LMM is inappropriate for mixed outcome types. Separate models ensure valid inference.
3. **Fixed Effects**: Formatting, Naming, Commenting, and all interactions.
4. **Random Effects**: Random intercept for `function_id` to account for repeated measures.
5. **Covariate**: `token_count` included in all models to control for length effects (FR-007).
6. **Multiple Comparisons**: Bonferroni correction applied to all pairwise post-hoc tests (SC-003).
7. **Effect Size**: Partial eta-squared ($\eta^2$) for LMM; Odds Ratios for GLMM (SC-004).
8. **Collinearity**: If `token_count` absorbs the variance of the "Formatting" factor, the "Formatting" effect is reported as "confounded with tokenization".

### Power & Sample Size (Pilot Study)
- **Simulation**: A power analysis simulation is run on a small pilot (N=10) to estimate variance components.
- **Target**: N=30 functions (240 variants) is the maximum feasible on CPU.
- **Stop Rule**: If the simulation indicates power < 0.5 for interaction effects at N=30, the study is declared **exploratory only**. No confirmatory p-values will be claimed; results will be reported as "indicative effect sizes".
- **Limitation**: The study cannot definitively confirm causal claims for interaction effects due to sample size constraints.

### Causal Inference Assumptions
- **Style vs. Semantics**:
 - **Formatting & Commenting**: Treated as "Style" factors (semantics preserved).
 - **Generic Naming**: Treated as a **"Semantic Obfuscation"** factor. The "Generic" condition is not a style variant of the *same* semantics; it is a different semantic object.
 - **Claim Restriction**: Causal claims regarding "style impact" are restricted to Formatting and Commenting. The "Naming" effect is interpreted as "semantic obfuscation".
- **Identification Strategy**: The orthogonal generation of variants ensures that style factors are uncorrelated with code semantics (except for Naming). The random intercept for `function_id` controls for inherent code difficulty.

## Compute Feasibility Strategy

The plan is constrained to the GitHub Actions free tier (CPU, 7GB RAM, no GPU).

1. **Model Selection**: Use `Salesforce/codegen-2B-mono` loaded in **default precision** (float32).
 - **Fallback**: If 2B model exceeds RAM, fallback to `Salesforce/codegen-350M-mono`.
 - **Constraint**: No `load_in_8bit`, no `device_map="cuda"`, no quantization.
2. **Data Sampling**:
 - Data will be subset to fit ~GB RAM.
 - Target: **30 functions** $\times$ 8 variants = 240 inference samples.
 - Inference will be batched (batch size 1) to avoid memory spikes.
3. **Runtime Management**:
 - Total runtime budget: a fixed duration allocated for the execution of the study.
 - Inference time per sample: on the order of tens of seconds (CPU).
 - 240 samples $\times$ 45s = 10,800s = 3 hours (Feasible).
 - **Decision**: Plan targets **30 functions** as a pilot. If runtime permits, expand to 40.

## Risk Mitigation

- **Syntax Errors in Variants**: FR-002 mandates a `ast.parse()` check. If a variant fails, it is discarded and logged.
- **Timeout/Resource Exhaustion**: Retry mechanism with a configurable maximum number of attempts and exponential backoff. If a sample fails all retries, it is marked as `timeout` and excluded from statistical analysis (with log).
- **Missing Docstrings**: Functions with empty `docstring` are skipped for the summarization task (US-2).
- **Model Failure**: If the model crashes on a specific code pattern, the sample is flagged and excluded, not used to crash the whole run.
- **Power Limitation**: If power < 0.5, the report will explicitly state the study is **exploratory** and interpret results as "pilot estimates".
