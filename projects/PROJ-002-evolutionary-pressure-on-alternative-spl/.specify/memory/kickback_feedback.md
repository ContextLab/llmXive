## Unresolved concerns

- The plan relies on 'Synthetic FASTQ files' and 'Simulated phyloP scores' to validate the pipeline logic. This approach cannot validate the causal claim that 'splicing divergence correlates with positive selection' because the synthetic data generator will likely lack the complex, non-linear biological signal (e.g., specific SRE motifs driving PSI changes) required to test the enrichment hypothesis. A pipeline that runs successfully on random/synthetic data proves only code execution, not scientific validity. The research question requires real biological signal to answer.
- The proposed PGLS model `accelerated_flag ~ lineage + (1 | phylogeny)` is statistically invalid for the stated goal. The dependent variable `accelerated_flag` is binary (derived from phyloP), and the independent variable `lineage` is categorical. PGLS is designed for continuous traits evolving under Brownian motion. Using PGLS on a binary outcome derived from the same phylogenetic tree used for correction creates a circular dependency and violates model assumptions. A phylogenetic logistic regression (e.g., `phyloglm` in R) or a phylogenetic Fisher's exact test extension is required for binary enrichment data.
- The plan states 'Sampling the input FASTQ files to a manageable size (e.g., first 2M reads per sample)' for CI. While acceptable for runtime validation, the spec requires detecting ΔPSI ≥ 0.1 with 80% power (FR-011). 2M reads may be insufficient to robustly quantify PSI for lowly expressed genes or complex splicing events in non-human primates where reference bias is higher. The power analysis in `research.md` assumes the synthetic data mimics real variance, but without real data, the power calculation is unverified. The methodology cannot confirm if the sample size is actually sufficient for the biological effect size.
- FR-010 mandates a 90-day retention policy followed by Zenodo deposition and local deletion. The plan.md and research.md explicitly state that the CI run uses 'synthetic data' and 'simulated metadata' to fit free-tier constraints. There is no corresponding plan element (phase, task, or contract) describing the implementation of the Zenodo upload logic, the 90-day timer mechanism, or the local deletion logic. The plan focuses entirely on the synthetic CI path, leaving the real-world data lifecycle requirement (FR-010) unbacked.

- The plan proposes using PGLS to 'adjust enrichment p-values' by regressing 'accelerated_flag ~ lineage'. This is a category error. PGLS requires a continuous response variable and estimates a phylogenetic signal (lambda) to correct standard errors. The outcome here is binary (accelerated vs. not) and the predictor (lineage) is the grouping factor. You cannot 'adjust a Fisher's p-value' via PGLS in this manner. The analysis conflates a contingency table test (Fisher) with a regression model, rendering the statistical validation circular or mathematically invalid.
- The plan defines 'Lineage-Specific Events' (LSE) using a threshold (ΔPSI > 0.1) and then tests for enrichment of these same events against 'accelerated' regions. If the definition of LSE and the definition of 'accelerated' (phyloP ≤ -2.0) are both derived from the same set of events without an independent null model or permutation test, the enrichment test may be tautological. Specifically, if the 'accelerated' flag is determined by the same evolutionary divergence that defines the 'lineage-specific' nature of the event, the test validates a definition rather than an empirical correlation.
- The plan states: 'The model regresses the binary presence/absence of accelerated events on lineage'. PGLS (via `caper`) is designed for continuous traits evolving under Brownian motion or OU models. Applying it to a binary trait (0/1) without using a phylogenetic logistic regression (e.g., `phyloglm` in `phylolm` or `ape`) is methodologically unsound. The resulting p-values will be invalid.
- The plan applies Bonferroni correction 'within lineage' (α / N_events) for the Fisher test. However, the Fisher test is performed on the *aggregate* set of events (LSE vs Non-LSE) to produce a *single* p-value per lineage. There is no 'N_events' to correct against within a single Fisher test; the test already accounts for the counts. This suggests a misunderstanding of the statistical test being performed. Correction should only apply across the 4 lineages (FDR), not within a single contingency table test.
