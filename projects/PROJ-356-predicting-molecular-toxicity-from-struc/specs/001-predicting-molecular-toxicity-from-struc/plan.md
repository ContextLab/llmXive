# Implementation Plan: Predicting Molecular Toxicity from Structural Alerts via Rule-Based Systems

**Branch**: `001-predicting-molecular-toxicity` | **Date**: 2026-06-28 | **Spec**: `specs/001-predicting-molecular-toxicity/spec.md`

## Summary

This feature implements a reproducible, CPU-tractable pipeline to compare two modeling paradigms for mutagenicity prediction: (1) a transparent, rule-based scoring system derived from curated structural alerts (SMARTS patterns), and (2) a data-driven Logistic Regression model using global molecular descriptors. The pipeline downloads a verified mutagenicity dataset, standardizes chemical structures, extracts both feature sets, trains models via stratified cross-validation, and performs a rigorous statistical comparison (DeLong's test) to determine if the added complexity of descriptors yields a statistically significant improvement in ROC-AUC.

**Scale/Scope**: Dataset subset to a manageable size (sampled if necessary) to fit memory; A set of structural alerts; A set of molecular descriptors.

## Constitution Check

*GATE: Must pass before Phase 0 research.*

- **I. Reproducibility**: The plan mandates pinned random seeds, deterministic data fetching from verified URLs, and a `requirements.txt` for environment isolation. All scripts will be runnable end-to-end. A custom implementation of DeLong's test is provided in code to ensure reproducibility without external package dependencies.
- **II. Verified Accuracy**: Dataset URLs are restricted to the verified list (ToxCast/Parquet). No external citations for dataset sources will be added without verification.
- **III. Data Hygiene**: Raw data downloads will be checksummed (SHA-256). Checksums will be recorded in `data/raw/.checksums.json` and mirrored in the project state file `artifact_hashes` map. No data may be modified in place; every transformation produces a new file with a documented derivation.
- **IV. Single Source of Truth**: All metrics (AUC, F1, p-values, Recall, motif counts) will be generated by code and stored in JSON reports, never hand-typed.
- **V. Versioning**: A dedicated script `code/scripts/update_state.py` (or a step in `run.py`) will generate SHA-256 hashes of all artifacts (data, models, reports) and update the `state/projects/PROJ-356-...yaml` file with these hashes and timestamps upon completion.
- **VI. Statistical Significance**: The plan explicitly includes DeLong's test for model comparison as mandated by the spec and constitution. The test is applied to **out-of-fold predictions** (one prediction per instance from the fold where it was held out), not averaged predictions, to satisfy statistical validity.
- **VII. Chemical Structure Validity**: RDKit standardization and Da mass filtering are mandatory pre-processing steps before any feature engineering.

## Detailed Implementation Phases

### Phase 1: Data Acquisition & Preprocessing (FR-001, FR-002)
1.  **Download**: Fetch `toxcast.csv` from the verified HuggingFace URL.
2.  **Checksum**: Calculate SHA-256 of the downloaded file. Record in `data/raw/.checksums.json` and `state/...yaml`.
3.  **Standardize**: Use RDKit to canonicalize SMILES.
4.  **Filter**: Remove molecules with MW >= 1000 Da or invalid structures. Log counts.
5.  **Deduplicate**: Remove duplicate SMILES. If labels conflict, discard the entry and log.

### Phase 2: Feature Engineering (FR-003, FR-004)
1.  **Alerts**: Load `config/structural_alerts.json`. Validate against `contracts/alerts.schema.yaml`. Generate binary vectors for multiple patterns.
2.  **Descriptors**: Compute a **fixed, pre-defined set** of 20 global descriptors (e.g., MW, LogP, TPSA, etc.) using `rdkit.Chem.Descriptors`. No data-dependent correlation filtering is performed to prevent leakage.
3.  **Scaling**: Apply `StandardScaler` (fit on train fold only) to descriptors.

### Phase 3: Model Training & Evaluation (FR-005, FR-006)
1.  **Cross-Validation**: k-fold stratified CV, repeated multiple times (total folds).
2.  **Rule-Based Model**: Calculate risk score (sum of weights).
    *   **ROC-AUC**: Sweep threshold over the **entire continuous range** of scores (0 to K) to generate the ROC curve.
    *   **F1/Recall**: Calculate at discrete thresholds (low, 0.5, 0.7) for error analysis.
3.  **Logistic Regression**: Train with L regularization.
    *   **ROC-AUC**: Sweep threshold over [0, 1].
    *   **F1/Recall**: Calculate at discrete thresholds.
4.  **Metrics**: Compute ROC-AUC, F1, Recall, and Confusion Matrix for both models.
5.  **Recall Comparison**: Calculate difference in Recall (Desc - Rule) and compare against a predefined significance threshold (SC-002).

### Phase 4: Statistical Analysis (FR-007, SC-001)
1.  **Prediction Collection**: Collect **out-of-fold (OOF)** predictions for every instance (one prediction per instance from the fold where it was held out). **Do NOT average** predictions across folds for this step.
2.  **DeLong's Test**: Perform DeLong's test on the paired OOF probability vectors.
3.  **Output**: P-value, 95% CI, and significance flag.
4.  **Correction**: No multiple-comparison correction (FR-009).

### Phase 5: Error Analysis (FR-008, SC-003)
1.  **Identify FNs**: Filter test set for Rule-Based model False Negatives.
2.  **Scaffold Extraction**: Compute Murcko scaffolds for FNs.
3.  **Count Comparison**: Count the number of FNs and the number of unique missing motifs. Compare these counts to determine feasibility of rule-set expansion (SC-003).
4.  **Report**: List top scaffolds

The research question is to identify the most prevalent scaffolds, using a method of ranking by frequency as described in [Author, Year] (DOI: XX.XXXX/XXXX). and specific chemical classes.

### Phase 6: Reporting & Versioning
1.  **Generate JSON**: Create `results/metrics.json` with all metrics, test results, and error analysis.
2.  **Update State**: Run `scripts/update_state.py` to hash all artifacts and update `state/projects/PROJ-356-...yaml`.

## Risk Assessment

1.  **Dataset Variable Mismatch**: If the verified dataset lacks specific labels or SMILES.
    *Mitigation*: Pre-flight check script validates column existence.
2.  **SMARTS Syntax Errors**: Invalid patterns in `config/structural_alerts.json`.
    *Mitigation*: Try-except blocks in `features/alerts.py` to log and skip invalid patterns.
3.  **Class Imbalance**: Severe imbalance may skew Logistic Regression.
    *Mitigation*: Stratified CV preserves distribution; class weights can be tuned if F1 is poor.
4.  **Runtime Exceedance**: Large dataset causing OOM.
    *Mitigation*: Sampling logic to cap dataset size at a computationally manageable scale if necessary.
5.  **Statistical Validity**: Incorrect application of DeLong's test.
    *Mitigation*: Strict adherence to OOF prediction collection; custom implementation validated against known examples.
