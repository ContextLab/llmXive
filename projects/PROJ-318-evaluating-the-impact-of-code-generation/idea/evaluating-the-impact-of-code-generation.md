# Evaluating the Impact of Code Generation Models on Code Documentation Completeness

**Field**: computer science

## Research question

Does documentation generated by open-weight code LLMs achieve functional parity with human-written documentation for public methods in Python open-source repositories?

## Motivation

Software maintainability depends on accurate documentation, which often lags behind code changes in practice. Understanding the completeness gap in LLM-generated docs is critical for adopting AI-assisted development tools safely without degrading codebase quality.

## Related work

- [DocAgent: A Multi-Agent System for Automated Code Documentation Generation (2025)](http://arxiv.org/abs/2504.08725v3) — Proposes multi-agent systems for documentation, relevant for comparing single-model vs. agent-based generation quality.
- [SIMCOPILOT: Evaluating Large Language Models for Copilot-Style Code Generation (2025)](http://arxiv.org/abs/2505.21514v1) — Introduces benchmarks for interactive coding assistants, providing metrics adaptable for documentation evaluation.
- [Execution-based Evaluation for Data Science Code Generation Models (2022)](http://arxiv.org/abs/2211.09374v1) — Demonstrates execution-based validation methods that can be adapted to verify docstring accuracy against code behavior.
- [JaCoText: A Pretrained Model for Java Code-Text Generation (2023)](http://arxiv.org/abs/2303.12869v1) — Investigates code-text generation models, offering architectural baselines for documentation tasks.
- [ReCode: Robustness Evaluation of Code Generation Models (2022)](http://arxiv.org/abs/2212.10264v1) — Focuses on robustness evaluation, relevant for testing documentation stability across prompt variations.

## Expected results

LLM-generated documentation will cover approximately 70–80% of public method signatures but frequently omit detailed parameter descriptions. A Wilcoxon signed-rank test will confirm that the completeness gap between human and LLM docs is statistically significant (p < 0.05).

## Methodology sketch

- **Data Acquisition**: Download source code for the top 20 Python repositories from the PyPI leaderboard using `curl`/`wget` via the GitHub API to ensure public dataset usage.
- **Preprocessing**: Use Python `ast` module to extract all public methods and their signatures; extract existing human-written docstrings for ground truth.
- **Model Setup**: Load `Salesforce/codegen-350M-mono` using 4-bit quantization (`bitsandbytes`) to fit within 7GB RAM constraints on the GitHub Actions runner.
- **Generation**: Prompt the model to generate docstrings for each extracted method using a fixed temperature (0.2) to ensure reproducibility.
- **Analysis**: Compute completeness scores by matching generated docstring parameters against AST-defined parameters; calculate semantic similarity using `sentence-transformers/all-MiniLM-L6-v2`.
- **Statistical Test**: Apply a Wilcoxon signed-rank test to compare human vs. LLM completeness scores across the 20 repositories.
- **Resource Control**: Limit generation to 1,000 methods per repo to ensure the full pipeline completes within the 6-hour job time limit.
