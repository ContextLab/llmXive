## Unresolved concerns

- FR-005 and US-2 require projecting 'external frequency distributions' onto the pseudo-inverse of W_U to compute a 'mean embedding' vector. However, the plan.md Summary explicitly states the methodology projects '*model token embeddings* (not frequency distributions)'. T020 and T021 implement the frequency distribution projection, creating a direct contradiction with the plan's stated methodology. The tasks do not cover the 'model token embeddings' projection described in the plan, leaving that plan element unimplemented.
- US-2 Acceptance Scenario 2 requires calculating the 'overlap ratio of the top-10 tokens' and reporting it. T022 implements this, but T023 (Integration) is marked as unchecked `[ ]`. Without T023, the pipeline cannot output the `token_attribution_report.json` required to verify the overlap ratio, leaving the US-2 independent test unexecutable.
- US-3 requires a permutation test and external validation. T030 (Integration) is marked as unchecked `[ ]`. Without T030, the `permutation_result.json` and `wals_validation.json` outputs are not generated, preventing the verification of the p-value and significance flag required by US-3 Acceptance Scenario 2.
- US-1 Independent Test requires outputting a JSON report with cosine similarity scores. T014 (Orchestrator) is marked as unchecked `[ ]`. Without T014, the `similarity_matrix.json` is not generated, preventing the verification of the geometric comparison pipeline.
- Plan.md Project Structure lists `tests/contract/test_schemas.py` as a Phase 1 output. T008 implements this skeleton but is marked `[ ]` (unchecked). Without T008, the contract tests for `similarity_report.schema.yaml`, `permutation_result.schema.yaml`, and `wals_validation.schema.yaml` cannot run, violating the plan's testing strategy.
- Plan.md mandates 'Contract tests validating JSON output schemas'. T009, T016, and T024 implement these contract tests but are all marked `[ ]` (unchecked). Without these, the plan's requirement for schema validation is not covered by any task.
- Plan.md Technical Context specifies 'Memory usage < 6 GB peak' and 'Stream real data'. T037 implements 'Data Streaming Verification' to ensure `streaming=True` is used, but it is marked `[ ]` (unchecked). Without T037, there is no task to verify the memory constraint compliance for large datasets.
- Spec.md Edge Cases ask 'What happens when the SVD computation fails due to numerical instability...'. T038 implements a 'Numerical Stability Guard' for this, but it is marked `[ ]` (unchecked). Without T038, the edge case handling is not implemented.
- T012b (Full SVD Execution) is marked as requiring T008a (Profiling) and T011a (Vocab Mapping), but T008a is listed as a dependency of T012, not T012b. More critically, T012b executes the 'complete unembedding matrix' SVD which is the primary consumer of the 'representative subset' logic in T012. The dependency chain T012 -> T008a -> T012b is circular or missing the explicit 'Profiling Pass' artifact that T012b must consume to validate the 6h constraint before execution. T012b should explicitly depend on the *output* of T008a (the pass/fail signal), not just the task completion.
- T021 (Token Ranking) depends on T020 (Mean Embedding) and T012b (Full SVD). However, T020 depends on T019a/b (Frequency Distributions) and T012b. The task T021 description states it ranks tokens based on the projection of the 'frequency-weighted mean embedding vector' onto the 'Edge Spectrum subspace'. The Edge Spectrum subspace is the output of T012b. The dependency T021 -> T012b is correct, but T021 also implicitly requires the *Vocabulary Mapping* layer from T011a to map the ranked token IDs back to the specific language vocabularies (English vs BLOOM) for the overlap calculation in T022. T021 should explicitly depend on T011a to ensure the token IDs are valid for the downstream overlap logic.
- T029 (WALS Correlation) depends on T013 (Cosine Similarity), T029a (Config), and T029b (WALS Acquisition). T013 computes the similarity between English models and BLOOM. T029 uses this to correlate with WALS features. However, the spec (US-3) and plan imply the WALS validation is an *external* validation of the *observed shift*. The dependency on T013 is correct for the 'observed' value, but T029 also requires the *null distribution* or the *statistical test results* (p-value) from T028 to contextualize the correlation (e.g., is the correlation significant?). While T029 might just output a correlation coefficient, the 'validation' logic in the spec implies a comparison against the statistical significance established in US-3. T029 should depend on T028 to ensure the validation is performed in the context of the statistical test results.
- T008a (MANDATORY PROFILING) is listed as a dependency for T012b, but T008a itself depends on T012 (SVD extraction on subset). The task T008a description says 'Implement ... to perform a profiling run ... to verify the computational time constraint'. This implies T008a *runs* the code from T012. The dependency T008a -> T012 is correct. However, T012b (Full SVD) depends on T008a. This creates a flow where T012 (subset) -> T008a (profile) -> T012b (full). This is logically sound, but the task T012b description says 'to be executed ONLY after T008a passes'. This is a runtime guard, not a build-time dependency. The ordering is correct, but the task description could be clearer that T008a produces a 'pass/fail' artifact that gates T012b.
- T037 (Data Streaming Verification) depends on T019a and T019b. T019a/b implement the frequency distribution computation. T037 adds logging/tests to verify `streaming=True` is used. This is a valid dependency. However, T037 is in the 'Polish' phase, while T019a/b are in US2. If T019a/b are implemented without streaming (violating the memory constraint), T037 will fail. The dependency is correct, but the risk is that the violation is caught too late (in the Polish phase) rather than during the US2 implementation. Consider moving T037's verification logic into the US2 implementation tasks or making T037 a prerequisite for T019a/b completion.

- Task T026a implements a 'Within-Language Baseline' (Llama-3 vs. Mistral) as a proxy for the 'model-seed null distribution' required by FR-004. The task description admits this is a 'Deviation' because seed-variant models are unavailable. This constitutes a silent relaxation of the functional requirement FR-004 ('System MUST perform a permutation test... using a model-seed null distribution'). A task cannot satisfy an FR by implementing a fundamentally different statistical method (comparing two different models vs. comparing seed variants of the same model) without an explicit amendment to the spec or a formal requirement relaxation in the plan. The current task text treats a methodological failure as a valid implementation.
- Task T029 includes a clause: 'If data is unavailable, the task must output a data_unavailable flag and graceful degradation logic (no mock validation allowed).' The task effectively allows the project to pass the 'validation' step without performing the validation, which is a semantic weakening of the success criterion. The spec/plan must explicitly define an alternative validation path or a 'fail' state for SC-004 if the primary data source is missing; 'graceful degradation' without a replacement metric is a constraint violation.
- Task T012b mandates 'Full SVD Execution... for the complete unembedding matrix' for Llama-3, Mistral, and BLOOM. The plan and spec (Assumptions) state the system must run on a 7GB RAM runner with a <6h CPU constraint. The task relies on T008a (profiling on a 'representative subset') to verify feasibility, but profiling a subset does not guarantee the full matrix SVD fits in memory or time. This task silently assumes feasibility without a mandatory verification step for the *full* matrix before execution, risking a violation of SC-005 (computational feasibility) and Constitution Principle I (Reproducibility) if the run OOMs or times out. The task should include a hard pre-check or fallback mechanism for the full matrix execution.

The task list separates RedPajama processing (T019a) but lacks a corresponding acquisition/validation task for RedPajama that mirrors T018a's strictness for Common Crawl, creating a gap in the FR-006 compliance chain.
