## Unresolved concerns

- The plan relies on 'external RL failure rates' provided as a separate CSV (data/external/rl_failure_rates.csv) to validate the hypothesis. However, the spec and plan do not define the source, generation method, or verification of this ground truth. If this data is synthetic or unverified, the correlation analysis (FR-005) lacks causal validity and cannot support the claim that the 'gap' predicts failure. The methodology assumes the existence of a verified ground truth without a plan to obtain or validate it.
- The plan proposes using K-Means clustering (k=2) on the divergence scores to define 'High' and 'Low' risk groups, then using these labels to train a logistic regression classifier. This introduces circular reasoning (data leakage): the clustering is unsupervised and driven solely by the divergence metric, so the resulting labels are mathematically guaranteed to correlate with the input. Training a classifier on these labels to 'predict' the same input's relationship to failure is methodologically unsound and will yield artificially high accuracy that does not generalize.
- The plan assumes N=500 is sufficient for statistical power without a formal power analysis. For a Pearson correlation test aiming to detect a moderate effect size (r=0.3) with 80% power at alpha=0.05, N=500 is adequate. However, if the expected effect is small (r=0.15), N=500 yields power < 0.5. The plan must explicitly state the minimum detectable effect size or perform a power analysis to justify the sample size, as the research question hinges on detecting a potentially subtle 'gap' effect.
- The plan uses DistilBERT embeddings for the 'thinking' prefix and BM25 for tool retrieval, then computes cosine similarity between them. BM25 is a keyword-based retrieval system that does not produce semantic embeddings. The plan states: 'centroid of retrieved tool descriptions' but does not specify if the tool descriptions are also embedded with DistilBERT before centroid calculation. If the tool centroid is calculated from raw text or BM25 scores, the cosine similarity calculation is invalid (comparing vectors of different dimensions or types). The methodology must explicitly state that tool descriptions are embedded using the SAME model (DistilBERT) as the thinking prefix.
- The plan cites 'iohadrubin/nq_reranking_bm25' as the 'BM25 Retrieval Corpus' for tool documentation. This dataset (Natural Questions) contains Wikipedia article titles and text, not the specific 'tool documentation' or 'function signatures' required for an agentic reasoning environment. Using general web text as a proxy for a specific tool API corpus creates a semantic mismatch: the 'thinking' trace (math reasoning) will be compared against a centroid of generic Wikipedia text, not the actual tools the agent is supposed to select. This invalidates the 'Semantic Divergence' metric as it measures divergence from a generic corpus rather than the specific tool distribution defined in the hypothesis.
- The plan assumes 'ground_truth_tools' exist in the MathVista dataset to drive the BM25 retrieval. MathVista is a multimodal math reasoning benchmark containing questions, images, and answers; it does not natively contain a 'ground_truth_tools' field or a list of executable tool identifiers for each problem. The plan fails to cite a source for where these tool identifiers are derived from (e.g., a specific tool-augmented subset, a separate mapping file, or a synthetic generation step). Without a verified source for the 'tool' vocabulary, the retrieval step cannot be constructed as described.
- The plan relies on an external CSV 'data/external/rl_failure_rates.csv' for the dependent variable (RL failure rates) but provides no source, URL, or generation method for this data. The spec assumes the project 'does not re-run RL training,' yet no alternative source for these specific failure rates (linked to MathVista problem IDs) is cited. If this data does not exist publicly or is not generated in a prior step, the correlation analysis (FR-005) is impossible to execute, rendering the validation of the hypothesis infeasible.
- Constitution Principle V (Versioning Discipline) is claimed as 'passed' in the plan, but the plan only mentions a script to 'generate content hashes'. It fails to explicitly state that these hashes MUST be recorded in the project state file (`state/projects/PROJ-849-llmxive-follow-up-extending-agent-explor.yaml`) as required by the Constitution. This is a direct violation of Principle V.
- Constitution Principle II (Verified Accuracy) is claimed as 'passed', but the plan states 'All citations in research.md will reference only the verified dataset URLs'. The Constitution requires that citations be verified by the 'Reference-Validator Agent' against the primary source with a title-token-overlap threshold. The plan describes a static constraint (using verified URLs) but omits the mandatory *process* (the Agent verification gate) required by Principle II.
- The plan states 'Target Platform: Linux (GitHub Actions Free Tier: 2 CPU, 7 GB RAM)' but the Spec (FR-008) and Constitution (Principle I) require the system to 'automatically sample data if the full dataset exceeds these constraints'. The plan does not explicitly describe the *mechanism* for this automatic sampling (e.g., streaming, batch size adjustment) in the Technical Context, only the constraint itself.
- The 'Source Code' structure lists `src/models/schemas.py` but the 'contracts/' directory contains `dataset.schema.yaml` and `output.schema.yaml`. The plan does not explicitly state that `schemas.py` is auto-generated from or strictly synchronized with the YAML contracts, creating a potential SSoT (Single Source of Truth) ambiguity between the code and the contract definitions.
- The plan uses 'ground_truth_tools' from MathVista to drive the BM25 retrieval (Section 2, 'Data Fit Verification'). The resulting 'tool_centroid' is then compared against the 'thinking_prefix' to compute 'Semantic Divergence'. This creates a circular validation: the 'external' tool distribution is definitionally determined by the ground truth of the problem the agent is solving. If the agent's thinking is correct, it likely aligns with the ground truth tools, making the divergence score a trivial function of correctness rather than an independent measure of a 'thinking-acting gap'. The validation target (divergence) is not independent of the predictor (ground truth used for retrieval).
- The plan correlates 'Semantic Divergence' (derived from ground-truth tools) with 'RL failure rates'. Since the divergence metric is mathematically tied to the ground truth (via BM25 on ground-truth tools), and RL failure is defined by deviation from ground truth, the correlation is tautological. High divergence (mismatch between thinking and ground-truth tools) will mechanically correlate with failure. This analysis cannot empirically validate the 'gap' hypothesis; it merely confirms that 'not matching the answer key correlates with failing the test'.
- The plan proposes K-Means clustering on 'divergence scores' to create 'High' and 'Low' risk groups, then uses these to predict failure. Since the divergence score is derived from ground-truth alignment, clustering it effectively clusters 'correctness' vs 'incorrectness'. Using this to predict failure (which is also defined by correctness) is a category error. The predictor and the outcome are measuring the same underlying latent variable (alignment with ground truth), rendering the logistic regression a trivial identity check rather than a predictive model.
- The plan assumes 'ground_truth_tools' are available in MathVista to drive BM25. MathVista is a multimodal QA dataset; it typically provides a final answer, not a structured list of 'ground truth tools' or API calls. If the plan infers tools from the answer or hallucinates them, the 'tool_centroid' becomes noisy and the metric invalid. If the plan relies on a different dataset for tools, the link between the specific problem instance and the tool set is broken. The methodology assumes a data structure (problem -> specific tool list) that may not exist in the cited source.
