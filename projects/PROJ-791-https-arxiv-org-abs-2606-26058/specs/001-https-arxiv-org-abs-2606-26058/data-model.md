# Data Model: DomainShuttle Reproduction & Validation

## Overview

This document defines the data structures used for the inference pipeline, artifact validation, and reporting. The data model is designed to be lightweight, avoiding complex database schemas in favor of in-memory objects and JSON/YAML configuration files, suitable for a stateless CI runner.

## Entities

### 1. InferenceConfig
Configuration for the video generation run.
- **Purpose**: Stores parameters for the `predict_s2v.py` script.
- **Format**: YAML or JSON.

```yaml
model_name: "wan2.2_fun"
device: "cpu"
precision: "float32" # or "float16" if supported
frame_count: 16
resolution:
  width: 512
  height: 512
num_inference_steps: 20
guidance_scale: 7.5
seed: 42
```

### 2. InputData
The input artifacts for the generation.
- **Purpose**: Holds the reference image and prompt.
- **Format**: File paths (strings) and string content.

```json
{
  "reference_image_path": "assets/sample_ref_image.jpg",
  "text_prompt": "A cat walking on a tightrope in the rain",
  "config_path": "configs/default_cpu.yaml"
}
```

### 3. OutputArtifact
The generated video file and its metadata.
- **Purpose**: Represents the result of the inference.
- **Format**: File system object + metadata dictionary.
- **Note**: This entity now includes quantitative scores from the CLIP proxy to replace subjective ratings. It also includes fields to track the cause of failure (hardware vs. model).

```json
{
  "file_path": "output/video_42.mp4",
  "format": "mp4",
  "duration_seconds": 2.5,
  "resolution": {
    "width": 512,
    "height": 512
  },
  "file_size_bytes": 1500000,
  "is_playable": true,
  "fidelity_score": 0.78,
  "motion_score": 0.45,
  "subject_present": true,
  "constraint_attribution": "HARDWARE_LIMIT" // or "MODEL_INCOMPATIBLE" or "NONE"
}
```

### 4. ReproductionReport
The final summary of the run.
- **Purpose**: Documents the success/failure and metrics.
- **Format**: Markdown + JSON summary.

```json
{
  "run_id": "2024-05-21-120000",
  "status": "SUCCESS", // or "FAILURE", "TIMEOUT", "OOM"
  "execution_time_seconds": 3600,
  "peak_memory_mb": 6500,
  "artifact_valid": true,
  "fidelity_score_avg": 0.76,
  "motion_score_avg": 0.44,
  "qualitative_assessment": "Subject appears in video, motion is smooth.",
  "errors": [],
  "constraint_attribution": "HARDWARE_LIMIT" // If 16-frame failed but 8-frame succeeded
}
```

## Data Flow

1.  **Config Loading**: `InferenceConfig` is loaded from `configs/default_cpu.yaml`.
2.  **Input Loading**: `InputData` paths are resolved. If `reference_image_path` is missing, the process fails with `MissingInputError`.
3.  **Inference**: The model generates `OutputArtifact`.
    - *Memory Check*: If peak memory > 7GB, process exits with `MemoryError`.
    - *Time Check*: If execution > 6h, process exits with `TimeoutError`.
4.  **Validation**: `OutputArtifact` is validated against `InferenceConfig` (resolution, duration) and **CLIP Proxy** (fidelity/motion scores).
    - *Sensitivity Check*: If `fidelity_score` < 0.5, trigger 8-frame run.
    - *Attribution*: If 8-frame succeeds, set `constraint_attribution` to "HARDWARE_LIMIT".
5.  **Reporting**: `ReproductionReport` is generated by consuming the `fidelity_score`, `motion_score`, and `constraint_attribution` fields from `OutputArtifact`.

## Constraints

- **Memory**: All data structures must fit within 7GB RAM. Large tensors (video frames) are processed in batches or discarded after writing to disk.
- **Disk**: Output artifacts must fit within the 14GB disk limit. A single 512x512, 16-frame MP4 is expected to be < 50MB.
- **Format**: All configuration and report data must be serializable to JSON/YAML for CI compatibility.
