\section{Experimental Setup}
\label{app:setup}

This section expands on \autoref{sec:setup}: training data and hyperparameters (\appref{app:training-setup}), the agentic system prompt and tool schemas (\appref{app:tools}), and per-benchmark descriptions (\appref{app:benchmarks}).

\subsection{Training}
\label{app:training-setup}

We expand on \autoref{sec:setup} with the full training data composition and hyperparameter list.

\paragraph{SFT data composition.}
SFT uses 64{,}274 trajectories drawn from three sources: ViRL~\citep{virl}, fvqa~\citep{fvqa}, and PyVision-RL~\citep{pyvision-rl}. All trajectories are generated by Qwen3-VL-32B-Thinking~\citep{Qwen3VL} acting as the teacher model. $\sim$25\% of the trajectories contain at least one tool call; the remainder solve the problem with internal reasoning alone. We retain only trajectories whose final answer is correct against the ground-truth label.

\paragraph{RL data composition.}
RL uses 15{,}591 problems obtained by filtering the SFT problem set: we remove problems that the SFT checkpoint solves on all 4 rollouts, and problems that the 32B teacher fails on all 4 rollouts. This filtering convention follows prior agentic RL work~\citep{deepeyesv2, prorl} and removes both trivial problems (zero learning signal) and unreachable problems (no positive signal achievable at this scale). We additionally include 22{,}000 hard questions from MMFineReason-hard\footnote{A question subset of the MMFineReason corpus that Qwen3-VL-4B entirely fails on 4 rollouts.} to expose the policy to problems unseen during SFT, bringing the RL training set to $\sim$37K problems.

\paragraph{Policy gradient objective.}
\label{app:clip-objective}
Both SFT\,+\,GRPO and SFT\,+\,AXPO \emph{maximize} the standard PPO-clip surrogate~\citep{PPO} regularized by a KL penalty toward a fixed reference policy $\pi_{\text{ref}}$ (the SFT-initialized checkpoint). Letting $\rho_t(\theta) = \pi_\theta(y_t \mid \vx, \vy_{<t}) / \pi_{\theta_{\text{old}}}(y_t \mid \vx, \vy_{<t})$ denote the per-token importance ratio against the rollout policy $\pi_{\theta_{\text{old}}}$ and $\hat{A}$ the trajectory-level advantage broadcast uniformly to every token of $\vy$,
\begin{gather}
\label{eq:clip-app}
    \mathcal{L}_{\text{clip}}(\vy;\, \hat{A}) \;=\; \mathbb{E}_{t}\!\big[\, \min\!\bigl(\rho_t(\theta)\, \hat{A},\; \mathrm{clip}(\rho_t(\theta),\, 1-\epsilon_{\text{low}},\, 1+\epsilon_{\text{high}})\, \hat{A}\bigr) \\
    \;-\; \beta\, D_{\mathrm{KL}}\!\left[\pi_\theta(\cdot \mid \vx, \vy_{<t}) \,\Vert\, \pi_{\text{ref}}(\cdot \mid \vx, \vy_{<t})\right] \,\big]. \nonumber
\end{gather}
The training objective is $\max_\theta\; \mathbb{E}_{\vy \sim \pi_{\theta_{\text{old}}}}\!\left[\mathcal{L}_{\text{clip}}(\vy;\, \hat{A})\right]$. GRPO instantiates $\hat{A} = A_i$ for trajectory $\tau_i$ from \autoref{sec:prelim-rl}; AXPO applies the same surrogate per stream in \autoref{eq:axpo-loss} (per-prefix advantage on continuations, recovery-indicator advantage on the source prefix). We use the asymmetric DAPO clipping range $\epsilon_{\text{low}} = 0.2,\, \epsilon_{\text{high}} = 0.4$ and KL coefficient $\beta = 1\times 10^{-3}$.

\paragraph{Hyperparameters.}
We adopt the ProRL~\citep{prorl} recipe for the RL stage. The key hyperparameters are:
\begin{itemize}[leftmargin=*, itemsep=2pt]
    \item Sampling temperature: $1.0$
    \item KL coefficient: $1\times 10^{-3}$
    \item Asymmetric clipping: $\epsilon_{\text{low}} = 0.2$, $\epsilon_{\text{high}} = 0.4$ (DAPO-style~\citep{DAPO})
    \item Group size: $N = 8$ rollouts per question
    \item Maximum number of turns per trajectory: $3$
    \item Hardware: $8 \times$ A100 80GB GPUs
\end{itemize}
All hyperparameters are held constant between SFT\,+\,GRPO and SFT\,+\,AXPO; only the rollout loop differs (see below).

For implementation of the training pipeline, we used verl~\citep{verl} and rllm~\citep{rllm} libraries.

\paragraph{AXPO-specific hyperparameters.}
AXPO adds two knobs on top of the GRPO recipe:
\begin{itemize}[leftmargin=*, itemsep=2pt]
    \item Branching budget: $r = 0.25$ of total rollout (i.e., extra continuations per training step are capped at $0.25 \cdot BN$).
    \item Per-prefix resample count: $K = 4$ continuations per selected prefix.
\end{itemize}

\subsection{System Prompt and Tool Interface}
\label{app:tools}

We use the same agentic system prompt for SFT, RL, and evaluation across all model sizes where the prompt is modified version of the system prompt of deepeyes v2~\citep{deepeyesv2}.
Tools are exposed via a unified \texttt{<tool\_call>} interface used in qwen-agent~\citep{Qwen3VL} and the model is free to invoke any subset of the three tools (\texttt{python}, \texttt{tavily-search}, \texttt{image\_zoom\_in}) on every question. 
The full text of the prompt is reproduced below.

\begin{promptbox}
You are an agent: please keep going until the user's query is completely resolved, before ending your turn and yielding back to the user. Only terminate your turn when you are sure that the problem is solved.

Solve the following problem step by step. In your reasoning process, if the answer cannot be determined, you can write Python code in a Jupyter Notebook to process the image and extract more information from it. The stdout and stderr content, along with the images generated by ``\texttt{plt.show()}'' will be returned to better assist with the user query.

You MUST use the python tool to analyze or transform images whenever it could improve your understanding. This includes but is not limited to zooming in, rotating, adjusting contrast, computing statistics, or isolating features. If you find you lack sufficient knowledge to confidently answer the question, you MUST conduct search to thoroughly seek the internet for information. No matter how complex the query, you will not give up until you find the corresponding information. You can use the tavily-search tool, which will search the web and return relevant information based on your query.

\textbf{Notes for tools:}

\textbf{\#\# image\_zoom\_in tool}
\begin{enumerate}[leftmargin=*, itemsep=0pt, topsep=2pt]
    \item \textbf{image\_zoom\_in} crops and magnifies a specific region of the image so you can inspect fine details.
    \item When you need to look at a specific region more closely, prefer \textbf{image\_zoom\_in} over writing python code to crop --- it is simpler, faster, and returns a high-resolution view directly.
    \item After a zoom-in, the cropped image is appended to \texttt{\_pil\_images} in the python tool (the observation will tell you the exact index), so you can further process it with code if needed.
\end{enumerate}

\textbf{\#\# python tool}
\begin{enumerate}[leftmargin=*, itemsep=0pt, topsep=2pt]
    \item \textbf{python} runs in a stateful Jupyter notebook kernel --- variables, imports, and state persist across calls.
    \item The conversation images are pre-loaded as \texttt{\_pil\_images} (list of PIL Image objects) and \texttt{\_images} (list of file paths). Images produced by other tools (e.g.\ \texttt{image\_zoom\_in}) are also appended to these lists.
    \item Use \texttt{plt.show()} to visualize processed images; generated plots are automatically captured and returned.
    \item Available packages include numpy, scipy, sympy, matplotlib, PIL/Pillow, and the standard library.
    \item \textbf{Always use Python for non-trivial computation} --- trigonometric calculations, multi-step arithmetic, equation solving, and area/volume formulas. Do NOT compute these mentally; write code to ensure numerical accuracy.
    \item Your program should always return in finite time. Do not write infinite loops.
    \item Writing file to disk is not allowed.
\end{enumerate}

\textbf{\#\# tavily-search tool}
\begin{enumerate}[leftmargin=*, itemsep=0pt, topsep=2pt]
    \item You MUST engage in many interactions, delving deeply into the topic to explore all possible aspects until a satisfactory answer is found.
    \item Before presenting a Final Answer, you will \textbf{cross-check} and \textbf{validate the information} you've gathered to confirm its accuracy and reliability.
    \item You will carefully analyze each information source to ensure that all data is current, relevant, and from credible origins.
\end{enumerate}

\textbf{\#\# general}

You MUST think and plan before each function call, and reflect on the outcomes of the previous function calls. DO NOT do this entire process by making function calls only, as this can impair your ability to solve the problem and think insightfully.

Remember to place the final answer in the last part using the format:
\begin{center}
\texttt{<answer>}\\
\texttt{\textbackslash boxed\{"The final answer goes here."\}}\\
\texttt{</answer>}
\end{center}
\end{promptbox}

\paragraph{Search domain blacklist.}
To prevent answer leakage during training and evaluation, \texttt{tavily-search} is configured with the API-level parameter \texttt{exclude\_domains = ["huggingface.co"]}, which filters out the site most likely to index benchmark questions or solutions. The exclusion is enforced uniformly across training and evaluation rollouts.

\subsection{Benchmark Details}
\label{app:benchmarks}

We evaluate on nine multimodal benchmarks grouped by which tool family is most expected to help: \textbf{Reasoning} (Python interpreter), \textbf{Perception} (image zoom-in), and \textbf{Search} (web search). All benchmarks are evaluated in the agentic setting with the same tool interface used in training; the model is free to use any subset of the three tools or none at all on every question.

\paragraph{Reasoning.} These benchmarks evaluate multi-step reasoning over text, math, and figures, where a Python interpreter is the natural tool for symbolic and numerical computation.
\begin{itemize}[leftmargin=*, itemsep=2pt]
    \item \textbf{MathVision}~\citep{mathvision}: competition-style math problems paired with diagrams (geometry, algebra, combinatorics). Answers are short numerical or symbolic expressions; we use the public mini test set containing 304 questions.
    \item \textbf{DynaMath}~\citep{dynamath}: a benchmark of programmatically generated math questions with multiple parametric variants per seed problem. We evaluate on a single-variant slice (501 seed problems). The benchmark  complements MathVision as it provides additional algorithmically generated problems spanning plane geometry, statistics, and other domains with diverse difficulty.
    \item \textbf{Math-VR}~\citep{mathvr}: a mathematical visual reasoning benchmark introduced with CodePlot-CoT, in which problems are paired with diagrams (figures, plots, geometric constructions) that the model must inspect to compute the answer. The benchmark targets the regime where the visual content is load-bearing for the math derivation rather than decorative.
\end{itemize}

\paragraph{Perception.} These benchmarks contain high-resolution images with information that is hard to read at the default vision encoder resolution. The image zoom-in tool is the natural recovery mechanism.
\begin{itemize}[leftmargin=*, itemsep=2pt]
    \item \textbf{V$^{\star}$}~\citep{vstar}: a fine-grained visual-search benchmark in which the answer depends on a small object embedded in a complex scene. Designed to expose the failure of monolithic image encoders on high-resolution detail.
    \item \textbf{VisualProbe}~\citep{mini-o3}: a multimodal probing benchmark introduced with mini-o3, where each question targets a specific small region (text, sign, label) within a high-resolution scene. We use the \emph{hard} subset, which selects questions Qwen3-VL-Thinking-8B fails without tools.
    \item \textbf{HR-Bench-4K and HR-Bench-8K}~\citep{HRBench}: high-resolution single-image perception at $4096^2$ and $8192^2$ resolutions, respectively. Questions probe small text, distant objects, and fine textures; the two splits share a common question pool with images rescaled to the target resolution.
\end{itemize}

\paragraph{Search.} These benchmarks require external knowledge that is unlikely to be in the model's parametric memory or that requires multi-hop lookup. The Tavily web search API is the natural tool.
\begin{itemize}[leftmargin=*, itemsep=2pt]
    \item \textbf{HR-MMSearch}~\citep{hrmmsearch}: high-resolution multimodal search questions whose answers are composed by chaining two or more web lookups (e.g., ``in which year was the mayor of the city where this fashion designer was born''). The benchmark stresses both perception (identifying entities from the image) and multi-hop search composition.
    \item \textbf{MMSearch}~\citep{mmsearch}: an earlier multimodal search benchmark covering current-events questions whose answers depend on web content released after the model's training cutoff. Predominantly single-hop, but answers are often time-sensitive.
\end{itemize}

\paragraph{Evaluation protocol.} For all benchmarks, we draw four rollouts per question and report Pass@1 (per-question accuracy averaged over four rollouts) and Pass@4 (a question is correct if any of the four rollouts is correct). Final-answer matching against the gold label is performed by an exact match for math and multiple-choice question and F1 score for search questions. For benchmarks that are hard to evaluate with exact match (e.g., MMSearch), we use a llm-as-a-judge using gpt-5-mini model. 

\paragraph{Evaluation hyperparameters.}
\begin{itemize}[leftmargin=*, itemsep=2pt]
    \item Sampling: temperature $0.7$, top-$p$ $0.8$, top-$k$ $20$, presence penalty $1.5$
    \item Rollouts per question: $N = 4$ (used for Pass@1 / Pass@4)
    \item Maximum agentic turns per trajectory: $5$
    \item Maximum response length: $32{,}768$ tokens
\end{itemize}

\paragraph{Why is post-training needed at all?}
Comparing Non-agent and Base at the same model size measures the effect of \emph{turning tools on without any training}. The gain is strongly scale-dependent: Pass@1 / Pass@4 deltas are \dAgentTwoB at 2B, \dAgentFourB at 4B, \dAgentEightB at 8B, and \dAgentThirtyTwoB at 32B. At 2B the Base (agentic) baseline is actually \emph{worse} than the Non-agent baseline on Pass@1, showing that small models pay an agent-prompt overhead that outweighs the gain from correct tool use. Larger models recover Pass@1 and accumulate a Pass@4 ceiling gain, but even at 32B the gap to SFT\,+\,AXPO at 8B on Pass@4 (\eightBvsAgentPassFour) shows that inference-time tool access alone does not close the agentic gap. Post-training is therefore necessary at every trainable scale.

\paragraph{Why SFT before RL?}
Comparing GRPO (RL from base) against SFT\,+\,GRPO measures the value of an SFT initialization before RL. Pass@1 gaps shrink with size (\dSFTinitPassOne), but Pass@4 gaps remain positive at every scale (\dSFTinitPassFour). RL from the base policy can find a single high-reward mode (Pass@1 partially recovered at 8B), but it consistently sacrifices diversity, capping the Pass@4 ceiling~\citep{rlvr}.
SFT broadens the cold-start distribution over tool-using trajectories before group-based exploration begins. Without it, all-wrong tool-using subgroups dominate from step~1 and the policy collapses onto a narrow single-mode solution~\citep{deepeyesv2}.

\paragraph{Why RL after SFT?}
SFT minimizes log-likelihood on a fixed set of teacher trajectories: it can make the student \emph{imitate} the teacher's tool-use patterns, but the optimization signal lives entirely on the teacher's distribution. At test time the student samples from its own distribution and small token-level deviations from the teacher accumulate into prefixes the SFT objective never scored, so imitation alone does not guarantee that on-policy rollouts inherit the teacher's correctness (\emph{exposure bias}).
RL after SFT narrows this gap by training on the policy's own rollouts under an outcome reward, adapting the policy to its own induced trajectory distribution rather than to the teacher's~\citep{rlgeneralize}.
Empirically, comparing SFT against SFT\,+\,GRPO isolates this on-policy adaptation effect given the SFT initialization: Pass@1 improves at every scale (\dRLAfterSFTPassOne), the regime where on-policy reliability matters most, while Pass@4 gains are smaller and uneven (\dRLAfterSFTPassFour), consistent with GRPO consolidating mass on trajectories the SFT policy can already reach rather than discovering new ones. RL is therefore necessary on top of SFT, but standard GRPO's adaptation is biased toward the SFT manifold, motivating AXPO's \toolresample.

\paragraph{Why AXPO on top of SFT, instead of GRPO?}
The previous two paragraphs leave a structural gap: SFT sets the Pass@4 ceiling and RL sharpens within it, so further gains require a mechanism that simultaneously extends the ceiling \emph{and} sharpens. AXPO does both. Pass@1 gains over SFT\,+\,GRPO are \dPassOneTwoB / \dPassOneFourB / \dPassOneEightB $\ $at 2B/4B/8B, and Pass@4 gains are \dPassFourTwoB / \dPassFourFourB / \dPassFourEightB. The Pass@4 lift is the diagnostic signature that AXPO operates on a different axis from SFT-then-GRPO sharpening: by recovering all-wrong tool-using subgroups whose gradient signal would otherwise vanish (\autoref{sec:gap}), AXPO turns previously dead continuations into trainable ones, expanding the policy's reachable correct-trajectory set rather than only redistributing mass within it.

\subsection{Generalization to an Unseen Tool: Image Search}
\label{app:image-search-gen}
\input{tables/tab_image_search}

Training in \autoref{sec:setup} uses three tool families: a Python interpreter, a Tavily web search engine, and an image zoom-in tool. None of these is an \emph{image-based} retrieval tool that takes an image as the query and returns visually similar pages, which we cannot directly utilize due to excessive API costs.
To probe whether AXPO's tool-using policy transfers beyond the training tool inventory, we extend the inference-time tool set with an image-search tool and re-evaluate on \textbf{MMSearch}~\citep{mmsearch}, the search benchmark with the largest fraction of image-grounded queries. Since we did not have access to a production image-retrieval endpoint at the budget required for full evaluation, we approximate the tool by prompting GPT-5.4 with the query image and instructing it to return the kind of structured response a visual-search API would emit (caption, candidate entity names, related URLs). This proxy is imperfect, but it preserves the key property the experiment probes: an entirely new tool interface, unseen at training time, whose output the model must integrate into its reasoning.

\autoref{tab:image-search-gen} shows the result. \emph{Without} image search at inference, SFT\,+\,GRPO and SFT\,+\,AXPO are essentially tied (Pass@1: $44.0$ vs.\ $45.0$; Pass@4: $59.3$ vs.\ $61.0$). \emph{With} image search at inference, the gap opens on Pass@1: AXPO gains $+4.6$\,pp ($45.0 \to 49.6$) while GRPO gains only $+2.8$\,pp ($44.0 \to 46.8$); on Pass@4 both methods gain similarly ($+6.0$ and $+5.7$). The Pass@1 asymmetry suggests AXPO not only \emph{can} use the unseen tool when given enough samples (Pass@4 captures this) but converts that capability into reliable first-shot answers, whereas GRPO's tool-using policy is more narrowly fitted to the training tool inventory. We read this as evidence that resampling-induced exposure to diverse tool-call resolutions during training transfers to tool families the model has never been trained on, although the claim is bounded to one benchmark and one new tool type.

\subsection{Detailed Ablation Analysis}
\label{app:ablation-details}

\autoref{tab:ablation} ablates each AXPO design choice on 8B at matched compute. All numbers below are 8B Pass@1 averages on the four-benchmark composition (MathVision, V$^{\star}$, VisualProbe, HR-MMSearch).

\paragraph{Prefix fixing.}
Removing prefix fixing reduces AXPO to $52.1$, near the SFT\,+\,GRPO baseline of $51.9$: each resample becomes a fresh rollout from $\pi_\theta(\cdot \mid \vx)$, so only a $q$-fraction of resamples are tool-using and the $(1{-}q)$ waste factor that \autoref{prop:branching} removes returns in full.

\paragraph{Tool-using resampling target.}
Targeting questions where all no-tool rollouts fail for resampling instead reaches $51.7$, since the policy already covers the no-tool subgroup under GRPO and additional samples there cannot recover the missing tool-using signal. Resampling pays off only when the resampling target is the under-covered tool-using subgroup.

\paragraph{Uncertainty-based prefix selection.}
Replacing uncertainty rank with random ranking drops to $51.0$ ($-2.9$\,pp): the prefix is still fixed so the coverage gain is preserved, but random ranking routes the budget to prefixes where the policy has already converged toward a wrong continuation, wasting resamples on determined commitments.

\paragraph{Prefix credit.}
Dropping the recovery-indicator credit on the source prefix yields $51.4$ ($-2.5$\,pp): prefix tokens receive no positive signal even when resampling recovers a continuation, so the policy is not steered toward producing high-yield prefixes in the first place.

\paragraph{Per-prefix advantage group.}
Merging resampled continuations into the source group instead of forming a per-prefix advantage yields $51.2$ ($-2.7$\,pp): the shared prefix then collects contradictory advantages from the failed source and successful resamples while no-tool successes in the source group dilute the recovered signal.

\subsection{Detailed Comparison Analysis}
\label{app:comparison-details}

\autoref{tab:comparison} compares AXPO to alternative training recipes. All numbers are 8B Pass@1 averages.

\paragraph{Reward shaping does not recover the gain of AXPO.}
A tool penalty~\citep{otc} drops to $46.0$ and a tool bonus~\citep{pyvision-rl, pixelreasoner, deepeyes} reaches only $50.8$, both \emph{below} the GRPO baseline of $51.9$. Reward-channel interventions either suppress tool use globally (penalty) or reward tool calls regardless of whether they help (bonus); neither addresses the coverage problem that AXPO targets, which is conditional on the prefix being tool-using and on resampling restoring at least one correct continuation.

\paragraph{Extra rollout budget does not close the gap.}
Doubling the GRPO rollout count ($\times 4$ exceeding AXPO's extra compute) reaches $52.4$, still $1.5$\,pp below SFT\,+\,AXPO. The extra budget re-derives non-tool-using prefixes and cannot break the all-wrong tool-using subgroups that resampling is designed to recover; throwing more samples at the same $(1{-}q)$ waste factor does not change the structural problem.

\paragraph{Alternative RL algorithms.}
We replace GRPO with three recently proposed alternatives. RLTF~\citep{rltf}, an algorithm that utilizes the text feedback from the external teacher, reaches only $43.3$ Avg with $40\%$ tool utilization, suffering tool-call collapse over training (consistent with reports of hint leakage degrading text-feedback RL on agentic tasks). CISPO~\citep{cispo}, an importance-sampling variant, reaches $47.8$ Avg: its clip rule reduces gradient bias but does not address the all-wrong subgroup whose advantage is zero by construction. ARPO~\citep{arpo}, an agent-specific branching method that draws additional continuations after a tool observation when post-observation entropy is high, is the closest comparison and the strongest competitor at $51.3$ Avg, yet still trails AXPO by $2.6$\,pp: post-observation branching cannot recover when the tool call itself is the failure mode, which is the regime \autoref{sec:gap} identifies as binding. The RL-algorithm sweep confirms AXPO's gain is not a generic effect of any agent-aware RL trick but specifically reflects \toolresample's coverage restoration on tool-using thinking prefixes.

\subsection{Confidence as a Tractable Proxy for Tool-Call Entropy}
\label{app:conf-vs-entropy}

\autoref{sec:method} ranks candidate prefixes by the mean probability the source policy assigned to the tool-call tokens of $\tau^{\text{src}}$ (henceforth \emph{confidence}). The natural alternative is the predictive entropy $H(\pi_\theta(\cdot \mid \vx, \vy_{<t}))$ of the next-token distribution at each tool-call position, but this choice is precluded by training-loop structure. Per-token sampling probabilities are emitted essentially for free by the rollout engine (vLLM~\citep{vllm}), whereas predictive entropy requires either an extra forward pass over the full vocabulary under the trained policy (FSDP) or vLLM-side full-vocabulary logprob expansion at decode time. Our RL pipeline under verl~\citep{verl} alternates rollout (vLLM) and gradient updates (FSDP), so inserting a per-prefix entropy computation would force an additional vLLM\,$\to$\,FSDP\,$\to$\,vLLM round-trip per training step (or equivalently expensive vLLM-side full-vocabulary logprob expansion), a cost we wished to avoid.

To verify that confidence preserves the ranking an entropy-based scheme would induce, we measured the per-step correlation between mean-token-confidence and exact policy entropy on tool-call positions of failed tool-using rollouts collected during training (\autoref{fig:conf-vs-entropy}). Across training steps, the two signals are strongly correlated (Pearson $\rho = 0.843$, Spearman $\rho = 0.835$, OLS fit $y = 0.41 x + 0.07$ on entropy vs negated confidence). Confidence therefore preserves the relative ordering an entropy-based scheme would produce, and we adopt it as the ranking signal in AXPO.

\section{Extended Related Works}
\label{app:related-works}

This appendix expands on \autoref{sec:related} with the longer descriptions that the main text abridges.

\paragraph{Two-stage SFT-then-RL recipe for agentic VLMs.}
Recent agentic VLM systems share a two-stage recipe: supervised fine-tuning on tool-using trajectories distilled from larger teachers~\citep{FireAct, AgentDistill}, followed by outcome-reward RL~\citep{pyvision-rl, deepeyesv2, deepeyes}. They differ mainly in the distillation corpus, the tool interface, and the choice of base model. Most prior agentic-RL VLM studies focus on a single tool family (typically image manipulation), whereas our setup combines code, search, and image zoom-in on a thinking-capable base (Qwen3-VL-Thinking)~\citep{Qwen3VL}, a configuration under which the RL stage becomes the primary bottleneck.

\paragraph{Group-relative RL and BroRL.}
RL with verifiable outcome rewards is the standard post-SFT stage for reasoning models~\citep{r1, o1}. GRPO~\citep{GRPO, DAPO, prorl} samples $N$ rollouts per question, normalizes rewards within the group, and assigns the resulting advantage uniformly to every token. BroRL~\citep{brorl} argues that increasing $N$ reduces unsampled blind spots at a rate determined by per-token baseline probability. Both analyses were developed for thinking-only trajectories, where the rollout structure is uniform across tokens and the per-token baseline probability decays slowly. \autoref{prop:branching} shows that AXPO's prefix-fixed resampling dominates raw scaling in the regime where tool-call tokens sit at low policy probability, exactly the failure mode \autoref{sec:gap} diagnoses.

\paragraph{Mid-trajectory branching for tool-using RL.}
ARPO~\citep{arpo} and AEPO~\citep{aepo} adapt group-relative RL to tool-using rollouts by branching \emph{after} a tool observation, drawing additional continuations from $\pi_\theta(\cdot \mid \vx, \vt_1, \va_1, \vo_1)$ when post-observation entropy is high. The exploration target is the continuation given a fixed tool call, which absorbs variance introduced by the tool output but cannot recover when the tool call itself is wrong (wrong sub-image, wrong python code, wrong search query). AXPO instead branches \emph{at} the tool-call boundary and triggers only on all-wrong tool-using subgroups, with a per-prefix advantage that credits the source prefix exactly when its branched continuation succeeds, making it a targeted intervention on the regime \autoref{sec:gap} identifies as binding rather than a general rollout-budget reshaper. Empirically, AXPO outperforms ARPO on the 8B comparison ($53.9$ vs.\ $51.3$ Avg, \autoref{tab:comparison}), consistent with the diagnosis that the tool call itself, not the post-observation continuation, is the dominant failure locus. The two strategies are compatible, and combining tool-call and post-observation branching is a natural extension we leave to future work.

\section{Coverage Argument for \Toolresample}
\label{app:coverage}

This appendix formalizes \autoref{prop:branching}: prefix-fixed resampling from a tool-committed prefix dominates raw sampling at recovering correct tool-using rollouts whenever the selected prefix satisfies $p(\vt_1) \geq q\, p^{\text{tool}}$, with strict inequality once $p(\vt_1) > q\, p^{\text{tool}}$. The mechanism is $q \to 1$ by construction: every resampled continuation from a tool-committed prefix is tool-using, so raw's $(1-q)$ waste factor vanishes and the threshold a prefix must clear sits at $q\, p^{\text{tool}}$, strictly below the average per-tool-using-rollout success $p^{\text{tool}}$.

\paragraph{Setup.}
Let $\pi_\theta$ be the policy, $\vx$ a question, and $R(\tau) \in \{0,1\}$ the outcome reward. Decompose $\tau = (\vt_1, \vy_{\geq 1})$ into a thinking prefix and its continuation (tool call, tool output, and the remainder of the trajectory). Let $\mathcal{T} = \{\vt_1 : \vt_1 \text{ includes the opening } \toolcall \text{ tag}\}$ be the set of tool-committed prefixes, and define
\[
    q := \Pr_{\pi_\theta}[\vt_1 \in \mathcal{T} \mid \vx], \qquad
    p(\vt_1) := \Pr_{\pi_\theta}[R=1 \mid \vx, \vt_1], \qquad
    p^{\text{tool}} := \mathbb{E}_{\vt_1 \sim \pi_\theta(\cdot \mid \vx,\, \vt_1 \in \mathcal{T})}[p(\vt_1)].
\]
So $q$ is the rate at which the policy commits to tool use from $\vx$, and $p^{\text{tool}}$ is the per-tool-using-rollout success rate. Both are directly measurable (\autoref{fig:analysis-a}, \autoref{fig:analysis-b}). We compare two samplers, each drawing $N$ trajectories, on the event \emph{``at least one sample is a correct tool-using rollout''}:
\begin{itemize}[leftmargin=*, topsep=2pt, itemsep=1pt]
    \item \textbf{Raw rollout}: $\tau \sim \pi_\theta(\cdot \mid \vx)$. Per-sample probability of a correct tool-using rollout is $\Pr[\vt_1 \in \mathcal{T}] \cdot \Pr[R=1 \mid \vt_1 \in \mathcal{T}] = q\, p^{\text{tool}}$.
    \item \textbf{Prefix-fixed resampling}: fix $\vt_1^{\text{src}} \in \mathcal{T}$ and draw continuations from $\pi_\theta(\cdot \mid \vx, \vt_1^{\text{src}})$. Every resampled continuation is tool-using by construction ($q \to 1$), and per-resample success is $p(\vt_1^{\text{src}})$.
\end{itemize}
Coverage, i.e., the probability that at least one of the $N$ samples is a correct tool-using rollout, equals $1 - (1-p_{\text{eff}})^N$ with $p_{\text{eff}} = q\, p^{\text{tool}}$ under raw rollout and $p_{\text{eff}} = p(\vt_1^{\text{src}})$ under resampling.

\paragraph{Proof of \autoref{prop:branching}.}
The function $f(p) = 1 - (1-p)^N$ has derivative $f'(p) = N(1-p)^{N-1}$, non-negative on $[0,1]$ and strictly positive on $[0,1)$, so $f$ is non-decreasing on $[0,1]$ and strictly increasing on $[0,1)$. Under the assumption $p(\vt_1^{\text{src}}) \geq q\, p^{\text{tool}}$, applying $f$ to both sides gives $1 - (1-p(\vt_1^{\text{src}}))^N \geq 1 - (1 - q\, p^{\text{tool}})^N$. Strictness follows whenever $p(\vt_1^{\text{src}}) > q\, p^{\text{tool}}$ and $q\, p^{\text{tool}} \in (0,1)$. The mechanism is $q \to 1$ by construction: resampling eliminates raw's wasted $(1-q)$ fraction of non-tool rollouts, so the baseline threshold is $q\, p^{\text{tool}}$ rather than $p^{\text{tool}}$.

\paragraph{Consequence for GRPO training.}
Under GRPO with binary rewards, coverage on correct tool-using rollouts determines whether tool-call tokens receive a \emph{positive} (reinforcing) gradient. If the tool-using subgroup contains no correct rollout, the tool-call tokens get no positive signal: the subgroup is either empty (no tool-call tokens at all in the group) or all-wrong with non-positive group-normalized advantage --- zero in fully all-wrong groups ($r_i - \mathrm{mean}(\{r_j\}) = 0$) and \emph{negative} in mixed-success groups where some no-tool rollouts succeed (which actively penalizes tool use). Hence $1 - (1-p_{\text{eff}})^N$ is exactly the probability that a group produces a positive (reinforcing) learning signal at the tool call. \autoref{prop:branching} then says that prefix-fixed resampling strictly breaks the all-wrong tool-using subgroups identified in \autoref{fig:analysis-b} at any fixed $N$.

\paragraph{Satisfying the threshold.}
By the definition of $p^{\text{tool}}$ as the conditional expectation over the tool-committed prefix distribution $\pi_\theta(\cdot \mid \vx,\, \vt_1 \in \mathcal{T})$, $\mathbb{E}_{\vt_1}[p(\vt_1)] = p^{\text{tool}} > q\, p^{\text{tool}}$ whenever $q < 1$, so the expected prefix lies strictly above the \autoref{prop:branching} threshold. All prefixes at or above the conditional mean satisfy the assumption, and so do below-average prefixes down to $q\, p^{\text{tool}}$; only prefixes with strongly subpar per-branch success ($p(\vt_1) < q\, p^{\text{tool}}$) can underperform raw sampling. Empirically, $p(\vt_1)$ is far from constant across $\mathcal{T}$: fixing a failed tool-using rollout's prefix and resampling its tool call recovers $\sim$15\% of originally-failed tool-using subgroups.

\paragraph{Relation to BroRL.}
Our target is tool use, and the diagnostics in \autoref{sec:gap} establish exactly the regime where naive rollout scaling fails to help: tool use is rare (\autoref{fig:analysis-a}: $q \in [0.2, 0.35]$) and all-wrong rates on tool-using subgroups are high (\autoref{fig:analysis-b}), so tool-call tokens sit in a low-probability regime under the policy. \citet{brorl} show that scaling $N$ reduces unsampled blind spots at rate $p^2(1-p)^N$ (Theorem 1, Lemma 2); this decay is per-token, fast for high-$p$ regions and slow for low-$p$ regions. Raw scaling therefore preferentially fills the thinking-only blind spot while leaving the tool-call blind spot under-sampled. \autoref{prop:branching} sidesteps this by relocating samples: conditioning on a tool-committed prefix eliminates the $(1-q)$ waste factor so the local per-branch success becomes $p(\vt_1^{\text{src}})$, and the same budget $N$ reaches the tool region effectively. BroRL's guarantee is uniform in token type; AXPO targets the specific region that the low tool-use attempt rate has made hardest to cover.

\section{Limitations and Future Work}
\label{app:limitations}

We expand on the limitations sketched in the main-text conclusion and discuss future directions.

\paragraph{Verifiable outcome rewards.}
AXPO's subgroup-level trigger (``all-wrong tool-using subgroup'') and per-prefix advantage in Eq.~\ref{eq:axpo-adv} both rely on a binary, automatically verifiable outcome signal $r \in \{0,1\}$. Tasks where verifiability is partial (open-ended generation~\citep{NEURIPS2025_e5849736, lee2025maskingteacherreinforcingstudent}, multi-criterion subjective evaluation) require a different definition of ``failed subgroup'' before \toolresample applies. The resampling mechanism itself is reward-agnostic, but the trigger and the all-wrong learning signal argument in \autoref{sec:gap} are not.

\paragraph{Compute scope of training.}
Trained models are at most 8B parameters; 32B is reported inference-only as the SFT teacher. This boundary reflects available compute rather than a methodological choice~\citep{kim2026visualtokenpruningfails, lee2025maskingteacherreinforcingstudent, yu2026hideseereasoningprefixmasking, lee2026genrecalgenerationrecalibrationlarge, Lee_2025_CVPR}.
Whether the Thinking-Acting Gap and AXPO's recovery generalize to 32B+ trainable scale, where the base policy is closer to the tool-use ceiling and the all-wrong-subgroup fraction may be smaller, is open.
Furthermore, although the teacher model can theoretically scale to 100B+ parameters, analyzing the effects of teacher scaling on downstream reinforcement learning (RL) remains an area for future work.

\paragraph{Future work: richer tool environments.}
Our tool inventory covers three families that span numerical, retrieval, and visual modalities (Python interpreter, web search, image zoom-in), each shown to be load-bearing in prior agentic VLM work~\citep{deepeyesv2, pyvision-rl, mini-o3}; this multi-tool setup is already broader than most prior agentic-VLM RL studies, which typically focus on a single tool family. One notable absence is an image-search tool, which prior works~\citep{deepeyesv2,hrmmsearch} report results with but which we could not include during training and full evaluation due to per-call API costs at our scale (we instead approximate it via a GPT-5.4 prompt for the unseen-tool study in \appref{app:image-search-gen}). Tools with longer execution latency (browser agents, GUI control, terminal sessions) and tools with high per-call cost (LLM-as-tool)~\citep{toolorchestra} introduce additional considerations that we do not address.
Extending AXPO to such richer~\citep{lee2025refinebenchevaluatingrefinementcapability, Lee_2025_ICCV, yu2024sparkmultivisionsensorperception}, longer-horizon environments is a natural next step.

\paragraph{Future work: generalization to other base models.}
All experiments use Qwen3-VL-Thinking, which is currently the strongest open thinking-capable VLM with reliable tool-using ability and the model best supported by our RL framework. The Thinking-Acting Gap is a property of RL on an agentic reasoning model rather than of any specific architecture~\citep{NEURIPS2024_473a9a75, 10.1007/978-3-031-72967-6_16, lee-etal-2024-collavo, lee2024phantomlatentlargelanguage}, and we expect the diagnostics and AXPO's coverage gain to transfer; replicating on the most recent Qwen3.5 models~\citep{qwen35}, once RL framework support matures, would help establish the gap as a general phenomenon and rule out base-specific confounds.


\section{Broader Impacts}
\label{app:broader-impacts}

AXPO is a methodological improvement to the RL training of agentic VLMs and is additive on top of a publicly released base model (Qwen3-VL-Thinking~\citep{Qwen3VL}), so it does not introduce a new capability class. Among the three tool families used in this work, the Python interpreter raises standard security concerns for code-executing agents. We can run all tool calls in a sandboxed execution environment during both training and evaluation, consistent with prior agentic-VLM RL work~\citep{pyvision}. Beyond this sandboxing, broader safety considerations~\citep{NEURIPS2021_8e5e15c4,Lee_2022_CVPR, Kim_2023_CVPR, Lee_2023_ICCV, lee2021towards, 10222502, KIM2026112173} (e.g., misuse of tool-grounded outputs, causality, robustness, deployment-time guardrails, downstream effects of cheaper agentic deployments) are out of scope for this work and were not addressed.
