# Implementation Plan: Investigating the Relationship Between Brain Network Dynamics and Musical Genre Preference

**Branch**: `001-brain-music-preference` | **Date**: 2024-05-24 | **Spec**: `specs/001-investigating-the-relationship-between-b/spec.md`
**Input**: Feature specification from `/specs/001-investigating-the-relationship-between-b/spec.md`

## Summary

This project implements a computational pipeline to investigate the relationship between resting-state fMRI brain network dynamics (static and dynamic functional connectivity) and musical genre preference. The system downloads data from OpenNeuro, preprocesses it using fMRIPrep (CPU-optimized), extracts time courses via the Schaefer-400 atlas, computes graph-theoretic metrics, and performs robust statistical correlation analysis with Benjamini-Hochberg correction. The plan strictly adheres to CPU-only constraints (GitHub Actions free tier) and ensures dataset-variable fit by validating the presence of required behavioral proxies (STOMP-R) before execution.

**CRITICAL FEASIBILITY NOTE**: Statistical power analysis indicates that for an effect size of |r| ≥ 0.3 with [deferred] power, a minimum sample size of N=85 is required. The spec's assumption of sampling down to N=50 for runtime feasibility directly contradicts the power requirement. **This plan will HALT with `ERR_UNDERPOWERED` if N < 85.** The spec's assumption regarding N=50 is flagged as a blocking contradiction requiring immediate resolution (see Constitution Check).

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| **Docker for fMRIPrep** | Required for consistent neuroimaging preprocessing (Constitution Principle VI). | Running fMRIPrep directly on host is non-reproducible and prone to environment drift. |
| **Sliding-Window Sensitivity** | Required by FR-011 and SC-005 to ensure dynamic metrics are robust. | Single-window analysis would fail the "Sensitivity Analysis" acceptance criteria and risk spurious results. |
| **Multiple-Comparison Correction** | Required by FR-007 and Constitution Principle VII. | Uncorrected p-values would inflate false positives, violating statistical rigor. |
| **Motion Regression** | Required to prevent dynamic metrics from being proxies for head motion. | Ignoring motion leads to false positives in dynamic connectivity analysis. |

## Phase Ordering (Compute Feasibility)

**CRITICAL ORDERING**: Validation MUST precede Preprocessing to avoid wasting compute on invalid data.

1.  **Data Ingestion & Validation (FR-001, FR-001b)**: Download OpenNeuro data. **CRITICAL**: Check for `musical_genre` or `STOMP-R` in `participants.tsv`. **If missing, HALT with `ERR_DATA_MISSING`**.
2.  **Power Check (FR-009)**: Verify available N ≥ 85. **If N < 85, HALT with `ERR_UNDERPOWERED`**. (Note: Spec assumption of N=50 is invalid for target power).
3.  **Preprocessing (FR-002)**: Run fMRIPrep (Docker) on the **validated** subset. (Note: N=85 may exceed 6h runtime; plan includes logic to split jobs or fail gracefully).
4.  **Time Series Extraction (FR-003)**: Extract 400-ROI time courses using Schaefer atlas.
5.  **Motion Control**: Regress out FD/DVARS from time series *before* dynamic analysis.
6.  **Metric Computation (FR-004, FR-005)**: Calculate static and dynamic metrics.
7.  **Sensitivity Analysis (FR-011)**: Re-run dynamic metrics with window sizes of varying magnitudes.
8.  **Statistical Analysis (FR-006, FR-007, FR-009, FR-010)**: Correlations, BH correction, Power analysis, Null distribution check (**1,000+ permutations**).
9.  **Visualization (FR-008)**: Generate figures.

*Note: If the full dataset exceeds 6 hours, the pipeline will attempt to split the job. If it cannot, the project will be marked as "Runtime Infeasible" and requires a spec amendment to lower power or increase compute resources.*

## Assumptions & Spec Conflicts

- **Assumption Conflict**: The spec assumes "if dataset size exceeds this, the analysis will be restricted to a random sample of 50 subjects." **This plan rejects this assumption.** N=50 is underpowered for the target effect size. The pipeline will **HALT** if N < 85. A spec amendment is required to either lower the power threshold (e.g., to 0.6) or accept a smaller effect size (e.g., |r| ≥ 0.4).
- **Data Availability**: The plan assumes a dataset with STOMP-R or musical preference exists. If not found, the project is untestable with current sources.
- **fMRIPrep Memory**: fMRIPrep may require >7GB RAM. The plan includes logic to downsample resolution if OOM occurs, but this may affect metric validity.
- **Permutation Count**: The plan requires 1,000+ permutations for robust null estimation, contradicting the spec's FR-010 (100 permutations). A spec amendment is required.

## Compute Feasibility (CPU-Only)

- **Environment**: GitHub Actions Free Tier (2 CPU, 7GB RAM).
- **Strategy**:
  - **fMRIPrep**: Run in Docker. If memory errors occur, the pipeline will downsample spatial resolution or process subjects sequentially (not in parallel) to stay within 6GB RAM.
  - **Data Sampling**: **NO SAMPLING TO N=50**. The pipeline requires N≥85. If N < 85, it halts.
  - **Libraries**: `nilearn` and `networkx` are CPU-optimized. No GPU/CUDA code will be used.
  - **Precision**: Default float64. No 8-bit quantization.
  - **Runtime Risk**: N=85 subjects may exceed the 6-hour limit. The pipeline will log a warning and may fail. This requires a spec amendment to increase compute resources or reduce the scope.
