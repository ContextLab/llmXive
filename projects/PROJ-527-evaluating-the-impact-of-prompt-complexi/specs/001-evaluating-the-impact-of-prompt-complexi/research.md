# Research: Evaluating the Impact of Prompt Complexity on LLM Code Generation Performance

## Research Question
How does prompt complexity, defined by token length and structural composition (examples, constraints, instructions), **associate** with the correctness (pass rate), efficiency, and stylistic quality of code generated by Large Language Models?

> **Note on Causality**: This study is **observational**. While the spec uses the term "Impact," the design does not involve random assignment of prompt structures to the LLM's internal state. All findings will be framed as **associational** relationships, not causal claims.

**Dataset Fit Verification**:
- The HumanEval dataset contains `problem_id`, `prompt` (starter code), `canonical_solution`, and `test`.
- **Missing Variables**: The dataset does *not* contain pre-calculated structural element counts or token counts. The system must derive these dynamically (FR-009, FR-011).
- **Missing Variables**: The dataset does *not* contain LLM-generated code. The system must generate this via an LLM query (FR-002).
- **Fit**: The dataset is a perfect fit for the *input* requirements (problems) and *evaluation* requirements (unit tests). The *independent variable* (prompt complexity) and *dependent variable* (generated code) are produced by the system.

## Methodological Approach

### 1. Prompt Generation Strategy
For each of the HumanEval problems, five prompt variants will be generated:
1.  **Simple**: Problem statement only.
2.  **Moderate**: Problem + 1 example.
3.  **Complex**: Problem + constraints.
4.  **Very Complex**: Problem + multi-step instructions.
5.  **Degenerate**: Problem + redundant constraints/examples (targeting >500 tokens).

**Token Counting**: `tiktoken` (cl100k_base) will be used.
**Structural Counting**: A dynamic parser will count examples, constraints, and instruction blocks.

### 2. Code Generation & Execution
- **LLM Source**: A CPU-tractable model (e.g., CodeLlama-7B via HuggingFace Inference API or a smaller quantized GGUF model if local) will be queried.
- **Execution**: Generated code is executed against HumanEval unit tests with a A timeout mechanism with a configurable duration will be implemented. (FR-003).
- **Static Analysis**: `ruff` will be used to extract cyclomatic complexity and lines of code (FR-004).

### 3. Statistical Analysis (Corrected Methodology)

**Primary Test**: **Linear Mixed Models (LMM)** will be used instead of ANOVA/Kruskal-Wallis.
- **Rationale**: The data structure is nested (5 prompt variants generated per problem). Treating the 820 rows as independent observations violates the assumption of independence (pseudoreplication).
- **Model Specification**:
  - **Fixed Effects**: Complexity Label, Prompt Token Count, Residualized Structural Score.
  - **Random Effects**: `problem_id` as a random intercept to account for problem-specific difficulty variance.
  - **Outcome**: Pass rate (binary or proportion) or Readability Score.
- **Collinearity Resolution**: Since token count and structural count are definitionally collinear, the plan will:
  1.  Regress `structural_element_count` on `token_count`.
  2.  Use the **residuals** of this regression as the "Residualized Structural Score" predictor.
  3.  This isolates the effect of "structure" independent of "length".
- **Covariate Control**: **Prompt Token Count** will be used as a covariate when analyzing readability metrics. **Lines of Code (LoC) is excluded as a covariate** for readability metrics to avoid circular validation (since readability metrics include LoC).
  - *Note*: This contradicts FR-012 in the source spec, which mandates LoC as a covariate. The spec must be amended to remove this requirement to ensure scientific validity.
- **Multiple Comparison Correction**: If post-hoc pairwise comparisons are performed (e.g., between complexity levels), Bonferroni or Holm-Bonferroni correction will be applied (FR-006).
- **Sensitivity Analysis**: Complexity boundaries will be shifted by ±10 tokens to test robustness (FR-010).

### 4. Addressing Reviewer Objection (alan-turing-simulated)
The reviewer notes that "token length" may not capture "state transitions."
- **Response**: This study treats token/structural count as an *observable proxy* for the complexity of the internal state. The "degenerate" tier (redundant instructions) is specifically included to test if structural redundancy (which likely increases memory/state load) has a different impact than simple length.
- **Limitation**: The analysis will explicitly state that findings are *associational* and do not prove causality or directly measure internal state transitions.

## Statistical Rigor & Power

- **Sample Size**: 164 problems × 5 variants = 820 observations (nested within 164 problems).
- **Power Analysis**: Based on literature estimates for similar LLM studies (effect size f ≈ moderate), a sample of 164 problems with 5 groups provides >80% power to detect medium effects in an LMM framework. However, the study is likely **underpowered for small effect sizes** (f < 0.15). This limitation will be explicitly stated in the final report.
- **Family-Wise Error**: Correction (FR-006) is mandatory to control Type I error rates across pairwise comparisons.
- **Effect Size**: Cohen's d (for t-tests) or Eta-squared (for ANOVA) or Marginal/R2 (for LMM) will be reported with standard interpretation thresholds (SC-004).
- **Unit of Analysis**: The unit of analysis is the **problem** (n=164), not the generation (n=820). The LMM handles the repeated measures structure.

## Computational Feasibility

- **Hardware**: GitHub Actions free-tier (multi-core CPU, several GB RAM).
- **Strategy**:
  - No GPU training.
  - LLM queries will be made via API (if available) or a small CPU model (e.g., TinyLlama, CodeLlama-7B-GGUF with `llama.cpp` CPU backend).
  - If local inference is too slow, the dataset will be sampled (e.g., a representative subset) to ensure the time limit is met.
  - Data will be processed in chunks to stay within 7 GB RAM.

## Decision Rationale

| Decision | Rationale |
|----------|-----------|
| Use `tiktoken` for token counts | Standard for LLMs; aligns with FR-001. |
| Use `ruff` for static analysis | Faster and more modern than `pylint`; aligns with FR-004. |
| Use **Linear Mixed Models (LMM)** | Mandatory to handle nested data (5 variants/problem) and avoid pseudoreplication. |
| Residualization of Structure | Necessary to disentangle collinear predictors (length vs. structure). |
| Exclude LoC as Covariate | Avoids circular validation (readability metrics include LoC). |
| CPU-only execution | Mandatory for GitHub Actions free-tier compliance. |
| Observational framing | No randomization of LLM internal state; claims must be associational. |
| Spec Amendment Required | FR-005 (ANOVA) and FR-012 (LoC covariate) must be updated to reflect corrected methodology. |
