# Implementation Plan: Evaluating the Impact of Prompt Complexity on LLM Code Generation Performance

**Branch**: `001-prompt-complexity-evaluation` | **Date**: 2026-06-26 | **Spec**: `spec.md`

## Summary

This project evaluates the **associational relationship** between prompt complexity (defined by token length and structural composition) and the correctness/efficiency/stylistic quality of code generated by LLMs. The implementation generates five distinct prompt variants (simple, moderate, complex, very complex, degenerate) for each problem in the HumanEval dataset, queries a CPU-tractable LLM, executes unit tests, and performs **Linear Mixed Model (LMM)** analysis to identify performance trade-offs and inflection points while controlling for problem difficulty and nested data structure.

> **Spec Amendment Required**: This plan corrects methodological flaws identified in the source spec.
> 1. **FR-005 Update**: The spec mandates ANOVA/Kruskal-Wallis, but the corrected methodology requires **Linear Mixed Models (LMM)** to handle the nested structure (5 variants per problem) and avoid pseudoreplication.
> 2. **FR-012 Update**: The spec mandates using "lines of code" as a covariate for readability metrics, which creates circular validation (since readability includes LoC). The plan uses **prompt token count** as the covariate instead.
> These changes are necessary for scientific validity and must be reflected in the spec before implementation.

## Technical Context

**Language/Version**: Python 3.11  
**Primary Dependencies**: `datasets`, `tiktoken`, `scikit-learn`, `statsmodels`, `pandas`, `ruff` (via subprocess), `requests` (for HuggingFace Inference API), `pyyaml`  
**Storage**: Local CSV/Parquet files in `data/` (checksummed); checksums recorded in `state/projects/PROJ-527-...yaml`.  
**Testing**: `pytest` with isolated virtualenvs for code execution.  
**Target Platform**: GitHub Actions free-tier runner (Linux, multiple CPUs, sufficient RAM, no GPU).  
**Project Type**: Research CLI / Data Analysis Pipeline.  
**Performance Goals**: Complete analysis of a subset of HumanEval (or full set if lightweight model) within 6 hours on CPU.  
**Constraints**: No GPU/CUDA; no 8-bit/4-bit quantization; memory usage < 7 GB; disk usage < 14 GB.  
**Scale/Scope**: 164 HumanEval problems × 5 variants = 820 observations (nested within 164 problems).

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

1.  **Principle I (Reproducibility)**: The plan mandates pinned `requirements.txt`, fixed random seeds in `code/`, and fetching HumanEval from the canonical HuggingFace URL provided in the verified datasets block.
2.  **Principle II (Verified Accuracy)**: All citations in `research.md` will be limited to the URLs in the "Verified datasets" block. The plan explicitly references the **Reference-Validator Agent** and the `CITATION_TITLE_OVERLAP_THRESHOLD` mechanism to ensure automated validation of all citations before they contribute to review points.
3.  **Principle III (Data Hygiene)**: Raw data (HumanEval) will be downloaded once and checksummed. Generated code and results will be written to new files in `data/` with no in-place modification. Checksums are recorded in the project state YAML file (`state/projects/...yaml`), not just local files.
4.  **Principle IV (Single Source of Truth)**: Statistical results in the final paper will be derived programmatically from `data/analysis_results.csv` generated by the scripts in `code/`.
5.  **Principle V (Versioning Discipline)**: The plan includes a `hash_artifacts.py` script in `code/` that computes SHA-256 hashes of all data artifacts and automatically updates the `state/projects/...yaml` file after each run, ensuring the versioning discipline is enforced programmatically.
6.  **Principle VI (Automated Evaluation Standardization)**: Code correctness will be measured *exclusively* via HumanEval unit test pass rates. Stylistic quality will use `ruff` (static analysis), prohibiting manual assessment.
7.  **Principle VII (Prompt Structure Control)**: The plan strictly adheres to the five defined complexity tiers and logs token counts/structural elements for every request.

**Reviewer Feedback Integration (alan-turing-simulated)**: The plan acknowledges the objection that "token length" may not fully capture "state transitions." While the primary independent variable remains token/structural count (per spec), `research.md` will explicitly frame this as a proxy for internal state complexity and note this limitation. The "degenerate" tier (redundant instructions) is specifically designed to test the impact of structural redundancy, which may better approximate "memory exercise" than simple length.
