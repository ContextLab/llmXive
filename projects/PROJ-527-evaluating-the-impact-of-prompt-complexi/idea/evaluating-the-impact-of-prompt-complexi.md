# Evaluating the Impact of Prompt Complexity on LLM Code Generation Performance

**Field**: computer science

## Research question

How does prompt complexity (measured by token length and structural elements such as examples and constraints) affect the correctness, efficiency, and stylistic quality of code generated by Large Language Models?

## Motivation

Prompt engineering is widely used to improve LLM code generation, yet most guidance focuses on prompt *content* rather than *complexity*. Understanding whether more elaborate prompts consistently improve code quality—or whether diminishing returns or degradation occur—is critical for developers and researchers designing efficient prompting workflows. This work addresses a gap in systematic empirical evidence on the relationship between prompt structural complexity and code generation outcomes.

## Literature gap analysis

### What we searched

We queried Semantic Scholar/arXiv/OpenAlex for papers combining terms such as "LLM code generation," "prompt complexity," "prompt length," and "prompt engineering" with "software" or "programming." Eight results were returned. Two papers directly address code generation evaluation with LLMs; one addresses prompting strategies for reasoning; one addresses code-text generation models. None systematically vary prompt complexity as an independent variable against code quality metrics.

### What is known

- [Evaluating Code Generation of LLMs in Advanced Computer Science Problems (2025)](http://arxiv.org/abs/2504.14964v1) — Establishes that LLMs are used by students for code generation but does not systematically vary prompt structure.
- [Code Prompting Elicits Conditional Reasoning Abilities in Text+Code LLMs (2024)](http://arxiv.org/abs/2401.10065v3) — Shows that specific prompting techniques (e.g., chain of thought) improve reasoning, but does not test complexity thresholds.
- [JaCoText: A Pretrained Model for Java Code-Text Generation (2023)](http://arxiv.org/abs/2303.12869v1) — Demonstrates code-text generation capabilities but does not study prompt complexity as a variable.

### What is NOT known

No published work has systematically measured how varying prompt token length and structural elements (examples, constraints, multi-paragraph instructions) affects code correctness pass rates, execution efficiency, and readability metrics. The optimal complexity threshold—beyond which additional prompt elaboration yields diminishing returns or degradation—remains unquantified.

### Why this gap matters

Developers and researchers need evidence-based guidance on prompt design to balance effort and performance. Without this knowledge, teams may over-invest in elaborate prompting or miss opportunities where simpler prompts suffice. Filling this gap enables cost-effective LLM deployment in educational and professional code-generation workflows.

### How this project addresses the gap

This project systematically generates prompts across controlled complexity levels for standardized programming problems, evaluates generated code using automated test suites and static analysis, and identifies complexity-performance tradeoff curves. The methodology directly produces the previously-unavailable evidence on optimal prompt complexity thresholds.

## Expected results

We expect to find a non-monotonic relationship between prompt complexity and code quality, with performance improving up to a moderate complexity threshold and then plateauing or declining. We will measure pass rate on HumanEval test suites and code readability metrics; a significant correlation (or lack thereof) at different complexity levels will confirm or falsify the optimal threshold hypothesis.

## Methodology sketch

- Download HumanEval dataset from HuggingFace Datasets (public, ~164 programming problems with unit tests).
- For each problem, generate 5 prompt variants: simple (1 sentence), moderate (1 paragraph), complex (multiple paragraphs + examples), very complex (constraints + chain-of-thought), and degenerate (overly verbose with redundant instructions).
- Query open-source LLM (e.g., CodeLlama-7B via HuggingFace Inference API or local GGUF model on CPU) for each prompt; record token counts and response quality.
- Execute generated code against HumanEval unit tests; record pass/fail rate per complexity level.
- Run static analysis (pylint/ruff) on generated code; extract readability and complexity scores.
- Perform ANOVA or Kruskal-Wallis test to compare performance metrics across complexity levels; compute effect sizes.
- Plot complexity vs. performance curves to identify inflection points and diminishing returns.
