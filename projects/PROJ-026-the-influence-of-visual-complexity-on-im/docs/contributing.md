# Contributing to the Visual Complexity Project

Thank you for your interest in contributing! This document outlines the workflow and standards for this research pipeline.

## Development Workflow

1. **Fork and Clone**: Create a fork of the repository and clone it locally.
2. **Branching**: Create a new branch for your task (e.g., `feature/us1-metrics` or `fix/permutation-bug`).
3. **Implementation**:
 - Follow the existing code structure in `code/`.
 - Ensure all imports match the `API surface` defined in the project.
 - Do not fabricate data; use real sources or the `--null-effect` mode for testing.
4. **Testing**:
 - Write unit tests in `tests/` for new logic.
 - Run `pytest` to ensure all tests pass.
5. **Documentation**:
 - Update `docs/README.md` or `docs/api_reference.md` if you change public interfaces.
 - Add comments to complex logic.

## Code Standards

- **Python Version**: 3.11+
- **Formatting**: Black (line length 88)
- **Linting**: Ruff / flake8
- **Type Hinting**: Use type hints for all function arguments and return values.
- **Logging**: Use the `utils.logging` module for all log messages. Do not use `print()`.

## Project Structure Guidelines

- **`code/`**: All source code.
- **`data/`**:
 - `raw/`: Never modify files here.
 - `processed/`: Generated by scripts.
 - `results/`: Final outputs.
- **`tests/`**: Mirror the structure of `code/`.

## Commit Messages

Follow the convention: `[T-ID] [Story] Brief description`
Example: `[T013] [US1] Implement edge density calculation`

## Pull Request Process

1. Ensure CI passes (tests, linting).
2. Verify that the pipeline runs end-to-end on synthetic data.
3. Update documentation if necessary.
4. Request review from maintainers.

## Methodological Integrity

- **No Data Fabrication**: Do not hard-code results or fake datasets for "demo" purposes.
- **Reproducibility**: Always set random seeds where applicable.
- **Transparency**: Document any changes to statistical methods in `docs/research.md`.
