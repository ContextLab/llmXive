# Evaluating the Robustness of LLM-Generated Code to Input Perturbations

**Field**: computer science

## Research question

How do minor input perturbations (e.g., rephrasing, typos, synonym substitution) affect the functional correctness of code generated by open-source large language models?

## Motivation

As LLMs become integral to software development pipelines, their reliability under noisy or adversarial input conditions is critical for safety and deployment. Current evaluations often focus on clean prompts, leaving a gap in understanding how fragile generated code is to real-world prompt variations.

## Related work

- [Evaluating Code Generation of LLMs in Advanced Computer Science Problems (2025)](http://arxiv.org/abs/2504.14964v1) — Establishes baseline metrics for LLM code generation performance on standard programming problems.
- [Evaluating perturbation robustness of generative systems that use COBOL code inputs (2025)](http://arxiv.org/abs/2511.18488v2) — Demonstrates sensitivity of generative systems to minor input variations, providing a methodological precedent for perturbation analysis.

## Expected results

We expect a statistically significant drop in pass@k scores when prompts are perturbed, with logical errors increasing more than syntax errors. The magnitude of degradation will correlate with the semantic distance of the perturbation from the original prompt.

## Methodology sketch

- **Data Acquisition**: Download the HumanEval dataset (164 programming tasks) from HuggingFace Datasets (`openai_humaneval`).
- **Prompt Engineering**: Create base prompts for each task and generate 3 variants per task using rule-based perturbations (character swaps, synonym replacement, word order shuffling).
- **Model Inference**: Run inference using a quantized (4-bit) StarCoder2-3B model via `transformers` on CPU to ensure fit within 7GB RAM constraints.
- **Code Execution**: Execute generated solutions in a restricted sandbox (using `docker` or `subprocess` with timeout) to capture runtime errors and test case failures.
- **Evaluation Metric**: Calculate pass@1 rates for original vs. perturbed prompts.
- **Statistical Analysis**: Apply McNemar's test to compare paired success/failure counts between original and perturbed conditions.
- **Error Classification**: Manually categorize a subset of failures into syntax, logic, or hallucination types.
- **Resource Management**: Limit total generations to 500 samples to complete within the 6-hour GitHub Actions time limit.
