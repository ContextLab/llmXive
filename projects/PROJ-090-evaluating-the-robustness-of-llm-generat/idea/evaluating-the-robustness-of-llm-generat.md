# Evaluating the Robustness of LLM-Generated Code to Input Perturbations

**Field**: computer science

## Research question

How does the functional correctness of code generated by open-source large language models degrade when the natural language prompt is subjected to semantically-preserving, minor perturbations (e.g., synonym substitution, typo injection, and syntactic rephrasing) that do not alter the underlying programming task?

## Motivation

As LLMs become integral to software development, their reliability under noisy, real-world user input is critical. Current evaluations rely on clean, canonical prompts, creating a gap in understanding whether generated code is robust to the inevitable variations in how developers phrase requirements. Addressing this gap is essential for deploying LLMs in safety-critical pipelines where input noise could lead to silent logical errors.

## Literature gap analysis

### What we searched

We queried Semantic Scholar, arXiv, and OpenAlex using two distinct search strategies: (1) specific queries combining "LLM code generation," "robustness," "perturbation," and "HumanEval"; and (2) broader queries on "adversarial robustness of generative models" and "input noise sensitivity in code synthesis." The search returned three relevant records in the provided literature block, but none directly address the specific interaction between natural language prompt perturbations and the functional correctness of generated code on standard benchmarks like HumanEval.

### What is known

- [Security Degradation in Iterative AI Code Generation -- A Systematic Analysis of the Paradox (2025)](https://arxiv.org/abs/2506.11022) — Establishes that iterative use of LLMs can introduce security vulnerabilities, highlighting the need to study code quality degradation over time, though it does not isolate single-prompt input noise.
- [Holistic Adversarial Robustness of Deep Learning Models (2022)](https://arxiv.org/abs/2202.07201) — Provides a general framework for studying worst-case performance and adversarial risks in deep learning, offering theoretical grounding for robustness metrics but lacking specific application to code generation tasks.
- [Certifying Joint Adversarial Robustness for Model Ensembles (2020)](https://arxiv.org/abs/2004.10250) — Discusses robustness certification for ensembles of deep neural networks, relevant for understanding theoretical bounds on model stability but not directly applicable to the empirical evaluation of LLM code generation under prompt noise.

### What is NOT known

No published work has empirically measured the sensitivity of LLM-generated code to *semantically-preserving* natural language perturbations on standard benchmarks. Specifically, there is no data on whether minor changes like synonym substitution or typo injection in the prompt lead to a statistically significant drop in pass@k scores, nor is there a consensus on which perturbation types are "safe" (preserving task intent) versus "destructive" (altering the problem definition) in the context of code generation.

### Why this gap matters

Filling this gap is crucial for distinguishing between model fragility and prompt misinterpretation. If a model fails only when the prompt is semantically broken, it is behaving correctly; if it fails on semantically equivalent prompts, the model lacks robustness. Understanding this distinction is necessary for developing reliable code-generation tools and for creating benchmarks that reflect real-world usage where user prompts are rarely perfect.

### How this project addresses the gap

This project addresses the gap by implementing a controlled perturbation framework on the HumanEval dataset that explicitly filters for semantic equivalence before evaluation. By comparing pass@1 rates between original and verified-semantically-equivalent perturbed prompts, we will provide the first empirical evidence on the magnitude of robustness degradation in LLM code generation and identify which types of input noise are most detrimental.

## Expected results

We expect to observe a statistically significant, albeit modest, drop in pass@1 scores for perturbed prompts compared to the baseline, with synonym substitution and typos causing less degradation than syntactic rephrasing. The results will demonstrate that while LLMs are robust to minor surface-level noise, their performance is sensitive to perturbations that subtly alter the logical flow of the natural language instructions, confirming that "robustness" is not uniform across perturbation types.

## Methodology sketch

- **Data Acquisition**: Download the HumanEval dataset (164 Python programming tasks) from HuggingFace Datasets (`openai_humaneval`), ensuring all test cases are locally accessible.
- **Perturbation Generation & Validation**: For each of the 164 tasks, generate exactly 3 perturbed prompts using rule-based transformations: (1) synonym replacement in non-keyword tokens, (2) random character typo injection in function descriptions, and (3) syntactic rephrasing of the problem statement. *Crucially*, use a secondary, frozen LLM (e.g., a smaller, quantized model) to score the semantic similarity between the original and perturbed prompts; only retain perturbations with a semantic similarity score > 0.95 to ensure the task intent is preserved.
- **Model Inference**: Run inference using a quantized (4-bit) StarCoder2-3B model via `transformers` on CPU. Limit generation to 500 total samples (164 original + ~336 perturbed, sampling tasks randomly if necessary to fit the 6-hour GHA limit) to ensure feasibility.
- **Code Execution**: Execute generated solutions in a restricted, timeout-enforced sandbox (using `docker` or `subprocess`) to capture runtime errors, timeouts, and test case failures against the original HumanEval test suite.
- **Evaluation Metric**: Calculate pass@1 rates separately for the original prompts and each valid perturbation type.
- **Statistical Analysis**: Apply McNemar's test to compare paired success/failure outcomes for (Original vs. Perturbation Type A), (Original vs. B), and (Original vs. C) separately to avoid aggregating distinct interventions. Use Bonferroni correction for multiple comparisons.
- **Error Classification**: Manually classify a stratified random sample of 50 failures (25 from original, 25 from perturbed) into "syntax error," "logical error," or "hallucination" to identify the nature of robustness failures.
- **Resource Management**: Implement strict timeouts for inference and execution (e.g., 30s per sample) and cap total runtime to 5 hours to allow for analysis within the 6-hour GHA window.
