# Data Model: llmXive Follow-up: Structural Mismatch Cost in Heterogeneous Retrieval

## Overview

This document defines the data structures used for the synthetic query generation, execution logging, and statistical analysis. All data is stored in JSON/Parquet formats within the `data/` directory.

## Entities

### 1. Query Instance
Represents a single retrieval request generated by the synthetic engine.

*   **query_id**: `str` (UUID)
*   **logical_plan**: `dict` (JSON-serializable representation of the plan tree)
*   **source_type**: `enum` ("text", "relational", "graph")
*   **complexity_level**: `int` (Plan depth: 1, 2, 3, 4+)
*   **ground_truth_plan**: `dict` (Output from the **independent, exact, rule-based solver** at runtime, using exhaustive search over the schema graph, not a static lookup table)
*   **generated_params**: `dict` (Randomized parameters: result-set size, syntax complexity)
*   **max_queries_constraint**: `int` (Hard limit of 500, enforced by a **hard-stop loop** in the generator logic: `if count >= 500: break`)

### 2. Execution Metric
Represents the outcome of a single query run.

*   **query_id**: `str` (FK to Query Instance)
*   **latency_ms**: `float` (Real measured time via `time.perf_counter` + complexity-proportional artificial delay injected by CPU-burner loop)
*   **translation_error**: `bool` (True if generated plan != ground truth)
*   **timeout_flag**: `bool` (True if exceeded the designated time limit.)
*   **cpu_throttled**: `bool` (Always True in this experiment)
*   **engine_version**: `str` (Version of the engine used)
*   **cpu_burner_ms**: `float` (Time spent in the deterministic CPU-burner loop, used for baseline subtraction and complexity scaling)

### 3. Analysis Result
Aggregated results for the LMM and Trend Test.

*   **cutoff**: `int` (Sensitivity analysis parameter)
*   **spike_point**: `float` (Estimated threshold where latency increases)
*   **slope_change**: `float` (Change in slope before/after spike)
*   **linear_slope_interaction_p**: `float` (P-value for the interaction term in LMM)
*   **jonckheere_p**: `float` (P-value for the monotonic trend test)
*   **confidence_interval**: `dict` (95% CI for the interaction term)
*   **is_significant**: `bool` (True if p < 0.05)

## Data Flow

1.  **Generation**: `synthetic_query_gen.py` -> `data/processed/queries.jsonl` (Max queries enforced by hard-stop loop)
2.  **Execution**: `main.py` reads queries, runs engines -> `data/processed/metrics.jsonl`
3.  **Analysis**: `lmm_model.py` reads metrics -> `data/processed/analysis_results.json`
4.  **Visualization**: `visualizer.py` reads metrics -> `data/processed/plots/interaction_plot.png`

## Storage Constraints

*   **Memory**: All data must fit within 7GB RAM. The synthetic dataset is capped at a feasible scale to ensure feasibility.
*   **Disk**: Total data size < 14GB (well within limits).
*   **Format**: Parquet for bulk data, JSON for logs and results.
*   **Constraint Enforcement**: The `max_queries_constraint` field in `Query Instance` and the **hard-stop loop** in the generator logic ensure the query limit is strictly enforced.
