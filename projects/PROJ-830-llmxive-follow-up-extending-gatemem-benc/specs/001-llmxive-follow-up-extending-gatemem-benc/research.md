# Research: GateMem Gatekeeper Extension

## 1. Problem Statement & Hypothesis

The "GateMem" benchmark evaluates memory governance in multi-principal shared-memory systems. Current baselines often lack a dedicated, decoupled governance layer, leading to potential leakage of sensitive data or failure to honor deletion requests.

**Hypothesis**: Implementing a lightweight, CPU-tractable gatekeeper (combining semantic intent classification with deterministic rule enforcement) will significantly reduce **semantic** unauthorized data leakage (improving the "Semantic Access Control Score") and increase deletion compliance (improving the "Forgetting Score") without significantly degrading retrieval utility.

**Note on Construct Validity**: The plan distinguishes between "Hard Leaks" (exact string matches, which are deterministic and trivial to block) and "Soft Leaks" (semantic paraphrasing, which are probabilistic). The primary hypothesis test focuses on the reduction of **Soft Leaks** to avoid circularity with ground-truth string definitions.

## 2. Dataset Strategy

### Verified Datasets
The research relies exclusively on the **GateMem** dataset.

- **Dataset Name**: GateMem
- **Source URL**: https://huggingface.co/datasets/Ray368/GateMem
- **Format**: JSONL (Aggregated across domains)
- **Relevance**: Contains annotated queries, user roles, memory chunks, and ground-truth labels. The dataset covers four domains: medical, office, education, household.

**Dataset Variable Fit**:
- **Required Variables**: Query text, User Role, Memory Chunks, `leak-target` annotations (ground truth), Deletion Log entries.
- **Fit Confirmation**: The plan includes a **Dataset Verification Step** (Phase 0.2) to confirm the presence of these fields. If `leak-target` is missing, the plan will fallback to a "Synthetic Leak Injection" strategy or rely on rule-log verification for the Forgetting metric, while using the semantic detector for Access Control.

### Data Loading Strategy
The implementation will use a custom loader script (`code/data/loader.py`) that aggregates all four domain subsets from the main repository URL into a unified `data/raw/` structure. The data will be cached locally with a checksum to ensure reproducibility (Principle III).

## 3. Methodology

### 3.1 Gatekeeper Architecture
The system consists of three sequential stages:
1.  **Intent Classification**: A frozen DistilBERT model (default precision, CPU) classifies the query's intent and associated sensitivity.
2.  **Rule Enforcement**: A deterministic regex/logic engine checks the query and retrieved chunks against:
    *   **Role-Based Policies**: Does the user role match the allowed access for the chunk category?
    *   **Deletion Logs**: Is the chunk ID present in the active deletion log? (Priority: Deletion > Role).
3.  **Filtering**: Chunks failing the checks are removed before the LLM receives the context.

### 3.2 Metric Calculation (Hybrid Approach)
To address construct validity concerns:
- **Blocking Efficiency (Hard Leaks)**: Measures the percentage of ground-truth `leak-target` strings successfully blocked. This is a deterministic metric ([deferred] expected if rules work). Reported separately.
- **Semantic Access Control Score (Soft Leaks)**: Measures the percentage of *semantic* leaks prevented.
    - **Leakage Detector**: Uses a hybrid approach:
        1.  **Exact Match**: Scans for ground-truth strings.
        2.  **Semantic Similarity**: Uses a lightweight `sentence-transformers` model (e.g., `all-MiniLM-L6-v2`) to compare the LLM output against the `leak-target` string and the original sensitive chunk. A score > 0.85 indicates a "Soft Leak".
    - **Calculation**: `(1 - (semantic_leaks / total_unauthorized_attempts)) * 100`.
- **Forgetting Score**: Measures the percentage of deleted data successfully suppressed (using the same Hybrid Leakage Detector).
- **Utility Score (Context-Aware)**:
    - **Reference Answer**: Generated by the baseline system (without gatekeeper) on the full context.
    - **Judge Model**: A small, CPU-tractable LLM evaluates the Gatekeeper's output against the Reference Answer.
    - **Rubric**: Explicitly instructed to ignore "missing context" if that context was flagged as sensitive/deleted. Focuses on "helpfulness given available safe context".
    - **Normalization**: Score 0.0–1.0.

### 3.3 Statistical Analysis
- **Unit of Analysis**:
    - **Domain Level (Primary)**: Aggregated scores per domain (n=4).
    - **Query Level (Secondary)**: Per-query semantic leak scores.
- **Tests**:
    - **Domain Level**: Wilcoxon signed-rank test (non-parametric, suitable for n=4) comparing Baseline vs. Gatekeeper aggregated scores.
    - **Query Level**: **Permutation Test**. Since queries share model weights (lack of independence), a standard t-test is invalid. The plan will use a permutation test (sufficient iterations) to shuffle the "Gatekeeper" label and calculate the distribution of score differences, providing a valid p-value for the observed effect.
- **Multiple Comparison Correction**: Bonferroni correction applied across the four domains.
- **Power Analysis**:
    - Given n=4 domains, the study is underpowered to detect small effect sizes (Cohen's d < 1.5).
    - **Action**: The plan will report **Effect Sizes** (Cohen's d) and **95% Confidence Intervals** (via bootstrapping) as the primary evidence, with p-values as secondary/exploratory.
    - **Minimum Detectable Effect Size (MDES)**: With n=4, α=0.05, power=0.8, MDES is approximately d=1.8. Effects smaller than this may not be statistically significant.

### 3.4 Computational Constraints
- **Hardware**: GitHub Actions free-tier (limited CPU, ~7 GB RAM, no GPU).
- **Model**: DistilBERT-base-uncased (frozen, default precision); `all-MiniLM-L6-v2` for semantic detection (CPU-tractable).
- **LLM**: A small, CPU-tractable model (e.g., DistilLlama) for the utility judge and generation.
- **Memory Management**: Data subsets will be loaded in batches; garbage collection will be enforced between runs.

## 4. Decision Rationale

| Decision | Rationale | Alternative Considered |
|----------|-----------|------------------------|
| **Hybrid Leakage Detector** | Exact match is deterministic (tautology risk); Semantic match measures true information leakage. Combined, they provide a complete picture. | Relying solely on exact match (invalidates hypothesis); Relying solely on semantic match (misses obvious hard leaks). |
| **Permutation Test** | Queries are not independent (shared model weights). Permutation test accounts for this dependency and works for small n. | Paired t-test (assumes independence, invalid); Wilcoxon (assumes independence). |
| **Context-Aware Utility Rubric** | Prevents the judge model from penalizing the gatekeeper for removing sensitive context, isolating "utility" from "compliance". | Standard utility rubric (conflates helpfulness with completeness). |
| **Effect Size Reporting** | With n=4, p-values are unreliable. Effect sizes and CIs provide a more robust measure of impact. | Relying solely on p-values (high risk of Type II error). |

## 5. Limitations & Risks

- **Statistical Power**: The GateMem test set size (n=4 domains) limits the power to detect small effect sizes. The plan mitigates this by reporting effect sizes and confidence intervals.
- **Utility Degradation**: The gatekeeper may over-filter. The sensitivity analysis (FR-009) is designed to mitigate this.
- **CPU Runtime**: Inference on CPU may be slow. The pipeline will include timeout detection to fail gracefully if the temporal limit is approached.
- **Leakage Detection Accuracy**: The semantic detector may have false positives/negatives. The hybrid approach (exact + semantic) provides a safety net.
- **Dataset Schema**: If the GateMem dataset lacks `leak-target` annotations, the plan falls back to synthetic injection or rule-log verification.

## 6. Ethical Considerations

- **Privacy**: The dataset contains synthetic or anonymized data. No PII will be committed to the repository (Principle III).
- **Bias**: The rule engine is deterministic, but the underlying training data for DistilBERT may contain biases. The sensitivity analysis helps identify potential over-blocking.
