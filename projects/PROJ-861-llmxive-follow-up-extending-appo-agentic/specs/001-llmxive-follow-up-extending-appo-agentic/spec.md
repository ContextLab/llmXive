## User Scenarios & Testing

### User Story 1 - Static Branching Score Computation (Priority: P1)

A researcher needs to compute the "Static Branching Score" for a corpus of reasoning traces (GSM8K/MATH) using a frozen LLM on CPU-only infrastructure to establish the baseline predictor variable without requiring expensive online rollouts.

**Why this priority**: This is the foundational data generation step. Without the static scores, the core correlation hypothesis cannot be tested. It directly addresses the motivation of eliminating online rollouts for initial scoring.

**Independent Test**: The system can be fully tested by running the static inference pass on a subset of 100 tasks and verifying that a score matrix is generated for every token position where the model outputs a probability distribution, without any interaction with an environment or reward signal.

**Acceptance Scenarios**:

1. **Given** a frozen decoder-only model (e.g., Phi-2) loaded in CPU-only mode and a dataset of 100 reasoning traces, **When** the system processes each token position, **Then** it outputs a "Static Branching Score" calculated as the Kullback-Leibler divergence between the model's next-token distribution and a uniform distribution over the top-k (k=5) alternatives.
2. **Given** a task where the model's next-token probability is highly concentrated (low entropy), **When** the system calculates the score, **Then** the resulting score reflects low divergence (indicating a predictable path), and no GPU acceleration is requested or utilized.
3. **Given** a dataset exceeding 1 GB, **When** the system processes the data, **Then** it streams or chunks the input to ensure peak memory usage remains below 7 GB, completing the pass for all 100 tasks within 30 minutes (derived from the 5-hour total pipeline limit to allow sufficient time for dynamic generation and analysis).

### User Story 2 - Dynamic Branching Score Generation (Priority: P2)

A researcher needs to generate the ground-truth "Dynamic Branching Score" for a subset of tasks using the APPO algorithm with online rollouts to serve as the dependent variable for correlation analysis.

**Why this priority**: This provides the "future-aware likelihood gains" ground truth. While the goal is to replace this, the validation of the static method requires a dynamic baseline to compare against.

**Independent Test**: The system can be tested by running the APPO algorithm on a fixed subset of 50 tasks, recording the Dynamic Branching Scores generated by the policy likelihood gains, and verifying that the scores vary based on the actual rollout outcomes and reward signals (final answer correctness), distinct from the static pre-computed values.

**Acceptance Scenarios**:

1. **Given** a random subset of 50 tasks from the GSM8K/MATH datasets, **When** the APPO algorithm executes with online rollouts, **Then** it records a "Dynamic Branching Score" for each decision point based on the actual policy likelihood gains and outcome variance.
2. **Given** a task where the agent encounters a dead-end during rollout, **When** the system calculates the dynamic score, **Then** the score reflects the high penalty associated with that specific trajectory, differing significantly from the static prediction if the static model failed to predict the dead-end.
3. **Given** the constraint of 5 hours for the generation phase (static + dynamic), **When** the APPO algorithm runs on the subset, **Then** it completes the generation of dynamic scores for all 50 tasks, ensuring the total generation runtime does not exceed 4 hours 30 minutes to allow for analysis time.

### User Story 3 - Statistical Correlation and Validation (Priority: P3)

A researcher needs to align the static and dynamic score sequences and compute the Pearson and Spearman correlation coefficients to determine if the static approximation is a valid predictor of dynamic value.

**Why this priority**: This is the final analytical step that answers the research question. It transforms the raw data into the scientific conclusion regarding the feasibility of CPU-only agentic optimization.

**Independent Test**: The system can be tested by feeding pre-generated static and dynamic score files and verifying that the output includes the correlation coefficients, p-values from the permutation test, and residual plots.

**Acceptance Scenarios**:

1. **Given** aligned sequences of static and dynamic scores for 50 decision points, **When** the analysis script runs, **Then** it outputs the Pearson correlation coefficient (r) and Spearman rank correlation (ρ) with at least 3 decimal places of precision.
2. **Given** the null hypothesis that scores are uncorrelated, **When** the permutation test (up to 10,000 iterations, minimum 5,000) is executed, **Then** it reports a p-value indicating whether the observed correlation is significantly different from zero (p < 0.05).
3. **Given** a correlation result, **When** the system generates the residual visualization, **Then** it produces a plot showing the difference between static and dynamic scores, highlighting specific reasoning patterns (e.g., algebraic vs. arithmetic) where the approximation fails.

### Edge Cases

- What happens when the frozen model assigns zero probability to the "correct" token in the static trace? (The system MUST handle log(0) by applying a small epsilon smoothing factor of 1e-9 to prevent NaN errors).
- How does the system handle tasks where the APPO rollout fails to converge or exceeds the token limit? (The system MUST exclude these specific tasks from the correlation analysis and log them as "dropped due to rollout failure" to prevent skewing the dataset).
- What happens if the memory limit (7 GB) is approached during the permutation test? (The system MUST reduce the number of permutation iterations to 5,000 or sample the permutation space, ensuring the job completes within the 5-hour generation window).

## Requirements

### Functional Requirements

- **FR-001**: System MUST compute the "Static Branching Score" as the Kullback-Leibler divergence between the model's next-token distribution and a uniform distribution over the top-5 most probable tokens for every token position in the input corpus (See US-1). *Note: This metric measures predictability (entropy) as a proxy for branching criticality; the hypothesis that predictability correlates with value is an empirical claim to be tested.*
- **FR-002**: System MUST execute the APPO algorithm on a random subset of 50 tasks to generate "Dynamic Branching Scores" based on actual rollout outcomes and policy likelihood gains, using "final answer correctness" (binary reward: 1 if correct, 0 otherwise) as the reward signal (See US-2).
- **FR-003**: System MUST align static and dynamic score sequences by semantic decision points using a sequence alignment algorithm (e.g., dynamic time warping or edit distance) to handle divergent token sequences between static and dynamic traces, preserving token position alignment within each task (See US-3).
- **FR-004**: System MUST calculate the Pearson correlation coefficient and Spearman rank correlation between the two score sets to quantify the linear and monotonic relationship (See US-3).
- **FR-005**: System MUST perform a permutation test with up to 10,000 iterations (minimum 5,000) to determine the statistical significance (p-value) of the observed correlation, ensuring the result is not due to chance (See US-3).
- **FR-006**: System MUST enforce a hard memory limit of 7 GB and a time limit of 5 hours for the generation phase by streaming data and optimizing batch sizes, exiting with code 1 and writing a log entry containing the string "RESOURCE_LIMIT_EXCEEDED" before terminating if these constraints are exceeded (See US-1, US-2).

### Key Entities

- **ReasoningTrace**: A sequence of tokens and intermediate steps from a task (e.g., GSM8K problem), containing the ground-truth solution path.
- **BranchingScore**: A numerical value representing the "criticality" of a decision point, computed either statically (divergence) or dynamically (likelihood gain).
- **CorrelationResult**: A structured object containing the Pearson r, Spearman ρ, p-value, and confidence intervals derived from the alignment of scores.

## Assumptions

- The GSM8K and MATH datasets are available via HuggingFace Datasets and contain sufficient reasoning traces to generate a statistically significant sample (n ≥ 50) within the 5-hour generation window.
- The frozen LLM (e.g., Phi-2) can be loaded and run in inference-only mode on CPU without requiring GPU-specific libraries (e.g., CUDA, bitsandbytes) or quantization.
- The "Static Branching Score" calculation (KL divergence) is computationally lightweight enough to be performed on the entire subset within 30 minutes on a 2-core CPU.
- The APPO algorithm, when run on a subset of 50 tasks with "final answer correctness" as the reward signal, will complete within 4 hours on the free-tier runner, assuming standard hyperparameters and no complex environment interactions.
- The correlation threshold of 0.7 is a community-standard benchmark for "strong" predictive power in regression tasks, serving as a design choice for feasibility testing (not a definitive scientific conclusion).
- The hypothesis that predictability (entropy) correlates with branching criticality (value) is an empirical claim to be tested, not a definition; the static score is a proxy for "ease of guessing" which may correlate with "value" in well-trained policies.
- If the dataset lacks specific variables required for a more granular analysis, the study will proceed with the available token-level data, treating the correlation as the primary metric of success.
