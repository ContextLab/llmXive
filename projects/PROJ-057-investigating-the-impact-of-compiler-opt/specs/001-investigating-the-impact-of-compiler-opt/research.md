# Research: Investigating the Impact of Compiler Optimizations on LLM Inference Latency

## Problem Statement & Hypotheses

**Research Question**: How do compiler optimization flags (specifically `-O3`, `-ffast-math`, `-march=native`) impact the latency and numerical stability of LLM inference kernels (MatMul, Softmax, LayerNorm) on x86_64 CPU?

**Hypotheses**:
1.  **H1 (Latency)**: Aggressive optimizations (`-O3 -ffast-math`) will yield significantly lower median latency compared to `-O0` or `-O2` baselines (p < 0.05, Welch's Independent Samples t-test).
2.  **H2 (Stability)**: Aggressive optimizations will introduce measurable numerical drift (L2 error > 1e-5) compared to the `-O0` baseline and the high-precision reference.
3.  **H3 (Trade-off)**: There exists a Pareto frontier where a subset of configurations offers the best latency without violating the stability threshold, but the full trade-off curve (including unstable points) will be visualized.

## Dataset Strategy

This project uses **synthetic data** generated on-the-fly to ensure reproducibility and eliminate external dataset dependencies. To address construct validity, we generate **5 distinct random tensor sets** per configuration (seeds 42-46) covering different distributions (Gaussian, sparse, dense) to ensure stability is measured across diverse activation patterns, not just random noise.

| Data Type | Description | Source/Generation Method | Purpose |
| :--- | :--- | :--- | :--- |
| **Input Tensors** | Random float32 matrices (512x512, 768x768) with varied distributions. | `numpy.random.Generator` with seeds 42-46. | Input for kernel execution; ensures robustness against specific data patterns. |
| **Reference Output** | Ground truth results computed in Python `decimal` (512-bit). | `code/benchmarks/reference.py`. | Baseline for absolute stability calculation. |
| **Baseline Output** | Output from `-O0` compiled kernel. | `code/benchmarks/executor.py`. | Comparative baseline for drift analysis (Constitution Principle VI). |
| **Benchmark Results** | Latency logs (median, p95) and stability metrics (L2, MaxDiff). | `code/benchmarks/executor.py` + `code/analysis/stability_check.py`. | Primary dataset for statistical analysis. |

**Note**: No external datasets (e.g., HuggingFace, UCI) are required. The "dataset" is the collection of synthetic runs generated by the benchmark script.

## Methodology

### 1. Kernel Implementation & Compilation
-   **Kernels**: Matrix Multiplication, Softmax, Layer Normalization implemented in C++17.
-   **Compilers**: GCC (v11+) and Clang (v14+).
-   **Flags**:
    -   Baseline: `-O0`
    -   Standard: `-O2`, `-O3`
    -   Aggressive: `-O3 -ffast-math`, `-O3 -march=native`, `-O3 -funroll-loops`
    -   Combinations: Dynamic generation of all valid permutations.
-   **Process**: The `compile_runner.py` script compiles each kernel with each flag set, producing a unique binary.

### 2. Execution & Measurement
-   **Input**: Fixed-size synthetic tensors (batch=1, seq=512, hidden=768) generated with 5 distinct seeds (42-46).
- **Iterations**: **Fixed [deferred] iterations** per configuration.
 - *Rationale*: Constitution Principle VII mandates [deferred] iterations. This allows for **100 blocks of 100 iterations**, providing sufficient degrees of freedom for statistical testing.
    -   *No Adaptive Stopping*: Iteration count is fixed to eliminate selection bias.
-   **Metric**: `std::chrono::high_resolution_clock` for nanosecond precision.
-   **Output**: JSON/CSV log containing median latency, 95th percentile, and binary output file (for stability check).
-   **Stability Check**: For each of the 5 seeds, calculate L2 error against the Python decimal reference. The final stability metric for a configuration is the **maximum** L error observed across the 5 seeds.

### 3. Stability Analysis
-   **Reference 1 (Absolute)**: Python `decimal` module (512-bit precision) computes the exact result.
-   **Reference 2 (Comparative)**: The `-O0` baseline output (as per Constitution Principle VI).
-   **Comparison**:
    -   **L2 Relative Error**: $\frac{||A_{opt} - A_{ref}||_2}{||A_{ref}||_2}$
    -   **Max Absolute Difference**: $\max(|A_{opt} - A_{ref}|)$
-   **Threshold**: Any configuration with L2 error > 1e-5 is flagged as "Stability Failure".
-   **Pareto Analysis**: Configurations with error > 1e-5 are **NOT excluded** from the Pareto plot. They are plotted as "unstable" points. The Pareto frontier is calculated on **ALL points** (stable and unstable) using Median Latency (minimize) vs. Max Absolute Difference (minimize). The frontier is defined as the set of non-dominated points, empirically visualizing the full trade-off landscape.

### 4. Statistical Analysis
-   **Method**: **Welch's Independent Samples t-test** on block-averaged latency distributions.
    -   *Rationale*: Each configuration uses a distinct binary (independent samples), making paired t-tests invalid.
-   **Block Size**: 100 iterations.
 - *Total Blocks*: [deferred] iterations / 100 = **100 blocks** per configuration.
    -   *Degrees of Freedom*: 100 blocks per group provide sufficient power for the t-test.
-   **Null Hypothesis ($H_0$)**: No difference in mean latency between configuration A and B.
-   **Significance**: $\alpha = 0.05$.
-   **Correction**: Bonferroni correction applied for multiple comparisons.

### 5. Visualization
-   **Pareto Frontier**: Scatter plot of Median Latency (x-axis) vs. Max Absolute Difference (y-axis).
    -   Stable points (error ≤ 1e-5): Blue markers.
    -   Unstable points (error > 1e-5): Red markers.
-   **Annotation**: Points on the frontier (Pareto optimal) are highlighted.
-   **Significance Markers**: Stars or colors indicating statistically significant improvements over the baseline.

## Decision Rationale (Compute Feasibility)

-   **CPU-Only**: The project explicitly avoids GPU usage. Kernels are small enough (768x768 float32 ≈ 2.3MB) to fit entirely in L2/L3 cache or RAM, making CPU execution efficient.
-   **No Deep Learning Libraries**: We do not use PyTorch/TensorFlow for inference. We use raw C++ and Python `numpy`/`decimal` to isolate the compiler's effect. This avoids the overhead and memory footprint of deep learning frameworks.
- **Runtime**: With 2 CPU cores, compiling 12 combinations × 3 kernels = 36 binaries is negligible. Running [deferred] iterations of a 2.3MB matrix multiply takes < 100ms total per config (approx 10-20µs per run). Total runtime for [deferred] iterations is estimated at < 1 hour, well within the 6h limit.
-   **Memory**: Peak memory usage is dominated by the input tensors (< 10MB) and the Python process (< 500MB). This is well within the RAM limit.

## Risk Assessment

-   **Risk**: Compiler version mismatch on runner.
    -   *Mitigation*: The script checks `gcc --version` and `clang --version` at startup. If < 11 or < 14, it fails with a clear error.
-   **Risk**: `-ffast-math` causing NaNs.
    -   *Mitigation*: The stability check script explicitly scans for NaNs. If found, the configuration is logged as "NaN Failure" and excluded from the stable set but included in the "unstable" plot.
-   **Risk**: Runtime exceeds 6h.
 - *Mitigation*: Iteration count is fixed at [deferred]. If the total estimated time exceeds 6h (unlikely given the small kernel size), the script will reduce the tensor size (e.g., 512x512) rather than the iteration count, preserving statistical validity.
