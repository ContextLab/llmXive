# Feature Specification: Phase 4 Pipeline Validation — Planner + Tasker (with Analyze loop)

**Feature Branch**: `014-phase4-plan-tasks-testing`
**Created**: 2026-05-21
**Status**: Draft
**Input**: User description: "Phase 4: issue 48 + all sub-issues + any related agents. Validate each step of the llmXive pipeline; examine the inputs and outputs produced by any agents related to this phase; use REAL projects as inputs. Currently we're using projects 261 and 262 as ideal for carrying forward into this next phase."

## Background

Phase 4 of the llmXive agentic pipeline (tracked in [#48](https://github.com/ContextLab/llmXive/issues/48), umbrella [#107](https://github.com/ContextLab/llmXive/issues/107)) covers the **Spec Kit Plan → Tasks (with Analyze loop)** transition. Two agents participate:

- **`planner`** ([#65](https://github.com/ContextLab/llmXive/issues/65)) — drives `/speckit.plan`; takes a project at stage `clarified` and produces **five** artifacts in a single multi-file LLM response, split on `<!-- FILE: <relative_path> -->` markers, written under `projects/<PROJ-ID>/specs/<n>-<slug>/`: `plan.md`, `research.md`, `data-model.md`, `quickstart.md`, and at least one `contracts/<schema>.yaml`. Advances the project to stage `planned`.
- **`tasker`** ([#66](https://github.com/ContextLab/llmXive/issues/66)) — drives `/speckit.tasks` (Mode A) to derive a numbered `tasks.md`, then `/speckit.analyze` in a **Mode-A → Mode-B revision loop** that resolves every finding by full-document rewrite of the affected upstream artifact (`spec.md` / `plan.md` / `tasks.md`). The loop is bounded by `TASKER_MAX_REVISION_ROUNDS` (default 5). Advances the project through `tasked` → `analyze_in_progress` → `analyzed`, or escalates to `human_input_needed` on cap-hit.

The validation must use **real projects** — `PROJ-261-evaluating-the-impact-of-code-duplicatio` (Computer Science) and `PROJ-262-predicting-molecular-dipole-moments-with` (Chemistry), both currently parked at stage `clarified` after Phase 3 (spec 011) handed them forward via its `carry-forward.yaml`. Each already owns a `projects/<PROJ-ID>/specs/001-<slug>/spec.md` (the Phase-3 output) — which is the **input** to the Planner, and MUST be preserved by Phase 4.

**Entry stage**: `clarified`. **Exit stage**: `analyzed` (the entry stage of Phase 5, Implementation, [#49](https://github.com/ContextLab/llmXive/issues/49)).

This feature follows the precedent set by spec 011 (Phase 3 validation): it is a **validation** effort that exercises each step on real inputs, captures inspection records of every agent's I/O, gates against silent shortcuts, and emits a carry-forward manifest. It does **not** rewrite the Planner's or Tasker's core logic — except to add two missing quality gates identified during clarification (FR-006 URL-reachability and FR-007 data-model↔contracts consistency) — plus any fix for a real bug found.

## Clarifications

### Session 2026-05-21

- Q: Where should Phase-4 gates not currently enforced by the deployed planner/tasker (research.md URL-reachability, data-model↔contracts consistency) be enforced? → A: Harden the agent classes — add these gates into the Planner so the production pipeline gates them at runtime (each a separate, justified commit per FR-017), not merely in the validation layer.
- Q: How strict should the research.md URL-reachability gate be on the real run, given real URLs can be transiently down/rate-limited? → A: Hard-fail any URL that does not return 2xx/3xx (including transient 5xx/timeouts/429); no transient-retry leniency. The determinism/strictness tradeoff is accepted (re-run when a legitimate source recovers).
- Q: If both real projects converge with 0 analyze rounds, is the Tasker Mode-B (analyze-resolve) path validated? → A: Prefer real-content coverage when a finding occurs naturally; guarantee Mode-B coverage via the synthetic-input regression tests regardless; the phase report records which path(s) provided coverage.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Run the full Phase 4 pipeline on a single fresh real project (Priority: P1)

A maintainer wants to know whether Phase 4 works end-to-end on a single real project. They invoke the pipeline runner against `PROJ-261` (or `PROJ-262`) with `--max-tasks 2` — one step for the Planner and one for the Tasker (whose Mode-A→Mode-B analyze loop runs internally within that single invocation, bounded by `TASKER_MAX_REVISION_ROUNDS`). The runner picks the project, runs the Planner (producing the five plan artifacts), then the Tasker (generating `tasks.md` and driving the analyze loop to convergence), and reports `analyzed` as the final stage. All artifacts — the five plan files, `tasks.md`, the project's state YAML, and run-log entries — are inspectable side-effects on disk.

**Why this priority**: This is the smallest end-to-end smoke test that proves Phase 4 is not silently broken. It is the only test that surfaces integration bugs between Planner and Tasker (e.g., the Tasker failing to parse a `plan.md` the Planner just wrote, or the analyze loop never converging) and the only one that exercises the multi-round analyze revision loop on real content.

**Independent Test**: Can be fully validated by running the pipeline runner against `PROJ-261-evaluating-the-impact-of-code-duplicatio` and confirming the project state transitions `clarified → planned → tasked → analyze_in_progress → analyzed`, with five non-template plan artifacts and a `tasks.md` containing ≥10 numbered `T###` tasks.

**Acceptance Scenarios**:

1. **Given** PROJ-261 at `clarified`, **When** the runner is invoked for PROJ-261 with `--max-tasks 2`, **Then** the project ends at `analyzed`, with `plan.md`, `research.md`, `data-model.md`, `quickstart.md`, at least one `contracts/*.yaml`, and `tasks.md` all written under `projects/PROJ-261-…/specs/001-…/`, and the pre-existing `spec.md` preserved.
2. **Given** PROJ-262 at `clarified`, **When** the same runner invocation is repeated for PROJ-262, **Then** PROJ-262 also ends at `analyzed` with a substantively different artifact set reflecting its chemistry domain (Graph Neural Networks for molecular dipoles), including domain-appropriate `contracts/` schemas.
3. **Given** a Phase-4 run completes (success OR failure), **When** the maintainer reads `state/run-log/YYYY-MM/<run-id>.jsonl`, **Then** every Planner invocation, every Tasker invocation, and every Mode-B revision round appears with `started_at`, `ended_at`, `outcome`, and (on failure) `error` or `human_input_needed` fields.

### User Story 2 — Inspect inputs + outputs at every Phase 4 step (Priority: P1)

A maintainer auditing the pipeline wants to see exactly what each Phase-4 agent received as input and what it produced as output, on each reference project — including each iteration of the Tasker's analyze loop. Each agent invocation produces a structured inspection record under `specs/014-phase4-plan-tasks-testing/inspections/<project_id>/<agent>.json`; the Tasker record captures one sub-record per analyze round (the `/speckit.analyze` report it received and the Mode-B JSON patch it produced).

**Why this priority**: The user's explicit ask is "examine the inputs and outputs produced by any agents related to this phase". Without capturable I/O records, a green pipeline can still be silently wrong — e.g., the Planner inventing dataset URLs, or the Tasker "resolving" an analyze finding by deleting the constraint that triggered it. The inspection records make the validation reproducible and reviewable without re-running the LLM.

**Independent Test**: Can be fully validated by running Phase 4 for PROJ-261 and PROJ-262, then opening the four resulting inspection records (Planner + Tasker × 2 projects) and confirming each contains: (a) the system+user prompt verbatim, (b) the raw LLM response(s), (c) the parsed/structured output applied to the project, (d) before/after file diffs for every file modified — and for the Tasker, one sub-record per analyze round.

**Acceptance Scenarios**:

1. **Given** PROJ-261 completes the Planner step, **When** `specs/014-…/inspections/PROJ-261-…/planner.json` is opened, **Then** it contains the full prompt the Planner sent to the LLM (including the `spec_text`, `plan_template`, and `project_constitution` inputs), the full LLM response, the FILE-marker split into five named artifacts, and the resulting content (or diff) of each artifact.
2. **Given** PROJ-261 completes the Tasker step, **When** the corresponding `tasker.json` is opened, **Then** it lists each analyze round in order, and for each round records: the `/speckit.analyze` report received, the Mode-B JSON patch produced (`issues_resolved`, `issues_remaining`, `verdict`), and a before/after diff of every artifact rewritten that round — terminating with the `verdict: clean` round (or the `escalate` round on cap-hit).
3. **Given** both PROJ-261 and PROJ-262 finish, **When** the maintainer compares the two pairs of inspection records, **Then** the prompts and produced artifacts visibly differ by domain (PROJ-261's code-duplication plan vs PROJ-262's GNN-dipole plan), confirming the inputs were real and project-specific rather than templated.

### User Story 3 — Quality gates that catch silent shortcuts (Priority: P1)

The pipeline must refuse to mark Phase-4 stages complete when the agents have produced template-only, fabricated, or evasive output. Specifically, the validation must demonstrate that each of the following is **rejected** (outcome `failed`, stage HELD, run-log records the reason): a plan artifact byte-equal (modulo whitespace) to its template; a `research.md` citing an unreachable/invented URL; a `tasks.md` that is a prose stub (fewer than the validator's minimum task IDs); a Mode-B patch that leaks a unified diff, clobbers markdown headers, or weakens a constraint/test to make analyze pass; and an analyze loop that exceeds `TASKER_MAX_REVISION_ROUNDS` without converging (which must escalate to `human_input_needed`, never loop forever or false-advance).

**Why this priority**: The Phase-4 issue's acceptance criteria and the sub-issues for `planner` (#65) and `tasker` (#66) call these out explicitly ("No silent shortcuts", "research.md cites REAL reachable datasets/code/papers", "Mode-B patches don't replace tasks.md with prose stubs", "Mode-B patches preserve markdown headers", "escalates to human_input_needed if exceeded"). Without these gates, the pipeline silently produces unusable plans and the maintainer only discovers the rot during implementation or review.

**Independent Test**: Can be fully validated by (a) confirming both reference projects at `analyzed` genuinely passed every gate, and (b) running targeted regression tests that inject each synthetic failure mode and assert the pipeline returns `failed`/`escalate` and the stage does not advance.

**Acceptance Scenarios**:

1. **Given** the Planner returns a plan artifact structurally identical to its template (still containing placeholder text), **When** `write_artifacts` runs, **Then** the artifacts are deleted, the outcome is `failed`, the project stays at `clarified`, and the run-log records an actionable error.
2. **Given** the Planner's `research.md` cites a URL that returns anything other than 2xx/3xx (invented, malformed, 4xx, 5xx, timeout, or rate-limited), **When** the Planner's plan-time reachability check runs, **Then** the Planner hard-fails the artifact, names the offending URL in the inspection record, and the project does not advance — with no transient-retry leniency.
3. **Given** the Tasker's Mode-A `tasks.md` contains fewer than the minimum task IDs (a prose stub), **When** the task validator runs, **Then** the output is rejected, the outcome is `failed`, and the project stays at `planned`.
4. **Given** a Mode-B patch (a) leaks a unified-diff prefix (`--- a/…` / `+++ b/…` / `@@`), (b) reduces the markdown header structure of `spec.md`/`plan.md`, or (c) shrinks `tasks.md` below the task-ID minimum, **When** the patch is applied, **Then** the patch is rejected and the round fails loudly rather than corrupting the file.
5. **Given** `/speckit.analyze` keeps surfacing an unresolved finding past `TASKER_MAX_REVISION_ROUNDS`, **When** the cap is hit, **Then** the Tasker writes `human_input_needed.yaml`, the project is HELD at `analyze_in_progress` (not advanced to `analyzed`), and the run-log records `escalated` with the unresolved finding.

### User Story 4 — Carry-forward checklist for Phase 5 (Priority: P2)

After Phase 4 validation is complete on both reference projects, a `carry-forward.yaml` file lists the project IDs and the exact stage they are parked at, ready for Phase 5 (Implementation) testing. The file follows the same shape as `specs/011-phase3-specify-clarify-testing/carry-forward.yaml`.

**Why this priority**: The user's direction (paraphrased): "we're using projects 261 and 262 as ideal for carrying forward into this next phase". Phase 4 must explicitly hand them to Phase 5 in the same canonical, machine-readable way the prior phases used.

**Independent Test**: Can be fully validated by opening `specs/014-…/carry-forward.yaml` after Phase 4 completes and confirming both projects appear with stage `analyzed` (or `human_input_needed`/`held` with a documented reason — the file MUST distinguish).

**Acceptance Scenarios**:

1. **Given** PROJ-261 and PROJ-262 successfully reach `analyzed`, **When** `carry-forward.yaml` is generated, **Then** both project IDs appear under a `passed` (or equivalent) key with `final_state: analyzed` and an `agents_run` summary listing the Planner and Tasker outcomes (and the Tasker's round count).
2. **Given** one project fails or is held during Phase 4, **When** `carry-forward.yaml` is generated, **Then** the failing project appears under a `failed`/`held` key with the stage it stopped at and a one-line reason citing the inspection record.

### Edge Cases

- **The plan is clean enough that `/speckit.analyze` reports zero findings on the first pass.** The Tasker MUST advance `planned → tasked → analyzed` with zero Mode-B rounds; the run-log MUST record `0-round` convergence as a success, not a failure or `no-op`.
- **`/speckit.analyze` surfaces a finding the Tasker cannot resolve within the cap.** The Tasker MUST escalate to `human_input_needed.yaml` and HOLD at `analyze_in_progress` — never loop indefinitely, never false-advance to `analyzed`.
- **The Planner emits fewer than five FILE markers, a duplicate marker, or a malformed `<!-- FILE: … -->` marker.** The multi-file split MUST fail closed: no partial artifacts are committed, the outcome is `failed`, and the project HOLDS at `clarified`.
- **`research.md` cites a plausible-but-dead URL (e.g., 404), a transiently-unavailable source (5xx/timeout/429), or a fabricated arXiv/dataset ID.** The Planner's reachability check MUST hard-fail the run and name the URL in all of these cases (no transient-retry leniency); it MUST NOT silently accept it. (Accepted tradeoff: a legitimate source that is temporarily down fails the run; re-run when it recovers.)
- **A Mode-B patch is returned as a unified diff instead of a full-document rewrite.** This is a regression of the spec-010 diff-leak bug; the diff-guard MUST reject it. The validation MUST include a regression test asserting the guard is still active.
- **A Mode-B patch "resolves" an analyze finding by deleting a constraint, requirement, or test to make analyze pass.** This violates the constitution's "fix the code, not the test"; the round MUST be rejected, not accepted as `clean`.
- **The Tasker is invoked on a project still at `clarified` (the Planner has not run, so there is no `plan.md`).** The Tasker MUST fail loudly or abstain — it MUST NOT fabricate `tasks.md` from `spec.md` alone.
- **A `contracts/*.yaml` schema references an entity absent from `data-model.md`, or vice versa.** The Planner's consistency check MUST hard-fail the artifact (not merely flag it) and name the mismatch in the inspection record.
- **Phase 4 is invoked on a project NOT at `clarified`** (e.g., already at `analyzed`). The runner MUST decline to re-run and either no-op cleanly or report "already past this phase". This is distinct from the reset semantic of FR-018, which fires only when the project is still at `clarified` but has leftover Phase-4 output artifacts.
- **Concurrent invocations on the same project.** Phase 4 acquires the project lock for the duration of each agent (and across the Tasker's loop). A second runner trying the same project MUST block (or skip) — never produce a torn write to any artifact.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The validation MUST be runnable end-to-end with a single command per project, with no manual steps between the Planner, the Tasker's Mode-A generation, and the Tasker's Mode-A→Mode-B analyze loop.
- **FR-002**: `PROJ-261-evaluating-the-impact-of-code-duplicatio` and `PROJ-262-predicting-molecular-dipole-moments-with` (current stage: `clarified`) MUST be the canonical real-project inputs; any future re-run MUST be reproducible from those same starting states.
- **FR-003**: For each agent invocation — and, for the Tasker, for each analyze revision round — the validation MUST capture and persist an inspection record containing: system prompt, user prompt, raw LLM response, parsed output, and a unified diff of every file the agent wrote.
- **FR-004**: Inspection records MUST be written under `specs/014-phase4-plan-tasks-testing/inspections/<project_id>/<agent>.json` and MUST be commit-safe (no secrets, no API keys, no full credentials — only model id + truncated request id). The Tasker record MUST nest one sub-record per analyze round.
- **FR-005**: The Planner MUST produce all five artifacts — `plan.md`, `research.md`, `data-model.md`, `quickstart.md`, and at least one `contracts/<schema>.yaml` — each non-empty, correctly separated by `<!-- FILE: <relative_path> -->` markers. A missing or empty artifact, or a failed marker split (e.g., a response with no FILE markers), MUST cause the Planner to hard-fail the artifact set (unlink anything written, `outcome: failed`, hold at `clarified`). This completeness check is the third Planner-side gate added by this feature (see FR-017), alongside FR-006 and FR-007.
- **FR-006**: The **Planner** MUST verify, at plan time, that every dataset/code/paper reference (URL/identifier) in `research.md` returns 2xx/3xx (HTTP HEAD/GET, or an otherwise verifiable identifier). Any reference that does not — invented, malformed, 4xx, 5xx, timeout, or rate-limited — MUST cause the Planner to hard-fail the artifact (outcome `failed`, stage HELD at `clarified`), with the offending reference named in the inspection record. There is NO transient-retry leniency (per clarification); a legitimate-but-temporarily-down source fails the run and is re-run later. This gate is new agent-side behavior added by this feature (see FR-017).
- **FR-007**: The **Planner** MUST verify a structural consistency between `data-model.md` and `contracts/`: (a) `data-model.md` actually defines entities (an attribute table, an ER/mermaid diagram, or entity headings) rather than empty prose, and (b) every emitted `contracts/*.yaml` parses as a real, non-empty schema (a YAML mapping/sequence, not an empty file or prose stub). A violation MUST cause the Planner to hard-fail the artifact, with the reason named in the inspection record. Entity↔schema **cardinality and naming are intentionally NOT constrained** — the Planner's contract requires *at least one* schema, not one per entity, and schema filenames legitimately differ from entity headings (a strict 1:1 name match produced false positives on real planner output during validation). This gate is new agent-side behavior added by this feature (see FR-017).
- **FR-008**: When any Planner artifact classifies as TEMPLATE (per `_real_only_guard.assert_real_or_raise` or equivalent), all artifacts from that invocation MUST be deleted, the outcome MUST be `failed`, the project stage MUST NOT advance from `clarified`, and the run-log MUST record the rejection reason.
- **FR-009**: The Tasker's Mode-A `tasks.md` MUST contain at least 10 numbered `T###` tasks in the canonical `- [ ] T### [P?] [USx?] <description with file path>` format. Output with fewer than the validator's minimum task IDs (a prose stub) MUST be rejected: outcome `failed`, stage HELD at `planned`.
- **FR-010**: Tasks in `tasks.md` MUST respect data-flow ordering — any task that consumes an artifact MUST appear after the task that produces it (e.g., download datasets before training; create a directory before writing into it). The validation MUST automatically check at least the dataset-download-before-use and directory-create-before-write invariants.
- **FR-011**: The Tasker's Mode-B patches are FULL-document rewrites, never unified diffs. A patch MUST be rejected (and the round MUST fail loudly, not silently corrupt the file) if it (a) leaks a unified-diff prefix (`--- a/…` / `+++ b/…` / `@@`), (b) reduces the markdown header structure of `spec.md` or `plan.md`, or (c) shrinks `tasks.md` below the task-ID minimum.
- **FR-012**: A Mode-B patch MUST NOT resolve an analyze finding by weakening or deleting a requirement, constraint, or test. The validation MUST detect this automatically: across any Mode-B rewrite of `spec.md`, the count of `FR-NNN` and `SC-NNN` lines MUST NOT decrease (a drop signals a deleted requirement/constraint). A patch that reduces the count MUST be flagged as a finding and the validation MUST treat the round as a violation rather than accepting it as `clean` (per the constitution's "fix the code, not the test"). This check lives in the validation layer (`scripts/validate_phase4.py`), not the Tasker agent.
- **FR-013**: The Tasker MUST run the `/speckit.analyze` Mode-A→Mode-B loop bounded by `TASKER_MAX_REVISION_ROUNDS` (default 5). On convergence (`verdict: clean`), the project MUST advance to `analyzed`. On cap-hit, the Tasker MUST write `human_input_needed.yaml`, HOLD at `analyze_in_progress`, and record `escalated` — it MUST NOT loop indefinitely or false-advance. The round count MUST be recorded (e.g., in `.specify/memory/tasker_rounds.yaml`).
- **FR-014**: Every agent invocation MUST appear in the per-run JSONL run-log with: `agent`, `project_id`, `started_at`, `ended_at`, `outcome ∈ {committed, abstained, failed, held, no-op, escalated}`, `error` (if any), and — for the Tasker — the `analyze_rounds` count and final `verdict`. The run-log format itself is NOT modified (see the Run-log Entry entity); the verbatim per-round detail (each round's analyze report + Mode-B patch + diffs) is captured in the Tasker's inspection record under `rounds[]` (FR-004), not as separate run-log lines. No silent advancements.
- **FR-015**: After Phase 4 completes, a `carry-forward.yaml` MUST be generated under the spec directory listing each reference project, its final stage, an `agents_run` summary (including the Tasker's round count), and (on failure/hold) a one-line reason. Format MUST match the precedent set in `specs/011-phase3-specify-clarify-testing/carry-forward.yaml`.
- **FR-016**: The validation MUST include regression tests under `tests/integration/test_phase4_plan_tasks.py` for: (a) the multi-file FILE-marker split, (b) invented/unreachable-URL rejection in `research.md`, (c) prose-stub `tasks.md` rejection (below task-ID minimum), (d) the Mode-B diff-leak guard, (e) the Mode-B markdown-header-preservation guard, and (f) the analyze-loop cap → `human_input_needed` escalation. These MUST run as part of the standard unit-test pass.
- **FR-017**: Three gates are in-scope agent hardening for this feature, all added to the Planner's `write_artifacts` path: the FR-005 artifact-set-completeness gate, the FR-006 URL-reachability gate, and the FR-007 data-model↔contracts consistency gate (per the clarification to harden the agent classes rather than gate only in the validation layer). They MAY ship as one cohesive commit (the canonical `_research_guard` module) since they share the same call site and failure semantics. Beyond these three additions, the validation MUST NOT change Planner or Tasker logic UNLESS a real bug is found; in that case the bug fix MUST be a separate, justified commit and MUST cite the failing inspection record by path.
- **FR-018**: At the start of every validation run on a reference project, if Phase-4 output artifacts (`plan.md`, `research.md`, `data-model.md`, `quickstart.md`, `contracts/`, `tasks.md`, and any analyze/tasker round state) already exist in the project's `specs/<n>-<slug>/` directory from a previous partial run, the validation MUST delete those Phase-4 outputs while PRESERVING the Phase-3 `spec.md` (the Planner's input). The deletion MUST be recorded under a `reset_artifacts` key in the inspection record.
- **FR-019**: If Phase 4 is invoked on a project whose state has already advanced past `clarified` (e.g., already `planned`/`analyzed`), the runner MUST decline to re-run and either no-op cleanly or report the project as "already past this phase". This is distinct from FR-018, which fires only when the state is still `clarified` but leftover Phase-4 output artifacts exist.
- **FR-020**: The Planner's `plan.md` MUST include a Constitution Check section that references every numbered principle in the project's `.specify/memory/constitution.md`; absence of this section, or a principle left unaddressed, MUST be flagged in the inspection record.
- **FR-021**: Per-agent wall-clock budget MUST be enforced at 900 seconds (the `wall_clock_budget_seconds` value from the registry); for the Tasker this budget applies PER analyze ROUND. A timeout MUST classify as `failed`, NOT `committed`, and the project stage MUST HOLD.
- **FR-022**: The validation MUST demonstrate the Tasker Mode-B (analyze-resolve) path is exercised. When a reference project triggers ≥1 Mode-B round naturally, that real round MUST be captured in the Tasker inspection record. Independently of whether the real runs trigger findings, the synthetic-input regression tests (FR-016 d/e/f) MUST cover Mode-B. The phase report MUST state which path(s) — real run and/or regression tests — provided Mode-B coverage, and for which project.

### Key Entities

- **Reference Project**: A real project ID + slug at a known starting stage. Used as the input to a phase-validation run. `PROJ-261` and `PROJ-262` (both at `clarified`) are the Phase-4 reference projects.
- **Plan Artifact Set**: The five Markdown/YAML documents the Planner produces in one multi-file response — `plan.md`, `research.md`, `data-model.md`, `quickstart.md`, `contracts/<schema>.yaml` — written under `projects/<PROJ-ID>/specs/<n>-<slug>/`.
- **Analyze Revision Round**: One Mode-A→Mode-B iteration of the Tasker's loop: the `/speckit.analyze` report received plus the Mode-B JSON patch produced (`issues_resolved`, `issues_remaining`, `verdict`). Bounded by `TASKER_MAX_REVISION_ROUNDS`.
- **Inspection Record**: A JSON file capturing the verbatim prompts, raw response(s), parsed output, and file diffs produced by a single agent invocation. Lives under the spec directory; one per `(project, agent)` pair. The Tasker record nests one sub-record per analyze round.
- **Carry-forward Manifest**: A YAML file listing each reference project, the stage it ended at after this phase, the agents run (with the Tasker's round count), and whether it `passed`/`failed`/`held`. Hands the projects off to Phase 5's validation.
- **Run-log Entry**: An existing pipeline concept — one JSONL line per agent invocation (and per Tasker round) under `state/run-log/YYYY-MM/<run-id>.jsonl`. Phase 4 validation reads these (does not modify the format).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Both PROJ-261 and PROJ-262 reach stage `analyzed` within a single Phase-4 run, with NO human intervention and the Tasker converging within `TASKER_MAX_REVISION_ROUNDS`.
- **SC-002**: Each Planner run produces all five artifacts non-empty, with at least one `contracts/*.yaml` schema, and a `plan.md` containing a Constitution Check section that addresses every numbered constitution principle — verified by automated check, not manual reading.
- **SC-003**: 100% of dataset/code/paper references in each `research.md` return 2xx/3xx, verified by the Planner at plan time; any reference not returning 2xx/3xx (including transient 5xx/timeout/429) hard-fails the run with no retry.
- **SC-004**: Each `tasks.md` contains at least 10 numbered `T###` tasks AND passes the data-flow ordering check (no consumer task ordered before its producer) — verified automatically.
- **SC-005**: 100% of agent invocations AND Tasker analyze rounds in the run produce an inspection record on disk. If any are missing, the validation FAILS.
- **SC-006**: At least one regression test exists and passes for each of the six known failure modes (FILE-marker split, invented/unreachable URL, prose-stub tasks.md, Mode-B diff-leak, Mode-B header-clobber, analyze-loop cap escalation).
- **SC-007**: For a clean plan, the Tasker converges to `analyzed` in ≤3 analyze rounds (the ideal target; the hard cap is `TASKER_MAX_REVISION_ROUNDS`=5). The recorded round count proves convergence within the cap.
- **SC-008**: The carry-forward manifest correctly identifies both projects as `passed` (final stage `analyzed`) — or accurately reports any failure/hold — with the recorded final stages matching the on-disk state YAMLs.
- **SC-009**: A maintainer with no prior context can read `specs/014-…/inspections/PROJ-261-…/planner.json` and `tasker.json` and reconstruct exactly what each agent was asked and what it returned (including every analyze round), without consulting any other file.
- **SC-010**: The validation surfaces (in a phase report) any silently-broken behavior caught — e.g., a Planner data-model/contracts mismatch, or a Tasker Mode-B patch that weakens a constraint — naming the offending inspection-record path.
